# Highnote 参考模式

本文提炼 Highnote 官方文档中的通用产品与系统设计模式，重点参考发卡、收单、资金账户、交易生命周期、授权控制、按需供资、事件通知和运营报表能力。它用于补充支付产品、卡产品和资金运营设计视角，不替代监管规则、卡组织规则、银行合作协议、合规意见或团队自研架构决策。

## 适用场景

- 设计发卡平台、企业卡、员工卡、车队卡、虚拟卡、共享额度卡和卡片生命周期管理。
- 设计资金账户、持卡人账户、Program/企业账户、卡账户、交易授权和清算入账关系。
- 设计卡交易事件流、预授权、捕获、撤销、退款、拒绝、争议和报表追踪。
- 设计主资金池与子账户之间的实时补给、预算控制、共享额度和按需供资场景。
- 设计 spend controls、额度、商户限制、MCC 限制、地理限制、时间窗口和协同授权。
- 设计收单场景中的在线支付、支付订单、商户资金归集和统一资金移动视角。
- 设计 webhook、事件通知、报表生成和数据共享等集成能力。
- 设计发卡/收单/资金移动一体化产品时，补足“卡支付产品层”的业务表达。

## 与 Formance 的互补关系

- **Formance 更偏资金事实基础设施**：钱包、账本、对账、连接器、规则表达和流程编排。
- **Highnote 更偏卡产品平台能力**：发卡对象、持卡人、卡生命周期、授权控制、交易事件和运营视图。

支付技能里可以把 Formance 当作底层资金能力参考，把 Highnote 当作卡产品与支付运营能力参考。不要把两者混成同一层，也不要直接把某个平台的数据结构照搬成团队领域模型。

## 发卡与账户模式

- 账户体系要区分至少四类对象：Program/平台主体、企业或客户账户、持卡人或团队成员、卡片及其资金/额度归属。
- 卡不是账户本身，而是访问支付网络和授权体系的支付介质；卡片状态、额度、控制规则和交易事件应与底层资金账户解耦。
- 共享额度卡、单卡单账户、主副卡、虚拟卡和一次性卡的资金归属、责任归属、对账口径和风控口径必须分别定义。
- 开户、KYC/KYB、持卡人入网、卡片创建、激活、冻结、挂失、补卡、关户和权限变更都应有明确生命周期。

## 账务多分录与资金事件模式

- 金融账户下应显式建模多个 ledger，而不是只有一个余额字段。不同 ledger 表达不同资金语义，例如 `CASH`、`AUTHORIZATION`、`AVAILABLE_CASH`、应收应付、费用或在途。
- 每个 ledger 都要有正常余额方向，不要只保存一个“正数余额”却不解释其借贷语义。正常余额与当前余额方向不一致时，应视为负余额或异常状态。
- 资金事件是账务变化的最小业务事实，ledger entry 是该事件在具体 ledger 上的落点，journal entry 则表达一次资金事件涉及的成组借贷分录。
- 一个交易事件可能影响多个 ledger：授权阶段体现为 hold 或 pending，清算阶段体现为 cash 与 available 变化，调整或退款阶段体现为信用回补或补扣。
- 账户余额更像某一时点的账务快照，不天然等于“可用余额”或“待处理余额”。待处理、可用、冻结等展示口径通常需要跨多个 ledger 计算。
- 对账和客服追踪都要能下钻到单笔 ledger entry 或 journal entry，而不是只看聚合余额。

## 授权控制模式

- spend controls 应在产品方案中显式建模，而不是散落在代码或运营后台备注里。
- 控制维度至少包括金额、次数、时间窗口、MCC、商户、地理位置、币种、卡状态、余额充足性和主体权限。
- 协同授权适合高风险支付场景，例如车队、采购、企业费用、代理操作和大额支付；要明确谁发起、谁审批、超时怎么处理、拒绝后如何收敛。
- 授权控制与账务记账不是一回事：授权通过只代表网络侧允许消费，不等于最终清算入账，也不等于真实资金已结算。

## On-demand Funding 模式

- 按需供资适合“主资金池 + 子金融账户 + 卡支付”的模式：子账户平时可维持低余额或零余额，在授权或消费前后由主账户实时补给。
- 预算额度和实际资金余额应分离建模。额度表达“最多能花多少”，余额表达“现在账上有什么”，二者不能混成一个字段。
- 共享预算卡、项目卡、车队卡、员工福利卡等场景，通常更适合用“主账户资金 + 子账户限额 + 实时补资”而不是给每张卡长期沉淀真实资金。
- 按需供资要明确触发条件：授权前补给、授权后补给、清算时补给，还是定时回补；不同触发点对应不同风险和对账口径。
- 若采用按需供资，必须额外说明失败路径：主账户余额不足、额度不足、补资超时、并发授权、重复补资、逆向交易回退和预算重置。
- 这类设计本质上是在做“实时额度控制 + 实时资金调拨”，要把授权控制、资金事件、ledger entry 和可用余额展示一起设计。

## 交易生命周期模式

- 卡交易至少要区分授权、清算、入账、退款、撤销、拒付/争议、费用和调账等不同事件。
- 同一笔交易会经历多个事件，事件之间既要能关联，又不能用一个状态字段粗暴覆盖全部阶段。
- 交易事件模型应保留原始网络事件、平台加工事件和账务事件三种视角，便于运营、客服、财务和审计各自追踪。
- 延迟清算、部分清算、增量授权、预授权释放、跨日退款和争议回退都要有独立处理路径。
- 对账粒度优先使用事件级标识，而不是只用交易主单号。一个 `transactionId` 下可能有多个授权、清算、撤销、退款或调整事件。
- 不要只依赖 network reference 做事件关联。网络标识在授权、清算、退款和批量处理阶段可能不稳定、不完整或一对多。
- 授权成功只代表资金被预占或挂起，不代表已清算。清算成功才会驱动正式入账；清算后的 reversal 更接近退款或 adjustment，而不是简单释放 hold。

## 卡产品运营模式

- 卡产品设计不能只有支付能力，还要有客户入网、发卡配置、卡片管理、额度管理、交易查询、争议处理、报表导出和告警视图。
- 运营后台要支持按 Program、企业、团队、持卡人、卡、商户、币种、交易状态和时间窗口追踪。
- 对卡类产品，报表至少要覆盖授权通过率、拒绝原因、清算成功率、退款时效、争议量、额度使用率、异常交易和余额风险。
- 事件通知和 webhook 设计要考虑幂等、顺序、重放、缺失补拉和运营留痕。

## Reporting 模式

- 报表不是交易 API 的附属输出，而是独立的数据面，面向运营、财务、风控、对账、客服和审计。
- 报表消费方应按字段名或显式 schema 读取，不要依赖列顺序；要能容忍平台新增字段而不破坏既有处理逻辑。
- 报表生成通常是异步任务，要有报表类型、参数、状态、生成时间、可下载地址、过期策略和重试机制。
- 报表模型至少应区分 summary metrics、transaction activity、ledger entry、negative balance、transfer/ACH event 等不同视角。
- 交易活动报表和账务分录报表应能通过事件级字段关联，而不是只靠交易主单号。
- 报表不是替代账本，而是把账本、交易、授权、资金移动和运营动作按消费场景重新投影。

## Webhook 与事件集成模式

- 事件通知是支付平台对外的正式接口面，不只是“附送消息”。必须定义事件类型、幂等键、顺序语义、重试策略、补拉机制和留痕要求。
- 消费方不得假设 webhook 绝对有序，也不能假设单次投递必达；设计时要支持去重、乱序重组、重放和最终一致收敛。
- 交易事件、资金事件、报表生成事件和运营动作事件要分层，不要把不同语义的事件混在一个泛化 topic 中。
- 事件失败后的恢复机制要清楚：重试、死信、人工补偿、补拉 API、状态回查和告警。

## 收单与支付订单模式

- 收单视角下要显式区分支付订单、支付尝试、网络事件、资金入账和商户结算，不要把“下单成功”当成“收款完成”。
- 在线支付产品需要单独设计 tokenization、敏感卡信息隔离、PCI 边界和商户前后端责任分工。
- 收单产品的核心对象通常不是 card 本身，而是 payment order / payment intent / payment attempt 一类业务承载体；它负责聚合多次尝试、状态推进和客户体验。
- 若平台同时有发卡和收单能力，必须额外说明内部账户转移、同平台闭环资金流和外部网络事件之间的界面边界。

## Data Share 与运营数据模式

- 数据共享能力适合作为报表之外的“分析数据面”，面向 BI、风控分析、经营分析、审计留档和离线数仓。
- 共享数据面要明确刷新时效、字段字典、历史回放能力、字段新增兼容性和权限边界。
- 分析数据面和运营 API 应分开设计：前者优先可查询和一致性语义，后者优先实时交互和业务操作语义。

## 收单与资金移动视角

- 若产品同时覆盖发卡、收单或资金移动，要拆清“谁是付款方、谁是收款方、谁控制卡、谁持有资金、谁承担通道和争议风险”。
- 付款网络事件、内部账本事件、外部银行流水和客户对账单不是同一份数据；必须建立映射与差异闭环。
- Program 管理费、卡费、交易费、通道成本、争议损失和补贴要分别建模，不要挂在一个“手续费”口径下。

## 设计落地检查

1. 是否区分 Program/平台主体、客户/企业账户、持卡人、卡片和底层资金账户。
2. 是否把卡片当成支付介质，而不是直接等同于账户余额。
3. spend controls 是否被配置化、可审查、可测试、可追踪。
4. 是否区分授权成功、清算成功、账务入账、真实资金结算和财务确认。
5. 交易模型是否支持多事件关联，而不是一个状态字段覆盖全部生命周期。
6. 是否覆盖虚拟卡、实体卡、共享额度卡、主副卡和一次性卡等差异场景。
7. 是否显式区分资金事件、ledger entry 和 journal entry，并能解释多分录的借贷关系。
8. 是否用事件级粒度做对账和追踪，而不是只依赖 `transactionId` 或网络参考号。
9. 是否区分预算额度、可用余额、授权占用余额和真实现金余额，避免按需供资场景下口径混乱。
10. webhook/事件流是否具备幂等、顺序、重放和补拉机制。
11. 是否覆盖争议、拒付、退款、撤销、补卡、挂失、冻结和权限变更的运营闭环。
12. 报表是否支持事件级追踪、分录级追踪、负余额监控和运营指标投影。
13. 收单场景下是否区分支付订单、支付尝试、网络事件、资金入账和商户结算。
14. 是否明确 tokenization、敏感数据隔离、PCI 边界和商户责任分工。
15. 分析数据面是否支持字段扩展兼容、时效说明和权限控制。
16. 是否把卡网络规则、银行合作协议、合规限制和资金账务规则列为待确认项。
