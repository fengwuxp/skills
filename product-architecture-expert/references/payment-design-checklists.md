# 支付与资金设计评审清单

## 使用时机

- 用户要求对支付、账户、账本、清结算、对账、收单、发卡、VCC、跨境、全球平台、争议、风控或业财一体化方案做 CR、触发验证、提交前检查或可用性评审。
- 已有 PRD、产品方案、架构方案或图形 brief，需要判断是否缺少金融业务主轴、资金安全、外部规则、账务证据和运营闭环。

## 不适用场景

- 需要从零设计支付产品时，先读 `payment-methodology.md`。
- 需要完整清结算、卡组织、ACH、VCC、争议或风控专项方法时，继续读对应 reference。
- 本文是检查清单，不替代监管、法务、财务、通道或持牌机构确认。

## 读取后必须产出

- 符合项、必改、建议、待确认项、专业确认方和阻断上线的金融业务红线。

## 需要继续读取的 reference

- 支付方法论读 `payment-methodology.md`；清结算读 `clearing-settlement.md`；监管规则读 `regulatory-baseline.md`；全球跨境读 `global-payment-emerging.md`；卡组织读 `card-network-and-card-rails.md`；VCC 读 `virtual-card-and-vcc.md`。

## 金融业务专家总检

- 是否能用一句话回答：谁的钱、因什么业务、在什么主体和账户下、沿哪条支付轨道、以什么规则流转、何时可用、谁承担风险、谁最终确认。
- 是否区分客户资金、商户待结算资金、平台自有资金、保证金、准备金、授信额度、预算额度、手续费收入和通道成本。
- 是否把业务确认、支付受理、授权成功、清算入账、清账完成、结算出款、资金到账、财务确认拆成不同状态或验收点。
- 是否输出四流、单据状态、账户/账务、清结算/对账、风险合规、数据证据和验收确认方，而不是只给页面功能或通道接口。
- 是否说明每个金额变化的业务来源、账务证据、外部依据、风险责任和异常回退/追偿路径。
- 是否把法域、主体资质、外部规则版本、数据边界和待专业确认项写清楚；不能确认时是否明确列为待确认，而不是写成上线结论。

## PRD 检查

- 是否完成场景识别：组织语境、业务动作、支付轨道、资金复杂度和目标产物。
- 是否按场景选择了正确参考路径，而不是把清结算、通道路由、争议证据、ACH、卡组织和 VCC 混成一个模板。
- 面向 Airwallex 类全球金融平台、嵌入式金融或公开平台文档分析时，是否先拆清 Connected Accounts、Accounts、Payments、FX、Payouts、Issuing、Spend 和 Embedded Finance 的能力族群，而不是照抄厂商产品目录。
- 是否说明组织类型、资质边界和资金归属。
- 嵌入式金融、BaaS、Payments for Platforms 或白标能力是否有平台责任边界矩阵，拆清平台、客户、商户、最终用户、银行/持牌机构、PSP、卡组织和本地合作方责任。
- 最终用户看到的品牌、合同主体、资金服务主体、隐私主体、账单主体和客服入口是否一致；不一致时是否写出用户解释视图。
- 是否完成法域、牌照、客户资金、反洗钱、数据安全和业务连续性基线检查。
- 是否列出金融业务红线、待确认项和不可上线条件。
- 是否标明外部规则来源、版本/发布日期、生效日期、适用法域、核验日期和待确认方。
- 是否画出正向支付、退款、撤销、结算、失败补偿流程。
- 是否定义业务单、支付单、通道单、账务流水、内部清算单、结算单的关系。
- 是否给出状态机，以及每个状态的进入条件、退出条件和幂等策略。
- 是否说明金额口径：订单金额、优惠金额、实付金额、手续费、分账金额、可结算金额、退款金额。
- 是否定义后台操作权限、审批流、操作日志和审计要求。
- 是否列出报表、查询、导出、告警、运营处理台。
- 涉及多通道时，是否说明路由规则、通道健康、降级、成本口径、对账和配置治理。
- 涉及商户或交易风险时，是否说明 KYB、风险等级、风控动作、证据闭环和结算影响。
- 涉及卡产品时，是否区分 Program、客户账户、持卡人、卡片、授权规则和底层资金账户。
- 是否把状态、事件、webhook、报表、沙盒模拟和 go-live checklist 写入验收，而不是只描述前台功能。

## 全球金融平台交付包检查

- 是否输出能力地图，并明确账户、收单、付款、发卡、FX、费用、报表、嵌入式金融的范围和非范围。
- 是否输出主体责任矩阵，覆盖平台、客户、商户、最终用户、持牌机构、PSP、银行、卡组织和本地合作方。
- 是否输出对象生命周期，覆盖 account、payment、refund、transfer、beneficiary、payer、cardholder、card、authorization、dispute 和 report。
- 是否输出四流与资金路径，区分业务流、支付信息流、账户/账务流、真实资金流；跨境场景是否补消息网络、账户网络和资金网络。
- 是否输出 FX 与费用模型，覆盖币种、报价、锁价、费用、舍入、收入成本确认和报表解释。
- 是否输出运营与报表面，覆盖配置、审批、工单、告警、报表、对账、审计和客户解释。
- 是否输出风险与合规门禁，覆盖 KYC/KYB/KYT、AML、制裁、PCI、外部规则核验和专业确认。
- 是否输出验证与上线包，覆盖 sandbox、状态模拟、webhook、幂等、异常演练、灰度、回滚和客户告知。
- 交付包若裁剪，是否说明本期非范围、承接方、后续阶段和风险影响。

## 生命周期与运营成熟度检查

- 是否能按“入网 -> 配置 -> 交易/资金动作 -> 状态事件 -> 清结算/账务 -> 报表/对账 -> 异常/争议 -> 退出/留存”串起主对象。
- 客户、商户、受益人、持卡人、卡、账户、付款能力和报表能力是否都有创建、激活、暂停、冻结、关闭、归档或留存口径。
- 配置生效是否有审批、灰度、回滚、版本、操作者和审计日志。
- 退出场景是否覆盖资金退回、未结争议、未完成清结算、历史报表、数据导出、证据留存和用户/商户告知。
- 是否评估当前能力成熟度处于接口接入、对象化、运营闭环、平台化还是可解释金融基础设施阶段，并指出下一阶段要补的控制面。

## 清结算方案检查

- 参与方和结算对象是否完整。
- 是否建立多账本视角，拆清客户可见账、平台内部账本、外部机构账、真实资金账和会计账的事实源、责任方和确认方。
- 支付、退款、撤销、清分、结算、调账、长短款、拒付或退汇是否能映射到各账本的加记、减记、冻结、解冻、冲正、挂账、单据、流水、分录、文件或回执。
- 是否区分内部可见额度、通道/清算机构账、银行/存管真实资金和财务确认，避免把任一层“成功”写成全链路完成；若先给客户或商户可用，是否说明垫资、授信、准备金、限额和风控承接。
- 跨机构支付是否识别往来户模式或代理结算模式，并说明支付指令、清算场次、轧差规则、结算指令、回执和失败回退。
- 代理结算、平台代收代付、多 PSP 或跨境收付是否说明内部账本与底层资金账户的映射规则、净额/全额口径、风险承担和客户解释视图。
- 大账户、备付金账户、清算账户、Nostro/Vostro 或资金池与客户余额、商户待结算、VA、子账户或钱包余额之间是否能核对一致。
- 多业务线清结算规划是否先盘点线上线下断点、报表不记账、对账无差错闭环、系统耦合、服务抽象度低和重复建设问题。
- 是否按算钱、记钱、装钱、结钱、对钱拆出清算中心、账务中心、账户中心、结算中心和对账中心，并说明每个中心的输入、输出、单据、状态和验收口径。
- 是否给出线上化先行、清算中心跟进、账务账户收口的分期迁移路线，而不是要求一次性推倒重来。
- 是否区分内部清分/内部清算、持牌清算业务、商户结算和真实资金结算。
- 是否先生成清账结果再进入结算，而不是用渠道到账金额、银行流水或上游结算文件直接反推商户应结金额。
- 交易/订单、手续费、退款、调账、冲正、补结、扣回、拒付或退单是否都进入清账模型并参与轧差。
- 渠道、银行或清算机构文件中的负项、调账项和非正常交易是否能映射到清账科目，而不是只在结算层人工扣减。
- 费率、分账、分润、补贴、税费是否可以追溯到配置。
- 清算日、自然日、交易日、账务日、结算日是否区分清楚。
- 清算文件未按预期到达、迟到、缺失或处理状态异常时，是否先核对交易日、清算日、入账/结算日、cut-off、周末/假期、processing calendar 和时区口径，而不是直接归因于网关慢。
- 是否区分 `accepted but delayed` 与 `rejected / held`，并记录 ACK、确认号、批次 ID、记录数、金额合计、reject reason / error code 和下一档预计处理时间。
- 清算文件延迟工单是否标明三段链路归因：商户/PSP 到收单处理、卡组织/清算网络、银行/代理行/币种中转。
- 是否把连续多批次缺失、同一 MID 大面积 reject、金额轧差突然偏离设为资金链路级升级信号。
- 批次生成、锁定、审核、出款、回写、重跑是否可控。
- 是否区分待结算、可结算、已结算、冻结、在途、长短款。
- 是否覆盖退款跨结算周期、部分退款、部分分账、结算后退款。
- 是否说明差错单如何生成、流转、核销和入账。

## 对账方案检查

- 是否区分清账对账和结算对账：先对业务/支付/退款/手续费/调账/渠道文件与清账结果，再对清账结果驱动的结算金额与银行、通道或存管实际划付/到账。
- 至少有平台交易对账、通道交易对账、资金到账对账、账务一致性核对。
- 是否明确内部账本账户与外部现金池、银行账户、通道账户或清算机构文件的映射关系。
- 文件获取、解析、验签、去重、落库、比对、差错生成有完整链路。
- 支持跨日、跨批次、延迟通知、补文件、重跑和版本追溯。
- 每个差异类型都有自动处理优先级和人工兜底流程。
- 对账结果能反向驱动补单、补入账、挂账、调账和财务凭证。

## 账务核心检查

- 是否按账户层、账务层、账簿层、总账层拆清职责，而不是只设计账户表、余额表或流水表。
- 是否区分客户可见余额、账务分录、真实资金到账、日余额、总账余额和财务确认口径。
- 是否说明科目、末级科目、外部账户、内部账户、分户账、账户模板和余额方向的关系。
- 每类业务事件是否能生成凭证规则、分录流水、余额变化、业务单据、系统跟踪号、幂等键和外部 reference。
- 是否区分日间联机记账和日终核算，明确会计日切、缓冲清理、渠道结转、明细汇总、总账汇总、试算平衡和总分核对。
- 实时记账、缓冲记账或单边余额更新是否说明适用场景、性能原因、最终一致性边界、失败补偿、重跑策略和审计证据。
- 高频内部户、即时分账、多分账方、担保交易或合单交易是否识别热点账户和批量汇总风险。
- 错账、长短款、待查错账、调账和核销是否通过凭证、冲正、反向分录、调账或挂账闭环处理，而不是直接修改历史余额。
- 账务后台是否覆盖科目管理、账户管理、分录流水、记账凭证、日终作业、权限隔离和操作审计。
- 借贷方向、科目设置、试算平衡、总分核对、财务报表口径是否列为财务或账务负责人确认项。

## 资金安全检查

- 客户资金和平台自有资金是否隔离。
- 是否存在未授权垫资、挪用/占用/借用客户备付金或待结算资金、透支结算或重复出款。
- 是否避免将普通待结算资金、保证金或平台应付款误称为客户备付金。
- 所有出款是否有审批、限额、复核、幂等和回单核验。
- 负余额、冻结、黑名单、争议订单、退款预留是否会影响结算。
- 是否能按商户、通道、币种、主体、账户维度追溯余额。

## 外部规则时效性检查

- 方案引用的法律法规、监管文件、卡组织规则、Nacha/ACH 规则、PCI DSS、银行/通道协议、税务和会计准则是否确认最新版本。
- 是否记录规则来源、版本或发布日期、生效日期、适用法域、适用主体、核验日期和确认人。
- 争议时限、return code、手续费、报送、数据留存、KYC/KYB、AML、PCI 范围和客户资金规则是否按最新外部规则复核。
- 无法确认最新版本或适用范围的内容，是否明确列为待法务/合规/财务/通道确认项，而不是写成确定结论。
- 外部规则变化是否会触发重新评审、回归测试、用户告知、合同更新或运营 SOP 更新。

## 技术实现检查

- 钱包层、账本层、对账层职责是否清楚；钱包 API 不得绕过账本、幂等、风控和审计直接改余额。
- 外部请求有幂等号，内部事件有唯一业务事件号。
- 金额使用整数最小货币单位或高精度十进制，禁止浮点。
- 状态推进可重试、可补偿、可审计。
- 批处理有批次号、游标、断点续跑、互斥锁和重入保护。
- 所有重要事件进入消息或事件表，支持失败重投。
- 关键账务写入使用事务、唯一索引和防重逻辑。
- 错账通过冲正、反向分录、调账或补记处理，不直接改历史账务流水。
- 报表口径来自可追溯明细，不依赖不可解释的汇总数。

## 通道与路由治理补充

- 是否定义通道、通道账户、能力、凭证、结算口径、健康信号和路由规则版本。
- 是否区分支付方式展示排序、支付尝试路由、资金出款路由和清结算路由。
- 是否记录路由命中原因、排除原因、候选通道、规则版本和审计日志。
- 通道超时、失败、回调延迟、文件延迟和资金到账异常是否有熔断、恢复和人工处理路径。
- fallback 前是否先完成幂等校验、状态回查和重复扣款/重复出款防护。
- 费率、限额、通道开关、密钥、证书和结算账户变更是否有审批、灰度、回滚和四眼复核。
- settlement report、payment activity export、fee / FX / rounding、response code 和 webhook event 是否能支撑运营、客服、对账和商户解释。

## FX、费用和舍入补充

- 是否区分交易币种、授权币种、账户币种、清算币种、结算币种、报表币种和记账本位币。
- 汇率来源、报价有效期、锁价时点、重新报价、报价撤销和异常兜底是否明确。
- 固定费、比例费、跨境费、网络费、processor fee、FX markup、退款费、拒付费、退汇费和人工处理费是否分别建模。
- 费用由用户、商户、平台、合作机构或内部成本中心中的谁承担，收入和成本何时确认，是否能进入账务矩阵。
- 舍入规则、最小货币单位、差额归属、批量轧差和报表展示精度是否一致。
- 用户可见价格、商户账单、平台收入、通道成本、财务凭证和对账报表是否能解释金额差异。

## 风控、欺诈与商户运营补充

- 商户准入是否覆盖 KYB、受益所有人、MCC、禁限售行业、履约模式、网站/App 和风险等级。
- 是否区分交易前、交易中、交易后、结算前和运营后的风控动作。
- 是否覆盖用户、商户、支付工具、设备、地理、金额、频次、退款、拒付、ACH return 和出款风险信号。
- 规则和模型是否有版本、审批、灰度、回滚、误杀复盘、指标监控和人工审核路径。
- 风控处置是否能影响结算暂停、准备金、限额、冻结、关闭商户和证据留存。
- 风控证据是否能关联交易、账务、争议、客服、对账、商户评级和审计。

## 争议、退款与拒付运营补充

- 是否区分 refund、void、reversal、return、dispute、chargeback、representment 和 arbitration。
- 是否区分卡争议与非卡争议，不把钱包、APM、本地支付、账户转账或平台仲裁套进卡 chargeback 模板。
- 是否能表达 Alert / Inquiry / Retrieval / Chargeback / Representment / Pre-arbitration / Arbitration 的对象、状态、资金动作、时限和证据包。
- 是否能把逆向事件关联到原交易、授权、清算、结算、账务分录、通道 reference 和争议工单。
- 是否覆盖部分退款、跨周期退款、结算后退款、退款失败、重复退款和退款/争议碰撞。
- 退款资金来源、商户准备金、负余额、结算暂停、追偿、争议费和损失确认是否明确。
- 是否把争议率/CB 率作为风控、结算、商户运营联动的红线指标；具体阈值、区域、生效日、排除项是否回到 Visa/Mastercard/收单行/通道最新官方规则确认。
- 拒付进入后是否先分型、止血、按原因码组证、判断 representment，再做结果回流。
- 是否按前置预防、预警处置、正式争议、反馈回流四层设计争议治理闭环。
- 是否有结构化证据模型、附件管理、提交时限、提醒、SLA、人工复核和状态闭环。
- 是否设计交易争议举证行为日志，覆盖账户、设备、授权、协议确认、履约、客服、退款和历史未争议交易。
- 争议举证行为日志是否区分可信服务端事件、前端辅助事件和人工补充材料。
- 证据日志是否追加写入、不可随意修改，并具备权限隔离、脱敏展示、访问审计、版本留痕和留存期限控制。
- EvidencePackage 是否按支付轨道、争议/退回原因码和银行/卡组织/通道模板生成，避免直接提交全量行为日志。
- ACH 场景是否单独生成 ACH evidence package，而不是套用卡 chargeback 证据包。
- 对外提交前是否剔除完整 PAN、CVC/CVV、敏感认证数据、未脱敏证件影像和无关隐私材料。
- 争议结果是否反哺风控、商户评级、对账差错、财务凭证和报表。

## 卡产品补充

- 是否区分授权、清算、入账、退款、撤销、争议/拒付等不同事件，而不是一个状态字段覆盖全部生命周期。
- 是否区分资金事件、ledger entry、journal entry 和余额快照，并能解释多分录借贷关系。
- 是否区分预算额度、授权占用、真实现金余额和可用余额，尤其是按需供资场景。
- spend controls 是否覆盖金额、次数、时间窗、MCC、商户、地理、币种、卡状态和主体权限。
- 协同授权、人工审批、超时处理、拒绝回退和事件通知机制是否完整。
- 虚拟卡、实体卡、共享额度卡、主副卡或一次性卡的资金归属和责任边界是否清楚。
- 对账与客服追踪是否能下钻到事件级和分录级，而不是只看交易主单号或聚合余额。

## VCC 与虚拟卡补充

- 是否明确 VCC 实际资金模式：credit、debit、prepaid、charge card、commercial card 或一次性 token。
- 是否定义 Program、客户账户、资金账户、持卡人、卡片、控制规则、授权和账务之间的关系。
- 是否覆盖创建、激活、展示、冻结、注销、重发、轮换、过期和注销后的延迟清算/退款/争议。
- 是否区分预算额度、可用额度、授权占用、真实现金余额、应收应付、费用、退款、争议和 FX。
- spend controls 是否覆盖金额、次数、MCC、商户、地理、币种、时间窗、线上/线下、recurring 和审批规则。
- PAN/CVC 展示、token、PCI 范围、日志脱敏、权限、密钥管理、审计和批量导出是否经过评审。
- 是否区分远程授权、控制规则、告警阈值、3DS、地址校验、overcapture、欺诈反馈和交易仿真的产品职责。

## 报表与数据面补充

- 是否区分交易活动报表、账务分录报表、汇总指标报表、负余额监控报表等不同消费场景。
- 报表字段消费是否按字段名/schema 而不是按列顺序。
- 报表是否支持异步生成、状态查询、下载、过期和失败重试。
- 报表是否能用事件级字段关联交易、资金事件和分录，而不是只靠交易主单号。

## 收单与事件集成补充

- 是否区分支付订单、支付尝试、网络事件、资金入账和商户结算。
- 是否明确 tokenization、敏感支付信息隔离、PCI 边界和商户前后端责任分工。
- webhook 是否支持幂等、重试、乱序容忍、补拉和状态回查。
- 平台事件、账务事件、运营事件和报表事件是否按语义分层，而不是混在一个通道里。

## ACH 与银行转账补充

- 是否区分发起方、发起行、接收行、清算网络和受益人。
- 是否区分提交成功、清算成功、资金到账、资金可用和对账完成。
- debit 场景下是否说明授权取得、授权留存、授权撤销和争议处理。
- debit 场景下是否保存授权文本版本、授权确认动作、授权时间、授权金额/频次/日期、撤销方式和账户验证结果。
- 是否区分 return、NOC、reversal、refund 和 retry，不把它们混成一个失败处理动作。
- 是否记录 return code、NOC 主数据更正、reversal 原因、retry 决策、用户通知和人工复核。
- ACH evidence package 是否按银行、ODFI/RDFI、通道或 Nacha 规则裁剪，并完成最小必要、脱敏和提交审计。
- Payouts 场景是否拆清 transfer、beneficiary、payer、batch transfer、approval、failure reason、confirmation、regulatory requirement 和出款对账。
- 受益人字段、银行字段、国家/地区要求、税务材料和监管材料是否作为动态 schema / ruleset 管理，并标明官方文档、银行协议或合规确认来源。

## 卡组织与卡支付轨道补充

- 是否区分 cardholder、merchant、issuer、acquirer、network 的职责边界。
- 是否区分 authorization、clearing、settlement、funds availability 和 merchant payout。
- 是否覆盖 preauth、incremental auth、partial capture、void、refund、chargeback 等多事件链路。
- 是否能画出标准卡交易时序，并解释 authorization hold、capture、presentment、issuer posting、network settlement 之间的关系。
- 是否区分 PAN、token 和各自生命周期、责任边界与 PCI 范围。
- 是否把 Clearing Core 视为 transaction lifecycle、accounting consistency、reference continuity、dispute traceability、replay / investigation 的治理系统。
- reconciliation 是否验证金额一致性、关系一致性、结构一致性和锚点一致性，而不是只比金额和状态。
- 清算异常是否按 Authorization -> Clearing、Clearing -> Posting、Posting -> Settlement、Reference chain -> Dispute、Time chain 分类并隔离处理。
- 多卡组扩展是否把 clearing event model、matching、amount / fee decomposition、posting、replay 和 investigation 放在核心层，把文件、引用、费用和争议规则差异放进 scheme adapter。

## 跨境与多币种补充

- 区分交易币种、清算币种、结算币种、记账本位币。
- 汇率来源、报价锁定、换汇时点、汇损益归属必须明确。
- 需要考虑 KYC/KYB、AML、制裁筛查、地区限制、退款路径和资料留存。
- 跨境通道的清算文件、资金到账和手续费结构通常不同步，必须单独对账。
- 是否区分全球支付能力的底层路径：新型轨道绕行、聚合包装传统基础设施、深控或自建本地金融基础设施。
- 是否穿透评估本地牌照/合作主体、银行关系、本地清算网络、代理行、流动性、换汇、对账文件、异常追踪和争议责任，而不是只看覆盖国家、币种、费率和 API。
- 是否说明合规、资金、清结算、风险和运营复杂性由平台、客户、商户、PSP、银行或本地合作机构中的谁承担。
- 是否验证客户侧确定性：交易完成率、资金可用时间、费用/汇率可解释性、异常闭环责任和赔付/告知口径。

## 新支付形态补充

- 是否先确认法域、面向客群、主体资质和监管许可，再讨论稳定币/Web3 或 AI 代理支付。
- 稳定币/Web3 支付是否区分链上交易、链下账务、托管钱包、非托管钱包和出入金通道。
- AI 代理支付是否有明确授权范围、金额限制、人工确认门槛、审计日志和撤销机制。
- 全球支付清算是否拆清消息网络、账户网络、资金网络和本地清算网络。
- 银行核心类方案是否覆盖客户、账户、产品参数、账务分录、日切、总分核对和监管报送。

```text
2. 组织语境与资质边界

4. 正向交易链路
5. 退款、撤销、退汇和异常链路

9. 账务分录/账户流水矩阵
10. 后台操作、审批、报表和告警
11. 风险、合规、审计和验收标准
```
