# 产品判断动作链

本文把产品专家已有的产品洞察、Backlog、PRD、合议评审和 AI Native 产品上下文能力串成可调用的判断动作链。它参考 `pm-skills` 的“让产品判断成流程”思路，但不安装、复制或照搬外部 Skill。

## 使用时机

- 用户提到 `pm-skills`、产品判断成流程、产品动作链、产品判断动作链、路线图取舍、发布复盘或增长实验。
- 用户的访谈、工单、竞品、数据、路线图、PRD、发布材料散在多处，需要判断现在做什么、为什么做、先不做什么。
- 用户希望从反馈 / 资料进入机会、Backlog、PRD、验收种子或 AI Native 工程交接，而不是只写一份漂亮文档。
- AI Native 要判断产品事实、取舍结论和验收种子是否足以进入系分、TDD、编码或质量门禁。

## 不适用场景

- 只写普通 PRD 或产品方案，直接读 `product-prd-template.md` 和 `product-design-and-prd.md`。
- 只做 Backlog 排序，直接读 `po-backlog-manager.md`。
- 只做产品洞察 / 机会雷达，直接读 `product-insight-analyst.md`。
- 只做系统设计、代码、测试或 CR，交给 `资深架构师`。
- 不安装 `phuryn/pm-skills`，不照搬其 68 个 skills、42 个 workflows、slash commands、plugins 或 PM 工具包。

## 读取后必须产出

- 当前处于产品判断动作链的哪一段，以及为什么。
- 已知事实 / 证据、合理推断、待确认项、范围外不做。
- 判断动作、取舍结论、不做项、下一产物、owner、验收 / 停止条件。
- 是否交给 `product-insight-analyst.md`、`po-backlog-manager.md`、PRD reference、合议评审 reference 或 AI Native 工程交接。

## 需要继续读取的 reference

- 反馈、访谈、工单、竞品、行业资料或知识库资料进入产品洞察时，读 `product-insight-analyst.md`。
- 机会清单、优先级、路线图、User Story 或 AC 决策时，读 `po-backlog-manager.md`。
- 需要正式 PRD / 产品方案时，读 `product-prd-template.md` 和 `product-design-and-prd.md`。
- 需要多角色产品评审或 PRD 预扫描时，读 `product-deliberation-workflow.md` 和 `product-prd-quality-gates.md`。
- 需要交给 AI Native 或架构师时，读 `ai-native-product-context.md`，端到端编排交给 `ai-native-engineering-workflow`。

## 1. 判断框架

用老祖宗的三层思维收敛复杂产品工作：

- **不易**：产品判断必须守住用户 / 主体、问题、证据、取舍、owner、验收和停止条件。
- **变易**：访谈、PRD、Backlog、路线图、发布和增长实验只是不同阶段的动作，不是孤立文档。
- **简易**：小团队优先固化四个高频入口：访谈 / 反馈总结、需求优先级、PRD / Product Context Card 准入、发布复盘 / 增长实验。

产品专家的职责不是把输入材料整理得更好看，而是把材料转成判断：现在解决什么问题，为什么值得做，为什么不是别的方案，先不做什么，下一步交给谁。

## 2. 七步动作链

| 阶段 | 输入 | 产品判断动作 | 输出 | 下一 owner |
| --- | --- | --- | --- | --- |
| 资料归集 | 访谈、工单、竞品、数据、政策、运营反馈 | 去重、分层、标注证据来源和置信度 | 资料资产表 / 问题候选 | 产品专家 |
| 问题地图 | 资料资产和业务目标 | 判断真实问题、影响面、失败成本和当前替代方式 | 问题地图 / 非标问题卡 | 产品专家 / 业务 owner |
| 机会判断 | 问题地图和证据链 | 判断机会是否足够强、是否有 owner、是否需要继续验证 | 机会雷达 | 产品专家 |
| 假设与风险 | 机会候选 | 写清解决方案假设、反例、风险、非目标和可证伪实验 | 假设卡 / 验证动作 | 产品专家 / UED / 业务 owner |
| Backlog 取舍 | 机会和验证结果 | 做 BV/EE、业务 / 用户 / 工程三桌校验、拒绝或延后理由 | Backlog 决策包 | 产品专家 / PO |
| PRD / 上下文 | 已选 Backlog | 补对象、流程、状态、规则、验收种子和待确认项 | PRD / Product Context Card | 产品专家 |
| 发布与回流 | 上线结果、数据、客服、运营、事故、复盘 | 判断是否达成目标、是否继续、收缩、废弃或进入下一轮 | 发布复盘 / 增长实验结论 | 业务 owner / AI Native |

## 3. AI Native 前置门禁

进入 AI Native 工程交接前，产品专家只回答产品侧是否足够：

- 业务目标和非目标是否能被验收。
- 核心对象、状态、流程、规则和异常是否能交给架构师消费。
- 验收种子是否覆盖正常、边界、异常、运营、数据和风险路径。
- 取舍结论是否说明了为什么做、为什么现在做、为什么不做其他项。
- 待确认项是否有 owner、影响范围和停止条件。

任一项不足时，输出补齐清单和下一 owner，不进入系分、TDD、编码或 Execution Grant。

## 4. 取舍规则

- 路线图是取舍账本，不是愿望清单；每个进入项必须有证据、owner、验收和拒绝 / 延后对照。
- 没有证据的机会只能进入待验证，不写成 PRD 结论。
- 没有 owner 的需求只能进入待确认，不交给研发排期。
- 没有验收种子的需求不能进入 TDD、编码或发布准出。
- 外部 workflow、文章观点或 AI 生成内容只能作为检查项，不能替代产品 owner 判断。

## 6. 回流规则

- 发布复盘必须回看最初目标、验收种子、上线结果、返工、缺陷、运营成本和用户价值。
- 增长实验必须说明假设、变量、样本、观察指标、停止条件和下一动作，不能只报数据涨跌。
- 已验证的产品经验可回流到 PRD 模板、Backlog 规则、Product Context Card 或 AI Native source-map；不得把一次性偏好、未确认推断或私有材料写入 Skill。

## 7. 不吸收边界

- 不把 `phuryn/pm-skills` 的 commands、plugins、slash command、PM toolkit、resume / NDA / privacy 等通用工具包搬入本仓库。
- 不把“AI 写 PRD 更快”当产品能力提升；判断链的价值在证据、取舍、owner、验收和回流。
- 不让 AI 替产品 owner 做最终路线图取舍、商业优先级、合规判断或上线承诺。
