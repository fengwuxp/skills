# 业务架构规划

业务架构规划用于把战略意图、业务能力、价值流、核心对象、项目组合、系统支撑和知识库回流连成可决策、可交接、可验证的结构。它不是组织架构图、系统清单或一次性大图，而是帮助团队回答“哪些能力最值得投、哪些项目重复建设、哪些差距阻塞战略、哪些知识必须沉淀”的决策工具。

## 使用时机

- 用户要求业务架构规划、业务能力地图、业务 IT 对齐、战略落项目、项目组合治理、投资取舍、重复建设识别或路线图规划。
- 用户希望从企业战略、业务目标、部门诉求、现有项目、系统清单、流程资料或知识库材料中整理业务架构。
- 用户要求把业务架构结论落到 PRD、Backlog、产品上下文包、AI Native 交接、知识库或上下文治理。
- 用户提到 capability map、value stream、capability owner、capability gap、portfolio、roadmap、能力-项目-系统映射。
- 产品方案还没有进入具体 PRD，但已经需要先统一业务语言、能力边界、优先级、依赖和 owner。

## 不适用场景

- 只写单个功能 PRD、页面说明、交互稿反推或运营后台细节时，优先使用 PRD / 产品设计 reference。
- 只做系统架构、技术选型、代码实现、测试或生产变更时，交给 `资深架构师`。
- 只画组织汇报关系、岗位分工或绩效责任时，不使用本文；业务架构关注业务能力与价值流，不替代组织设计。
- 没有战略意图、业务问题或决策场景时，不做全公司全量地图；先输出准入缺口和 Round 0 补齐清单。
- 不把单篇文章、行业框架、顾问方法或厂商案例写成当前组织事实、投资结论、系统设计授权或上线审批。

## 读取后必须产出

- 业务架构准入卡：战略意图、真实问题、决策场景、范围边界、业务 owner、验收方式和资料缺口。
- 业务能力地图：能力分层、能力定义、目标主体、owner、输入输出、关键规则和能力边界。
- 价值流 / 关键业务流程：从触发到价值实现的主链路、异常链路、人工节点、数据和风险节点。
- 核心对象与规则卡：术语、业务对象、生命周期、状态、关系、不变量、规则 owner 和验收种子。
- 能力-项目-系统映射：每个能力对应现有项目、系统、数据源、接口事件、运营动作和已知重复建设。
- 差距 / 依赖 / 优先级矩阵：能力差距、业务影响、风险、依赖、投入候选、停止条件和待确认项。
- 项目组合 / 路线图建议：保留、合并、延后、停止或新建的项目候选，以及阶段目标和准出证据。
- 知识库回流计划：按业务域或模块规划保存位置、权威文件、更新 owner、证据来源和复审机制。

## 1. 准入原则

业务架构先问决策，不先问图。只有当团队需要做跨业务域取舍、项目组合治理、能力重复建设识别、战略落地路径或知识库权威化时，才进入业务架构规划。

四个基本原则：

- 正名：先统一业务术语、主体、能力名和对象名；名称不清，能力地图会变成口号堆叠。
- 体用：战略目标、业务责任、风险和验收是体；能力图、流程图、项目表、系统图和知识库是用。
- 循名责实：每个能力必须能追到业务 owner、目标用户、可观察结果、核心规则和验收口径。
- 知止：不知道战略意图、决策问题或 owner 时停止扩写，只给补齐清单；不要把不确定性画成确定图。

业务架构规划准入卡：

```text
业务架构准入卡

战略意图：
真实问题：
决策场景：投资取舍 / 重复建设 / 路线图 / 系统支撑 / 知识库治理 / 其他
范围边界：
业务域 / 模块：
关键主体：
业务 owner：
资料来源：
当前已有项目 / 系统：
验收方式：
待确认项：
本轮不做：
```

## 2. 规划 Loop

业务架构规划按从问题到决策的顺序推进：

```text
真实问题
-> 战略意图
-> 业务能力地图
-> 价值流 / 关键流程
-> 核心对象与规则
-> 能力-项目-系统-数据映射
-> 差距 / 依赖 / 优先级
-> 项目组合 / 路线图
-> 产品 / 工程交接
-> 知识库回流
```

每一步都要分层表达：

- 已知事实：来自战略材料、访谈、PRD、系统清单、数据报表、工单、流程文档或代码证据。
- 合理推断：从事实推导出的能力边界、重复建设、依赖或风险。
- 待确认项：需要业务 owner、产品 owner、架构师、合规、财务或运营确认。
- 范围外不做：本轮不讨论的业务域、项目、系统、组织问题或实现细节。

## 3. 业务能力地图

能力不是部门、页面、系统、接口或项目名。能力是某个业务主体为了实现业务目标而稳定需要的业务结果和行为集合。

能力定义至少包含：

```text
能力卡

能力名：
上层目标：
目标主体：
可观察业务结果：
输入：
输出：
关键对象：
关键规则：
能力 owner：
现有项目 / 系统支撑：
验收口径：
边界外：
```

能力分层建议保持三层以内：

- L1 业务域能力：例如客户经营、交易履约、资金结算、风险控制、运营服务。
- L2 核心业务能力：例如客户准入、合同管理、订单履约、商户结算、争议处理。
- L3 可交付能力单元：能进入 PRD、Backlog、系统设计或运营流程的最小能力块。

不要为了显得完整把能力拆成过细的功能点。若一个能力无法说明 owner、业务结果、对象、规则和验收，它还不是可规划能力。

## 4. 能力-项目-系统映射

业务架构必须能发现投入浪费和交接断点。做完能力地图后，把能力映射到项目、系统、数据和运营动作：

```text
能力-项目-系统映射

能力：
现有项目：
现有系统：
数据源 / 报表：
接口 / 事件：
运营动作：
重复建设：
支撑缺口：
风险：
owner：
证据来源：
```

判断口径：

- 一个能力对应多个项目：检查是否存在重复建设、边界不清或阶段性接力。
- 一个项目覆盖多个能力：检查是否需要拆分 owner、验收和路线图。
- 一个系统支撑多个能力：检查公共能力、变更风险和数据口径 owner。
- 能力没有系统支撑：判断是人工流程、外部系统、临时方案还是战略缺口。
- 系统没有能力归属：判断是否是历史包袱、内部技术资产、可退役能力或未命名业务能力。

## 5. 差距和项目组合

业务架构规划的价值在于支持决策，而不是产出漂亮图。差距和项目组合输出应直接回答投什么、不投什么、先做什么、停什么。

差距 / 依赖 / 优先级矩阵：

```text
能力：
现状证据：
目标状态：
差距：
业务影响：
风险等级：
依赖：
候选项目：
建议动作：保留 / 合并 / 延后 / 停止 / 新建 / 继续调研
优先级：
owner：
准出证据：
停止条件：
```

优先级不要只按老板声音、项目大小或系统难度排序。至少同时看业务价值、失败成本、风险暴露、依赖阻塞、复用价值、验证难度和可逆性。

## 5A. 图形化辅助路由

业务架构可以画图，但图只是辅助决策和沟通，不是业务架构本身。命中图形化表达时，继续读取 `diagram-output.md`；正式图形化交付默认只生成 SVG，Mermaid / Markdown 草图、PNG、PDF 或截图等其他格式必须由用户明确提出。

按问题选择最小图形：

- 战略到能力：业务能力地图。
- 端到端价值实现：价值流 / 跨角色流程图。
- 能力落地现状：能力-项目-系统-数据映射图。
- 投资取舍：差距 / 依赖 / 路线图。
- 工程交接：产品到系统上下文图。
- 高风险对象：对象生命周期 / 状态机 / 规则决策图。

每张图必须能回到事实来源、能力 owner、业务规则、待确认项和下一步决策。图不能替代业务确认、产品判断、工程设计、Execution Grant 或上线审批。

## 6. 学习结果与知识库规划

业务架构结论必须能沉淀到项目知识库或上下文中，并且按业务域或模块分区保存。不要把所有学习结果塞进一个总文档，也不要把一次会议纪要当作长期知识库。

推荐目录结构：

```text
knowledge-base/
  README.md
  <业务域或模块>/
    能力地图.md
    术语与对象表.md
    流程状态与判断规则.md
    能力-项目-系统映射.md
    项目组合与路线图.md
    接口事件与数据.md
    待确认与决策记录.md
```

若项目已有知识库约规，优先遵守本地入口，不强行创建新目录；但仍应明确每类学习结果的权威落点。

知识库规划卡：

```text
知识库规划卡

业务域 / 模块：
权威入口：
能力地图保存到：
术语与对象保存到：
流程状态规则保存到：
能力-项目-系统映射保存到：
项目组合与路线图保存到：
接口事件数据保存到：
待确认与决策保存到：
更新 owner：
证据来源：
复审触发：
不得写入：
```

保存规则：

- 事实和证据保存到对应业务域或模块，不保存在跨域总文档里。
- 跨域共识只在总入口做索引，链接到各业务域权威文件。
- 未确认内容进入 `待确认与决策记录.md`，不得写成已确认知识。
- 外部文章只保存来源 metadata、可迁移方法和吸收边界，不保存原文、图片或付费内容。
- 涉及客户、商户、交易、合同、密钥、生产配置或内部敏感数据时，必须脱敏或不写入。

## 8. 反模式和停止条件

常见反模式：

- 从组织架构图开始，把部门职责误当业务能力。
- 从系统清单开始，把已有系统合理化为业务架构。
- 从全量大图开始，缺少真实决策问题和 owner。
- 把战略口号拆成项目清单，却没有能力差距、依赖和停止条件。
- 把能力地图画成页面功能、菜单、接口或技术组件。
- 只输出图，不输出能力 owner、项目组合、路线图和知识库落点。
- 把外部文章或顾问框架直接套进当前组织，不区分事实、推断和待确认。

停止条件：

- 战略意图、真实问题或决策场景缺失。
- 关键业务 owner 不明确，且无法从材料中判断。
- 能力无法追到目标主体、业务结果、对象规则或验收口径。
- 项目组合建议会影响预算、组织、公共契约、合规、生产或数据口径，但缺少确认方。
- 知识库落点不清，可能导致结论散落在聊天记录或一次性文档里。
