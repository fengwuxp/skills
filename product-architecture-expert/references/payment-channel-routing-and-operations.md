# 支付通道与路由运营参考

本文用于设计支付通道接入、路由、降级、运营治理和通道对账能力。它不替代支付机构、银行、收单行、发卡行、卡组织或本地清算网络的正式协议。

## 适用场景

- 设计收单、代扣、代付、出入金、发卡、钱包或本地支付方式的多通道接入。
- 设计支付路由、支付方式展示、失败降级、成本优化、成功率优化和通道运营后台。
- 设计通道账户、商户号、密钥、能力开通、结算周期、费率、限额和 SLA 管理。
- 设计通道文件、通道流水、外部 reference、手续费、结算批次和资金到账对账。

## 通道对象模型

- **Channel Provider**：支付机构、银行、收单行、发卡处理商、开放银行、钱包、本地支付网络或聚合服务商。
- **Channel Account**：平台在通道侧的账户、商户号、子商户号、结算账户、资金池或 program 账户。
- **Capability**：通道支持的支付方式、币种、国家/地区、交易类型、capture、refund、void、dispute、payout、tokenization 等能力。
- **Credential**：API key、证书、密钥、webhook secret、文件传输凭证和回调地址。
- **Routing Rule**：按场景、主体、支付方式、币种、金额、地域、风险、成本、成功率和可用性选择通道。
- **Settlement Profile**：清算文件、结算周期、手续费口径、资金到账账户、结算币种和对账字段。
- **Health Signal**：成功率、延迟、错误码、超时、回调延迟、文件延迟、人工公告和资金到账异常。

## 路由决策维度

- **业务维度**：业务线、商户、订单类型、履约方式、订阅/一次性、B2B/B2C、风险等级。
- **支付维度**：支付方式、卡品牌、BIN、卡类型、ACH debit/credit、wallet、本地支付方式、是否支持 recurring。
- **资金维度**：币种、金额、汇率、手续费、结算周期、退款路径、拒付责任和资金可用时点。
- **风险维度**：用户风险、商户风险、设备/IP/地理、MCC、退款率、拒付率、ACH return 率、异常频次。
- **体验维度**：支付方式展示顺序、跳转/内嵌、3DS/强认证、到账时效、失败原因可解释性。
- **合规维度**：法域、牌照、地区限制、制裁名单、数据跨境、通道协议和卡组织/清算网络规则。
- **运营维度**：通道稳定性、SLA、维护窗口、人工处理能力、客服可追踪性和成本预算。

## 路由执行原则

- 先判定资质、法域、支付方式可用性和客户资金边界，再谈成功率和成本优化。
- 路由应在发起外部请求前形成可审计决策，记录候选通道、命中规则、排除原因、版本号和操作者。
- 同一支付尝试只允许一个明确的外部语义；切换通道前必须确认前一通道的授权、扣款、出款或撤销语义。
- 对授权类、代扣类、出款类交易，不得因超时就盲目换通道重试，必须先做幂等、状态回查和重复扣款/重复出款防护。
- fallback 要区分用户可感知重试、后台补偿重试和人工处理，不要把它们混成一个“重试次数”字段。
- 支付方式展示排序可以做实验，但真实扣款、清算、结算和争议责任必须按可审计规则落地。

## 通道健康与降级

- 健康指标至少覆盖成功率、拒绝率、超时率、平均延迟、回调延迟、文件延迟、资金到账延迟和异常错误码。
- 降级策略要区分限流、暂停新增、只允许查询、只允许退款、人工审核和完全熔断。
- 通道恢复不能只看 API 探活，要结合交易成功率、回调恢复、对账文件恢复和资金到账恢复。
- 维护公告、接口变更、证书过期、密钥轮换、费率调整和协议变更应进入配置变更治理。

## 成本与收益口径

- 不要只用单笔费率做路由。还要考虑汇率、固定费、跨境费、拒付费、退款费、失败成本、人工处理成本和资金占用。
- 收单场景下要区分 MDR、interchange、scheme fee、processor fee、平台加价和商户收费。
- 代付或银行转账场景下要区分提交费、成功费、退汇费、加急费、同城/跨行/跨境费用和银行附加费。
- 成本优化不能牺牲资金安全、合规边界、用户告知、争议处理和财务可解释性。

## 通道对账与结算

- 通道层必须保存外部订单号、外部交易号、网络参考号、批次号、资金到账账户、手续费和原始状态。
- 对账要区分交易对账、手续费对账、退款对账、拒付/争议对账、资金到账对账和结算文件对账。
- 同一个业务单可能对应多次支付尝试、多个通道事件或多个清算事件，不能只靠业务单号做一对一匹配。
- 通道差错应能驱动补单、补回调、补入账、冲正、挂账、调账和人工工单。

## 配置治理

- 路由规则、费率、限额、商户能力、通道开关和密钥变更必须有版本、审批、灰度、回滚和审计日志。
- 高风险配置应四眼复核，例如出款通道、结算账户、费率、限额、白名单、风控绕过和资金调拨。
- 路由规则要能解释“为什么命中这个通道”，否则出现资金差错时无法复盘。

## 设计落地检查

1. 是否定义通道、通道账户、能力、凭证、路由规则、结算口径和健康信号。
2. 是否区分支付方式展示排序、支付尝试路由和资金清结算路由。
3. 是否说明路由决策字段、规则版本、审计日志和命中/排除原因。
4. 是否覆盖超时、回调延迟、重复请求、通道切换和重复扣款/重复出款防护。
5. 是否定义通道健康、熔断、恢复、维护公告和证书/密钥治理。
6. 是否能按通道、商户、币种、支付方式和批次追踪成本、成功率、差错和资金到账。
7. 是否把通道协议、费率合同、卡组织/清算网络规则和合规限制列为待确认项。
