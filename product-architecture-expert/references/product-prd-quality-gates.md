# PRD 质量门禁

本文是 `product-prd-template.md` 的评审与验收门禁。用于生成、补全、评审 PRD 时判断文档是否可评审、可开发、可测试、可运营。

## 使用时机

- PRD 进入正式评审、提交前检查、CR、触发验证或符合性评审。
- 用户要求完善、补全、改写、审查已有 PRD，或要求判断 PRD 是否可用。
- 用户要求多个 AI、PM / Reviewer、产品大师、MAGI、多视角或合议式评审 PRD、AI 生成方案、HTML Demo 或产品方案。

## 不适用场景

- 只需要快速草稿或一页摘要时，可先使用 `product-prd-template.md` 主骨架。
- 涉及支付资金时，还必须读取 `product-prd-financial-appendix.md` 和支付专项 reference。

## 读取后必须产出

- 符合项、必改、建议、可选，以及每条问题的章节/位置、影响、建议改法和验证方式。
- 需要评审会前置扫描时，先输出“疑似问题清单”，再由产品 owner、QA、架构 owner 或业务 owner 判断优先级；不得把 AI 扫描结果直接写成已确认缺陷。

## 需要继续读取的 reference

- 可复制 PRD 主模板读 `product-prd-template.md`；PRD 写作标准读 `product-design-and-prd.md`；结构完整性用 `scripts/check_product_deliverable.py --kind prd`。
- 合议式产品评审读 `product-deliberation-workflow.md`；合议评审报告结构完整性用 `scripts/check_product_deliverable.py --kind product-review`。

## 1. PRD 质量门禁

提交评审前至少确认：

- 目标、范围、非目标和验收标准清楚。
- 用户、主体、角色、权限和责任边界清楚。
- 核心概念、业务抽象、核心对象、状态机、业务规则和字段口径可以解释所有关键流程。
- 术语歧义、新业务对象、多主体协作、平台化能力或工程交接会影响理解时，必须有核心概念与业务抽象章节；若裁剪不展开，应说明“不需要单独展开”的依据。
- 主流程、逆向流程、异常流程和人工处理流程闭环。
- 运营后台、审计日志、报表、指标和数据追溯已覆盖。
- 风险、依赖和待确认项有负责人和截止时间。
- 涉及支付、资金、合规、隐私、安全或其他强监管场景时已进入专项审查，并保留外部规则时效性待确认项。

## 2. 文档治理门禁

提交评审前还要判断文档是否可读、可执行、可维护：

- 文档目标、目标读者和文档类型清楚；PRD、MRD、BRD、一页摘要或专项附录的边界没有混用。
- 正式 PRD 是最终交付标准版本，不是讨论纪要、迭代草稿或 AI 生成过程记录；正文不得保留多轮争论、被拒方案展开、prompt/回答轨迹、旧版本流水和无主建议。
- 主文档可扫读，复杂细节拆到附录、矩阵或图；不要用篇幅堆砌替代目标、范围、规则和验收。
- 图形化视图选择合理：多角色、多对象、多状态、多团队交接或高风险评审时，应至少有用例图、流程图、泳道图、状态机图或能力地图之一；轻量 PRD 可明确“不需要单独画图”。
- 图文一致：图中的角色、对象、状态、流程、规则、异常、人工兜底、风险和待确认项必须能回到正文、规则矩阵和验收种子；图不能表达正文没有确认的结论。
- 信息传递顺序清楚：事实和证据先于判断，判断先于决策，决策先于承诺、验收和交接。
- 表格只用于比较、矩阵或追踪；单个问题、规则、验收和待确认项优先用卡片、短段落或编号流程表达。
- 描述可靠性清楚：已确认、假设、待确认和不适用分开标注；无法确认的信息不得写成确定结论。
- 不能只写功能名；功能需求至少能说明入口、流程、交互、规则、异常和验收样例。
- 不能用功能名、页面名或系统字段替代核心概念和业务抽象；关键概念必须说明定义、适用范围、反例、责任方和与对象/规则/验收的追踪关系。
- 版本治理保留当前版本号、状态、owner、评审结论摘要、影响范围、通知对象和过程记录链接；完整变更原因、讨论过程、修订流水和被拒方案进入任务计划、产品合议评审报告、Decision Log 或中间任务文档。
- 文档、原型、规则矩阵、验收标准和工程交接口径一致；不一致时必须先列入必改或待确认项。
- P0/P1 需求能通过主模板需求链回到问题/证据、目标、场景、能力/功能、规则和验收种子；无法追踪的条目先列为必改或待验证。
- 合议式评审能说明触发原因、阶段门、共识、分歧、必改、建议、待确认、owner、验证方式和下一步去向；缺失时不能作为准出结论。

## 3. 可验收性门禁

需求进入评审前，还必须确认：

- 每条功能需求都有需求 ID、优先级、角色、场景、前置条件、输入、处理、输出、异常和验收样例。
- 涉及核心概念或业务抽象的需求，验收标准能回到概念定义、抽象边界、不变量、对象状态或规则编号。
- 用户故事包含角色、能力和价值，不只写“作为用户，我想要”。
- 成功指标有口径、基线、目标值、观察窗口和负责人，避免停留在“提升体验”“优化效率”等不可验证表述。
- 验收标准可以回到需求 ID、业务流程、规则矩阵、状态机或数据指标。
- 图形化验收路径清楚：关键流程图、状态机图或泳道图能帮助测试和研发识别正常、边界、异常、运营和风险路径；没有图时，文本流程必须同样可追踪。
- 验收种子能回到需求 ID、场景编号、规则编号或质量属性 ID；不能只有独立的“测试通过”。
- 验收描述必须包含可观察结果，避免只写“完成开发”“测试通过”“页面正常”等不可复核表述。
- 缺少章节、不可测试需求、不可衡量指标、范围边界不清、验收标准无法追溯和强监管确认方缺失，提交评审时都归入“必改”。

## 3.1 AI 预扫描四维度

评审会前可先让 AI 做广度扫描，目标是提前暴露“该问什么”，不是替代正式需求评审。扫描结果统一标为疑似问题，进入人工过滤、排序和 owner 决策。

四个维度：

- 完整性：目标、范围、非目标、用户/主体/权限、前置条件、输入输出、异常、性能、数据、运营、风险和验收是否缺失。
- 一致性：同一对象、字段、状态、规则、权限、流程和验收在不同章节、模块、原型、规则矩阵或接口说明中是否互相矛盾。
- 可测试性：成功指标、功能需求、非功能要求和验收标准是否可观察、可测量、可设计测试用例；“体验好”“及时”“大量”“稳定”等表述必须补口径。
- 二义性：句子、术语、角色、触发条件、次数、时长、范围、例外和失败处理是否存在多种合理解释。

输出格式：

```text
AI 预扫描维度：完整性 / 一致性 / 可测试性 / 二义性
锚点：<章节 / 图 / 表 / 需求ID / 原型位置>
疑似问题：
影响：评审 / 开发 / 测试 / 上线 / 运营 / 风险
建议追问：
建议改法：
决策：ACCEPT / REJECT / PENDING
owner：
验证方式：
```

使用红线：

- 预扫描只负责找疑似问题和追问点；是否采纳必须由 owner 判断。
- AI 预扫描不能替代正式需求评审、QA 测试设计或产品 owner 决策；正式评审仍需要人工判断、排序和确认。
- 格式越统一，扫描越可靠；缺少团队 PRD 模板或章节锚点时，先列“源材料缺失”，不要脑补。
- 对业务术语、历史约定、合规例外和领域黑话，AI 可能误判二义性；必须用事实锚点和人工确认过滤。
- 预扫描结果属于过程资产；进入正式 PRD 时，只吸收已确认的最终修改、风险、待确认和验收，不复制整段扫描过程。

## 4. 已有 PRD 符合性评审输出

评审已有 PRD 时，输出固定分为：

- 符合项：已经满足模板、可验收性或专项门禁的内容。
- 必改：会阻断评审、研发、测试、上线或专业确认的缺口。
- 建议：不阻断交付，但会影响协作效率、理解一致性或后续维护的问题。
- 可选：不影响当前版本核心目标的表达润色、附录补充或后续增强。

每条评审项必须说明章节或位置、问题、影响和建议改法；正式评审同时运行 `scripts/check_product_deliverable.py --kind prd`，并把脚本发现的结构缺口并入“必改”。

## 5. 合议式产品评审输出

评审 AI 生成方案、复杂 PRD、原型候选或多方争议需求时，输出固定分为：

- 触发原因：为什么需要合议式评审，而不是直接生成正文。
- 当前阶段：Round 0、方案共识、交互/图形验证、PRD 定稿或准出检查。
- 共识：已经确认且可进入下一步的目标、范围、对象、流程、规则或验收。
- 分歧：尚未统一的取舍，必须包含影响、备选方案、owner 和未决期间处理方式。
- 必改：阻断评审、研发、测试、上线或专业确认的问题。
- 建议：不阻断但影响可读性、可用性、协作效率或后续维护的问题。
- 待确认：问题、影响章节、确认方、未确认前处理和验证方式。
- 下一步：进入 PRD、Backlog、图形化、AI Native 编排、架构师交接或暂停补证据。

合议式产品评审报告是过程资产。进入正式 PRD 时，只把已确认的共识、决策、风险、待确认和验收改动吸收进正文；分歧过程、被拒理由长文、角色争论和多轮草稿保留在评审报告或 Decision Log，通过链接追溯，不复制到 PRD 主体。

正式合议评审报告运行 `scripts/check_product_deliverable.py --kind product-review`；脚本通过只代表结构完整，不代表产品判断、业务确认、合规审查或工程验证已经通过。
