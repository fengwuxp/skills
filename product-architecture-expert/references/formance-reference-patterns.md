# Formance 参考模式

本文提炼 Formance 官方 Wallets、Ledger、Reconciliation、Connectivity、Numscript、Flows 文档中的通用产品与系统设计模式。它只作为支付产品、账务、连接器、流程编排和对账设计参考，不替代监管规则、财务制度、机构接口规范或团队自研架构决策。

## 使用时机

- 设计电子钱包、商户余额、预付账户、平台内部账户、补贴账户或多币种账户。
- 设计账务核心、产品账本、资金流水、余额查询、冻结解冻、调账和审计追踪。
- 设计通道、银行、存管账户、客户备付金账户、商户结算账户与平台内部账之间的对账。
- 设计外部支付服务商、银行、开放银行、虚拟账户、现金池和交易导入的连接器层。
- 设计可配置账务规则、分账规则、手续费规则、退款规则和跨模块资金流程编排。
- 需要把支付产品从“交易状态机”提升为“可解释、可核对、可追溯的资金事实系统”。

## 不适用场景

- 只做卡产品、VCC 或收单运营方案时，优先读 `highnote-reference-patterns.md`、`virtual-card-and-vcc.md` 或 `payment-channel-routing-and-operations.md`。
- 本文只抽象公开产品模式，不替代团队自研账务架构、监管规则、财务制度或机构接口规范。

## 读取后必须产出

- 钱包层、账本层、对账层、连接器层、记账规则、流程编排、事实源、幂等审计和待确认项。

## 三层 + 三个支撑能力

Formance 的模块划分可抽象成三层核心能力和三个支撑能力：

1. **Wallets：业务钱包层**。面向用户、商户、服务商或平台内部对象，表达余额、冻结、收付、转账和业务权限。钱包层是有业务语义的账户产品，不应直接暴露底层账本细节。
2. **Ledger：账本层**。面向不可变账务事实，记录账户、交易、分录、余额和元数据。账本层负责资金事实的可审计、可重放和可查询，不承载页面流程。
3. **Reconciliation：核对层**。面向内部账本与外部现金池、银行流水、通道文件、清算机构文件之间的差异识别、策略匹配、漂移处理和闭环核销。

三个支撑能力分别是：

- **Connectivity：连接器与外部账户层**。连接外部支付服务商、银行、开放银行、虚拟账户和交易数据，形成可管理的外部账户、现金池和同步状态。
- **Numscript：记账规则表达层**。用受限、可读、可审查的脚本表达资金移动、分账、手续费、退款、留存和约束，避免把账务规则散落在业务代码里。
- **Flows：流程编排层**。把钱包、账本、连接器、审批、事件等待和补偿动作编排成可追踪流程，适合长事务、异步通道和跨系统资金流程。

产品设计时可采用“钱包表达业务、账本表达事实、连接器接入外部世界、对账表达可信度、脚本表达规则、流程表达编排”的分层。不要把钱包余额、数据库余额字段、通道返回状态和银行真实到账混为一谈。

## Wallets 模式

- 钱包是业务对象，不只是余额字段；必须定义持有人、币种、余额类型、权限、冻结规则、收付方向和可用性。
- 钱包操作要映射到底层账本交易：充值、支付、退款、冻结、解冻、转账、提现、调账都应能追溯到分录。
- 钱包余额通常需要区分 `available`、`pending`、`held/frozen`、`settled`、`in_transit` 等口径；具体命名按业务语境调整。
- 一个业务钱包可以映射多个账本账户，例如可用余额、冻结余额、待结算余额、费用账户、在途账户。
- 钱包 API 不应绕过幂等、权限、额度、风控、审计和对账约束直接修改余额。

## Ledger 模式

- 账本账户应按主体、业务类型、资金属性、币种和余额方向建模，而不是按页面字段建模。
- 每一笔账务交易由多条分录构成，借贷或入出方向必须平衡；任何单边余额变化都应被视为风险信号。
- 账本记录应不可变；更正错误使用冲正、反向分录、调账或补记，不直接改历史流水。
- 分录必须带业务事件号、幂等键、外部参考号、币种、金额、发生时间、记账时间、来源系统和元数据。
- 账本应支持按账户、交易、业务参考、时间、币种、主体和批次查询，支撑客服、运营、财务和审计追踪。
- 设计时区分产品账本、会计总账和银行真实资金流水。产品账本可服务业务实时性，但最终财务口径需由财务制度确认。

## Bi-Temporality 模式

账务系统要区分至少两类时间：

- **业务发生时间**：交易、退款、费用、清算或外部事件实际发生的时间。
- **账本记录时间**：系统把该事实写入账本、进入报表或参与对账的时间。

跨日通知、补单、延迟文件、退款跨结算周期、汇率变动和人工调账都会造成两类时间不一致。方案中必须说明交易日、账务日、清算日、结算日、自然日和批次日如何取值，以及重跑时按哪个时间维度归属。

## Reconciliation 模式

- 对账对象应明确为“内部账本账户”与“外部现金池/银行账户/通道账户/清算机构文件”的关系。
- 对账不能只比订单号；要定义数据源、时间窗口、匹配键、金额容差、手续费拆分、状态映射、币种和汇率口径。
- 对账策略应能配置，例如一对一、一对多、多对一、延迟到账、批量结算、手续费净额、部分退款和跨日归属。
- 差异要形成可流转对象，至少包含差异类型、来源、责任方、金额、币种、账龄、处理动作、审批、核销和审计日志。
- 对账结果必须能反向驱动补单、补入账、冲正、挂账、调账、重跑、人工确认和财务凭证。
- 对账不只是发现差异，更要定义差异的生命周期和关闭条件。

## Connectivity 模式

- 连接器层负责接入外部支付服务商、银行、开放银行、虚拟账户和清算/通道数据，不承载内部账务规则。
- 外部账户要映射为内部可识别对象，至少包含机构、主体、币种、账户类型、账户标识、权限范围、同步状态和资金归属。
- 现金池或外部资金池要按机构、主体、币种、账户和用途拆分，不能把多个资金性质不同的账户混成一个余额。
- 外部交易导入要记录来源、原始流水号、对方账户、金额、币种、状态、发生时间、入账时间、拉取批次和原始报文摘要。
- 连接器应具备幂等拉取、断点续跑、去重、验签、重试、限流、降级、监控和人工补录/补文件机制。
- 连接器输出的是外部事实，是否记账、如何记账、是否可结算，必须交由账本、对账、风控和业务规则判断。

## Numscript 模式

- 记账脚本适合表达可审查的资金移动规则，如充值、支付、分账、手续费、退款、冻结、释放、结算和调账。
- 脚本必须是受限 DSL 或配置化规则，不应具备任意代码执行、外部调用、数据库查询、权限判断或副作用能力。
- 每条脚本规则都要说明输入变量、资金来源账户、目标账户、金额口径、币种、舍入规则、余额约束和失败语义。
- 复杂规则应版本化，脚本版本、参数快照和执行结果要写入账务元数据，保证历史交易可复现。
- 脚本不能替代财务确认；借贷方向、科目、收入成本确认、税费和财务凭证仍需财务制度确认。
- 对高风险资金规则，必须提供测试用例、样例交易、边界金额、余额不足、部分退款、跨日和重跑场景。

## Flows 模式

- 流程编排适合长事务和跨系统资金流程，如入金、出金、商户结算、退款退汇、差错处理、人工审批和跨通道补偿。
- 流程步骤应可追踪、可恢复、可重试、可暂停、可取消，并记录每一步的输入、输出、外部引用和责任方。
- 流程中可以等待外部事件，例如通道通知、银行回单、对账结果、审批结果、KYC/KYB 完成或风控放行。
- 编排层不应直接绕过账本、风控和权限修改余额；资金事实仍由账本交易表达。
- 每个流程都要定义幂等键、超时、重试、补偿、人工介入、告警、状态机和终态。
- 对资金流程，必须区分“业务流程完成”“账务已入账”“真实资金到账”“可结算”“财务已确认”等不同完成口径。

## 设计落地检查

1. 钱包层是否只表达业务能力，而不直接篡改底层账本。
2. 账本层是否能用分录解释每一笔余额变化。
3. 账户模型是否覆盖主体、币种、资金属性、余额方向和业务约束。
4. 所有写账动作是否有业务事件号、幂等键、唯一约束和审计元数据。
5. 错账是否通过冲正、反向分录或调账处理，而不是改历史流水。
6. 是否区分业务发生时间、账本记录时间、清算日和结算日。
7. 对账是否覆盖内部账本、外部通道文件、银行流水和资金池余额。
8. 差异是否有自动处理、人工兜底、账龄告警和核销闭环。
9. 外部连接器是否有账户映射、幂等同步、去重、断点续跑、原始报文留痕和异常补录机制。
10. 记账脚本是否受限、版本化、可测试、可审查，且不承担外部调用和权限判断。
11. 流程编排是否支持事件等待、超时、重试、补偿、人工介入和终态定义。
12. 报表和财务凭证是否来自可追溯明细，而不是不可解释的汇总数。
13. 监管、财务、税务和外部机构口径是否被列为待确认项。
