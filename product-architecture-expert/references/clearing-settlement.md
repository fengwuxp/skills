# 内部清结算与账务设计

## 概念语境

- **内部清分/内部清算**：平台或机构内部按合同、业务规则、费率和分账关系计算参与方应收应付。不是持牌清算业务。
- **持牌清算业务**：清算机构按金融基础设施或清算规则处理机构间支付业务和债权债务关系。非清算机构不得从事或变相从事。
- **结算**：基于内部清分/清算结果或外部清算结果完成资金划拨、出款或余额入账。
- **清结算**：口语中常指“把账算清，把钱给对”，正式方案中必须拆成内部核算、对账、资金划拨和财务入账。

国内不同企业会混用这些词。讨论前先说清语境：平台内部清结算、第三方支付机构清结算、银行间清算、跨境清算不是同一件事。

## 清算、结算、清结算辨析

清算和结算必须先定语境，再下定义：

- **理论二元语境**：清算负责支付指令交换、清分、轧差和债权债务计算；结算负责最终资金转移。
- **机构命名语境**：清算机构、结算账户、清算账户、登记结算系统等名称可能来自不同监管或行业习惯，不能只凭名称判断职责。
- **平台产品语境**：很多企业把清分、记账、结算、出款、对账都统称“清结算”，正式方案必须拆开。
- **内部核算语境**：平台内部清算通常是按合同和规则算清楚应收应付，不等同于持牌清算业务。

写 PRD 或方案时，首次出现“清算”“结算”“清结算”必须说明：主体是谁、对象是谁、算的是什么、钱是否真的动了、最终资金在哪个账户完成转移、由谁确认。

## 支付清算生态分层

参考陈天宇宙公开文章《图解支付清算生态全景》的宏观拆解，支付清结算方案不能只看平台内部表和结算单，还要先把生态参与者和机构层级说清楚：

1. **交易平台层**：提供交易场景和用户入口，负责选购、计价、订单、账单、收银台、履约、售后和商户结算诉求。
2. **支付服务层**：通常由支付机构、PSP、收单机构或平台支付中台提供，负责收款、付款、退款、路由、风控、账务、清算和商户服务。
3. **清算服务层**：清算机构或支付网络负责机构间信息转接、交易清算、清算文件、轧差、结算指令和参与方额度管理。
4. **金融服务层**：商业银行提供账户、银行卡、协议支付、企业账户、资金存放、扣款和入账等金融基础。
5. **央行/金融基础设施层**：提供清算账户、大小额支付系统、网银跨行清算系统、会计核算和支付信息管理等最终结算基础设施。

输出方案时，必须标明当前产品属于哪一层、依赖哪一层、资金最终在哪一层完成清算或结算；不要把平台内部清分、支付机构清算、清算机构轧差和央行最终结算混成一个“结算成功”状态。

## 典型链路

```text
业务订单
  -> 支付单
  -> 通道请求
  -> 支付结果
  -> 客户/商户入账
  -> 日切
  -> 通道/清算机构文件
  -> 交易对账
  -> 银行/存管/客户备付金账户资金到账
  -> 资金对账
  -> 商户结算单
  -> 出款/代付
  -> 财务凭证/报表
```

## 交易平台七段全链路

交易平台型产品要把支付放回完整交易链路中检查：

1. **选购环节**：商品、购物车、权益、优惠和价格基础。
2. **交易处理环节**：计价、订单、账单、账单支付明细和卡券冻结。
3. **支付处理环节**：收银台、支付核心、支付单、通道请求和支付结果。
4. **履约处理环节**：发货、上门服务、到店核销、虚拟权益发放或其他交付事实。
5. **清结算处理环节**：基于最终交易和履约结果完成清分、手续费、分账、分润和结算计算。
6. **账务处理环节**：把交易、支付、清分、结算和调账转为账户流水、账务分录和余额事实。
7. **财税票处理环节**：形成会计核算、税务计提、发票开具或接收、财务报表和审计材料。

这七段不是每个系统都要独立建设，但 PRD、产品方案和图形化交付必须说明本期覆盖到哪一段、上下游交接条件是什么、缺失段落由谁负责。

## 跨机构清算四步

当付款人与收款人的资金账户不在同一机构，或支付机构、清算机构、银行之间需要跨机构处理时，按四步拆解：

1. **联机交易**：交易平台或支付机构发起支付指令，清算机构或支付网络转接至付款行或相关参与方，完成签约校验、扣款、受理或拒绝。
2. **实时清算**：清算机构登记交易明细，按场次或实时规则更新参与机构可用额度、清算明细或内部清算结果，并回传交易处理结果。
3. **定时结算**：清算机构按结算场次对参与方净额轧差，把净额提交最终结算基础设施或银行账户完成资金划拨。
4. **日终处理**：各机构获取清算/结算文件、动账通知、对账文件和汇总账单，完成内部对账、备付金或银行账户入账、商户结算和财务处理。

产品文档要区分“联机交易成功”“清算明细确认”“机构间结算完成”“商户可结算”“财务已确认”。这几个状态的时间、文件、责任方和失败处理都不同。

## 备付金与额度口径

涉及第三方支付机构、清算机构或客户备付金语境时，必须把以下口径拆开：

- **集中存管账户余额**：客户备付金集中存放账户中的真实余额，具体监管和账户安排以最新法规、机构协议和专业确认结果为准。
- **映射额度**：支付机构向清算机构或网络分配并被确认的额度，用于限制或支持清算场次内的收付处理。
- **可用额度**：基于映射额度、入金、出金、清算结果实时或准实时变化的业务处理额度。
- **清算净额**：某个场次或周期内收款、付款、退款、手续费等轧差后的净流入或净流出。

不要把可用额度当成真实现金余额，也不要把映射额度变化写成真实资金划拨。涉及备付金、集中存管、清算账户、特许参与者或央行支付系统时，必须进入 `regulatory-baseline.md`，并标注待法务、合规、财务、通道或持牌机构确认。

## 两条线

- **收款线**：用户付款、通道或清算机构处理、商户待结算、手续费收入、商户结算。
- **付款线**：平台/商户付款、代付通道、付款通道处理、退汇/失败、成本和资金回补。

收支两条线独立建模，再在财务核算层汇总。不要用一套状态和一组账户勉强覆盖所有方向。

## 四段数据

清结算方案至少说明四类数据如何产生和互相核对：

1. **账务数据**：账户系统或账务核心产生的余额流水、冻结流水、会计事件。
2. **支付数据**：支付平台记录的收款、退款、付款、打款退回等业务记录。
3. **清分/清算数据**：内部规则计算结果，或通道、清算机构给出的交易文件、手续费、应收应付。
4. **结算数据**：银行入账、存管/客户备付金账户到账、出款回单、资金流水。

## 清账、结算、对账三段门禁

参考公开文章《清结算和「对账」不是一回事——我们曾因此对错账、赔过钱》的案例化提醒，平台内部清结算不能把清账、结算和对账混成一个总金额校验。

- **先清账，再结算**：交易/订单、手续费、退款、调账、冲正、补结、扣回、拒付或退单等所有影响应收应付的事件，都必须进入清账模型并参与轧差；渠道、银行或清算机构文件中的负项、调账项和非正常交易不能只在结算层处理。
- **结算只认清账结果**：结算输入应是清账批次、结算对象、净应收/应付和风险限制；渠道到账金额、银行流水或上游结算文件只能用于合理性校验和资金对账，不能直接反推商户应结金额。
- **先对清账，再对结算**：第一层对“数”，检查业务数据、支付/退款/手续费/调账数据、渠道文件和清账结果是否一致；第二层对“钱”，检查清账结果驱动的结算金额和银行、通道或存管实际划付/到账是否一致。
- **异常先回清账归因**：如果结算金额、到账金额或商户账单不一致，先判断是清账漏项、重复项、费率/规则错误、跨日归属错误还是执行出款错误；不得只调结算金额绕过清账差异闭环。

## 三类在途

- **客户在途**：支付记录与客户/商户账务入账之间的差异。
- **支付在途**：平台记录与通道/清算机构文件之间的单边或金额差异。
- **资金在途**：通道/清算机构处理结果与银行、存管账户或客户备付金账户实际到账之间的长短款、挂账或未达账。

每一类在途都要有来源、责任方、处理状态、核销规则、超时告警和报表。

## 七个环节

设计清结算时按以下环节逐个验算：

1. 支付交易记账。
2. 通道/清算机构文件与交易对账。
3. 交易差错处理。
4. 商户内部清分与结算单生成。
5. 客户账调账或补入账。
6. 通道/清算机构结算与资金对账。
7. 长短款核销与财务结转。

## 跨业务线清结算全局规划

参考陈天宇宙公开文章《头部大厂，怎么做清结算全局规划，分享一个真实案例！》提炼，遇到多业务线、多套支付清结算体系、线下 Excel 核算、财务人工打款或中台规划场景时，先按“问题盘点 -> 资金路径 -> 能力切分 -> 分期迁移”做全局规划，不要一上来画微服务或页面功能。

### 现状盘点

先从所有业务线的资金流从头到尾梳理一遍，至少检查六类问题：

- 线上线下两层皮：奖惩、收入、补贴、扣款、提现或打款仍在线下处理。
- 系统只出报表不记账：有计算结果，但没有结算记录、账务流水、凭证规则或账户变化。
- 对账只对不处理：发现差异后没有差错单、处理状态、责任方、核销规则和财务入账动作。
- 计算、账务和资金耦合：费用规则、账务记录、账户余额、出款动作混在一个系统或一张大表里。
- 服务抽象度低：能力按单个业务需求堆叠，无法复用，只能复制粘贴。
- 业务线体系分散：不同业务线重复建设清分、账务、账户、结算和对账能力，标准不统一。

这类问题说明当前不是补几个字段或页面能解决，而是需要做资金能力边界重构。

### 五中心能力切分

从“一笔钱从用户支付到参与方收到，中间经历哪些动作”倒推能力中心：

| 动作 | 中心 | 产品职责 |
| --- | --- | --- |
| 算钱 | 清算中心 | 抽佣、分成、奖惩、补贴、税费、规则引擎和任务调度，目标是算得准、规则可配置。 |
| 记钱 | 账务中心 | 账务明细、费用类型、交易类型、凭证规则和流水追踪，目标是记得清、可审计。 |
| 装钱 | 账户中心 | 商户、骑手、平台、营销、保证金等资金容器，可用、冻结、待结算、在途等余额口径。 |
| 结钱 | 结算中心 | 结算单、结算批次、自动结算、手动结算、自主提现、出款和回写，目标是结得对。 |
| 对钱 | 对账中心 | 系统账、通道账、银行账、财务账之间的对账、预警、差错处理和核销。 |

五中心是产品能力边界，不强制要求五个独立系统。小团队可以合并实现，但 PRD 和系统方案必须把职责、对象、单据、状态、权限、数据源和验收口径拆清楚。

### 建设原则

- 专业化：账务、账户、结算和对账要符合财务确认、审计追溯和资金安全要求，不把清结算做成临时报表。
- 模块化：按能力中心降低耦合，每个中心有清楚的输入、输出、单据、状态和异常处理。
- 配置化：金额规则、费率、分成、奖惩、结算周期和对账策略要可配置、可审批、可灰度、可回滚、可追溯。

### 分期迁移策略

全局规划不能要求一次性推倒重来。建议按风险和收益分三步：

1. **线上化先行**：优先把线下金额动作搬上系统，例如押金、奖惩、提现、补贴、扣款和人工打款；用灰度城市、业务线或费用项控制风险。
2. **清算中心跟进**：先建设统一计算规则、任务调度、结算对象和规则版本，再逐步接入多业务线。
3. **账务账户收口**：将旧系统计算逻辑、账户余额、流水、账务分录和对账差错逐步迁入统一模型；旧系统保留只读或过渡能力，直到对账和财务确认稳定。

### 全局架构表达

输出清结算全局规划图时，至少分四层表达：

- 业务接入层：各业务线、交易系统、履约系统、售后系统、运营后台。
- 场景解决方案层：钱包、薪资、佣金、商家结算、保险、押金、补贴、分润等标准方案。
- 核心能力层：清算、账务、账户、结算、对账五中心。
- 基础连接层：规范体系、监控告警、银行通道、第三方支付、财务系统、税务系统和数据仓库。

图形目标不是好看，而是让评审方看清楚：不同业务线如何统一接入，哪些能力被沉淀为通用中心，哪些金额动作还在线下或旧系统，迁移路径和验收口径是什么。

## 账户和科目

常见账户/科目类型：

- 客户/商户余额账户、可用余额、冻结余额、待结算余额。
- 平台内部过渡账户、在途账户、差错账户。
- 存管账户、客户备付金账户（仅限适用主体）或银行存款账户，按通道、机构、币种或账户主体拆分。
- 手续费收入、通道成本、分润应付、营销补贴、税费、退款挂账。
- 应收通道款、应付商户款、应付分账方款、长短款。

账户设计要回答三个问题：余额归谁、为什么变化、什么时候可动用。

复杂钱包或账务核心可进一步区分钱包账户和账本账户：钱包账户面向业务持有人和产品能力，账本账户面向不可变分录和余额事实。一个钱包账户可以映射可用、冻结、待结算、在途、费用等多个账本账户，避免用一个余额字段承载所有资金状态。

## 账务核心与会计基础

支付的底层是账本，账务核心不是余额字段集合，而是把业务事件变成可审计账务事实的系统。设计时至少说明：

- **外围驱动**：支付核心登记渠道清算往来，交易核心登记商户待结算和营销类记账，清结算核心登记分账分润、手续费、通道成本和已结算，打款核心登记出款及退回，对账核心登记已清算、已结算和差错。
- **凭证规则**：每类业务事件必须映射凭证规则、账户科目、借贷方向、金额口径、摘要、业务单据和可重放条件。
- **账户结构**：账户名称、会计科目、凭证号、余额、流水和交易能力要分层表达；不要把账户主体、账户用途和会计科目混成一个字段。
- **会计循环**：业务发生、凭证生成、账簿登记、试算平衡、对账、结账和报表输出要有明确时点。
- **总分核对**：总账、明细账、银行日记账、平台账本和外部资金流水之间要能平行登记、汇总核对和差异追踪。

产品经理不需要替代财务做最终会计结论，但必须能把交易、支付、退款、清分、结算、调账、长短款和费用转成可让财务确认的账务矩阵。

## 对账设计

对账不是简单比订单号，至少定义：

- 数据源：平台支付表、账务流水、通道/清算机构文件、银行流水、结算回单。
- 关键字段：业务单号、支付单号、通道单号、商户号、金额、币种、状态、交易时间、清算日、结算日。
- 匹配规则：一对一、一对多、多对一、金额容差、手续费拆分、跨日归属。
- 差异类型：平台单边、通道单边、金额差、状态差、手续费差、重复记录、延迟记录。
- 处理动作：补单、冲正、补入账、调账、挂账、人工确认、重跑对账、核销。

若采用账本系统，应把内部账本账户与外部现金池、银行账户、通道账户、清算机构文件建立明确映射。对账结果不是只生成报表，还要形成差异对象，驱动补单、补记账、冲正、挂账、调账、审批和核销。

对账顺序必须区分清账对账和结算对账：清账对账先证明订单、支付、退款、手续费、调账和渠道文件已经完整进入应收应付计算；结算对账再证明清账结果对应的出款、到账和回单已经完成。只对结算总额不对清账明细，无法定位漏单、重复、手续费错误或调账漏进的问题。

## 结算系统设计

结算系统必须明确：

- 结算对象：商户、门店、达人、司机、供应商、代理商、分账方。
- 周期：T+1、D+1、D+0、实时、手动、自主提现、自动结算。
- 余额口径：交易金额、可结算金额、冻结金额、退款预留、手续费、税费、已结算、待出款。
- 出款方式：到银行卡、到账户余额、内部转账、代付通道、跨境汇款。
- 风控限制：黑名单、冻结、超限、资料过期、争议订单、负余额、商户风险等级。
- 失败处理：退票、卡异常、通道失败、部分成功、重复出款拦截、人工复核。

结算系统不得把渠道或银行到账金额当作商户应结金额的直接来源。结算批次必须记录清账批次、结算对象、净额构成、扣减项、风险冻结、审核动作、出款指令、执行结果和回单锚点，确保结算动作可以回放到清账结果。

## 外卡收单结算补充

外卡收单场景下，Clearing、Settlement、商户到账必须分开表达：

- **Clearing / Financial Presentment**：让授权后的交易进入卡网络正式账务和责任链路，形成网络可识别的 financial claim。
- **Network member settlement**：issuer、acquirer 等成员之间按网络规则完成应收应付和资金义务。
- **Platform allocation and netting**：收单机构或平台基于上游结算结果、商户、门店、币种、费率、退款、拒付、保证金和风险规则计算商户净额。
- **Merchant settlement / payout**：平台按商户账期、结算模板、risk reserve、rolling reserve、delayed settlement 和最低出款门槛释放可结金额。
- **Bank arrival**：银行或本地清算网络完成最终入账，商户看到银行到账或可提余额更新。

产品方案中要区分 `cleared amount`、`gross settlement basis`、`deductions`、`net settlement amount`、`withheld amount`、`available amount`、`in-transit amount` 和 `paid amount`。多币种场景还要区分交易币种、清算币种、商户结算币种、出款币种、锁汇汇率、结算汇率和银行实际出款汇率。

## 清算文件异常延迟排查门禁

清算文件未按预期到达、迟到、缺失或处理状态异常时，不能先归因于网关慢。外卡收单、跨境收单、多 PSP 或多地区清结算场景，先对齐五件事：

- **日期口径**：交易日、清算日、入账/结算日分别来自哪个字段、哪个系统、面向谁展示；对外“结算日”必须映射到具体清算文件字段、资金文件字段或银行入账字段。
- **Cut-off 与日历**：清算窗口、提交截点、周末、公众假期、多国假期和通道 `Processing Calendar / Holiday Schedule` 是否已进入运营公告、财务日历和大促日历。
- **时区口径**：后台、报表、清算文件、银行流水和通道日志使用 UTC、GMT+8、美国东部或本地时区时，必须统一展示口径，并标明 cut-off 所属时区。
- **文件受理状态**：区分 `accepted but delayed` 和 `rejected / held`，不能只看网关 API 200；必须记录 ACK、确认号、批次 ID、记录数、金额合计、reject reason / error code 和下一档预计处理时间。
- **延迟分段**：每个迟到工单必须标明卡在哪一段：商户/PSP 到收单处理、卡组织/清算网络、银行/代理行/币种中转，并给出证据日志、ACK 或银行/通道回执。

单批迟到不等于钱丢了；连续多批次缺失、同一 MID 大面积 reject、金额轧差突然偏离，才应升级为资金链路级事故信号。升级时必须同时通知产品、支付运营、财务、通道对接和风险负责人，按交易影响、商户影响、资金影响和对账影响分层处理。

## 账务矩阵模板

```text
业务事件 | 触发条件 | 会计借方科目 | 会计贷方科目 | 余额账户变化 | 真实资金动作 | 金额口径 | 对账来源
支付成功 | 通道成功通知 | 存管/客户备付金或在途资产 | 应付商户/客户负债 | 商户待结算增加 | 用户资金进入通道或存管账户 | 订单实付金额 | 通道/清算机构文件
退款成功 | 通道退款成功 | 应付商户/平台垫付科目 | 存管/客户备付金或在途资产 | 商户待结算减少或平台垫付增加 | 原路退回或通道退款 | 退款金额 | 通道退款文件
商户结算 | 结算批次审核通过 | 应付商户负债 | 银行存款/存管出款科目 | 商户待结算减少 | 银行或代付通道出款 | 可结算金额 | 银行回单
```

根据主体、科目类型和财务制度替换借贷方向。借方/贷方不等同于资金流入/流出；任何账务矩阵都必须由财务或账务负责人确认。
