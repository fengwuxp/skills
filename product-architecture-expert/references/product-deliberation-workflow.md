# 产品合议评审工作流

本文用于把复杂 PRD、AI 生成方案、原型候选或多方争议需求，从一次性生成改成可控的多视角合议评审。它不是独立 Skill，也不要求真实启动多个 Agent；它是产品架构专家在复杂场景下使用的流程、角色和门禁参考。

## 使用时机

- 用户要求多个 AI、PM / Reviewer、产品大师、MAGI、三方互审、多视角评审或合议式评审产品方案、PRD、Backlog、AI 原型或 HTML Demo。
- PRD、原型、AI Demo 或产品方案存在明显争议：目标不清、证据薄弱、隐含假设多、功能发散、业务/用户/工程/运营视角冲突。
- 用户给出一句话需求，但希望分阶段确认前置依赖、方案概要、交互或 PRD，而不是一次性写完整文档。
- AI 生成结果需要做产品侧 CR，判断是否值得进入 PRD、Backlog、AI Native 产品上下文包或交给架构师。
- 复杂 PRD 需要分批产出，并在每批后保留共识、分歧、待确认、owner 和验证方式。

## 不适用场景

- 只写一版轻量 PRD 或普通产品方案，且目标、范围、角色、对象、流程、规则和验收已经清楚；直接读 `product-prd-template.md` 与 `product-design-and-prd.md`。
- 只做产品洞察、资料资产化或机会雷达；优先读 `product-insight-analyst.md`。
- 只做 Backlog 决策、BV/EE、User Story 或 AC；优先读 `po-backlog-manager.md`。
- 只做系统设计、代码评审、测试、Bug 修复或生产变更；优先交给 `资深架构师`。
- 不把外部多 Agent 工具、平台专属命令、watchdog、车载专项模板或第三方长 prompt 直接复制进本技能。

## 读取后必须产出

- 当前是否需要合议式评审，以及触发原因。
- 合议角色分工、阶段门、用户确认点和停止条件。
- PRD / 方案 / 原型 / Backlog 的共识、分歧、必改、建议、待确认项、owner 和验证方式。
- 若进入正式 PRD 或产品架构交付，说明还需要读取哪些 reference，并在正式评审时运行 `scripts/check_product_deliverable.py --kind product-review` 或对应交付物检查。

## 需要继续读取的 reference

- PRD 正文与写作标准读 `product-prd-template.md` 和 `product-design-and-prd.md`。
- PRD 提交前检查、CR 和符合性评审读 `product-prd-quality-gates.md`。
- AI Native 原型、MVP harden 和产品侧交接读 `ai-native-product-context.md`。
- 产品洞察读 `product-insight-analyst.md`；Backlog 决策读 `po-backlog-manager.md`。
- 图形化表达读 `diagram-output.md`。
- 支付资金、监管、隐私、安全或外部规则场景继续读取支付资金和监管专项 reference。

## 按任务读取索引

| 任务 | 优先读取 | 跳过 |
| --- | --- | --- |
| 一句话需求进入分阶段 PRD | `1. 合议定位`、`2. 角色分工`、`3. 阶段门` | 不直接写完整 PRD |
| AI 生成方案或 Demo 评审 | `4. 证据与边界门禁`、`5. 合议评审输出` | 不把 Demo 实现细节翻译成 PRD |
| 多方争议或方向不一致 | `2. 角色分工`、`4. 证据与边界门禁`、`6. 分歧处理` | 不用多数意见替代 owner 决策 |
| 复杂 PRD 分批产出 | `3. 阶段门`、`7. 分批产出` | 不一次性生成长文 |
| PRD / 产品方案 CR | `5. 合议评审输出`、`8. 准出检查` | 不只给笼统建议 |

## 1. 合议定位

合议式评审的目标不是让多个角色表演争论，而是降低 AI 一次性生成放大误差的概率。它把产品工作拆成阶段，把不确定性提前暴露出来，把硬规则交给流程门禁，把发散审查交给挑战视角，最后由业务 owner 或用户确认决策。

适合进入合议的典型信号：

- 输入很短，但期望产物很重。
- 方案中有多角色、多对象、多状态、多团队协作或高风险规则。
- PRD 或原型已经由 AI 生成，但业务依据、边界和验收不稳。
- 用户明确希望先确认方案，再确认原型或交互，最后再写 PRD。
- 团队希望 AI 并行产出，但需要人工关键检查点控制风险。

## 2. 角色分工

合议可以由同一个 Agent 在输出中模拟工作位，也可以由外部编排工具分派真实子任务。无论实现方式如何，职责边界保持一致。

| 工作位 | 职责 | 不做什么 |
| --- | --- | --- |
| 流程控制位 | 判断是否进入合议，控制阶段、确认点、停止条件、质量门禁和最终准出。 | 不替业务 owner 拍板，不把多数意见写成已确认事实。 |
| 产品主笔位 | 整理问题地图、前置依赖、方案概要、PRD、Backlog 或产品上下文包。 | 不跳过证据、对象、流程、规则和验收直接堆功能。 |
| 挑战者位 | 审查证据薄弱点、逻辑矛盾、隐含假设、异常边界、可验收性和风险。 | 不负责改写正文，不用主观偏好替代业务证据。 |
| 用户 / owner | 确认关键假设、范围、取舍、风险接受和下一阶段准入。 | 不需要审查所有细节，但必须确认会影响方向和责任的结论。 |

## 3. 阶段门

合议默认采用四段式，不按固定轮次硬跑；按复杂度和 PRD 类型裁剪。

1. Round 0：前置依赖确认
   - 明确用户、主体、业务目标、范围、非目标、风险等级、证据来源和验收方。
   - 输出缺口、待确认问题和未确认前的默认处理。
2. 阶段一：方案共识
   - 先判定 PRD 类型和交付路径，再输出方案概要、关键对象、主流程、范围取舍和 2-3 个决策点。
   - 用户确认后再进入下一阶段。
3. 阶段二：交互、图形或原型验证
   - 仅当涉及前端交互、多角色流程、多状态流转或评审沟通成本高时触发。
   - 可以输出 HTML Demo、用例图、流程图、泳道图、状态机或能力地图，但图形和原型不能替代正文规则和验收。
4. 阶段三：PRD / Backlog / 上下文包定稿
   - 产品主笔位生成正文，挑战者位做证据、边界和可验收性审查，流程控制位做准出检查。
   - 正式交付前把共识、分歧、待确认项、owner、验证方式和下一步去向写清楚。

## 4. PRD 类型分流

先判断类型，再决定是否需要原型、图形、实验或专项附录。

| 类型 | 典型场景 | 合议重点 | 阶段二 |
| --- | --- | --- | --- |
| 功能型 | 新功能、页面、交互、运营后台能力。 | 场景、对象、流程、状态、规则、异常和验收。 | 需要交互或流程复杂时触发。 |
| 策略型 | 算法、规则、排序、实验、风控策略。 | 成功指标、实验设计、回滚条件、数据口径和误伤风险。 | 通常跳过 Demo，补实验或指标视图。 |
| 修复型 | Bug、体验问题、流程缺口。 | 问题证据、影响范围、修复边界、回归影响。 | 一般跳过，除非交互变化明显。 |
| 架构型产品需求 | 模块拆分、系统打通、平台能力。 | 业务对象、边界、依赖、非目标和交给架构师的验收种子。 | 用能力地图、流程图或状态机替代交互 Demo。 |
| 运营数据型 | 后台、报表、任务、通知、埋点。 | 人工处理、SLA、权限、指标口径、审计和发布运营。 | 视流程复杂度决定。 |
| 支付资金型 | 账户、账本、清结算、对账、VCC、跨境。 | 四流、资金归属、账务矩阵、外部规则和专业确认方。 | 图形化优先，必须进入专项 reference。 |
| AI Native 型 | AI 原型、MVP、业务 dogfooding、产品上下文包。 | 证据、不可复用实现、验收种子、产品侧交接条件。 | 可用原型或上下文包验证，但不判定 Execution Grant。 |

## 5. 合议评审输出

评审输出优先短、清楚、可行动。默认结构如下：

```text
合议评审结论

触发原因：
当前阶段：
评审对象：
PRD / 方案类型：

共识：
- 已确认的目标、范围、对象、流程、规则或验收。

分歧：
- 分歧点：
- 影响：
- 备选方案：
- 建议决策 owner：

必改：
- 位置：
- 问题：
- 影响：
- 建议改法：
- 验证方式：

建议：
- 不阻断，但会影响可读性、可用性、协作效率或后续维护的问题。

待确认：
- 问题：
- 影响章节：
- owner / 确认方：
- 未确认前处理：

下一步：
- 进入 PRD / 进入 Backlog / 进入图形化 / 交给 AI Native 编排 / 交给架构师 / 暂停补证据。
```

## 6. 证据与边界门禁

挑战者位优先审查以下问题：

- 目标、收益、成功指标是否有证据，还是只是假设。
- 用户、主体、业务 owner、风险承担方和验收方是否混用。
- 功能是否能回到问题、场景、能力、规则和验收种子。
- 是否把原型代码、Demo 实现、UI 细节或技术方案写成 PRD 事实。
- 是否缺少空状态、加载态、错误态、异常流程、人工兜底、回滚或发布后验证。
- 是否涉及支付资金、合规、隐私、安全、外部规则或生产运营专业确认。

分歧不能简单投票解决。分歧必须写清影响面、证据强度、决策 owner、未决期间的默认处理和复议条件。

## 7. 分批产出

复杂 PRD 或多模块方案不要一次性写完。先输出框架和共识，再按模块追加。

- 简单需求：一次性输出主文档，但仍保留待确认项。
- 中等复杂度：先输出方案概要和关键流程，再生成 PRD 主文档。
- 高复杂度：按对象、流程、规则、数据、运营、风险和验收分批，每批结束做一次小评审。

每批产出后都检查：

- 本批是否改变了前文目标、范围或对象口径。
- 是否新增分歧、风险或待确认项。
- 是否需要更新图形、规则矩阵、验收种子或 Backlog 决策。

## 8. 准出检查

正式、完整、可评审、提交前、CR 或触发验证场景中，合议评审报告应运行：

```bash
scripts/check_product_deliverable.py --kind product-review --file <评审报告.md>
```

该脚本只检查结构完整性，不替代产品判断、业务确认、合规审查或工程验证。脚本通过后仍要人工确认：

- 共识是否足以支撑下一阶段。
- 分歧是否都有 owner 和处理方式。
- 待确认项是否不会被误写成已确认事实。
- 验证方式是否可执行。
- 下一步去向是否清楚。

## 9. 反模式

- 把“多个 AI 互审”写成热闹的角色扮演，但没有阶段门、owner 和验证。
- 让 Reviewer 只做挑刺，不把问题落到章节、影响、建议改法和验证方式。
- Controller 自己拍板业务方向，绕过用户或业务 owner 确认。
- 为所有 PRD 强制生成 Demo、流程图和完整模板，导致文档变重。
- 把外部仓库的工具调用、平台机制、看门狗脚本或领域专项规则当作本仓库默认能力。
- 把多数共识写成事实，忽略证据强度、风险和待确认项。
