# AI Native 产品上下文包

本文定义 AI Native 场景下从业务意图、原型、MVP 或粗糙想法进入产品架构和工程交接的产品侧上下文包。它不是要求放弃 PRD，也不替代 AI Native 研发编排；它只把 PRD 从静态翻译文档升级为可运行证据、对象规则、验收种子和责任边界的组合。

## 使用时机

- 用户要求从 AI 原型、业务 dogfooding、MVP、Product Builder 工作方式或“放下 PRD”输入中提炼产品侧上下文、验收种子和工程交接条件。
- 用户已有原型、低代码页面、AI 生成 Demo、MVP、页面截图、HTML 或业务操作流程，希望进入可评审产品方案、PRD、系统设计或 CAD。
- 业务方、产品、运营或老板已能用 AI 快速做出候选方案，但需要判断哪些内容值得进入工程化。
- 产品方案要交给 `delivery-collab` 编排或 `资深架构师` 承接，需要先说明产品侧哪些事实已确认、哪些仍是假设。

## 不适用场景

- 用户明确只要求传统 PRD 模板、PRD 补全或 PRD 质量评审；此时优先读 `product-design-and-prd.md`、`product-prd-template.md` 和 `product-prd-quality-gates.md`。
- 用户只要求工程实现、代码 Review、测试或排障；此时优先交给 `资深架构师`。
- 用户要求设计端到端 AI Native 产品到研发流程、Harness/GSD/CAD 准入、角色编排或 AI 原生工具治理；此时优先使用 `delivery-collab`，本文件只提供产品侧输入。
- 目标、用户、主体、风险承担方、验收方完全不清楚时，不直接进入 MVP harden；先回 `product-scenario-routing.md` 做产品语义门禁。
- 支付、资金、账户、账本、清结算、对账、VCC、跨境、隐私、安全或监管合规场景，仍必须读取支付资金和监管专项 reference。

## 读取后必须产出

- 当前输入处于哪一层：意图、问题地图、可运行 MVP、产品架构候选、PRD 草案、工程交接包或待废弃噪声。
- AI Native 产品上下文包：目标/非目标、业务 owner、证据、对象状态、流程规则、数据审计、验收种子、风险确认和工程交接条件。
- Product Context Card / 产品上下文交接卡：产品侧最终确认的事实、对象、规则、验收种子、风险和待确认项，供 AI Native 编排和架构师消费。
- 是否具备交给 AI Native 编排或架构师继续判断的产品侧条件；若不具备，列出缺口和 Round 0 补齐动作。不得直接判定 GSD/CAD 准入或 Execution Grant。

## 需要继续读取的 reference

- 产品场景和语义门禁读 `product-scenario-routing.md`。
- PRD 正文、模板、质量门禁读 `product-design-and-prd.md`、`product-prd-template.md`、`product-prd-quality-gates.md`。
- 产品架构主线读 `product-architecture-methodology.md`。
- 运营、数据和发布读 `product-prd-operations-and-data.md`。
- 支付资金、外部规则和监管确认读 `payment-scenario-routing.md`、`regulatory-baseline.md`。
- 端到端产品到研发流程、GSD/CAD 准入、Harness/Agent 编排和角色边界读 `delivery-collab/references/product-to-engineering-lifecycle.md` 与 `delivery-collab/references/engineering-governance.md`。

## 1. 定位：PRD 从翻译文档变成可执行上下文

AI 降低了做出候选软件、原型和页面的成本，但没有降低产品判断、风险承担、验收和工程化成本。产品架构专家的职责不是维护“先写长 PRD 再动工”的旧流程，也不是让业务方直接把 AI Demo 扔给研发，而是把低成本产物转成可评审、可验证、可追责的上下文。

AI Native 产品侧工作链路：

```text
业务意图 / 问题地图
-> 可运行 MVP / 原型 / 页面
-> 产品架构候选
-> AI Native 产品上下文包
-> 产品侧工程交接包
```

OpenSpec、Harness/GSD、CAD 候选和 Execution Grant 由 `delivery-collab` 编排，并由 `资深架构师` 在工程侧继续判断。本文件只回答产品侧材料是否足够可靠、可追溯、可验收。

这条链路中的 PRD 仍然存在，但它不再只是“业务到研发的翻译稿”。它应承载：

- 业务 owner 的方向判断和结果责任。
- 原型、MVP、用户反馈或数据证据。
- 对象、状态、流程、规则和数据口径。
- 验收种子、非目标、风险和待确认项。
- 交给 AI Native 编排和架构师继续设计、TDD 和 CAD 判断的产品侧边界。

## 2. Product Builder 与传统 PM 能力校准

Product Builder 不是“会让 AI 写 PRD 的 PM”，而是能对业务结果负责、用 AI 快速验证方向、再把有效方案交给产品架构和工程 harden 的角色。

评估 Product Builder 候选时优先问：

1. 是否真正 own 一块业务结果，而不是只负责协调和文档交付。
2. 是否能说明用户价值、商业/运营目标、成功指标和失败成本。
3. 是否能用 MVP、原型、运营动作或数据验证主张，而不是只整理需求。
4. 是否能做取舍：本期做什么、不做什么、错了谁负责、何时停止。
5. 是否能把验证结果转成对象、流程、规则、数据和验收，而不是把 Demo 当需求。

传统 PM 的需求整理、逻辑表达和协调能力仍有价值，但在 AI Native 流程里必须升级为业务判断、证据组织、上下文设计、验收生成和结果复盘。只提升文档生成速度，不算产品能力升级。

## 3. AI Native 产品上下文包结构

上下文包应比长 PRD 更紧，但必须比一句 prompt 或一个 Demo 更完整。

```text
1. 业务意图：为什么做，业务 owner 是谁，结果责任是什么。
2. 证据来源：用户反馈、数据、工单、访谈、原型试用、MVP dogfooding 或运营观察。
3. 目标 / 非目标：本期验证什么，不验证什么，停止条件是什么。
4. 用户与主体：用户、客户、商户、运营、风险承担方、验收方和外部参与方。
5. 领域术语与核心对象：统一命名、对象定义、字段口径、状态、关系、不变量和权限。
6. 流程与规则：主流程、逆向流程、异常流程、人工流程、规则版本和优先级。
7. 原型 / MVP 说明：入口、页面、操作、限制、已知缺陷、不可复用实现和观察结果。
8. 数据与审计：指标、埋点、报表、导出、审计日志、数据源和口径负责人。
9. 验收种子：正常、边界、异常、运营、数据和风险路径的可观察验收样例。
10. 风险与待确认：合规、资金、隐私、安全、生产运营、外部规则和专业确认方。
11. 工程交接条件：哪些内容已确认，哪些只能作为候选，哪些需要架构师 harden。
```

上下文包可以被写进 PRD、产品架构方案、OpenSpec、Harness Plan 或项目已有任务文档，不强制新建固定文件名。关键是让后续架构师、测试和 CAD 能追踪每个实现任务服务哪个目标、对象、规则和验收。

### 3A. Product Context Card / 产品上下文交接卡

当产品专家被 AI Native 或架构师调用时，优先输出一张短卡，而不是把完整 PRD、过程讨论或工程计划塞给下游：

```text
Product Context Card:
业务目标 / 非目标:
业务 owner / 验收 owner:
事实证据:
核心用户 / 被影响主体:
领域术语 / 业务命名建议:
核心对象 / 字段口径 / 状态:
业务不变量:
主流程 / 逆向 / 异常 / 人工兜底:
关键规则 / 权限 / 数据口径:
验收种子:
风险 / 待确认 / 专业确认方:
交给 AI Native / 架构师的边界:
```

使用规则：

- 只写已确认事实、原型 / MVP / dogfooding 观察、合理假设、待确认项和范围外不做；不要混入讨论过程、被拒方案或 AI 推理轨迹。
- 领域术语和业务命名建议只服务下游建模、代码命名、测试命名和规则追踪；不得替代架构师的模块/类/接口设计，也不得把未确认的新名词包装成需求。
- 验收种子必须可被架构师转成 TDD、契约测试、集成验证、监控指标或人工验收；不能只写“体验更好”“更智能”。
- 产品专家不输出 Engineering Handoff Card、生产交付卡、Plan Grant、Execution Grant、CAD Grant 或上线批准。
- 产品事实不足时，退回 Round 0 补用户、主体、目标、对象、不变量、验收 owner 和风险确认方；不把 Demo、页面或功能清单包装成工程可执行输入。

## 4. MVP / 原型 harden 门禁

AI 生成原型或业务方 dogfooding MVP 进入工程前，先分级：

| 分级 | 含义 | 动作 |
| --- | --- | --- |
| 噪声 | 只是快速想法或视觉/交互尝试，没有目标、证据和 owner。 | 放入观察池或丢弃，不进入 PRD。 |
| 发现证据 | 暴露了用户问题、流程阻塞或规则缺口，但方案未确认。 | 先补问题地图、待确认项和验证动作。 |
| 产品候选 | 目标、主体、对象、流程和验收已有草案。 | 写 AI Native 产品上下文包或 PRD 草案。 |
| Hardened Candidate | 已有业务 owner、验证证据、对象规则、验收种子和风险确认路径。 | 可以形成产品侧交接包，由 AI Native 编排和架构师继续判断系统设计、GSD/CAD。 |

Hardened Candidate 最低门禁：

- 业务 owner 明确，结果责任和验收方明确。
- 目标、非目标和停止条件可验证。
- MVP / 原型的证据来源、使用范围、已知限制和不可复用实现已说明。
- 核心对象、状态、流程、规则、权限和数据口径能支撑主链路。
- 正常、边界、异常、人工兜底和数据验收至少有种子样例。
- 涉及资金、合规、隐私、安全、外部规则或生产运营时，已标专业确认方和进入研发前门禁。

## 5. 从上下文包到 PRD

当用户需要正式 PRD 时，不要把上下文包机械扩写成长文。按复杂度裁剪：

- 轻量需求：保留业务意图、目标/非目标、核心流程、功能范围、验收种子和待确认项。
- 复杂业务：补齐对象模型、状态机、规则矩阵、运营后台、数据报表、权限、审计、风险和发布。
- 高风险业务：补支付资金、监管、隐私、安全、专业确认方、人工兜底、回滚和上线门禁。

PRD 章节必须能回指上下文包：

| 上下文包字段 | PRD 承接 |
| --- | --- |
| 业务意图、目标、非目标 | 背景、目标、范围、成功指标 |
| 证据来源 | 用户反馈、问题地图、数据依据、MVP 观察 |
| 用户与主体 | 用户角色、参与方、权限和责任 |
| 核心对象 | 对象模型、字段口径、状态机 |
| 流程与规则 | 功能需求、业务流程、规则矩阵、异常处理 |
| 原型 / MVP 说明 | 页面/交互说明、已知限制、不可复用假设 |
| 数据与审计 | 指标、埋点、报表、导出、审计日志 |
| 验收种子 | 验收标准、测试场景、上线后观察 |
| 风险与待确认 | 风险、依赖、专业确认方、进入研发门禁 |

## 6. 与 AI Native 编排和架构师的交接

产品侧上下文包只回答“这个产品候选是否值得工程化、工程化时必须保持哪些业务事实”。它不是端到端流程设计，不是系统设计，不是 Harness Plan，也不是 Execution Grant。

交给 `delivery-collab` 编排或 `资深架构师` 承接前至少输出：

| 交接项 | 产品侧必须给出 | 架构侧继续处理 |
| --- | --- | --- |
| 目标与非目标 | 业务目标、成功指标、停止条件 | OpenSpec、质量属性、技术非目标 |
| 术语、对象与状态 | 领域术语、业务命名建议、核心对象、字段口径、状态、不变量 | 领域模型、接口契约、数据模型、代码/测试命名 |
| 流程与规则 | 主/逆/异常/人工流程、规则矩阵 | 模块边界、事务、一致性、幂等、可靠性 |
| 验收种子 | 正常、边界、异常、运营、数据验收 | TDD 场景、契约测试、集成验证、监控指标 |
| 风险与确认 | 资金、合规、隐私、安全、外部规则确认方 | 生产门禁、发布回滚、审计、安全架构 |
| MVP 证据 | 原型/MVP 观察、已知限制、不可复用实现 | AI Native 编排输入、GSD 任务包输入、CAD 候选判断 |

GSD/CAD 准入结论由 `delivery-collab` 编排，并由架构师在系统设计、OpenSpec、Harness Plan、验证矩阵和 Execution Grant 中确认。产品上下文包、Hardened Candidate 或 GSD Roadmap 都不是 Execution Grant；产品侧不得把“业务方能跑通 MVP”解释为“可以自动编码上线”。

## 6A. AI Native 调用产品专家的细化能力

当 AI Native 按 Skill 类型路由把任务交给产品专家时，产品侧只补产品事实和验收种子，不接管流程编排、测试实现或工程授权。

| 被调用能力 | 产品专家必须补齐 | 交回 AI Native / 架构师的证据 |
| --- | --- | --- |
| 产品验证 | 用户旅程、业务事实、正常/边界/异常/运营/数据/风险验收种子。 | AC、Given-When-Then、可观察结果、验收 owner。 |
| 业务流程自动化 | 参与角色、流程节点、审批/通知/人工兜底、异常恢复和运营动作。 | 流程图、责任边界、人工确认点、失败处理口径。 |
| 数据分析 | 指标定义、字段业务口径、数据 owner、样本解释和决策用途。 | 指标口径、数据源、复核人、不可用于决策的限制。 |
| 模板脚手架输入 | 业务对象、字段、规则、状态、不变量和样例数据。 | 结构化输入、非目标、覆盖风险、验收种子。 |
| 发布业务验收 | 灰度对象、运营观察、业务指标、客服/运营反馈和停止条件。 | 验收窗口、放量口径、回滚业务条件、复盘问题。 |

产品侧输出必须区分已确认业务事实、MVP / 原型 / dogfooding 观察、合理假设、待确认项和范围外不做。如果缺少业务 owner、验收方、核心对象、不变量或验收种子，产品专家应把任务退回 Round 0 补齐，不把 Demo、页面或功能清单包装成工程可执行输入。

产品侧还应尽量减少工程注释债：重要业务概念、状态名、规则名、异常场景、人工兜底和不变量必须有可复用命名。命名不清时，下游代码往往只能靠注释解释“这个规则到底是什么”；这类语义缺口应先回到产品上下文补齐，再交给 AI Native 或架构师。

## 7. 反模式

- 把“放下 PRD”理解为不写目标、对象、规则和验收。
- 把 AI Demo、页面或低代码实现直接当需求本身。
- 只有业务 owner 的主张，没有用户证据、数据证据或 MVP 观察。
- 只有原型点击路径，没有对象状态、规则版本、异常流程和人工兜底。
- 只衡量 AI 生成速度，不衡量验证周期、返工率、决策质量和上线后结果。
- 把产品侧上下文包当成架构设计、测试计划、Execution Grant 或上线批准。
- 涉及资金、合规、隐私、安全或外部规则时，用 AI 生成内容替代专业确认。

## 8. 外部 AI Native 产品文章来源边界

微信公众号文章《放下 PRD：写给AI Native时代的产品经理朋友们》可作为 AI Native 下传统 PRD 翻译职能弱化、Product Builder、业务 owner + Agent、MVP dogfooding 和 PRD 可执行上下文转型的公开参考来源。

本技能只吸收可迁移方法：

- PRD 从静态翻译文档转为可运行证据、对象规则、验收种子和责任边界的上下文包。
- Product Builder 以业务 owner、结果责任、业务判断和快速验证为核心，而不是只会协调和整理需求。
- MVP / 原型进入工程前必须经过 harden 门禁。
- 业务 dogfooding 结果需要产品架构整理成工程可承接材料。

本技能不吸收：

- 不把传统 PM 或 PRD 价值全盘否定。
- 不把“放下 PRD”写成跳过产品语义、评审、留痕、合规和验收。
- 不复制文章原文、标题表达、作者判断、引用案例、播客来源转述或传播性措辞。
- 不把公开文章观点当作团队组织调整、岗位评价或裁员建议。
