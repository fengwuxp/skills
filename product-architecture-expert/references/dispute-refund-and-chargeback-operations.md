# 争议、退款与拒付运营参考

本文用于设计退款、撤销、冲正、争议、拒付、证据提交和逆向资金处理能力。它不替代卡组织、ACH、钱包、本地支付网络或通道的正式 dispute/chargeback 规则。

## 使用时机

- 设计银行卡、ACH/银行转账、钱包、本地支付方式、企业卡和虚拟卡的逆向交易。
- 设计 refund、void、reversal、chargeback、dispute、representment、arbitration、return 和退汇。
- 设计争议运营台、证据链、时限管理、商户准备金、负余额追偿和财务入账。
- 设计客服、风控、运营、财务、法务与通道之间的协作流程。

## 不适用场景

- 只做正向支付、通道路由或清结算主流程时，优先读 `payment-methodology.md`、`payment-channel-routing-and-operations.md` 或 `clearing-settlement.md`。
- 具体卡组织、ACH、钱包、本地支付网络或通道 dispute/chargeback 规则必须按最新官方规则和协议确认。

## 读取后必须产出

- 逆向交易类型、责任方、资金动作、证据模型、时限、账务影响、运营动作、结果回流和待确认项。

## 需要继续读取的 reference

- 卡争议读 `card-network-and-card-rails.md`；ACH/银行转账争议读 `payment-rails-ach-and-bank-transfers.md`；商户风险读 `payment-risk-fraud-and-merchant-operations.md`；清结算和账务读 `clearing-settlement.md`；监管边界读 `regulatory-baseline.md`。

## 概念边界

- **Refund**：商户或平台主动退还交易金额，通常沿原支付路径或约定替代路径处理。
- **Void / Reversal**：在授权或未最终清算阶段取消或反转，语义上不同于已清算后的退款。
- **Return / Returned Payment**：ACH、银行转账或本地清算网络的退回结果，通常由网络规则和返回码驱动。
- **Dispute / Chargeback**：付款方通过发卡行、银行或支付网络发起的争议，可能导致资金先行扣回和证据提交。
- **Representment / Defense**：商户或平台提交证据反驳争议，争取资金回补。
- **Arbitration / Compliance Case**：争议升级到网络或规则层处理，费用、时限和责任更严格。

不要把以上对象统一叫“退款失败”或“交易失败”。每一种对象的责任方、时限、资金动作和账务处理都不同。

## 标准处理链路

1. **事件进入**：用户申请退款、商户发起退款、通道返回 reversal、银行返回 return、网络创建 dispute 或客服创建争议工单。
2. **归因识别**：关联原交易、支付尝试、授权、清算、结算单、账务分录、商户、用户和通道 reference。
3. **资金判断**：判断原交易是否已授权、已清算、已结算、是否部分退款、是否跨结算周期和是否存在准备金。
4. **证据收集**：收集订单、履约、物流、签收、服务记录、用户沟通、退款政策、AVS/CVV/3DS、descriptor 和风控记录。
5. **运营动作**：接受、反驳、补退款、冻结结算、扣准备金、生成负余额、发起追偿或升级人工处理。
6. **账务处理**：按事件生成冲正、退款、拒付扣回、争议费、回补、损失确认、应收应付或调账分录。
7. **闭环复盘**：更新 dispute case、客服结果、商户风险、规则模型、对账差错和财务报表。

## 外卡争议治理链路

外卡争议不是售后工单，而是资金归属权的后半场。系统和报表必须先区分卡争议与非卡争议：

- **卡争议**：Visa / Mastercard 等卡网络规则、原因码、时限、举证、资金扣回和 representment 链路。
- **非卡争议**：钱包、APM、本地支付、账户转账或平台仲裁链路，按对应 provider / 本地网络规则设计，不能套用卡 chargeback 模板。

常见卡争议链路可表达为 Alert -> Inquiry / Retrieval -> Chargeback -> Representment -> Pre-arbitration / Arbitration。不是每笔争议都会走完整链路，但对象、状态、资金动作、时限和证据包必须能逐段表达。

拒付或 chargeback 进入后先做五件事：

1. **分型**：按 fraud、fulfillment、billing、subscription、duplicate、credit not processed 或其他原因码识别问题类型。
2. **止血**：检查同商户、同 BIN、同设备、同邮箱、同履约模式和同路由是否正在放大风险；必要时暂停 capture、收紧结算、提高准备金或调整交易规则。
3. **按原因码组证**：围绕授权、3DS/AVS/CVV、descriptor、履约、物流、服务使用、订阅协议、退款政策和沟通记录构造证据包。
4. **判断 representment**：根据金额、胜率、证据强度、处理成本、商户等级和网络费用决定接受、反驳或部分处理。
5. **结果回流**：把败诉原因、胜诉证据、商户行为、产品缺口和规则误差回流到准入、风控、收银台展示、descriptor、客服 SLA、结算策略和对账差错。

争议治理至少四层：前置预防、预警处置、正式争议、反馈回流。争议率、CB 率、fraud/dispute monitoring program 只能作为红线指标框架；具体阈值、区域、生效日、排除项必须按 Visa/Mastercard/收单行/通道最新官方规则确认。

## 资金与账务口径

- 退款资金来源要明确：原交易待结算金额、商户余额、平台垫付、准备金、银行账户或其他约定来源。
- 结算后退款和拒付要说明是否形成商户负余额、是否从后续结算抵扣、是否触发追偿或准备金扣减。
- 争议费、拒付费、网络费、退款费和汇兑损益要分开，不要统一放在“手续费”字段。
- 同一交易同时出现退款和争议时，必须防止重复退款、重复扣回和账务双重损失。
- 证据提交、争议结果和资金回补要能驱动账务状态变化，而不是只更新运营工单状态。

## 证据模型

- 交易证据：原始支付请求、授权结果、清算结果、账务分录、网络参考号、通道回执和 webhook。
- 客户证据：身份验证、登录设备、IP、地理、历史交易、3DS/强认证结果和付款授权。
- 履约证据：商品/服务描述、发货、物流、签收、使用记录、电子交付、服务完成证明和订阅协议。
- 沟通证据：客服记录、用户确认、退款政策、取消政策、商户条款和投诉处理记录。
- 风控证据：规则命中、风险评分、人工审核、黑白名单、异常行为和后续欺诈确认。

证据字段应结构化存储，附件只作为补充。提交给通道或网络的证据格式、页数、大小、时限和语言要求必须按最新规则确认。

## 交易争议举证行为日志

交易争议举证行为日志（Dispute Evidence Activity Log）用于在 dispute、chargeback、ACH unauthorized return、disputed debit、reversal、representment 或银行调查中证明用户、账户、设备、授权、履约和沟通的事实链。它不是普通运营埋点，也不是客服备注；它是争议证据链、合规审计日志和风控证据留存的交叉能力。

这套日志能力可以跨卡、ACH/银行转账、钱包和本地支付方式复用，但证据包不能混用。卡争议更关注授权、3DS/AVS/CVV、履约、descriptor 和 chargeback reason code；ACH debit 更关注授权取得、授权留存、授权撤销、账户验证、return code、NOC、reversal、retry 和 unauthorized return 处理。

核心对象建议拆成：

- **EvidenceActivityLog**：用户、商户、运营、系统或外部通道产生的可举证行为事件。
- **EvidencePackage**：按支付轨道、争议/退回原因码、银行/卡组织/通道要求和提交窗口生成的证据材料包。
- **CardDisputeEvidencePackage**：卡交易争议材料包，按卡组织、收单行或 PSP 模板组织证据。
- **ACHEvidencePackage**：ACH debit、unauthorized return、disputed debit、NOC、reversal 或银行调查材料包，按 Nacha、ODFI/RDFI、银行或通道规则组织证据。
- **EvidenceExportAuditLog**：证据查看、导出、脱敏、提交、撤回和版本变更的审计日志。
- **DisputeEvidenceRepository**：按交易、账户、用户、商户、支付工具、设备和争议工单关联的证据仓库。

证据行为日志至少覆盖以下事件：

- **账户与身份**：注册、登录、登出、密码修改、邮箱/手机号验证、MFA/二次验证、KYC/KYB 状态变化。
- **会话与设备**：session、设备 ID 或 fingerprint hash、IP、user agent、粗粒度地理位置、风险设备标记。
- **交易授权**：订单、支付单、支付尝试、授权时间、3DS/强认证结果、AVS/CVV 结果、授权响应和拒绝原因。
- **ACH 授权**：debit authorization 展示版本、授权确认动作、授权金额、扣款频次、扣款日期、撤销方式、账户验证结果和授权撤销记录。
- **用户确认**：协议勾选、订阅授权、退款政策确认、价格/币种/税费/手续费展示版本、checkout 页面版本。
- **履约与服务**：发货、签收、物流、电子交付、服务开通、下载、登录使用、会员权益消费或 API 调用。
- **沟通与售后**：客服记录、退款申请、取消申请、用户确认、投诉处理、补偿方案和争议沟通。
- **历史交易关联**：同账户、同支付工具、同设备/IP、同邮箱、同地址的历史成功且未争议交易。
- **ACH 退回处理**：return code、NOC、reversal、retry 决策、主数据更正、通知用户、人工复核和资金/账务处理结果。

日志设计原则：

- 关键证据必须由服务端可信事件产生；前端埋点只能作为辅助，不得作为唯一授权、履约或协议确认依据。
- 日志应追加写入、不可随意修改；需要更正时只能追加更正事件、说明和操作者，不能覆盖历史记录。
- 每条日志至少包含事件 ID、主体、事件类型、事件时间、服务器接收时间、来源系统、关联交易/支付/争议 ID、结果、证据版本和最小必要上下文。
- 证据字段要按最小必要原则采集和提交。不要为了举证无限扩大用户画像、设备画像或行为追踪范围。
- 完整 PAN、CVC/CVV、敏感认证数据、未脱敏证件影像、无关聊天记录和无关隐私材料不得进入普通证据包；确需提交时必须经法务/合规/通道确认并脱敏。
- 证据包生成要按支付轨道、原因码和通道模板选择字段，不要把全量行为日志直接提交给银行、发卡行、RDFI/ODFI、卡组织或通道。
- 证据查看、导出和提交本身也要审计，记录操作者、时间、用途、材料版本、接收方和脱敏策略。

ACH 场景的反模式：

- 不要把 ACH return 当成卡 chargeback，也不要把 ACH reversal 当成 refund。
- 不要在缺少 debit authorization evidence、账户验证或状态回查的情况下重试扣款。
- 不要用前端点击日志替代 ACH debit 授权凭证；授权凭证必须能解释用户/企业授权了什么、何时授权、授权给谁、金额/频次/日期是什么、如何撤销。
- 不要忽略 NOC 对账户主数据和后续扣款的影响；NOC 是主数据更正信号，不是普通失败。
- 不要把所有 return code 都归类为“付款失败”；不同 return code 对重试、通知、风控、资金和账务处理要求不同。

## 时限与 SLA

- dispute/chargeback 通常有严格回应窗口，具体天数会随卡组织、通道、国家/地区、争议类型和合同而变化。
- 内部系统不应写死“统一 N 天”。应保存外部 deadline、内部处理 deadline、提醒策略、超时动作和责任人。
- 运营台要支持待处理、待补证、已提交、待结果、胜诉、败诉、已入账、已核销和已关闭等状态。

## 设计落地检查

1. 是否区分 refund、void、reversal、return、dispute、chargeback、representment 和 arbitration。
2. 是否能把逆向事件关联到原交易、授权、清算、结算、账务分录和通道 reference。
3. 是否覆盖部分退款、跨周期退款、结算后退款、退款失败、重复退款和退款/争议碰撞。
4. 是否说明商户准备金、负余额、结算暂停、追偿和损失确认规则。
5. 是否有结构化证据模型、附件管理、提交时限、提醒和人工复核。
6. 是否设计交易争议举证行为日志，并区分可信服务端事件、前端辅助事件和人工补充材料。
7. 证据日志是否满足追加写入、不可随意修改、最小必要、脱敏、权限隔离、访问审计和版本留痕。
8. 证据包是否按支付轨道、原因码和通道/卡组织/银行模板生成，而不是直接提交全量行为日志。
9. ACH 场景是否覆盖 debit authorization evidence、账户验证、授权撤销、return code、NOC、reversal 和 retry 决策。
10. 是否能把争议结果反哺风控、商户评级、对账差错和财务凭证。
11. 是否把卡组织、ACH、本地支付方式、通道协议和最新时限列为待确认项。
