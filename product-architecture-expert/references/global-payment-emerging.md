# 全球支付与新支付形态

本参考基于公众号公开索引中呈现的主题方向提炼，用于补足传统内部清结算之外的银行核心、全球支付清算、跨境支付、稳定币、Web3 和 AI 代理支付问题。不要复刻公众号原文，只使用通用方法和公开主题脉络。涉及真实业务时，先按 `regulatory-baseline.md` 确认法域、主体资质和监管红线。

## 使用时机

- 用户要求设计、评审或补全全球收款、跨境付款、多币种账户、本地支付网络、全球金融平台、Embedded Finance、稳定币/Web3 支付或 AI 代理支付方案。
- 支付方案涉及多法域、多主体、多币种、多清算网络、多 PSP、代理行、Nostro/Vostro、FX、费用或全球平台能力地图。

## 不适用场景

- 只做普通本地收单、VCC、ACH 或清结算局部方案时，优先读对应专项 reference。
- 稳定币、Web3、跨境合规、外汇、税务和本地支付网络规则必须按最新法规、协议和专业确认处理。

## 读取后必须产出

- 全球支付网络分层、主体责任、账户/资金路径、清结算复杂性、FX/费用、风险合规、运营报表、客户解释视图和待确认项。

## 需要继续读取的 reference

- 支付总纲读 `payment-methodology.md`；清结算读 `clearing-settlement.md`；监管基线读 `regulatory-baseline.md`；银行转账轨道读 `payment-rails-ach-and-bank-transfers.md`；通道编排读 `payment-channel-routing-and-operations.md`。

## 银行核心视角

讨论银行核心或类银行账户系统时，先区分：

- 客户信息、客户账户、账户余额、账务流水、会计分录不是同一层。
- 存款账户、贷款账户、内部账户、清算账户、过渡账户、挂账账户用途不同。
- 交易核心关注业务事件和状态推进，账务核心关注余额变化和会计一致性。
- 日终、日切、批量、计息、对账、总分核对、监管报送是核心系统稳定性的关键约束。

输出银行核心方案时，至少包含账户模型、产品参数、交易事件、账务分录、日切批处理、对账和异常处理。

## 全球支付清算

全球支付清算方案要画清三层网络：

1. **消息网络**：支付指令、报文、状态回执和追踪信息，如 Swift/GPI 或本地支付网络报文。
2. **账户网络**：代理行账户、清算账户、Nostro/Vostro、本地银行账户和支付机构账户。
3. **资金网络**：真实资金划拨、轧差、清算、结算、退汇和手续费扣收。

产品分析要回答：

- 汇款人和收款人分别在哪个国家、哪个币种、哪个账户体系。
- 中间行、代理行、本地清算网络和收款行各自承担什么职责。
- 费用采用 SHA、OUR、BEN 还是平台自定义收费，谁承担中间行费用。
- 状态如何追踪：已受理、已发报、处理中、已入账、退汇、合规拦截、资料补充。
- 到账时效如何承诺，以及失败、退汇、长时间未达如何赔付或解释。

## 跨境支付五层分析法

跨境支付不能只画“付款人 -> 收款人”。参考陈天宇宙公开文章中对全球支付信息流和资金流的拆解，跨境或全球本地支付方案先按五层分析：

1. **交易层**：交易平台、商户、买家、订单、履约、售后和本地支付入口。
2. **支付处理层**：收单机构、账户侧机构、钱包或本地 PSP，处理支付方式、授权、扣款、退款和商户侧状态。
3. **代理结算层**：代理行、合作机构、资金池、Nostro/Vostro、内部结算账户和本币/外币头寸管理。
4. **清算网络层**：本地清算网络、跨境报文网络、卡组织、Swift/GPI、CIPS、CHIPS、Fedwire 或目标国本地网络。
5. **最终清算层**：央行、清算账户、最终结算系统和监管层确认的资金完成口径。

每层都要分别说明信息流、账务流和真实资金流：哪一层只是传递支付指令，哪一层做内部记账，哪一层发生真实资金划拨，哪一层决定最终清算完成。

## Nostro/Vostro 与跨境三模式

跨境资金路径至少要识别账户结构和清算模式：

- **Nostro/Vostro**：从本机构视角看，Nostro 是“我方在你方开立的账户”，Vostro 是“你方在我方开立的账户”。产品方案要写清账户视角，避免把同一个账户在不同机构眼中叫反。
- **清算行模式**：通过具有清算能力或本币接入能力的清算行完成对应币种的本地清算。
- **代理行模式**：通过代理行账户和 correspondent banking 网络转发报文、扣收费用、处理退汇和状态追踪。
- **NRA/非居民账户模式**：通过境内或境外非居民账户承接跨境收付、结算或归集，必须结合当地账户、外汇和合规规则确认。

跨境方案要区分“外清、内结”：跨境报文和机构间清算是一层，平台或合作机构给商户、用户、子账户做内部结算是另一层。不要把商户看到到账等同于跨境清算网络最终完成。

## 国内、跨境、境外支付对比

- **国内支付**：核心在本地支付网络、实名/KYC、商户入网、客户备付金（支付机构适用）、收单、内部清结算和监管报送。
- **跨境支付**：核心在两地账户体系、换汇、收付汇合规、路由、代理行、本地清算和反洗钱。
- **境外本地支付**：核心在本地支付方式、本地牌照/合作机构、商户结算、本地税费和本地风控。

不要把跨境支付简单理解成“多一个币种”。真正复杂的是主体、账户、资金路径、合规材料、清算周期和失败退回。

## 全球支付三路径与底层控制力

全球支付平台在前台能力上很容易相似：都可能声称覆盖多国家、多币种、全球收款、快速开户、低费率和统一 API。产品架构评估不能停在功能清单，要下钻到平台如何获得这些能力，以及谁承担底层复杂性。

常见路径可先分为三类：

1. **新型轨道绕行**：用稳定币、Web3 或链上清结算缩短部分路径。它可能改善速度和成本，但必须先确认法域、出入金、托管、KYC/KYB、AML、Travel Rule、制裁筛查、税务、账务和用户客群边界；不得把链上到账直接等同于合规完成、法币可用或最终清算。
2. **聚合包装传统基础设施**：通过 PSP、网关、代理行、银行伙伴或本地机构快速扩展覆盖面。它能降低接入门槛，但底层仍可能存在多层转接、规则不一致、文件延迟、异常归因不清、对账割裂和伙伴依赖；不能只看前台体验轻不轻。
3. **深控或自建本地金融基础设施**：通过牌照、本地团队、银行连接、本地清算网络、账户体系、合规运营、流动性和对账能力提高控制力。它投入重、周期长，但可能把更多不确定性留在平台侧，而不是转嫁给客户。

评估时不要把三类路径写成绝对优劣。正确问题是：当前客户、法域、币种、支付方式、时效、风控、资金规模和运营能力，需要平台控制到哪一层。

## 复杂性承接与客户侧确定性

全球支付产品的价值不只是“能收能付”，而是平台是否替客户承接跨法域、跨轨道、跨币种、跨机构的复杂性。方案要明确复杂性归属：

- **合规复杂性**：牌照、主体、KYC/KYB、AML、名单筛查、地区限制、报送和资料补充由谁负责。
- **资金复杂性**：本地收单、本地清算、代理行、换汇、流动性、Nostro/Vostro、资金池、退汇和资金冻结由谁承接。
- **清结算复杂性**：交易日、清算日、入账日、结算日、cut-off、时区、假期、批次文件、手续费和轧差规则是否可解释。
- **风险复杂性**：拒付、退款、欺诈、争议、准备金、负余额、制裁命中和异常出款如何影响资金可用。
- **运营复杂性**：多市场对账、异常工单、状态追踪、客户告知、赔付口径和人工兜底是否形成闭环。

客户侧确定性至少看四个结果：交易能否稳定完成、资金何时可用、费用和汇率能否解释、异常由谁闭环。若平台只提供统一入口，却要求客户自己处理本地法域、银行关系、资金追踪、对账差错和争议责任，就不是完整的全球支付产品架构。

## 全球金融平台能力地图

参考 Airwallex 等公开产品文档时，不要把全球金融平台理解成单个“支付 API”。更稳定的产品架构视角是能力族群：

| 能力族群 | 产品问题 | 典型对象与交付物 |
| --- | --- | --- |
| Connected Accounts / 平台账户 | 平台如何为客户、子商户或合作方开户、入网、授权和隔离数据/资金权限 | connected account、客户主体、KYB/KYC、权限、平台关系、入网状态 |
| Accounts / Global Accounts | 客户如何在多币种、多地区接收资金、管理账户和授权扣款 | global account、wallet/account、bank transfer deposit、direct debit mandate、账户状态、入账状态 |
| Payments / 收单 | 如何让买家用卡、本地支付方式、钱包或银行转账完成一次性或周期性付款 | payment、payment attempt、payment method、refund、cancellation、webhook、settlement report |
| Transactional FX / 换汇 | 跨币种交易如何报价、锁价、换汇、记账和解释汇损益 | quote、rate、conversion、交易币种、清算币种、结算币种、FX fee |
| Payouts / 付款 | 平台如何向供应商、商户、用户或内部主体付款，并处理失败、审批和监管材料 | transfer、beneficiary、payer、batch transfer、approval、failure reason、confirmation |
| Issuing / 发卡 | 如何发行和管理企业卡、虚拟卡、消费控制和卡交易生命周期 | cardholder、card、issuing balance、spend control、authorization、3DS、card transaction |
| Spend / 费用与 AP | 企业支出、费用报销、应付账款和对账如何闭环 | expense、bill、approval、reconciliation、accounting export |
| Embedded Finance / 嵌入式金融 | 平台如何把账户、收款、付款、发卡或财资能力嵌入自己的客户产品 | global treasury、BaaS、payments for platforms、组件、API、客户责任分配 |

使用这张图时，先判断当前需求落在哪些能力族群，再回答对象、状态、资金流、数据流、风控、报表和验收。不要因为一个厂商把能力放在同一平台，就在自己的方案里混同账户、收单、出款、发卡和费用管理的主体责任。

## AI 出海全球资金管理

AI 出海企业的全球支付问题不能只按传统跨境电商收单处理。产品方案要先确认它的收入形态、成本形态和自动化支付主体，再组合全球收款、多币种账户、FX、付款、VCC 和嵌入式金融能力。

典型场景可按下表拆分：

| 场景 | 资金问题 | 设计检查 |
| --- | --- | --- |
| AI SaaS / API / 模型服务 | 全球订阅、按用量或 token 计费、多币种收入和开发者客户对账 | 本地支付方式、订阅和用量计费、余额/预充值、计量账单、退款、税费、收款币种、settlement report |
| AI 硬件与渠道型 SaaS | 海外分销商收款分散、渠道结算、跨币种利润和库存/履约对账 | Global Accounts、多市场收款归集、渠道伙伴结算、FX 锁价或报价、履约和售后对账 |
| GPU / 云资源 / 算力平台 | 大额 B2B 预充值、月结、上游供应商付款和内部采购支出 | 企业客户账户、授信/预付边界、受益人、批量付款、审批、回执、企业卡/采购卡、成本中心对账 |
| AI Agent 平台 | Agent 自动购买 API、算力、SaaS 或数据集，不能暴露企业主账户 | 每个 Agent / 任务独立 VCC 或受控支付凭证，单笔/周期额度、商户/MCC/币种/有效期限制、任务审计、撤销和异常告警 |

可以用“收、管、付、控”做产品检查，但不要把它写成单一厂商能力清单：

- **收**：目标市场、本地支付方式、卡/钱包/银行转账、支付成功率、退款和争议责任，以及收入如何进入对应账户或报表。
- **管**：多币种账户、资金归集、现金池、流动性、FX、费用、汇损益、余额可用性和财务解释口径。
- **付**：供应商、开发者、创作者、分销商或内部主体的受益人管理、批量付款、审批、失败退回、回执和出款对账。
- **控**：Agent 支付、企业支出和平台客户支出的授权范围、VCC / card control、额度、商户限制、有效期、审计、冻结和撤销。

涉及 AI Agent 自主支付时，VCC 或一次性受控凭证只是控制手段，不是完整方案。还必须说明：谁授权 Agent 花钱、模型能否自动决策、哪些交易必须人工确认、Agent 失控或提示注入时如何止损、支付凭证如何隔离敏感卡信息，以及主账户、预算额度、真实余额和卡授权占用如何对账。

涉及厂商案例时，只吸收场景拆解和检查项。覆盖国家、币种、到账时效、牌照数量、AI 外汇预测能力、产品可用性、费率和接口能力都属于强时效或商业承诺，必须回到最新官方文档、合同、通道规则或专业确认，不能写成通用产品事实。

## 状态、事件、报表、沙盒四件套

全球金融产品不是只提供前台动作，还要能解释动作之后发生了什么。参考公开平台文档时，优先吸收四件套：

- **状态**：账户、入账、支付、退款、付款、批量付款、卡、授权、交易和争议都应有自己的生命周期；不要用一个“成功/失败”覆盖多轨道语义。
- **事件**：webhook、异步通知、文件、批次和外部 reference 要能支持幂等、乱序、补拉、重放和审计。
- **报表**：交易活动、settlement、fee、FX、账务分录、对账差异和运营指标应按消费方拆分；报表不是导出按钮，而是业务解释面。
- **沙盒**：状态模拟、测试卡、测试入账、测试付款、交易模拟和 go-live checklist 应进入验收设计，用来证明方案可验证而不是只可描述。

这四件套适合写入 PRD、评审清单和验收标准；具体状态名称、错误码、测试卡号和国家字段必须回到最新官方文档确认。

## 平台责任边界矩阵

全球金融平台、BaaS、Payments for Platforms 或嵌入式金融方案，最容易把“平台能提供的能力”和“客户仍然承担的责任”混在一起。产品方案必须用责任矩阵把边界拆开：

| 责任面 | 需要回答的问题 | 常见确认方 |
| --- | --- | --- |
| 客户/商户入网 | 谁负责 KYC/KYB、受益所有人、业务真实性、禁限售行业、资料更新和定期复核 | 合规、风控、平台运营、合作机构 |
| 账户与资金 | 谁持有客户资金、谁能发起收付、谁能冻结/解冻、资金归属和破产隔离如何解释 | 法务、财务、持牌机构、银行 |
| 支付与出款 | 谁选择支付方式或付款轨道，谁承担失败、退汇、重复出款、拒付和退款责任 | 支付运营、通道、财务、客服 |
| 换汇与费用 | 谁报价、谁锁价、谁承担汇损益、手续费如何展示和入账、四舍五入差额归谁 | 财务、资金、定价、合规 |
| 数据与报表 | 谁可以看哪些数据，报表字段、刷新时效、下载、留存和删除由谁负责 | 数据、法务、安全、运营 |
| 风控与争议 | 谁配置规则、谁处理告警、谁承担争议损失、谁负责证据收集和反馈回流 | 风控、运营、法务、商户管理 |
| 集成与上线 | 谁完成沙盒测试、webhook、状态回查、异常演练、go-live checklist 和回滚预案 | 产品、工程、运营、客户成功 |

若平台能力是白标、组件化或 API 嵌入式交付，还要额外说明：最终用户看到的品牌主体、合同主体、资金服务主体、客服入口、隐私告知、账单/报表主体和争议处理主体是否一致。不一致时，PRD 必须把用户解释视图单独写清楚。

## FX、费用和舍入口径

全球支付、全球账户、收单、出款和发卡都可能涉及多币种与费用。不要把 FX 和 fee 当成财务事后处理，产品方案至少拆清：

- **币种层次**：交易币种、授权币种、账户币种、清算币种、结算币种、报表币种和记账本位币。
- **报价层次**：实时汇率、锁定汇率、牌价来源、报价有效期、报价撤销、重新报价和异常兜底。
- **费用层次**：固定费、比例费、跨境费、网络费、processor fee、FX markup、退款费、拒付费、退汇费和人工处理费。
- **舍入层次**：最小货币单位、四舍五入规则、差额归属、批量轧差和报表展示精度。
- **归属层次**：费用由用户、商户、平台、合作机构或内部成本中心中的谁承担，收入与成本何时确认。
- **解释层次**：用户可见价格、商户账单、平台毛利、财务凭证和对账报表必须能解释同一笔金额为什么不同。

若无法确认汇率来源、锁价时点、费用承担和舍入差额，方案只能标为待确认，不能写成“按通道返回为准”。

## 全球金融平台交付包

当用户要求设计、评审或对标 Airwallex 类全球金融平台时，输出不能停在能力地图。可评审方案至少包含八个交付包：

1. **能力地图**：Connected Accounts、Accounts、Payments、FX、Payouts、Issuing、Spend、Embedded Finance 的范围、非范围和组合关系。
2. **主体与责任矩阵**：平台、客户、商户、最终用户、银行/持牌机构、PSP、卡组织和本地合作方的合同、资金、数据、风控和客服责任。
3. **对象与生命周期**：account、wallet、payment、refund、transfer、beneficiary、payer、cardholder、card、authorization、dispute、report 等对象的状态、事件和终态。
4. **四流与资金路径**：业务流、支付信息流、账户/账务流、真实资金流；跨境场景还要拆消息网络、账户网络和资金网络。
5. **FX 与费用模型**：币种层次、报价、锁价、费用、舍入、收入成本确认和报表解释。
6. **运营与报表面**：入网、配置、审批、异常工单、对账差错、settlement report、activity export、费用报表、审计日志和告警。
7. **风险与合规门禁**：KYC/KYB/KYT、AML、制裁筛查、数据边界、PCI、当地牌照、外部规则核验和专业确认项。
8. **验证与上线包**：sandbox、状态模拟、webhook 验证、幂等/乱序/补拉、异常演练、go-live checklist、灰度、回滚和客户告知。

这八个交付包是产品方案结构，不要求一次性全部上线。轻量方案也应明确哪些包是本期范围、哪些是后续阶段、哪些是待专业确认。

## 生命周期主轴

全球金融平台的对象很多，但生命周期可以按一条主轴组织：

```text
入网 -> 配置 -> 交易/资金动作 -> 状态事件 -> 清结算/账务 -> 报表/对账 -> 异常/争议 -> 退出/留存
```

- **入网**：客户、商户、持卡人、受益人、平台账户完成资质、合同、权限和风险等级确认。
- **配置**：账户、支付方式、付款轨道、发卡项目、费率、限额、webhook、报表和风控规则生效。
- **交易/资金动作**：收款、付款、退款、撤销、换汇、授权、清算、结算、调账和冻结。
- **状态事件**：外部状态、内部状态、webhook、批次文件、报表事件和人工操作形成可追踪时间线。
- **清结算/账务**：金额拆分、费用、FX、清账、结算、账务分录、资金到账和财务确认。
- **报表/对账**：交易报表、费用报表、结算报表、账务报表、资金对账和差错闭环。
- **异常/争议**：失败、退汇、重复、拒付、争议、欺诈、冻结、负余额和人工工单。
- **退出/留存**：客户停用、商户关闭、卡注销、受益人停用、资金退回、数据导出、证据留存和报表保留。

产品方案可以从任何业务动作切入，但必须能回到这条主轴，说明动作前置条件、后置状态、责任方、证据和关闭条件。

## 运营成熟度阶梯

全球金融平台能力不能只按“有/没有”判断。可以用五级成熟度评估当前方案：

| 等级 | 特征 | 风险 |
| --- | --- | --- |
| L1 接口接入 | 能调用外部 API 或文件，人工解释状态和金额 | 依赖人肉排查，异常不可规模化 |
| L2 对象化 | 有账户、支付、付款、卡、报表等对象和状态 | 对象能查，但责任、费用和账务可能仍割裂 |
| L3 运营闭环 | 有后台、工单、报表、对账、告警、审批和审计 | 能运营，但跨地区、跨币种扩展成本高 |
| L4 平台化 | 能按客户/商户/地区/币种配置能力、费率、限额、webhook 和报表 | 需要强治理，配置错误会放大资金风险 |
| L5 可解释金融基础设施 | 状态、资金、账务、费用、风险、合规和客户解释视图一致 | 成本高，必须持续规则核验和专业确认 |

产品专家评审时应指出当前方案处在哪一级、下一阶段要补哪个控制面。不要用“接了全球支付平台”直接宣称达到平台化或基础设施能力。

## 稳定币与 Web3 支付

先确认法域和客群。中国境内语境下，虚拟货币不具有与法定货币等同的法律地位，相关交易炒作、兑换、支付结算等活动存在明确监管红线；不要面向境内用户设计虚拟货币支付、兑换或撮合产品。境外或多法域场景也必须先确认当地支付、汇款、虚拟资产、托管、证券/商品、AML 和制裁要求。

稳定币/Web3 场景要区分五件事：

- **钱包**：托管钱包、非托管钱包、多签、MPC、地址归属和私钥责任。
- **链上交易**：链、代币合约、Gas、确认数、重组、失败交易和链上浏览器证据。
- **链下账务**：平台内部账户、余额、冻结、商户待结算、法币应收应付。
- **出入金**：法币入金、换币、铸造/赎回、OTC、支付通道或银行账户。
- **合规**：KYC/KYB、AML、制裁筛查、Travel Rule、地区限制和可疑交易监控。

设计重点是链上链下双账本一致性：链上交易不可撤销，但链下业务可能需要退款、争议、冻结、人工审核和财务核算。

## AI 代理支付

AI 代理支付不是让模型直接“花钱”，而是把支付能力封装为受控授权：

- 授权主体：用户、企业、管理员、代理账户。
- 授权范围：商户、品类、金额、次数、时间、币种、支付方式。
- 决策边界：模型可建议、可填单、可触发预授权，但高风险支付应有人类确认或策略审批。
- 支付凭证：每次代理行为都要记录任务、上下文、授权、风控结果、支付单和审计日志。
- 失败处理：支付失败、重复扣款、退款、争议、超额、商户欺诈和用户否认。

输出 AI 代理支付方案时，必须给出“可自动执行”和“必须确认”的分界线。

## 方案模板：全球/新支付

```text
1. 场景与用户目标
2. 参与方、国家/地区、主体和资质边界
3. 币种、账户、钱包或银行账户模型
4. 消息流、资金流、账务流和合规流
5. 路由、费用、汇率、时效和状态追踪
6. 对账：平台、通道、银行/链上、本地清算、账务
7. 异常：退汇、链上失败、合规拦截、长短款、人工审核
8. 风控、KYC/KYB、AML、审计和报表
9. 用户告知、后台处理台和验收标准
```
