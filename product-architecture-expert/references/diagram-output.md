# 产品图形化交付

本文用于把产品架构、PRD、业务流程、状态机、规则、指标和支付资金方案中的关键关系转化为可评审、可维护、可验证的图形化产物。图形化交付是产品表达的一部分，不替代业务确认、合规确认、工程设计或验收标准。

## 使用时机

- 用户明确要求画图、流程图、能力地图、状态机、关系图、资金流图、产品架构图、运营后台结构图或可视化产物。
- PRD、产品方案或评审文档中存在多角色、多对象、多状态、多规则或多系统协作，仅靠文字会降低评审效率。
- 支付、账户、账本、清结算、对账、争议、VCC、ACH、卡组织、跨境或业财一体化方案需要表达业务流、支付信息流、账户/账务流和真实资金流。
- 需要把复杂产品方案沉淀成默认可导出的 SVG，或可由后续脚本稳定生成的图形资产；Mermaid/Markdown 草图、PNG/PDF/截图等其他格式仅在使用者明确提出时处理。

## 不适用场景

- 需求、主体、目标、对象、规则或验收口径还不清楚时，不要用精美图形包装未确认假设；先回到 `product-scenario-routing.md` 和产品语义门禁。
- 用户只需要文字评审、字段表、规则矩阵或验收矩阵时，不强行生成图片文件。
- 涉及监管、合规、卡组织、ACH、本地清算网络、外部通道或第三方 API 最新规则时，图形不能替代来源、版本、生效日期、核验日期和确认方。
- 不复制外部项目的图形、模板、Logo、专有配色或未审查脚本；只吸收可公开复用的方法思想。

## 读取后必须产出

- 图形目标：这张图解决什么评审问题，读者是谁，和 PRD/方案中的哪一节对应。
- 图形类型：角色关系、能力地图、业务流程、状态机、对象关系、规则决策、指标链路、运营后台结构或支付资金四流图。
- 输出形态：正式图形化交付默认只生成 SVG 并验证；Mermaid/Markdown 草图、PNG/PDF/截图等其他格式只有在使用者明确提出时才额外生成。
- 待确认项：图中哪些节点、状态、规则、资金路径、外部机构或数据口径仍需业务、产品、合规、财务、技术或外部机构确认。
- 验证结论：说明是否已检查语法、文件可读性、远程资源、文本溢出、关键节点和与正文口径的一致性。

## 需要继续读取的 reference

- 场景识别和产品语义门禁读 `product-scenario-routing.md`。
- PRD 和产品方案读 `product-prd-template.md`、`product-design-and-prd.md`。
- 通用产品架构读 `product-architecture-methodology.md`。

## 图形类型路由

| 场景 | 首选图形 | 适合表达 | 注意事项 |
| --- | --- | --- | --- |
| 产品架构方案 | 能力地图、模块关系图 | 前台、后台、运营、风控、数据、审计和外部系统边界。 | 能力地图必须能回到业务目标和验收，不只是功能堆叠。 |
| 复杂业务流程 | 跨角色流程图、泳道式流程 | 用户、运营、风控、财务、外部机构和系统之间的动作顺序。 | 必须覆盖主流程、逆向流程、异常流程和人工兜底。 |
| 生命周期或状态流转 | 状态机 | 对象状态、触发事件、守卫条件、终态和回滚路径。 | 状态名、事件名和 PRD/接口/数据字段保持一致。 |
| 规则体系 | 决策图、规则矩阵附图 | 额度、计费、审批、通知、风控、优先级和版本。 | 图只表达决策路径，完整条件和口径仍放规则矩阵。 |
| 数据指标和报表 | 指标链路图、数据流图 | 事件、埋点、数据源、指标口径、报表和验收。 | 标清数据 owner、口径归属和数据校验方式。 |
| 运营后台 | 信息架构图、操作流程图 | 查询、审核、复核、导出、告警、工单、审计和权限。 | 高危操作必须体现复核、审计、回滚或人工处理。 |
| 支付与资金 | 四流图、账户/账务流、对账闭环图 | 业务流、支付信息流、账户/账务流、真实资金流、对账和异常。 | 必须标注主体、资金归属、法域、外部机构和待合规确认项。 |

## 卓越图形能力要求

高质量产品图不能只是把文字搬进框里。输出前必须同时完成六个判断：

1. 图形意图：用于业务评审、PRD 说明、开发交接、运营培训、合规确认还是高层沟通。
2. 图形范围：一张图只解决一个主问题；超过 12 个核心节点时优先拆成主图 + 子图。
3. 视觉风格：选择适合评审语境的风格，不用随机配色或装饰性渐变掩盖复杂度。
4. 语义形状：同类对象使用同类形状和命名，例如角色、能力、对象、状态、规则、数据、资金和外部机构。
5. 语义箭头：箭头必须表达业务动作、状态迁移、审批、通知、数据流、账务流、资金流或异常回退。
6. 验证闭环：图形必须能回到 PRD、规则矩阵、验收标准、支付资金 reference 或待确认项。

## 视觉风格路由

| 风格 | 使用场景 | 视觉要求 |
| --- | --- | --- |
| Product Doc Clean | PRD、能力地图、业务流程、状态机和评审文档。 | 白底、克制颜色、清晰分组、短标签、强调业务对象和验收路径。 |
| Operations Console | 运营后台、人工处理、审批复核、告警和工单流程。 | 信息密度略高，使用查询、审核、导出、告警、审计等操作分组。 |
| Finance Ledger | 支付、账本、清结算、对账、分账分润和资金四流。 | 区分业务流、支付信息流、账户/账务流、真实资金流；标注主体、资金归属和外部机构。 |
| Risk Review | 合规、安全、风控、争议、权限和高危操作。 | 突出风险节点、人工确认、审批、审计、回滚和专业确认方。 |
| Executive Map | 高层评审、范围边界、阶段规划和跨团队协作。 | 低节点数、强分层、少细节，只保留目标、主体、能力、风险和决策点。 |

## 语义形状和箭头

正式交付生成 SVG 时使用更丰富的形状和图例表达语义；只有用户明确要求 Mermaid/Markdown 草图时，才用稳定节点名、`subgraph` 分层和箭头标签表达语义。PNG/PDF/截图等派生格式不默认生成。

| 语义对象 | Mermaid 表达 | SVG 表达 |
| --- | --- | --- |
| 用户、客户、商户、运营、风控、财务 | 角色节点 + 所属泳道 | 人形/角色卡片，颜色只表示角色类别。 |
| 产品能力、后台模块、页面入口 | 矩形节点 | 模块卡片或分组容器。 |
| 业务对象、订单、账户、工单、凭证 | 对象节点，必要时列关键字段 | 数据卡片，带对象名、状态、owner。 |
| 状态、生命周期节点 | `stateDiagram-v2` 或带状态标签的节点 | 圆角状态块，终态和异常态明显区分。 |
| 决策、审批、规则判断 | 菱形或 `条件?` 节点 | 菱形决策点，规则编号和 owner 可追踪。 |
| 数据源、报表、指标 | 数据/报表节点 | 圆柱或表格形态，标注口径和数据 owner。 |
| 外部机构、通道、银行、卡组织 | 外部泳道或边界框 | 外部边界容器，必须标注待确认规则。 |
| 账户、账本、资金账户 | 账户/账本节点 | 双边框或账本形态，区分账务余额和真实资金。 |

箭头语义保持一致：

- 实线箭头：同步业务动作、页面操作、审批流或主路径。
- 虚线箭头：异步通知、回调、人工补录、延迟处理或非主路径。
- 粗箭头或明确标签：真实资金流、账务分录、状态迁移等高风险路径。
- 回环箭头：补正、重试、退回、争议、撤销、冲正或人工兜底。
- 不同类型箭头必须在图例或正文中解释，不让颜色成为唯一语义来源。

## 稳定提示配方

当用户要求直接画图但上下文足够时，可用这些配方组织输出：

- 产品能力地图：按“用户入口、核心业务能力、运营后台、数据报表、风控审计、外部系统”分层；每层 3-6 个节点，箭头只表达主要协作关系。
- SaaS 运营后台：按“查询定位、审核处理、复核审批、导出报表、告警工单、审计追溯”组织，标出高危操作和权限边界。
- 业务状态机：先列对象、状态、触发事件、守卫条件、终态和异常态，再画 `stateDiagram-v2`。
- 支付资金四流：分四条泳道表达业务流、支付信息流、账户/账务流、真实资金流；对账、差错、挂账、冲正和人工处理必须入图。
- 对账闭环：按“内部账本、外部通道/银行、对账规则、差异分类、处理动作、审计验收”组织。

## 输出形态选择

默认使用 SVG 作为正式图形化交付物。Mermaid 仅在用户明确要求 Markdown 内嵌、文档草图、便于 diff 的文本图，或项目已有 Mermaid 维护约定时使用。

- `flowchart`：角色关系、能力地图、业务流程、系统协作和资金四流。
- `stateDiagram-v2`：状态机、生命周期和审批状态。
- `sequenceDiagram`：跨系统交互、通知回调、人工复核和异步流程。
- `erDiagram`：产品对象、核心字段和关系草图；生产表结构仍需工程/DBA 确认。
- Markdown 表格：规则矩阵、指标口径和验收矩阵；不要把复杂规则强行塞进图。

正式图形化交付默认只生成 SVG。Mermaid/Markdown 草图、PNG/PDF/截图等其他格式必须由使用者明确提出，或在已有 SVG 无法满足其指定消费场景时，经说明后再处理。生成文件时应优先使用本地、可审查、无网络依赖的脚本或工具；新增脚本必须说明输入、输出、写入范围、失败行为和是否访问网络。

## 图形内容规则

- 每张图只表达一个主问题：能力边界、流程顺序、状态变化、规则决策、资金路径或数据口径。
- 图中节点使用业务对象和角色名，不使用含糊的“系统”“平台”“服务”等泛称；确需使用时，必须补边界。
- 箭头必须有语义：动作、事件、数据流、资金流、审批、通知、回调、对账或异常。
- 对关键风险节点使用短标签标注，例如待确认、人工复核、合规确认、资金归属、外部规则、异常处理。
- 图形不得和正文冲突；如果图和 PRD/规则矩阵不一致，先修正文权威口径，再更新图。
- 金融、合规和外部规则场景必须把“图中结论仍待专业确认”写入正文待确认项。

## 验证清单

- SVG：文件能打开，SVG XML 可解析，包含 `viewBox` 或稳定宽高，不引用远程资源，不嵌入敏感信息。
- Mermaid/Markdown 草图：只有用户明确要求时才生成；生成后检查代码块类型、节点 ID、状态和箭头语义。
- 派生格式：只有用户明确要求 PNG/PDF/截图等其他格式时才生成；生成后必须检查尺寸、裁切、文本截断、图例和关键路径可读性。
- 可读性：节点数量不过载，长文本拆行，箭头不表达多重含义，关键路径从左到右或从上到下清晰。
- 一致性：图中角色、对象、状态、规则、指标和风险与 PRD、产品方案、规则矩阵、支付资金 reference 保持一致。
- 可追溯：交付时报告生成或修改的文件路径、图形用途、关键假设、验证动作和剩余待确认项。

正式、完整、可评审、提交前、CR 或触发验证场景中，提交图形 brief 给人工或 `$fireworks-tech-graph` 前必须运行 `scripts/check_product_deliverable.py --kind diagram-brief` 检查图形目标、目标读者、图形类型、节点/分组、箭头语义、假设/待确认项和输出格式是否齐全。该脚本只检查本地文本结构完整性，不判断图形美观度、业务正确性或渲染质量；无法运行时必须说明原因、人工检查结果和残余风险。
