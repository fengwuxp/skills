# 产品概念生命周期与退役

本文用于治理产品概念膨胀、新旧规则并存、需求只加不减和事实源分裂。它是产品侧概念治理 reference，不是工程删除授权、数据迁移方案或上线审批。

## 使用时机

- 用户提到概念膨胀、新旧概念并存、需求只加不减、事实源分裂、旧规则退役、旧入口下线或产品概念生命周期。
- 新概念、新规则、新页面、新状态、新报表或新运营口径进入方案前，需要判断旧概念如何保留、合并、迁移、归档或下线。
- PRD、产品方案、运营口径或数据报表中同一业务事实出现多个名称、多个状态口径、多个规则版本或多个事实源。

## 不适用场景

- 只做工程删除、数据迁移、公共契约变更、发布回滚或生产下线：交给 `资深架构师` 和对应工程流程。
- 只做新功能 PRD 且不存在旧概念、旧入口或旧规则影响面：使用 `product-prd-template.md` 和 `product-design-and-prd.md`。
- 涉及真实资金、外部支付规则、合规或监管口径时，先进入 `payment-scenario-routing.md` 和 `regulatory-baseline.md`。

## 读取后必须产出

- Concept Lifecycle Card：说明核心概念、事实源、新增 / 替代关系、旧概念关系、净增概念数、收敛 / 合并 / 废弃规则、迁移路径、用户 / 运营 / UED 影响、验收种子、下线 owner、复审日期、退役条件和待确认项。
- 明确哪些内容属于产品概念治理，哪些必须交给架构师、数据、运营、法务、合规或发布 owner 确认。
- 不把“没人投诉”“暂时兼容”“后面再说”写成旧概念可删除、可迁移或可下线的证据。

## 需要继续读取的 reference

- 需要先定义概念名称、目标主体、价值 / 意义边界和需求止损时，读 `product-architecture-methodology.md` 的“1.3 概念定名与需求止损”。
- 需要写完整 PRD、迁移期产品说明或验收标准时，读 `product-prd-template.md`、`product-design-and-prd.md` 和 `product-prd-quality-gates.md`。
- 需要画入口迁移、状态收敛、概念关系或用户路径图时，读 `diagram-output.md`。

## 1. 触发信号

- 同一业务对象有多个名称、多个状态口径、多个规则版本或多个事实源。
- 新能力上线后，旧入口、旧文案、旧报表、旧审批、旧字段和旧运营流程长期保留。
- 需求每次只加不减，用户或运营要靠经验知道“现在应该用哪个”。
- 产品、UED、研发、运营和数据团队对同一概念的理解不一致。
- PRD 只写新增功能，不说明旧概念如何迁移、合并、废弃或下线。

## 3. 产品侧处理规则

- 新概念必须说明谁拥有、替代了什么、复用了什么、与旧概念并存多久、何时停止旧入口；不能立即退役旧概念时，要标明净增概念数、复审日期和退役条件。
- 不确定是否可下线时，先补使用证据、用户影响面、运营兜底和数据对比，不直接删。
- 旧概念确实需要长期保留时，必须写清保留理由、适用场景、边界提示、owner 和复审日期。
- 对研发交接时，明确事实源、迁移口径、兼容周期、验收种子和风险 owner；产品专家不把概念退役写成工程删除授权。
- UED 需要重点验证信息层级、入口迁移、状态反馈、用户学习成本和错误路径，确保旧入口收敛不会变成体验断崖。

## 4. 常见偏差

- 用新名词包装旧需求，导致产品和代码都增加一套概念。
- 只在 PRD 里写“兼容旧版本”，没有兼容截止条件、数据口径和 owner。
- 为了降低短期风险保留所有旧入口，长期把复杂度转嫁给用户、运营和研发。
- 把“没人投诉”当成旧概念可删除证据，忽略数据、客服、运营和关键客户影响。
