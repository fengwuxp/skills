# 支付产品方法论

## 先识别场景再设计

开始设计前，先读取或参考 `payment-scenario-routing.md` 完成场景识别：组织语境、业务动作、支付轨道、资金复杂度和目标产物。不要把清结算、通道路由、争议证据、ACH、卡组织和 VCC 混成一个模板。

如果场景涉及真实资金、牌照、客户资金、个人信息、金融数据、跨境、稳定币/Web3 或 AI 代理付款，先读取 `regulatory-baseline.md`，再进入具体方案。

## 三种底层能力

1. **支付流分析能力**：从一个业务场景中分清谁向谁付款、谁收款、谁记账、谁实际持有资金、谁承担通道成本和风险。
2. **模式化设计能力**：把新场景归入已有支付模式，如收单、代收、代付、分账、担保交易、钱包、会员账户、预付、补贴、营销、退款、调账。
3. **架构能力**：把产品能力沉淀成交易核心、支付核心、账务核心、内部清结算、对账、结算、财务处理、通道网关、风控、报表等模块。

## 金融业务总纲

金融业务专家的交付目标不是堆支付概念，而是把业务事实、资金归属、账务证据和风险责任统一起来。任何涉及支付、账户、账本、清结算、对账、发卡、收单、跨境或业财一体化的方案，先用一条主轴收敛。

这里的金融业务默认指支付与资金产品语境，不扩展为信贷、证券、保险等全域金融方案；相关内容只作为主体、资金、风险、数据和外部规则边界处理。

```text
谁的钱，因什么业务，在什么主体和账户下，沿哪条支付轨道，以什么规则流转，何时可用，谁承担风险，谁最终确认。
```

围绕这条主轴检查六个控制面：

1. **主体与资质**：平台、商户、客户、支付机构、银行、清算机构、卡组织或处理商分别是什么角色，是否有对应合同、授权、牌照或待确认项。
2. **资金归属与账户**：客户资金、商户待结算资金、平台自有资金、保证金、准备金、授信额度、预算额度和费用收入必须分开表达，不能用一个“余额”覆盖。
3. **业务事实与支付事件**：订单、账单、授权、扣款、退款、撤销、拒付、return、调账、清账和结算是不同事实，必须有单据、状态、幂等键和责任方。
4. **账务与清结算**：每个资金事件都要能落到账户、分录、批次、清账结果、结算结果和对账来源；支付成功、清算入账、清账完成、结算出款和资金到账不得混写。
5. **风控合规与外部规则**：KYC/KYB/KYT、AML、名单筛查、限额、商户风险、争议、数据安全、卡组织/ACH/银行/本地清算网络规则必须说明确认方和核验状态。
6. **数据证据与运营闭环**：报表、审计、差错、工单、证据包、财务凭证和验收样例必须能解释金额为何这样变化、责任为何这样归属、异常如何回退或追偿。

输出金融业务方案时，至少说明四流、单据状态、账户/账务、清结算/对账、风险合规、数据证据和验收确认方。缺项必须标成待确认或非范围，不得用“后续对接”“财务处理”“通道返回”掩盖关键口径。

## 支付系统五层拆解

参考陈天宇宙公开文章《3.5万字：一文搞懂“支付系统”》的支付系统全景拆解，支付方案评审时先从五层看边界，再进入具体流程和页面：

1. **支付渠道层**：回答“接谁的什么产品”，按收付类型、终端、金额、对象、行业、地区、成本、限额和资质约束管理通道画像。
2. **支付网关层**：把外部多样化接口第一次统一为内部协议，处理通信、安全、签名、加密、返回码、协议转换、限流和原始报文留痕。
3. **支付核心层**：承载支付请求处理、参数校验和补全、风控、路由、支付单/流水、通道指令、结果处理和业务通知。
4. **统一支付能力层**：把支付核心包装成外部可理解的产品能力，如收款、付款、退款、代扣、分期、绑卡、合单支付、线下汇入等。
5. **支付接入层**：面向业务系统和终端提供收银台、API、SDK、后台操作或文件接入；收银台不是页面本身，而是交易和支付的衔接产品。

五层可以合并实现，但产品方案必须说明每层职责、对象、单据、状态、权限、运营入口和验收指标。不要让支付核心直接适配每个渠道差异，也不要把收银台展示策略、通道路由、账务处理和真实资金到账混成一个功能。

## 十二字能力地图

支付与资金产品能力可先用“买、收、付、退、充、提、转、调、算、结、管、对”做全景覆盖检查：

- **买**：交易体系能力，定义订单、账单、履约、售后和支付触发点。
- **收 / 付 / 退**：支付核心能力，覆盖收款、付款、退款、撤销、冲正和结果通知。
- **充 / 提 / 转**：钱包或账户能力，覆盖入金、出金、账户间转移、限额和余额口径。
- **调**：资金管理能力，覆盖调拨、调账、人工修正、补偿和审批。
- **算 / 结 / 管**：清结算能力，覆盖清分规则、结算规则、出款批次、商户/参与方管理和结算运营。
- **对**：对账能力，覆盖平台交易、通道交易、账务一致性、银行/存管资金和财务总分核对。

输出支付产品方案时，用这 12 类能力做缺口扫描：本期做什么、不做什么、由哪个系统负责、产生哪些单据、触发哪些状态、影响哪些资金和财务口径。

## 需求分析顺序

1. 业务为什么要支付：交易闭环、履约闭环、资金闭环、财务闭环分别是什么。
2. 支付对象是谁：个人、商户、平台、服务商、代理商、分账方、境外主体、链上地址。
3. 资金性质是什么：自有资金、客户备付金（仅限适用主体）、待结算资金、存管资金、营销补贴、保证金、押金、应收应付、手续费收入、通道成本。
4. 产品边界在哪里：只做支付受理，还是包含账户、账务、结算、发票、财务凭证和报表。
5. 成功标准是什么：支付成功率、到账时效、账平率、差错处理时效、资金安全、财务可解释性。

## 四流建模

- **业务流**：订单、履约、售后、分账对象、结算对象。
- **支付信息流**：支付请求、通道请求、支付结果、通知、补单、退款/撤销。
- **账户/账务流**：入账、冻结、解冻、扣减、冲正、调账、结转。
- **资金流**：用户付款、通道/清算机构处理、银行或存管账户到账、商户结算、提现/代付。

设计时不要只画用户支付页面。支付产品的价值通常藏在支付成功后的账务、对账、结算和异常处理里。

## 支付单据地图

常见单据及职责：

- **业务单**：业务系统的交易事实，如订单、账单、充值单。
- **支付单**：支付核心受理和状态推进的主单。
- **通道单**：支付机构或银行返回的通道维度记录。
- **账务流水**：账户余额变化的不可篡改记录。
- **内部清算单**：按合同和业务规则计算参与方应收应付、手续费、分账和分润；不是持牌清算业务。
- **结算单**：对结算对象进行周期性汇总并触发出款。
- **打款单/代付单**：真实资金出款请求。
- **对账差错单**：平台、通道、银行或账务之间不一致的处理单。
- **会计凭证**：面向财务核算和审计的会计语言。

## 支付核心主流程检查

支付核心不是“调通道接口”的薄层。设计支付核心能力时，至少检查四段主流程：

1. **接入请求处理**：权限、必填参数、格式、枚举、业务单据、交易金额、支付方式、终端、商户和风险上下文是否完整。
2. **支付报文处理**：交易信息补全、风控调用、路由选择、通道商户号、协议参数、成本和限额是否可解释。
3. **请求通道发起支付**：生成支付单和支付流水，封装通道报文，完成签名/加密/幂等，记录通道请求号和原始响应摘要。
4. **支付应答处理**：处理同步返回、异步通知、主动查询、超时、乱序、重复通知和补单，并通知交易、账务、路由、运营和风控。

复杂支付方式还要拆出“主流程、子流程、环节”：主流程承载共同语义，子流程承载终端、支付方式、通道和产品差异，环节承载可验证动作。任何一个环节都应能回答触发条件、输入、输出、状态变化、异常分支、幂等键和验收样例。

## 收银台与接入产品检查

收银台是交易环节和支付环节的衔接点，不只是一个页面。设计收银台或支付接入时，至少检查：

- 入口形态：PC、H5、App、小程序、SDK、API、硬件、后台操作或文件接入。
- 展示元素：商品/服务、收款方、金额、支付有效期、支付方式、优惠、发票、协议和结果页。
- 展示策略：终端、商品品类、商户要求、用户习惯、营销活动、渠道合作、限额、风控和可用通道。
- 支撑系统：交易核心、收银台后台、支付核心、路由、风控、渠道管理、账务、订单/账单和通知。
- 指标：用户支付体验、支付时效、支付成功率、支付转化率、失败原因、重试率和客服工单。

支付方式展示排序不等于通道路由；收银台可见支付方式、支付核心选择通道、资金清结算规则和对账规则必须分别建模。

## 广义通道与支付创新

广义通道用于识别非标准但能完成价值交换闭环的接入点：只要能发起收付指令、接收收付结果、确认价值交换结果或推动线上业务单据完成，就可能作为一种“通道”建模。

常见场景包括线下汇入流水查询、银企直联流水匹配、人工凭证导入、人工结果确认、内部账户余额增减接口、卡券/积分/仓储权益等内部价值载体。采用广义通道时必须补充：

- 它发起什么指令或接收什么结果。
- 如何和业务单、支付单、账务分录、银行流水或人工凭证关联。
- 如何保证幂等、权限、审计、异常处理和人工兜底。
- 如何进入对账、差错、客服、运营和财务确认闭环。

广义通道不能成为绕过合规、风控、账务、对账和权限的捷径；它只是把非标准价值交换纳入统一支付产品架构的一种建模方式。

## 产品方案骨架

```text
1. 背景与目标
2. 业务范围与非范围
3. 参与方与资质边界
4. 核心流程图：正向、退款、结算、异常
5. 单据与状态机
6. 账户与账务规则
7. 计费、分账、分润与结算规则
8. 对账、差错、调账和补偿
9. 权限、风控、合规、审计
10. 数据报表、告警和运营后台
11. 接口清单与字段字典
12. 验收标准和测试用例
```

## 常见抽象

- 收款和付款分开建模，退款和冲正不要偷懒复用正向流程。
- 信息流成功不等于资金到账，资金到账不等于可结算。
- 账户余额变化必须有业务事件、流水、凭证或差错来源可追溯。
- 所有批处理任务都要有批次号、日切日期、重跑策略和幂等键。
- 产品文档必须解释“为什么这样算钱”，而不仅是“页面怎么点”。

## 钱包、账本、对账三层模式

复杂账户型支付产品可以参考 `formance-reference-patterns.md` 的三层拆法：

- **钱包层**：表达用户、商户、平台或服务商可理解的账户产品能力，如余额、冻结、转账、提现、限额和权限。
- **账本层**：表达不可变资金事实，用交易、分录、账户、币种、业务事件号和元数据解释每一次余额变化。
- **对账层**：表达内部账本与通道、银行、清算机构、存管账户或客户备付金账户之间的可信度校验和差异闭环。

三层可以在系统实现上合并或拆分，但产品方案必须把职责说清。不要让钱包 API 直接改余额，不要让账本承载页面流程，也不要把对账做成离线报表而没有差异处理生命周期。

## 卡产品与授权控制模式

若场景涉及企业卡、员工卡、车队卡、采购卡、虚拟卡或共享额度卡，可参考 `highnote-reference-patterns.md`：

- **卡片是支付介质，不是账户本身**：先定义 Program/平台主体、客户账户、持卡人、卡片和底层资金账户的关系。
- **授权和清算分离**：授权成功不等于账务入账，不等于真实资金结算；产品方案必须拆开事件链路。
- **控制规则产品化**：额度、MCC、商户、地理、时间窗、协同授权等限制应作为显式规则，而不是散落在后台备注或代码分支里。
- **多分录账务显式化**：卡交易下至少要分清授权占用、清算入账、可用余额回补、费用、退款和争议回退分别落在哪些 ledger 或科目上。
- **按需供资口径分离**：预算额度、授权占用、真实现金余额和可用余额必须拆开表达，避免“明明账户为 0 却仍可消费”这种现象在设计文档里无法自解释。

这类产品的核心不只是“能刷”，而是“谁能刷、在什么条件下能刷、刷完如何解释、出问题后如何追责和回退”。

## 报表与运营数据面

支付或卡产品进入运营阶段后，报表不是“导出功能”，而是正式数据面：

- 交易活动报表面向运营、客服和风控，强调事件级追踪。
- 分录或账务报表面向财务、对账和审计，强调借贷关系和余额解释。
- 指标报表面向管理与经营分析，强调汇总口径、时点定义和维度切片。

产品方案至少要说明：谁看什么报表、基于哪类源数据、刷新时效如何、是否允许异步生成、字段是否可扩展、异常时如何补拉和追溯。

## 收单与事件集成补充

若产品覆盖在线收单、支付订单或统一资金移动，还应额外说明：

- 支付订单、支付尝试、网络事件、资金入账、商户结算分别是什么对象。
- tokenization、敏感支付信息隔离、PCI 边界和商户前后端职责如何划分。
- webhook/事件通知如何保证幂等、乱序容忍、补拉和最终一致。

这部分决定的不只是“能不能收款”，而是“支付网络事件、内部账务事件和商户可见状态能不能长期对得上”。

## 通道路由与运营治理补充

若产品涉及多个 PSP、银行、收单行、本地支付方式、发卡处理商或资金通道，应继续读取 `payment-channel-routing-and-operations.md`，并额外说明：

- 通道、通道账户、商户号、能力、凭证、结算口径和健康信号如何建模。
- 路由基于哪些字段决策：支付方式、币种、地区、金额、成本、成功率、风险、SLA、合规限制。
- 超时、回调延迟、通道失败、通道切换和重复扣款/重复出款如何防护。
- 路由规则、费率、限额、通道开关和密钥变更如何审批、灰度、回滚和审计。
- 通道交易、手续费、退款、争议、结算文件和资金到账如何对账。

路由不是“失败就换一家”。任何 fallback 都必须先确认前一通道的外部语义、幂等状态和资金风险。

## 风控、欺诈与商户运营补充

若产品涉及商户入网、异常交易、拒付/争议、ACH return、出款风险或风控后台，应继续读取 `payment-risk-fraud-and-merchant-operations.md`，并额外说明：

- 商户准入、KYB、受益所有人、MCC、行业限制、风险等级和资料变更。
- 交易前、交易中、交易后、结算前和运营后的风控动作。
- 用户、商户、支付工具、设备、地理、金额、频次、退款、拒付和出款等风险信号。
- 风控规则/模型的版本、审批、灰度、回滚、误杀复盘和指标监控。
- 风控证据如何反哺争议、对账、结算、客服、商户评级和审计。

风控不能只写成“接入风控系统”。产品方案必须解释哪些动作会影响支付、结算、资金可用和用户/商户告知。

## 争议、退款与拒付运营补充

若产品涉及售后、退款、撤销、退汇、ACH return、dispute 或 chargeback，应继续读取 `dispute-refund-and-chargeback-operations.md`，并额外说明：

- refund、void、reversal、return、dispute、chargeback、representment 和 arbitration 的边界。
- 原交易、授权、清算、结算、账务分录、通道 reference 和争议工单如何关联。
- 退款资金来源、跨周期退款、结算后退款、退款失败、部分退款和退款/争议碰撞。
- 商户准备金、负余额、结算暂停、追偿、争议费、损失确认和财务入账。
- 证据模型、提交时限、人工复核、SLA 和运营台状态。
- 交易争议举证行为日志如何沉淀账户、设备、授权、协议确认、履约、客服、退款和历史未争议交易证据。
- EvidencePackage 如何按支付轨道、争议/退回原因码和银行/卡组织/通道模板生成，并完成脱敏、版本留痕和提交审计。
- ACH 场景下，debit authorization evidence、账户验证、授权撤销、return code、NOC、reversal 和 retry 决策如何进入证据链。

逆向交易不是正向支付的附属功能。它直接决定资金损失、客服体验、商户风险和财务可解释性。

## VCC 与虚拟卡补充

若产品涉及 VCC、虚拟卡、企业卡、一次性卡、共享额度卡、旅行卡或供应商付款卡，应继续读取 `virtual-card-and-vcc.md`，并额外说明：

- VCC 实际资金模式：credit、debit、prepaid、charge card、commercial card 还是一次性 token。
- Program、客户账户、资金账户、持卡人、卡片、授权控制和账务之间的关系。
- 卡片创建、激活、展示、冻结、注销、重发、轮换、过期和交易后事件。
- 预算额度、可用额度、授权占用、真实现金余额、费用、退款、争议和 FX 的账务口径。
- PAN/CVC 展示、PCI 范围、日志脱敏、权限、审计和批量发卡/自动用卡风险。

虚拟卡的关键不是“没有实体卡”，而是“卡号、额度、资金、授权、清算和责任边界都能被解释”。

## 支付轨道专项补充

若场景明确落在 ACH、银行转账或卡组织网络，应继续下钻对应轨道：

- `payment-channel-routing-and-operations.md`：适合多通道支付、通道降级、路由配置、成本/成功率优化和通道运营治理场景。
- `payment-risk-fraud-and-merchant-operations.md`：适合商户风险、交易欺诈、ACH return、拒付风险、准备金和风控运营场景。
- `dispute-refund-and-chargeback-operations.md`：适合退款、撤销、拒付、争议、证据提交、商户准备金和逆向账务场景。
- `payment-rails-ach-and-bank-transfers.md`：适合代扣代发、批量付款、银行出入金、RTP/FedNow、ACH 授权举证、unauthorized return、NOC、reversal、跨境银行转账等场景。
- `card-network-and-card-rails.md`：适合银行卡收单、企业卡、虚拟卡、预授权、清算结算、拒付和 tokenization 场景。
- `virtual-card-and-vcc.md`：适合企业虚拟卡、一次性卡、共享额度卡、旅行卡、供应商付款卡和按需供资场景。

不要把不同支付轨道的状态、失败语义、清算时序和争议规则混成统一模板。轨道差异本身就是产品方案的一部分。
