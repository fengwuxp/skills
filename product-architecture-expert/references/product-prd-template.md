# 产品 PRD 文档模板

本文是产品架构专家的轻量 PRD 主模板，用于把需求整理成可评审、可开发、可测试、可运营的产品文档。默认目标是让人先读懂问题、判断和方案，不是把 AI 工作流字段铺成全量检查手册；复杂对象、完整状态机、规则矩阵、运营数据、金融资金和执行控制按需进入专项附录或执行计划。

使用前先明确文档目标、目标读者和文档类型。主文档只回答：为什么做、要达到什么目标、这是什么性质的问题、总体与详细方案是什么、流程如何运转、遵循什么规则、向角色或上下游提供什么产品接口、还有哪些风险及如何判断结果。生成前先裁剪，不能把轻量需求写成无人评审的长文，也不能只用功能名替代范围、规则和验收。

模板占位符使用 `〈...〉` 表示。实际输出时，正文段落和表格单元格默认保持普通文本；只有接口路径、技术字段、配置键、协议字段、风险等级或需要强调的结论才使用加粗或代码背景。

## 文档标识与文件命名

创建正式产品文档前，先确定规范主题、文档类型和目标路径，优先级为：用户明确指定 > 项目 `AGENTS.md` / `README.md` > 仓库已一致执行的约定 > 本模板回退规则。仓库现有文件名相互冲突时，不把单个历史文件当成规范；将冲突列为待确认并询问 owner。只有用户明确授权项目约规治理时才修改 `AGENTS.md` / `README.md`；未授权时，更新既有文档保持原路径，新建文档使用本模板回退规则并说明。

- PRD、产品需求文档和需求说明书统一使用 `〈主题〉-产品设计.md`，一级标题使用 `# 〈主题〉产品设计`。
- 其他正式产品产物使用 `〈主题〉-〈明确产物类型〉.md`，产物类型优先使用 `产品架构`、`业务架构`、`产品评审报告` 或 `产品上下文卡`，不得只写模糊的“方案”“设计”或“说明”。
- 主题和文档类型之间只使用一个 ASCII 连字符 `-`；主题内部保留稳定领域术语和 `API`、`DSL` 等既有缩写，不用空格、下划线或中文破折号做结构分隔。
- 文件名默认不包含序号、日期、版本、`草稿`、`最终版`、`新版` 等状态；顺序、版本和状态进入索引、文档头与 Git 历史。
- 产品专家负责给出规范主题；同一能力交给架构师时必须传递“规范主题 + 产品文档精确路径”，不得让下游自行改写业务概念。
- 更新既有文档默认保持原路径。只有用户明确进入命名迁移任务时才改名，并同步 Markdown 链接、任务材料、脚本、测试和其他硬编码路径。

## 使用时机

- 用户要求写作、生成、完善、补全、改写或评审 PRD、产品需求文档、需求说明书、需求文档模板、PRD 模板或产品方案模板。

## 不适用场景

- 仅做产品架构方法分析时读 `product-architecture-methodology.md`。
- 支付资金、PRD 评审、运营数据发布等专项内容按需读取独立附录，不在主模板展开。

## 读取后必须产出

- 可复制的 PRD 主骨架、裁剪依据、待确认项和需要继续读取的专项附录。

## 需要继续读取的 reference

- PRD 写作标准读 `product-design-and-prd.md`。
- 产品方案交给架构师继续设计、业务驱动架构交接或 TDD 承接时读 `product-architecture-methodology.md`。
- 正式评审、提交前检查、CR、触发验证或符合性评审时读取 `product-prd-quality-gates.md`。
- 支付资金读 `product-prd-financial-appendix.md`。
- 运营后台、通知、数据指标、埋点、报表、导出、发布后验证读 `product-prd-operations-and-data.md`。

## 信息传递流程

PRD 的正文按读者理解顺序组织：背景与问题 -> 目标与非目标 -> 定性与范围 -> 概要设计 -> 详细设计 -> 关键流程 -> 业务规则 -> 接口抽象 -> 风险与待确认 -> 验收摘要。设计生成仍可按“名 -> 事 ->（图）-> 法 -> 器 -> 验”由虚入实检查，但完整性框架不等于可见目录：

1. 事实：原始反馈、数据线索、业务背景、现状约束和真实问题。
2. 判断：问题是否成立、影响多大、证据强度如何、哪些仍是假设。
3. 决策：本期做什么、不做什么、优先级为什么这样排。
4. 承诺：产品能力、用户场景、流程规则、权限数据和风险处理。
5. 验证：验收种子、可观察结果、确认方、交给架构师或测试的依据。

其中“名”定义产品名相、主体、对象和边界；“事”说明真实场景与流程；“图”按复杂度用最小关键图暴露关系、状态、边界和冲突；“法”说明状态、不变量、规则和异常；“器”说明产品能力、前后台和数据载体；“验”说明可观察结果。它只用于作者和评审者检查是否贯通，不直接变成章节名、卡片数量或表格数量。

描述可靠性必须可见：已确认、假设、待确认、不适用要分开写；无法确认的信息不得写成确定结论。

同时按角色分层消费：决策层服务业务 / 管理者，说明为什么做、做什么和成功标准；方案层服务产品 / 设计 / 架构，说明方案取舍、范围、成本和风险；实现层服务开发 / 测试 / 发布，说明输入输出、边界异常、接口事件、验收和验证证据。

## 表达结构选择

- 背景、定性、方案和取舍优先用短段落讲清因果；结论、范围和风险用项目符号帮助扫读。
- 单个问题、规则、验收和待确认项直接写成段落或列表；只有同类条目反复出现、需要稳定引用或横向比较时才使用卡片和 ID。
- 业务流程优先用编号步骤或文本流程，便于读者理解先后关系。
- 多角色、多对象、多状态、多团队协作或评审者难以凭文字建立共同理解时，优先补一张最小关键图；图形是 PRD 的阅读入口，不替代规则、验收和待确认项。
- 表格只用于比较、矩阵或追踪；不要把所有信息都塞进表格。
- 多需求、多角色、多规则、多验收种子需要横向比较时，再把卡片合并成表格。

## 图形化视图选择

图不是新的产品事实源。图形按问题选择，不要求每份 PRD 固定画全套；普通轻量 PRD 可以只保留文本流程并明确“不需要单独画图”。命中复杂度或评审沟通成本较高时，在正文第 6 节或附录补充 1-3 张关键图：

- 用例图：用于说明角色、系统边界、核心用例和责任主体，适合评审“谁能做什么、谁负责验收”。
- 流程图：用于说明主流程、逆向流程、异常流程和人工兜底，适合评审“事情按什么顺序发生、哪里分支”。
- 泳道图：用于说明用户、运营、风控、财务、外部机构和系统之间的交接，适合评审“跨团队动作和 SLA 是否闭环”。
- 状态机图：用于说明对象生命周期、状态迁移、触发事件、守卫条件、终态和回滚，适合评审“对象何时能被操作”。
- 能力地图/产品架构图：用于说明本期能力边界、前后台能力、数据能力、风控审计和外部系统依赖，适合评审“范围是否清楚”。

每张图必须说明图形目标、目标读者、对应 PRD 章节、关键假设、待确认项和与验收/规则的追踪关系。需要正式图形化交付时读取 `diagram-output.md`；默认输出 SVG，Mermaid/Markdown 草图、PNG/PDF/截图等其他格式需用户明确提出。

## 使用与裁剪原则

- 轻量需求默认 5-8 个小节即可，不补全无关表格。
- 正式正文必须先形成背景、目标、定性、概要与详细设计、流程、规则和接口抽象的连续叙述；不得以问题卡、追踪表、需求 ID 或验收矩阵开篇。
- 多角色、多对象、多状态、多团队依赖或高风险需求，才展开核心概念、业务抽象、对象、流程、状态、规则、权限、审计和附录。
- 术语含义可能冲突、新业务对象首次进入 PRD、业务抽象会影响研发建模、或需要交给架构师/TDD 承接时，展开“核心概念与业务抽象”；轻量 PRD 可合并为 3-5 行术语说明。
- 未知信息写“待确认”，同时说明影响、确认方和未确认前处理方式。
- 正式 PRD 以最终标准版本为主，只保留当前有效的目标、范围、规则、风险、待确认、验收和必要版本元信息；讨论过程、迭代草稿、AI 推理轨迹、多角色争论、被拒方案和修订流水进入任务计划、产品合议评审报告、Decision Log 或中间任务文档，不复制进正文。
- 文档控制只保留当前版本号、状态、owner、发布日期、评审结论摘要和过程记录链接；主模板不默认展开完整修订历史。
- 需求变更必须补影响清单，覆盖目标/范围、流程/交互、规则/字段、接口/事件、验收种子、测试用例、发布风险和通知对象；影响清单进入过程资产，正式 PRD 只保留当前有效结论和链接。
- 验收摘要放在正文末尾，只保留业务结果、关键边界、异常和红线。详细验收矩阵进入执行计划；法规、审计、合同或项目约规要求单文档追踪时，才作为末尾按需附录。

优先级口径：

- P0：没有它不能上线、不能验收或会造成关键业务/资金/合规/安全风险。
- P1：核心体验或主流程必须具备，但可以通过灰度、人工兜底或明确限制降低上线风险。
- P2：增强体验、运营效率或后续扩展能力，不应阻塞当前版本核心目标。

## 默认 PRD 主模板：读者主线

### 1. 背景与问题

用连续短段落说明为什么现在要做、谁受到影响、现有处理方式在哪里受阻，以及判断依据来自哪里。先讲事实，再讲产品判断；不要以追踪表、问题 ID 或功能清单开篇。

- 现状与背景：〈当前业务、用户或运营处境〉
- 真实问题：〈用户任务、业务流程或经营目标受到什么阻碍〉
- 影响对象与代价：〈影响谁，不处理会发生什么〉
- 证据与可信度：〈访谈、数据、工单、事故、会议结论；已确认 / 假设 / 待确认〉

不把“用户要某功能”直接写成真实问题。来源较多、观点冲突或需要审计时，再在按需附录中建立反馈证据与问题地图。

### 2. 目标与非目标

先用一段话说明本期希望改变什么，再列必要边界：

- 业务目标：〈收入、效率、成本、风险、合规或经营目标〉
- 用户目标：〈用户能够完成什么此前难以完成的任务〉
- 成功口径：〈业务、用户、运营或数据上的可观察变化〉
- 本期非目标：〈明确不解决什么，以及原因〉
- 停止或拆期信号：〈出现什么情况需要重新评审范围〉

### 3. 定性、范围与总体判断

本节回答“这究竟是什么问题，应该在什么边界内解决”，避免未定性就堆功能：

- 产品定性：〈新产品 / 既有能力增强 / 流程治理 / 风险控制 / 运营提效 / 平台能力等〉
- 目标用户、业务主体与责任主体：〈谁使用、谁拥有规则、谁承担风险、谁验收〉
- 本期范围与上下游边界：〈产品负责、合作方 / 外部系统负责、人工承接、本期不做〉
- 能力提供者视角：〈本产品要稳定提供什么能力，而不是逐条实现哪些诉求〉
- 共性与核心要素：〈跨场景共同的目标、主体、对象、流程、规则和不变量〉
- 特殊性与变化轴：〈哪些差异有事实与验收依据，应进入场景、规则、参数或适配边界〉
- 稳定点与变化点：〈长期不变量、会按主体 / 渠道 / 规则版本变化的部分〉
- 核心名相：〈仅解释会影响共同理解的概念、对象及其“不是什么”〉
- 总体判断与可信度：〈推荐方向、关键依据、仍待确认的前提〉

察同用于发现稳定能力，辨异用于保护真实场景边界。一次性、局部或尚未证明复用价值的需求直接做最小方案，不强行抽象成平台能力。

### 4. 概要设计

用一段方案概述和一张按需关键图，让读者先建立整体认识，再进入细节：

- 方案概述：〈产品如何解决前述问题，为什么选择这条路径〉
- 核心能力：〈面向用户、运营、上下游提供哪些稳定能力〉
- 核心对象与关系：〈对象、归属主体、生命周期和关键不变量〉
- 关键交互与边界：〈用户端、运营端、外部参与方如何协作〉
- 方案取舍：〈真实备选、选择理由、放弃理由和复审条件〉
- 关键图：〈用例图 / 流程图 / 泳道图 / 状态机 / 能力地图 / 无需单独画图〉

图不是新的产品事实源。多角色、多对象、多状态、多团队或跨系统时，选择一张最小关键图暴露关系、状态、边界和冲突；轻量任务可明确不需要单独画图。

### 5. 详细设计

按场景展开能力如何被使用，不按页面控件或 AI 卡片展开：

#### 5.1 用户、主体与场景

- 角色与目标：〈谁在什么情境下要完成什么任务〉
- 前置条件：〈权限、对象状态、数据或外部依赖〉
- 产品行为：〈产品提供什么能力、反馈和状态变化〉
- 异常与人工兜底：〈失败、重复、超时、权限不足或人工处理〉

#### 5.2 产品能力与对象设计

- 能力职责：〈解决什么稳定业务问题，不负责什么〉
- 核心对象：〈定义、归属、字段口径、生命周期、状态和不变量〉
- 用户端与运营端：〈入口、查询、处理、复核、通知和人工兜底〉
- 扩展边界：〈真实变化轴如何扩展；无证据的未来需求不预埋〉

同类能力、对象或需求较多，需要稳定引用时再增加 `REQ-001` 等 ID；每个 P0/P1 需求仍须能回到问题、目标、场景、规则和验收，不要求正文处处展示追踪编号。

### 6. 关键流程

用编号步骤或流程图分别讲清：

1. 主流程：〈触发、校验、处理、反馈和完成〉
2. 逆向流程：〈撤销、取消、退回、冲正或恢复〉
3. 异常流程：〈失败、重复、超时、中断、外部不可用和状态不明〉
4. 人工流程：〈入口、责任人、SLA、复核和完成标准〉

每条流程应说明参与者、前置状态、输入、关键状态变化和可观察结果。流程图只降低理解成本，不替代规则和异常说明。

### 7. 业务规则与接口抽象

#### 7.1 业务规则

先用业务语言说明不变量和裁决逻辑；规则较多或有冲突优先级时再使用规则卡或矩阵：

- 适用对象与生效范围：〈主体、渠道、时间、版本〉
- 触发与判断条件：〈何时、依据什么判断〉
- 允许、禁止与必须：〈可做什么、不得发生什么〉
- 状态与冲突处理：〈状态守卫、规则优先级和版本裁决〉
- 异常与兜底：〈失败、冲突或外部不可用时怎么处理〉

#### 7.2 产品接口抽象

这里的“接口”是产品能力对角色或上下游暴露的稳定业务契约，不是提前设计 HTTP、RPC、字段或数据库：

- 接口名称与使用方：〈谁通过什么业务能力完成什么任务〉
- 输入与前置条件：〈业务材料、对象状态、权限和规则前提〉
- 业务输出与副作用：〈返回什么业务结果，改变什么对象或状态〉
- 失败语义：〈拒绝、挂起、重试、人工处理或无副作用〉
- 责任边界：〈本产品、调用方、外部机构或运营分别负责什么〉

技术协议、字段、错误码和幂等实现由系分继续设计；产品侧只固定会影响用户、业务规则和跨团队承诺的语义。

### 8. 数据、权限、风险与待确认

- 数据与指标：〈口径、数据源、留存、报表、埋点和 owner〉
- 权限、安全与审计：〈可见范围、操作权限、复核、日志和脱敏〉
- 运营可用性：〈后台、通知、告警、导出、客服和人工处理〉
- 依赖与风险：〈业务、合规、运营、数据、技术或外部依赖〉
- 待确认：〈问题、当前判断、影响、确认方、未确认前处理〉

### 9. 验收摘要与产品到架构交接

验收摘要放在正文末尾，使用业务和产品读者可以确认的语言：

- 正常结果：〈用户和业务能观察到的成功结果〉
- 关键边界：〈权限、状态、额度、时间、重复或临界条件〉
- 异常与兜底：〈失败后不得发生什么，如何恢复或人工处理〉
- 数据与运营结果：〈指标、报表、审计、通知或后台可观察结果〉
- 红线：〈资金、合规、隐私、安全或业务不变量〉
- 验收 owner：〈业务 / 产品 / 运营 / 合规等确认方〉

详细验收矩阵进入执行计划，承载验收种子 ID、Given / When / Then、测试映射、验证命令、执行 / 验证 owner 和任务状态。只有法规、审计、合同或项目约规要求单文档追踪时，才作为正文末尾附录；不得把它提前到主阅读路径。

业务驱动架构交接包保留已确认的目标、非目标、产品 DNA、核心对象、关键流程、关键规则、接口抽象、验收摘要、风险和待确认项。仍属假设的内容必须标注确认方和未确认前处理方式。

Product Context Card 只提取当前 PRD 的已确认事实，不形成第二份产品事实源：

```text
Product Context Card:
规范主题 / 产品文档路径:
业务目标 / 非目标:
产品定性 / 范围 / 业务 owner:
核心名相 / 用户 / 主体 / 对象:
状态 / 不变量 / 场景与关键流程:
关键业务规则 / 产品接口抽象:
验收摘要 / 风险 / 待确认:
交给架构师的边界 / 执行计划路径:
```

## PRD 按需组件库

以下组件只在命中复杂度或风险时展开；主文档保留组件摘要，避免默认模板过长。

### 核心概念与业务抽象卡

核心概念卡 C-001：

- 概念名称：〈业务概念〉
- 一句话定义：〈它是什么，用一句产品语言说明〉
- 适用范围：〈在哪些场景、对象、流程或规则中使用〉
- 不等于：〈容易混淆但不是它的概念、功能、对象或技术字段〉
- 主体/责任方：〈谁拥有定义、规则、数据和验收责任〉
- 同义词/系统名：〈业务别名、运营说法、英文名、API/字段名，按需填写〉
- 关联项：〈关联需求ID、对象、规则编号、验收种子ID或待确认项〉
- 可靠性标记：已确认 / 假设 / 待确认 / 不适用

业务抽象卡 A-001：

- 抽象名称：〈能力、对象、规则或关系抽象〉
- 抽象目的：〈把哪些业务现象、角色动作或规则差异归一，为什么需要抽象〉
- 归属边界：〈属于哪个业务主体、能力域、流程阶段或系统交接面〉
- 关键关系：〈与用户、主体、对象、状态、规则、数据或外部参与方的关系〉
- 生命周期/状态锚点：〈是否有创建、变更、冻结、关闭、失效、回滚等状态〉
- 不变量：〈任何实现、页面或流程都不能破坏的业务约束〉
- 不抽象范围：〈本期不归一、不平台化、不泛化的内容，防止过度抽象〉
- 验收/交接：〈关联验收种子、架构交接项、待确认项或专项附录〉

产品 DNA 卡 D-001：

- 核心对象：〈围绕哪些业务对象提供价值，它代表什么事实、不代表什么〉
- 业务不变量：〈任何页面、流程、规则、实现都不能破坏的业务事实〉
- 生命周期 / 状态：〈创建、变更、冻结、完成、失败、取消、归档及禁止流转〉
- 责任边界：〈本产品负责 / 合作方负责 / 外部系统负责 / 人工运营负责 / 本期不做〉
- 演化规则：〈新增角色、渠道、策略、规则、状态或上下游协作时如何扩展〉
- 验收方式：〈关联 AC、流程图、状态机、规则卡、数据口径、运营后台、审计或架构交接〉
- 可靠性标记：已确认 / 假设 / 待确认 / 不适用

### 对象、流程与图形视图组件

对象口径：

- 核心对象：〈对象定义、所属主体、关键状态、关键字段、不变量〉
- 对象关系：〈对象之间如何关联，是否有 1:1、1:n、n:m 或外部引用〉
- 不变量：〈任何状态下都必须成立的业务规则〉

主流程：

1. 触发：〈用户、系统或外部事件触发〉
2. 校验：〈权限、状态、数据、外部依赖或规则校验〉
3. 处理：〈系统处理、人工处理或外部调用〉
4. 反馈：〈页面、通知、状态、数据、报表或日志变化〉
5. 收口：〈完成、失败、取消、超时、回滚或人工兜底〉

图形视图：

- 首选图形：〈用例图 / 流程图 / 泳道图 / 状态机图 / 能力地图 / 不需要单独画图〉
- 图形目标：〈这张图解决什么评审问题〉
- 目标读者：〈业务 / 产品 / 运营 / 测试 / 架构师 / 开发 / 合规〉
- 覆盖内容：〈角色、对象、状态、流程、规则、异常、人工兜底或外部依赖〉
- 追踪关系：〈关联需求ID、规则编号、验收种子ID或待确认项〉
- 输出形态：〈正文草图 / SVG / Mermaid 草图，默认 SVG 仅在正式图形化交付时生成〉

### 规则卡与专项附录组件

规则卡 R-001：

- 适用对象：〈对象〉

- 验收样例：〈可观察的成功、失败或边界样例〉

附录按需展开：

## 精简输出规则

- 默认不展开文档控制、完整状态机、字段字典、规则矩阵、权限矩阵、非功能、发布运营和金融资金全表。
- 默认不强制每份 PRD 生成图片；多角色、多对象、多状态、多团队交接或高风险评审时，至少选择一张关键图帮助建立共同理解。
- 命中多团队协作、高风险、强合规、生产发布、用户明确要求或正式评审时，再展开相关附录。
- 主文档优先用短段落讲清因果，用项目符号和编号流程帮助扫读；重复条目才使用卡片，历史过程和明细矩阵移入附录或执行计划。
- 表格只用于比较、矩阵或追踪；单个问题、规则、验收和待确认项优先用段落或列表。
- 复杂并不等于写长，必须让评审者能在 5 分钟内看懂背景、目标、定性、总体方案、关键流程、规则、风险和验收摘要。
- 最终 PRD 不保留“第一版/第二版怎么改”“某角色建议过什么”“哪些方案曾被讨论后拒绝”等过程内容；只在必要时用一句当前结论或链接指向过程记录。
