# 产品 PRD 文档模板

本文是产品架构专家的轻量 PRD 主模板，用于把需求整理成可评审、可开发、可测试、可运营的产品文档。默认目标是快速阅读、准确传递、可靠描述和便于执行，不是全量检查手册；复杂对象、完整状态机、规则矩阵、运营数据、金融资金和发布计划按需进入专项附录。

使用前先明确文档目标、目标读者和文档类型。主文档只回答：为什么做、给谁做、做什么、不做什么、怎么流转、按什么规则验收、还有哪些风险。生成前先裁剪，不能把轻量需求写成无人评审的长文，也不能只用功能名替代范围、规则和验收。

模板占位符使用 `〈...〉` 表示。实际输出时，正文段落和表格单元格默认保持普通文本；只有接口路径、技术字段、配置键、协议字段、风险等级或需要强调的结论才使用加粗或代码背景。

## 使用时机

- 用户要求写作、生成、完善、补全、改写或评审 PRD、产品需求文档、需求说明书、需求文档模板、PRD 模板或产品方案模板。

## 不适用场景

- 仅做产品架构方法分析时读 `product-architecture-methodology.md`。
- 支付资金、PRD 评审、运营数据发布等专项内容按需读取独立附录，不在主模板展开。

## 读取后必须产出

- 可复制的 PRD 主骨架、裁剪依据、待确认项和需要继续读取的专项附录。

## 需要继续读取的 reference

- PRD 写作标准读 `product-design-and-prd.md`。
- 产品方案交给架构师继续设计、业务驱动架构交接或 TDD 承接时读 `product-architecture-methodology.md`。
- 正式评审、提交前检查、CR、触发验证或符合性评审时读取 `product-prd-quality-gates.md`。
- 支付资金读 `product-prd-financial-appendix.md`。
- 运营后台、通知、数据指标、埋点、报表、导出、发布后验证读 `product-prd-operations-and-data.md`。

## 信息传递流程

PRD 的阅读路径按“事实 -> 判断 -> 决策 -> 承诺 -> 验证”组织：

1. 事实：原始反馈、数据线索、业务背景、现状约束和真实问题。
2. 判断：问题是否成立、影响多大、证据强度如何、哪些仍是假设。
3. 决策：本期做什么、不做什么、优先级为什么这样排。
4. 承诺：产品能力、用户场景、流程规则、权限数据和风险处理。
5. 验证：验收种子、可观察结果、确认方、交给架构师或测试的依据。

描述可靠性必须可见：已确认、假设、待确认、不适用要分开写；无法确认的信息不得写成确定结论。

同时按角色分层消费：决策层服务业务 / 管理者，说明为什么做、做什么和成功标准；方案层服务产品 / 设计 / 架构，说明方案取舍、范围、成本和风险；实现层服务开发 / 测试 / 发布，说明输入输出、边界异常、接口事件、验收和验证证据。

## 表达结构选择

- 结论、范围和风险用短段落或项目符号，便于扫读。
- 用户问题、规则、验收和待确认项优先用卡片，便于补充上下文。
- 业务流程优先用编号步骤或文本流程，便于读者理解先后关系。
- 多角色、多对象、多状态、多团队协作或评审者难以凭文字建立共同理解时，优先补一张图；图形是 PRD 的阅读入口，不替代规则、验收和待确认项。
- 表格只用于比较、矩阵或追踪；不要把所有信息都塞进表格。
- 多需求、多角色、多规则、多验收种子需要横向比较时，再把卡片合并成表格。

## 图形化视图选择

图形按问题选择，不要求每份 PRD 固定画全套；普通轻量 PRD 可以只保留文本流程。命中复杂度或评审沟通成本较高时，在正文第 6 节或附录补充 1-3 张关键图：

- 用例图：用于说明角色、系统边界、核心用例和责任主体，适合评审“谁能做什么、谁负责验收”。
- 流程图：用于说明主流程、逆向流程、异常流程和人工兜底，适合评审“事情按什么顺序发生、哪里分支”。
- 泳道图：用于说明用户、运营、风控、财务、外部机构和系统之间的交接，适合评审“跨团队动作和 SLA 是否闭环”。
- 状态机图：用于说明对象生命周期、状态迁移、触发事件、守卫条件、终态和回滚，适合评审“对象何时能被操作”。
- 能力地图/产品架构图：用于说明本期能力边界、前后台能力、数据能力、风控审计和外部系统依赖，适合评审“范围是否清楚”。

每张图必须说明图形目标、目标读者、对应 PRD 章节、关键假设、待确认项和与验收/规则的追踪关系。需要正式图形化交付时读取 `diagram-output.md`；默认输出 SVG，Mermaid/Markdown 草图、PNG/PDF/截图等其他格式需用户明确提出。

| 任务 | 优先读取 | 跳过 |
| --- | --- | --- |
| 轻量 PRD 或小功能需求 | 一页摘要、问题证据、目标范围、场景功能、验收交接 | 不展开完整对象、流程、规则和专项附录 |
| 标准复杂业务 PRD | 主模板 1-8 节，按需读“PRD 按需组件库” | 不把组件库全部塞进正文 |
| 术语歧义、新业务对象或跨团队交接 | 主模板 4-8 节、核心概念卡、业务抽象卡 | 不直接堆功能点或页面控件 |
| 运营后台、权限、安全、审计 | 主模板 7-8 节、`product-prd-operations-and-data.md` | 不把运营后台写成普通前台功能 |
| 数据指标、埋点、报表 | 主模板 7-8 节、`product-prd-operations-and-data.md` 指标和报表部分 | 不用泛化指标替代口径 owner |
| 支付资金、清结算、VCC、跨境 | 主模板 7-8 节、`product-prd-financial-appendix.md` 和支付专项 reference | 不把通用 PRD 模板当金融规则结论 |
| 产品到架构交接或 TDD 承接 | 主模板 5-8 节、`product-architecture-methodology.md` | 不把产品上下文包当 Execution Grant |
| PRD 评审或提交前自检 | 当前 PRD 相关章节、`product-prd-quality-gates.md` | 不用模板完整度替代评审结论 |

## 使用与裁剪原则

- 轻量需求默认 5-8 个小节即可，不补全无关表格。
- 多角色、多对象、多状态、多团队依赖或高风险需求，才展开核心概念、业务抽象、对象、流程、状态、规则、权限、审计和附录。
- 术语含义可能冲突、新业务对象首次进入 PRD、业务抽象会影响研发建模、或需要交给架构师/TDD 承接时，展开“核心概念与业务抽象”；轻量 PRD 可合并为 3-5 行术语说明。
- 未知信息写“待确认”，同时说明影响、确认方和未确认前处理方式。
- 正式 PRD 以最终标准版本为主，只保留当前有效的目标、范围、规则、风险、待确认、验收和必要版本元信息；讨论过程、迭代草稿、AI 推理轨迹、多角色争论、被拒方案和修订流水进入任务计划、产品合议评审报告、Decision Log 或中间任务文档，不复制进正文。
- 文档控制只保留当前版本号、状态、owner、发布日期、评审结论摘要和过程记录链接；主模板不默认展开完整修订历史。
- 需求变更必须补影响清单，覆盖目标/范围、流程/交互、规则/字段、接口/事件、验收种子、测试用例、发布风险和通知对象；影响清单进入过程资产，正式 PRD 只保留当前有效结论和链接。

优先级口径：

- P0：没有它不能上线、不能验收或会造成关键业务/资金/合规/安全风险。
- P1：核心体验或主流程必须具备，但可以通过灰度、人工兜底或明确限制降低上线风险。
- P2：增强体验、运营效率或后续扩展能力，不应阻塞当前版本核心目标。

## 默认 PRD 主模板：需求主线

### 1. 需求概览

用 6-8 行先让读者知道这份 PRD 的核心结论、交付范围和需要支持的评审决策：

- 目标读者：〈产品 / 业务 / 运营 / 设计 / 测试 / 架构师 / 开发 / 合规，按需填写〉
- 需要决策：〈评审通过 / 确认范围 / 确认风险 / 进入设计 / 进入开发候选〉
- 本期结论：〈一句话说明要做的能力和价值〉
- 交付范围：〈本期做什么〉
- 非目标：〈本期明确不做什么〉
- 成功标准：〈业务、用户、运营、数据或质量指标〉
- 最大风险：〈业务、技术、合规、运营、数据或依赖风险〉
- 当前可信度：〈已确认 / 假设 / 待确认 / 不适用，并说明原因〉

### 2. 背景、问题与证据

用 3-6 行说明为什么现在要做、现有流程哪里受阻、依据来自哪里。需求来自反馈、工单、访谈、销售转述、数据或会议纪要时，保留反馈证据与问题地图；非反馈驱动需求写“不适用”并说明依据。

问题卡 P-001：

- 假设/问题ID：P-001
- 原始反馈/证据：〈原始反馈、数据线索或会议结论〉
- 真实问题：〈任务阻塞或业务问题〉
- 影响对象：〈用户 / 客户 / 运营 / 系统〉
- 证据强度：高 / 中 / 低，依据是〈数据、访谈、工单、事故、会议确认等〉
- 处理结论：进入 PRD / 待验证 / 非目标 / 观察池

检查要求：

- 不把“用户要某功能”直接写成真实问题；必须说明功能诉求背后的任务、流程或业务阻塞。
- 证据不足时进入待验证或待确认项，不把低证据反馈包装成已确认需求。

### 3. 目标、范围与非目标

目标说明：

- 业务目标：〈收入、效率、成本、风险、合规、体验等目标〉
- 用户目标：〈用户希望完成什么〉
- 运营/数据目标：〈后台、客服、财务、风控、数据或报表目标〉

范围决策：

- 本期做：〈本期交付的能力、页面、后台、规则、报表或流程〉
- 本期不做：〈明确排除项，以及为什么排除〉
- 上下游边界：〈系统、团队、合作方、外部依赖和失败处理责任〉
- 范围失控信号：〈出现哪些诉求时必须回到评审或拆版本〉

### 4. 用户、主体与角色/场景

主体口径：

- 用户：〈直接使用者〉
- 业务主体：〈客户、组织、商户、平台或内部团队〉
- 责任主体：〈谁拥有规则、数据、运营和验收责任〉
- 外部参与方：〈合作方、通道、监管或第三方平台，按需填写〉

场景 S-001：

- 角色：〈谁〉
- 触发：〈在什么业务背景或状态下发生〉
- 目标：〈想完成什么任务〉
- 成功结果：〈用户、业务或运营看到什么变化〉
- 失败/异常：〈失败、重复、超时、权限不足或人工兜底〉

### 5. 场景、能力与功能：需求链卡片

用需求链把问题、目标、场景、能力、规则和验收串起来，避免另起长篇追踪索引。

需求ID REQ-001：

- 来源/问题ID：P-001
- 场景编号：S-001
- 角色：〈角色〉
- 用户故事/任务：〈在什么场景下完成什么任务，获得什么价值〉
- 能力/功能：〈能力或功能〉
- 优先级：P0 / P1 / P2
- 规则编号：R-001
- 验收种子ID：AC-001
- 状态：草稿 / 待确认 / 已确认 / 非目标

要求：

- 每个 P0/P1 需求必须能回到问题、目标、场景、规则和验收。
- 每个进入开发候选的需求必须能关联场景、能力或功能、关键规则和验收种子。
- 无法关联目标或验收的条目先进入观察池、待验证或非目标。

## 默认 PRD 主模板：规则、风险与交接

### 6. 核心概念、业务抽象、对象、流程、图形视图与规则

轻量需求用短段落和编号步骤说明；复杂需求再按“PRD 按需组件库”展开核心概念、业务抽象、关键图形视图、完整状态机、生命周期、字段口径和规则矩阵。

按需启用条件：

- 业务术语、运营口径、系统命名或既有文档存在冲突。
- 本期引入新的业务对象、主体关系、能力抽象、规则抽象或产品平台化能力。
- 多团队、多系统、多角色、多状态协作，仅靠功能列表无法建立共同语言。
- PRD 需要交给架构师继续系统设计、TDD、验收种子或工程交接。
- 涉及支付资金、合规、隐私、安全、审计、生产运营或外部规则。
- 对象、状态、规则或责任边界会长期演化，需要先写产品 DNA，避免功能先行、规则后补。

本节默认只保留 3 类正文：

- 核心概念：说明容易误解的业务词。
- 业务抽象：说明需要归一的能力、对象、规则或关系。
- 产品 DNA：说明核心对象、不变量、生命周期 / 状态、责任边界、演化规则和验收方式。
- 最小流程/规则：说明主路径、关键约束和验收样例。

需要完整卡片、图形 brief、对象口径或规则卡时，继续读“PRD 按需组件库”。

### 7. 数据、权限、风险与待确认

数据与权限说明：

- 数据/指标：〈指标口径、数据源、留存、报表或埋点〉
- 权限/安全/审计：〈可见范围、操作权限、日志、复核、脱敏要求〉
- 运营可用性：〈后台、人工处理、通知、告警、导出或客服支撑要求〉

风险与待确认卡 Q-001：

- 问题/假设：〈问题、假设、外部规则或专业确认项〉
- 当前结论：〈已确认 / 假设 / 待确认 / 不适用〉
- 影响：〈对评审、开发、测试、运营或上线的影响〉
- 确认方/owner：〈负责人或确认团队〉
- 截止时间：〈日期〉
- 未确认前处理：〈限制范围、人工兜底、延后、不得上线或进入专项评审〉

### 8. 验收与产品到架构交接

验收种子交接矩阵可以先用卡片表达；多需求需要横向比较时再转成表格。

验收种子 AC-001：

- 需求ID：REQ-001
- 验收种子ID：AC-001
- 类型：正向 / 异常 / 权限 / 数据 / 运营 / 风险
- 前置条件：〈条件〉
- 触发：〈动作或事件〉
- 可观察结果：〈页面、数据、状态、日志、报表或通知结果〉
- 质量属性ID：QA-001
- 确认方：〈产品 / 业务 / 测试 / 运营 / 合规〉

业务驱动架构交接包至少保留：

- 已确认的目标、非目标、需求链、产品 DNA、核心对象、关键流程、关键规则、验收种子、风险和待确认项。
- 仍属假设的内容必须标注确认方和未确认前处理方式。
- 交给架构师继续系统设计、TDD 或生产验证时，补充 `product-architecture-methodology.md`，并说明哪些内容需要可代码化、可观测化、可评审化。

## PRD 按需组件库

以下组件只在命中复杂度或风险时展开；主文档保留组件摘要，避免默认模板过长。

### 核心概念与业务抽象卡

核心概念卡 C-001：

- 概念名称：〈业务概念〉
- 一句话定义：〈它是什么，用一句产品语言说明〉
- 适用范围：〈在哪些场景、对象、流程或规则中使用〉
- 不等于：〈容易混淆但不是它的概念、功能、对象或技术字段〉
- 主体/责任方：〈谁拥有定义、规则、数据和验收责任〉
- 同义词/系统名：〈业务别名、运营说法、英文名、API/字段名，按需填写〉
- 关联项：〈关联需求ID、对象、规则编号、验收种子ID或待确认项〉
- 可靠性标记：已确认 / 假设 / 待确认 / 不适用

业务抽象卡 A-001：

- 抽象名称：〈能力、对象、规则或关系抽象〉
- 抽象目的：〈把哪些业务现象、角色动作或规则差异归一，为什么需要抽象〉
- 归属边界：〈属于哪个业务主体、能力域、流程阶段或系统交接面〉
- 关键关系：〈与用户、主体、对象、状态、规则、数据或外部参与方的关系〉
- 生命周期/状态锚点：〈是否有创建、变更、冻结、关闭、失效、回滚等状态〉
- 不变量：〈任何实现、页面或流程都不能破坏的业务约束〉
- 不抽象范围：〈本期不归一、不平台化、不泛化的内容，防止过度抽象〉
- 验收/交接：〈关联验收种子、架构交接项、待确认项或专项附录〉

产品 DNA 卡 D-001：

## 精简输出规则

- 默认不展开文档控制、完整状态机、字段字典、规则矩阵、权限矩阵、非功能、发布运营和金融资金全表。
- 默认不强制每份 PRD 生成图片；多角色、多对象、多状态、多团队交接或高风险评审时，至少选择一张关键图帮助建立共同理解。
- 命中多团队协作、高风险、强合规、生产发布、用户明确要求或正式评审时，再展开相关附录。
- 主文档优先用短句、卡片和编号流程帮助扫读；长解释、历史背景和明细矩阵移入附录。
- 表格只用于比较、矩阵或追踪；单个问题、规则、验收和待确认项优先用卡片。
- 复杂并不等于写长，必须让评审者能在 5 分钟内看懂目标、范围、风险和验收。
- 最终 PRD 不保留“第一版/第二版怎么改”“某角色建议过什么”“哪些方案曾被讨论后拒绝”等过程内容；只在必要时用一句当前结论或链接指向过程记录。
