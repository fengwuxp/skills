# 问题台账与决策快照

## 使用时机

当 `grill-me` 进入多轮盘问、需要复用历史结论、可能跨上下文恢复、出现红线或需要形成下一阶段输入时读取。

## 不适用场景

- 一个问题即可关闭且不需要跨轮恢复的轻量确认。
- 只有可查 Facts、没有未决方案或设计分叉的任务。
- 已确认计划的执行与验证；此时直接交给对应执行能力。

## 读取后必须产出

- 每个问题、裁决动作及最终结论的可追溯记录。
- 已确认、已排除、待确认、冲突和 red_lines 的当前集合。
- 去重或重开依据，以及供下一阶段消费的决策快照。

## 需要继续读取的 reference

- 跨阶段执行前对账读 `wise-agent/references/delivery-execution-control.md`，仅在项目仓库上下文可用时读取。
- 需要经典决策校准时装载 `huaxia-practical-wisdom`，不在本文件复制经典框架。

## 按任务读取索引

| 任务 | 优先读取 | 跳过 |
| --- | --- | --- |
| 每轮记录问题与结论 | `问题记录`、`历史恢复与语义去重` | 不新建语义重复问题 |
| 从既有材料自行决定是否提问 | `证据先行的问询裁决`、`历史恢复与语义去重`、`自决边界` | 不把资料名称当作已核验 |
| 处理红线或结论冲突 | `红线`、`决策快照` | 不静默覆盖历史结论 |
| 跨阶段或跨上下文交接 | `台账载体`、`决策快照` | 不只依赖聊天记忆 |

## 台账载体

按已有权威载体就地更新：用户指定位置 -> Issue / Goal Ledger / Spec / Decision Log -> 项目已有任务状态或知识库。没有获准载体时，在当前任务中维护并在退出前输出；不得为了台账自动创建 `CONTEXT.md`、ADR、知识库目录或新的任务系统。

## 证据先行的问询裁决

每次提问前必须完成以下裁决；目的不是多查资料，而是只问证据仍无法关闭的剩余 Decision。

1. 把 blocker 改写为“对象 + 待判断属性或取舍 + 作用范围或阶段”的待裁决命题。
2. 标注命题类型并读取匹配的一手来源：
   - 意图与约束：用户当前明确结论、PRD / Spec、ADR、项目约规和可追溯知识库。
   - 当前实现：源码、配置、Schema、测试、日志和工具回读。
   - 历史决策：问题台账、Decision Log、Issue、Goal、ADR 及其 Owner 状态。
   - 风险与红线：Owner 原意、权限边界、强制约规以及适用的安全、合规或生产证据。
3. 实际读取内容和锚点并交叉核验，不得只列资料名。意图证据与实现证据回答不同问题，不能互相覆盖；代码和测试只能证明当前行为，不能自动定义正确业务语义。
4. 知识库只有能回链来源、Owner、适用范围和更新时间时才能作为已确认依据，否则只作检索线索；聊天摘要、模型记忆和无锚点转述也只作线索。
5. 形成唯一裁决动作：
   - `fact-confirmed`：匹配证据足以确认事实，直接自答。
   - `decision-reused`：存在仍适用的 Owner 结论或项目规则，直接复用。
   - `self-decided`：低风险、可逆、在授权内且不触碰 red_lines，采用默认选择并写明回退与验证。
   - `ask-owner`：证据缺失、冲突或失效，或剩余项属于新的价值取舍、公共契约、高风险、不可逆和红线冲突；只问这一个剩余 Decision。
6. 按“裁决动作到最终结论的确定映射”写入唯一状态机：`fact-confirmed` -> `confirmed`；`decision-reused` -> 沿用原最终结论；`self-decided` -> `self-decided`；`ask-owner` -> `pending`，若因证据冲突触发则在 Owner 裁决前为 `conflict`。Owner 回答后只能更新为 `confirmed`、`rejected` 或继续 `pending`。
7. 记录待裁决命题、命题类型、证据锚点、证据冲突、裁决动作和置信边界。前三类不得再问；`ask-owner` 必须说明已查什么、为何仍不能自决。历史恢复、语义去重和决策快照只读取最终结论；裁决动作只说明结论如何得到，不作为第二套状态。
8. 对外台账无论完整还是压缩，每条记录都必须原样包含 `裁决动作：<action>` 和 `最终结论：<state>`；不得省略任一字段，也不得用“裁决”“状态”等近义标签替代。

## 问题记录

```text
问题 ID：稳定编号，例如 Q-001。
决策主题：对象 + 取舍 + 作用范围或阶段，用于语义去重。
待裁决命题 / 命题类型：本轮要关闭的判断，以及意图与约束 / 当前实现 / 历史决策 / 风险与红线。
父主题：当前设计树分支；没有则为空。
问题：本轮只保留一个主 blocker。
为何现在问：它将改变的方向、范围、契约、验收或风险。
已查证据：历史问答、文档、代码、测试、日志、规则、知识库及锚点。
证据冲突 / 置信边界：不一致、失效、缺口及当前结论不能覆盖的范围。
裁决动作：fact-confirmed / decision-reused / self-decided / ask-owner。
推荐答案：Agent 建议、依据、代价和反证。
用户回答：原意摘要；不伪造原话。
最终结论：confirmed / rejected / pending / self-decided / conflict / superseded。
red_lines：本题命中的红线、底线、禁止或必须项。
影响范围：受影响的对象、文档、模块、任务或阶段。
下一阶段输入：可直接消费的约束、选择和待办。
重开原因：仅在证据变化、结论冲突、风险升级或 Owner 要求时填写。
写回位置 / 更新时间：真实载体和当前时间；未知则标待确认。
```

问题提出后立即建立记录；用户反馈后在同一记录更新最终结论，不另建一个语义重复问题。自决也必须建记录，不能只在回答里一带而过。

## 历史恢复与语义去重

提问前依次执行：

1. 恢复最近决策快照和未关闭问题，再读取关联 Issue、Goal、Spec、ADR、PRD、代码、测试和知识库。
2. 用“对象 + 取舍 + 范围或阶段”生成决策主题；比较含义，不比较措辞。
3. 只按最终结论恢复：命中 confirmed、rejected 或 self-decided 时复用结论，不再提问；命中 pending 时继续原问题，不新建编号；命中 conflict 时只问原冲突的裁决问题。
4. 发现语义重复但措辞不同，合并到原记录并标注 duplicate；不得重问。
5. 只有证据变化、历史冲突、风险升级或 Owner 明确要求时重开，保留旧结论并填写重开原因。

上下文不足以确认历史是否存在时，先读取允许的载体；仍无法恢复时说明缺口并请求用户提供位置，不假装这是一个全新问题。

## 自决边界

| 情况 | 动作 | 留痕 |
| --- | --- | --- |
| 可从文档、代码、测试、日志或工具确认的 Facts | 自答，不问用户 | 证据锚点与事实结论 |
| 已有 Owner 批准的决策或明确项目规则 | 复用，不重问 | 来源、适用范围和更新时间 |
| 低风险、可逆、在当前授权内且不触碰 red_lines | 可 self-decided | 默认选择、回退和验证条件 |
| 新价值取舍、公共契约、高风险、不可逆或红线冲突 | 一次问 Owner 一个 blocker | 推荐答案、影响和停止点 |
| 证据彼此冲突或来源失效 | 停止覆盖与执行 | conflict、冲突证据和裁决 Owner |

“自行决定”只减少不必要提问，不扩大写文件、Git、联网、发布、生产或不可逆权限。

## 红线

命中“红线、底线、不能碰、不可、禁止、不得、必须”等表达时：

1. 原意摘要写入 red_lines，并标记来源与适用范围。
2. 后续问题先检查是否触碰；触碰时不得用默认答案绕过。
3. 新结论若可能削弱红线，必须重开原决策主题并交 Owner 裁决。
4. 决策快照和执行前对账必须逐项复述，未对账不得执行。

## 决策快照

```text
已确认选择：关联问题 ID 与证据。
被排除方案：关联问题 ID 与排除原因，不得复活。
待确认项：未关闭问题、Owner 和停止影响。
red_lines：不可静默弱化的约束及来源。
下一阶段输入：目标、范围、约束、验收、验证和允许动作。
写回位置：Issue / Goal / Spec / Decision Log / 任务状态 / 当前任务输出。
```

退出条件不是问题数量，而是关键分叉均已 confirmed、rejected、self-decided 或明确 pending，且用户确认 shared understanding。连续两轮没有新证据或新决策时停止扩问，输出剩余待确认项。
