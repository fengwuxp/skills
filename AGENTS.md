# AGENTS.md

本仓库用于维护可安装到 Codex 的 Skills。创建、修改或审查技能时，必须遵守本文件约定。

## 技能设计原则

- `SKILL.md` 保持精简，承载角色定位、核心流程、场景路由和必要红线。
- 详细知识、模板、清单和方法论放入 `references/`，由 `SKILL.md` 明确何时读取。
- 不创建无必要的 README、安装指南、变更日志等辅助文件；仓库级说明统一放在根目录 `README.md`。
- 更新技能时同步检查 `agents/openai.yaml` 是否仍匹配技能定位。
- 修改后至少执行 YAML、引用和同步 dry-run 等轻量校验。
- 正式 PRD、系分、OpenSpec/SDD 等交付文档以最终标准版本为主，不保留讨论过程、迭代草稿、AI 推理轨迹或被拒方案展开；过程内容进入任务计划、评审报告、Decision Log、Goal Ledger、ADR 或中间任务文档。

## Skill 三级加载机制

Skill 必须按渐进式加载设计，不得把所有知识一次性塞进 Agent 上下文。Agent 的上下文窗口有限，Skill 维护者必须让 Agent 先用轻量元数据识别技能，再在技能触发后读取正文，只有需要细节时才读取 bundled resources。

| 层级 | 内容 | 规范 | 设计要求 |
| --- | --- | --- | --- |
| Level 1：Metadata | `SKILL.md` frontmatter 中的 `name` + `description`，以及与其一致的 `agents/openai.yaml` 展示信息。 | 约 100 词，始终在上下文中。 | 用于触发识别，必须短、准、覆盖核心场景；不得放长流程、细节清单、模板或领域知识。 |
| Level 2：Skill Body | `SKILL.md` 正文。 | 建议 500 行以内，Skill 触发后加载。 | 承载角色定位、核心工作流、场景路由、何时读取 reference、必要红线和不适用场景；不得复制 `references/` 的长知识。 |
| Level 3：Bundled Resources | `references/`、`scripts/`、`assets/`。 | 无固定大小限制，按需读取或执行。 | 承载详细方法论、模板、清单、示例、确定性脚本和输出资产；必须由 `SKILL.md` 明确读取时机。 |

三级加载落地规则：

- 新建或重构 Skill 时，先判断内容属于 Level 1、Level 2 还是 Level 3，再决定写入位置。
- `description` 只服务触发和识别，不承担完整说明书职责；过长、过窄或关键词堆砌都会降低触发质量。
- `SKILL.md` 只保留 Agent 执行当前任务必须立即知道的内容；超过 500 行或出现多个专题细节时，应拆入 `references/`。
- `references/` 必须一层直连，可由 `SKILL.md` 找到；长文件应在开头说明使用时机或目录，方便按需读取。
- 可确定、可重复、容易写错或需要真实解析/生成/校验的能力，应优先沉淀到 `scripts/`，而不是写成长段自然语言步骤。
- 同一规则只保留一个权威来源；其他层只做摘要和链接，避免 Level 2 与 Level 3 内容漂移。
- 审查 Skill 时必须检查三级加载是否成立：Metadata 能否触发、Skill Body 是否可执行、Bundled Resources 是否按需可发现。

## Agent 运行时分层

本仓库中的 Skill 不是一次性提示词，而是 Agent 运行时资产的一部分。维护技能时必须先判断经验、规则或工具应该进入哪一层，避免把所有内容都塞进 `SKILL.md`。

| 层级 | 放什么 | 不放什么 |
| --- | --- | --- |
| `AGENTS.md` | 每个会话都应该知道的默认行为、仓库级规则、安全边界、验证要求和经验迁移原则。 | 具体领域长知识、专题 checklist、单个技能的细节流程。 |
| `SKILL.md` | 特定任务才需要的角色定位、触发场景、核心流程、场景路由、必要红线和不适用场景。 | 大段背景知识、完整模板、可用脚本替代的手工步骤。 |
| `references/` | 领域知识、详细方法论、评审清单、模板、案例和复杂分支说明。 | 必须每次强制执行的校验动作。 |
| `scripts/` | 需要确定性、可重复、可审计执行的生成、验证、同步和安全检查。 | 只靠自然语言即可表达的判断原则。 |
| 目录级规则 | 只对某个技能目录或某类文件长期生效的维护约定。 | 已经适用于全仓库的通用规则。 |
| `~/.skill-learning/` | 用户明确同意后的本地长期偏好、业务背景和协作习惯。 | 仓库技能定义、团队公共规则、需要提交到远程的内容。 |

经验归位原则：

- 凡是每个会话都应该知道的内容，进入 `AGENTS.md`。
- 凡是多步流程、专题检查、分支决策，进入对应 `SKILL.md` 或 `references/`。
- 凡是只对某个技能目录生效的规则，才考虑目录级规则；不得污染全局。
- 凡是必须每次执行、不能依赖模型自觉的动作，进入 `scripts/` 或外部 Hook。
- 凡是需要真实查询、生成、解析或校验的能力，优先沉淀为脚本；Skill 负责决策和参数推断，脚本负责执行。
- 凡是从专题经验演化为所有相关任务都应遵守的通用约束，应从 Skill 上移到 `AGENTS.md`。
- 凡是用户个性化偏好、业务背景或协作习惯，按本地协作学习机制处理，不写入仓库。

Skill 经验抽象可按三类归位：

- 规划层经验：任务识别、触发路由、阶段顺序、依赖关系、分支门禁和停止条件，进入 `SKILL.md` 的路由摘要、`references/` 的 workflow/scenario routing，或触发验证 fixture。
- 功能层经验：可复用子任务流程、模板、评审清单、输出结构、领域方法论和组合模式，进入 `references/`，并由 `SKILL.md` 说明读取时机。
- 原子层经验：工具调用约束、脚本参数、输入输出 schema、常见失败模式、负例、fixture 和确定性校验，进入 `scripts/`、fixtures 或验证脚本。

沉淀经验前必须过滤一次性探索、回退、试错、临时偏好和未验证结论；相似经验应合并，过大的经验应拆分，引用不存在工具、无效参数或外部未审查能力的内容不得进入 Skill。

## Skill 供应链安全

Skill 可以携带脚本、引用外部资料并影响 Agent 行为，因此按可执行资产治理。

- 不安装、同步或复制来源不明的外部 Skill；引入外部 Skill 前必须审查 `SKILL.md`、`references/`、`scripts/` 和默认权限边界。
- Skill 脚本不得默认联网、上传文件、读取密钥、读取用户私有目录、修改 Git 历史或执行不可逆操作；确需此类能力时必须由用户明确授权。
- 任何新增脚本都应说明输入、输出、写入范围、失败行为和是否会访问网络。
- `sync-skills.sh` 使用 `rsync --delete`，正式同步前必须先 dry-run，并确认 `CODEX_HOME`、目标技能和备份位置正确。
- 不得把使用者学习记录、密钥、token、客户数据、内部合同、生产配置或不可公开组织信息提交到本仓库。
- 修改技能后必须执行统一验证；新增脚本或外部资料引用时，还应执行安全审计。

## 外部文章读取约规

- 分析、提炼或吸收外部文章时，必须以实际读取到的正文为依据；不得只凭标题、索引页、历史摘要、模型记忆或已有 `source-map.md` 条目推断文章内容。
- 常规抓取、网页搜索或 `curl` 无法取得正文，出现验证页、空正文、截断正文、动态渲染或跳转异常时，必须改用 Playwright 或等价浏览器自动化加载页面，提取标题、作者、发布时间和正文后再分析。
- Playwright 仍无法取得正文时，必须说明已尝试的读取方式、失败证据和残余风险；不得把未读取到正文的文章写成已吸收结论。
- 吸收到 Skill 时只保留可迁移的方法、检查项、路由和边界；不复制文章原文、原图、付费内容、作者口吻或未经核验的时效性结论。

## 外部文章本地证据归档约规

- 对高价值且易失效的外部文章，可在实际读取后保存本地私有证据；证据归档只用于后续复核，不用于公开再分发。
- 本仓库只保存公开索引、读取日期、读取状态、结构化提炼、核验边界、`archive_id` 和 `evidence_sha256` 等轻量 metadata；不得提交文章全文、原图、截图包、MHTML、PDF、付费内容或大段摘录。
- 原始证据必须保存到仓库外目录，默认 `~/.skill-source-archive/`，可用 `SKILL_SOURCE_ARCHIVE_HOME` 覆盖；不得放入本仓库、单个 Skill 目录或 Codex 技能安装目录。
- 归档脚本只处理已经通过浏览器、Playwright 或人工方式取得的本地证据文件，不默认联网、不上传、不读取密钥、不扫描用户私有目录；输入文件必须由使用者或当前任务明确指定。
- 使用 `scripts/archive-source-evidence.py` 归档前后应检查输出 metadata；需要提交到仓库时，只提交 `source-map.md` 中的轻量索引和归因边界，不提交私有归档目录。

## 本地协作学习机制

Skills 可以在用户明确同意后启用本地协作学习机制，本地沉淀和复用使用者稳定的行为习惯、决策流程、判断方式、行为方式、使用背景和业务知识，用于提升使用者与技能的配合度，持续优化后续输出，并在合适场景下协助使用者改进判断、表达和设计质量。

本地协作学习机制默认关闭。只有用户明确同意启用后，才允许创建学习目录和文件；如果用户明确拒绝，不得创建目录或文件，也不得再次主动提示，除非用户主动重新提及长期学习、偏好记录或技能演进。

同意范围遵循最小授权原则：用户在某个技能场景下简单回答“可以”“同意”“开启”时，默认只开启当前技能；只有用户明确表达“所有技能”“全局开启”“以后都可以记录”等含义时，才对所有技能生效。

### 存储位置

本仓库是技能维护仓库，只保存技能定义、参考资料和学习协议，不保存使用者个人学习数据。

学习记录不得放入本仓库、单个技能目录内，也不得写入 Codex 的技能安装目录。技能同步脚本使用 `rsync --delete`，安装目录中的额外文件可能在同步时被删除。

用户同意启用后，统一使用用户目录下的持久化目录：

```text
~/.skill-learning/
  consent.md
  global.md          # 可选：仅全局授权或明确跨技能约定时创建
  skills/
    <skill-id>.md
  archive/
```

如需自定义位置，可以设置 `SKILL_LEARNING_HOME`；未设置时默认使用 `~/.skill-learning/`。下文中的 `~/.skill-learning/` 均可由 `SKILL_LEARNING_HOME` 指向的目录替代。

- `~/.skill-learning/consent.md`：本地协作学习机制的启用标记，记录用户是否同意、同意时间、适用范围、适用技能、记录交互模式、学习模型和撤销方式。
- `~/.skill-learning/global.md`：跨所有技能通用的长期约定，仅在全局授权或明确跨技能约定时创建，例如 Git 操作默认由用户执行、重要修改前先评估等。
- `~/.skill-learning/skills/<skill-id>.md`：单个技能的专属学习记录，例如领域术语、业务背景、偏好的输出方式、常用评审路径。
- `~/.skill-learning/archive/`：历史观察、废弃偏好、迁移记录或已失效约定。

### 启用与拒绝

- 默认不得读取学习内容、创建或写入 `~/.skill-learning/`；只允许读取 `consent.md` 来判断授权状态。
- 不要求用户第一次使用技能时必问；只有通过“学习时机判定算法”判断当前任务已经出现稳定偏好、团队约规、业务背景、反复决策方式等长期沉淀价值，且询问不会打断关键任务时，才可以用一句话询问是否启用本地协作学习机制。
- 首次问询应说明默认只对当前技能生效，并提示用户可以明确要求全局生效。
- 首次问询示例：`是否允许我把当前技能中的稳定偏好保存到本地 ~/.skill-learning/，用于后续优化？默认只对当前技能生效；你也可以明确要求对所有技能生效。`
- 只有用户明确同意后，才创建 `~/.skill-learning/`、`consent.md` 和对应的 `skills/<skill-id>.md`；`global.md` 仅在用户明确全局授权，或明确确认某条约定应跨技能生效时创建。
- 如果用户明确拒绝，不创建任何学习目录或文件，不把拒绝写入磁盘，并且在当前会话或当前任务链中不再主动提示。
- 由于拒绝不落盘，跨会话不能假定已永久拒绝；但如果用户未主动提及长期学习、偏好记录或技能演进，原则上不主动追问。
- 如果本地学习目录存在但缺少 `consent.md` 或启用状态不明确，视为未启用；不得读取其中内容，应先询问用户是否继续使用这些记录。
- 用户可以随时要求暂停、清理或迁移本地学习记录；执行前必须说明影响范围。

### 授权范围

- `scope: skill`：默认范围。只允许当前技能读取和写入对应的 `skills/<skill-id>.md`；跨技能通用约定只有在用户明确确认后才写入 `global.md`。
- `scope: global`：全局范围。只有用户明确授权所有技能启用本地协作学习机制时才可使用；仍然不得记录敏感信息，不得未经确认上传、提交或共享到远程。
- 当用户表达含糊，例如“可以记录一下”，按 `scope: skill` 处理；不得扩大解释为全局授权。
- 当用户在一个技能内确认的领域偏好明显适用于所有技能，也应先写入当前技能的 `Pending Observations`，再询问是否提升为全局约定。
- 判断当前技能是否启用时，必须满足 `consent.md` 标记为启用，且 `scope: global` 或 `scope: skill` 的 `skill_id` 与当前技能一致。

### 记录交互模式

本地协作学习机制启用后，记录交互模式默认采用 `mode: hybrid`。未启用前，不存在静默记录。

- `mode: silent`：静默模式。仅用于低风险、不特殊、重复出现、可撤销、不会改变未来关键决策的稳定观察；只能写入 `Pending Observations`，不得直接写入 `Confirmed Agreements` 或 `global.md`。
- `mode: explicit`：显示用户确认模式。用于可能影响后续行为、改变团队约规、跨技能复用、涉及业务/合规/金融判断、权限边界、隐私偏好、输出风格强约束、用户身份或组织背景的记录；写入前必须向用户说明要记录什么、记录到哪里、影响范围是什么。
- `mode: hybrid`：混合模式。默认模式；低风险常规观察按静默模式进入 `Pending Observations`，需要确认的观察按显示用户确认模式处理。
- 用户可以随时要求切换为静默模式、显示确认模式或混合模式；切换前应说明影响范围，切换结果写入 `consent.md`。

### 学习时机判定算法

每次协作中都可以做轻量观察，但不是每个观察都值得学习。记录前必须先完成“候选识别 -> 价值评分 -> 风险门禁 -> 动作决策”的判定；未启用本地协作学习机制时，只能用该算法判断是否适合发起启用问询，不得写入学习目录。

#### 1. 候选识别

只有出现以下候选信号时，才进入学习时机判定；普通一次性任务、临时口味、情绪表达和过程噪音不进入学习流程。

- 稳定偏好：用户反复表达或明确强调的工作方式、输出结构、审查顺序、交付边界，例如“先评估再修改”“Git 操作我来”。
- 团队约规：团队模板、命名规范、编码约规、业务评审流程、上线标准、文档约定。
- 决策方式：用户在同类问题中反复采用的取舍原则、风险偏好、证据标准、审查路径。
- 业务背景：可复用的领域术语、产品边界、资金/账户/架构上下文、组织内通用知识。
- 纠偏结果：经过讨论后用户确认采纳的改进方式、风险边界或更准确表述。
- 禁止项：用户明确要求不要做的行为、不要记录的内容、不能触碰的业务/工程红线。

#### 2. 价值评分

对候选观察进行 0-2 分评分，分数越高越适合学习。分数不足时不记录，只在当前任务中使用。

| 维度 | 0 分 | 1 分 | 2 分 |
| --- | --- | --- | --- |
| 稳定性 | 单次表达、临时要求 | 同一任务链出现 2 次或用户语气明确 | 跨任务反复出现，或用户明确声明为长期约定 |
| 复用价值 | 只影响当前输出 | 可影响同类任务 | 可长期影响该技能或跨技能协作质量 |
| 证据强度 | 只是推断 | 有明确文本证据 | 用户明确确认或多次行为一致 |
| 可执行性 | 难以转化为行为 | 可转为提醒或偏好 | 可转为明确规则、检查项或默认流程 |
| 安全性 | 含敏感/隐私/合规不确定 | 可记录但需脱敏或确认 | 低敏、可撤销、不会扩大权限 |

建议阈值：

- 总分 `0-4`：不学习。仅作为当前任务上下文使用。
- 总分 `5-7`：可作为 `Pending Observations` 候选；是否写入取决于授权状态和记录模式。
- 总分 `8-10`：具备较强长期价值；若低风险可进入 `Pending Observations`，若涉及强约束或高风险必须显示确认。

#### 3. 风险门禁

价值评分不能覆盖风险门禁。出现以下任一情况时，不得静默学习，必须显示确认；红线内容不得沉淀为可执行偏好。

- 涉及个人隐私、账号、密钥、客户敏感数据、合同条款、内部权限或不可公开组织信息。
- 涉及金融、合规、法律、安全、生产上线、资金、风控、审计或数据删除等高风险判断。
- 会改变未来默认行为，例如默认提交代码、默认跳过测试、默认采用某个架构选择、默认记录跨技能约定。
- 观察来自用户可能错误的判断、过度泛化、证据不足的结论，或与现有红线冲突。
- 需要推断用户身份、能力、组织角色、业务权限或专业程度。

风险门禁动作：

- 可纠偏：先说明问题、风险和依据，和用户讨论修正；只有用户确认后的修正结果才可作为候选观察。
- 不可记录：包含敏感信息或红线冲突时，不写入学习记录；必要时仅记录脱敏后的通用规则，并显示确认。
- 可记录但需确认：说明记录内容、位置、范围、影响和撤销方式，等待用户明确同意。

#### 4. 动作决策

结合授权状态、记录模式、价值评分和风险门禁决定动作：

| 状态 | 条件 | 动作 |
| --- | --- | --- |
| 未启用 | 总分 < 8，或当前任务仍在关键路径上 | 不询问、不记录 |
| 未启用 | 总分 ≥ 8，且不打断任务，且低敏或可明确说明 | 询问是否启用本地协作学习机制 |
| 已启用 + silent | 总分 5-10，且低风险、低敏、可撤销 | 静默写入 `Pending Observations` |
| 已启用 + hybrid | 总分 5-7，低风险 | 可静默写入 `Pending Observations` |
| 已启用 + hybrid | 总分 ≥ 8，或可能影响未来行为 | 显示确认后写入 |
| 已启用 + explicit | 总分 ≥ 5 | 显示确认后写入 |
| 任意状态 | 涉及风险门禁 | 显示确认、纠偏或不记录 |

#### 5. 时间窗口与去重

- 不在用户等待关键产物、测试运行中断、生产故障处理、紧急修复等关键路径上发起学习问询。
- 优先在阶段性总结、审查完成、用户确认“继续/按建议来/以后都这样”之后进行学习判断。
- 同一观察不要重复写入；新证据应合并到已有 `Pending Observations`，补充来源、更新时间和置信度。
- 如果观察连续多次被验证，才建议询问是否提升为 `Confirmed Agreements`。
- 如果后续行为与旧观察冲突，应降低置信度或移入 `archive/`，不得强行沿用过期偏好。

### 学习模型

本地协作学习机制启用后，学习模型默认采用 `learning_model: coach`。用户可以随时选择或切换学习模型；切换前应说明影响范围，切换结果写入 `consent.md`。

- `learning_model: capture`：记录型。偏重忠实记录用户已确认的稳定偏好、术语、业务背景和工作方式；只对明显风险或明显错误做显示提醒。
- `learning_model: coach`：协作型。默认模型。在记录用户偏好的同时，对可能存在误判、逻辑漏洞、金融/合规/安全风险、架构或设计缺陷的内容进行适度纠偏，并与用户讨论改进方式。
- `learning_model: review`：审查型。更严格地审查用户判断和设计，主动指出不一致、证据不足、风险外溢、不可验证或不符合业务/工程标准的内容；适合金融、架构评审、生产上线前检查等高风险场景。

### 专业程度感知与协作校准

协作过程中应尝试判断用户在当前讨论话题上的专业程度，用于调整解释深度、纠偏强度、提问方式和是否沉淀为学习记录。专业程度是话题相关、阶段性的判断，不得固化为用户全局标签。

- 可参考的信号：用户对术语、边界、异常场景、权衡、风险、上下游约束和验证方法的掌握程度；问题是否能明确主体、法域、技术栈、目标产物和验收标准；是否能主动指出约束、反例和取舍。
- 不得仅凭职位、语气、自信程度、表达流畅度或单次失误判断专业程度。
- 置信度不足时，应通过 1-2 个澄清问题或轻量解释校准，不应直接学习为稳定偏好，也不应强行纠偏。
- 对高专业度用户，优先用证据、边界条件、反例和风险假设讨论，不做基础概念堆叠；纠偏应聚焦逻辑漏洞、标准冲突和生产风险。
- 对低专业度或语境不明的用户，先补必要背景和判断框架，再提出纠偏建议；避免把探索性提问误判为错误判断。
- 专业程度判断本身属于敏感推断，不应静默写入 `Confirmed Agreements`；只有用户明确要求记录协作方式时，才可作为带置信度的 `Pending Observations`。

纠偏规则：

- 发现用户判断或设计可能不正确、过度泛化、证据不足、违反业务/金融/合规/安全/架构红线时，必须使用显示确认模式提示用户，不能静默纠偏或静默写入为偏好。
- 纠偏应先说明观察到的问题、可能风险和依据，再给出可选改进方式；不得把用户未经确认的错误判断沉淀为 `Confirmed Agreements`。
- 如果用户坚持保留某个有风险的偏好，只能记录为带风险说明的 `Pending Observations`，并标明需要再次确认；涉及红线时不得记录为可执行约定。
- 学习机制应学习用户的有效经验和稳定偏好，同时保留专业判断；不得把“迎合用户”作为学习目标。
- 纠偏强度必须结合当前话题专业程度和风险等级校准：高风险问题即使用户专业度高也必须提示；低风险分歧应优先讨论取舍，不把不同风格误判为错误。

静默模式适用示例：

- 用户多次要求“先审查再修改”“Git 操作由用户执行”这类低敏稳定协作偏好。
- 用户反复使用某个固定输出结构，且该结构不涉及合规、隐私或业务重大判断。
- 用户在同一技能内重复使用固定术语或模板，记录后仅用于减少后续解释成本。

显示确认模式适用示例：

- 将观察提升为 `Confirmed Agreements`。
- 写入或修改 `global.md`，或把当前技能观察提升为跨技能约定。
- 记录金融业务红线、合规判断、组织制度、客户/商户/用户相关背景、权限边界或安全偏好。
- 记录任何可能让未来输出默认采取某种强约束行为的偏好。

### 记录原则

- 只记录稳定偏好、可复用业务知识和团队约定，不记录偶发情绪或一次性任务要求。
- 仅在用户同意启用本地协作学习机制后，才可以把使用过程中的观察自动保存到本地学习目录，优先进入 `Pending Observations`。
- 未经用户明确确认，不得把 `Pending Observations` 提升为 `Confirmed Agreements`。
- 静默写入的内容必须保持可审查、可删除、可撤销；用户要求查看、暂停或清理时，应优先处理。
- `~/.skill-learning/` 是用户本地学习记录，不属于本仓库默认提交范围；提交、上传或共享到远程前必须取得用户确认。
- 记录时区分事实、偏好、规则和推断。
- 不记录个人隐私、账号、密钥、token、客户敏感数据或无法安全共享的信息。
- 学习记录必须可追溯、可修正，建议包含来源、适用范围、更新时间、置信度和是否已确认。

## 新技能接入要求

新增或重构技能时，`SKILL.md` 应加入统一入口：

```markdown
## 本地协作学习机制

本地协作学习机制遵循仓库 `AGENTS.md`；本技能不保存学习数据，学习记录只允许在用户明确同意后写入 `~/.skill-learning/` 或 `SKILL_LEARNING_HOME`。
```

技能本身只保存短指针，不复制完整学习机制。完整学习时机、授权、风险门禁、记录模式和撤销规则由仓库或项目级 `AGENTS.md` 统一维护；学习数据保存在用户目录 `~/.skill-learning/`。这样技能更新、重命名、重构或重新同步时，长期学习记录不会随技能目录被覆盖或删除。

## Git 操作约定

本仓库默认由用户执行 Git 操作。除非用户明确要求，助手只做验证、审查、确认，并告诉用户是否可以提交。

Commit message 原则上使用中文说明变更意图、范围和验证结果；`type(scope):`、模块名、类名、方法名、脚本名、工具名、协议字段和外部规范要求保持原文。
