# 代码简洁之道（Clean Code）核心实践

## 命名

- **名副其实**：名称直接表达意图，消除注释。
- **避免误导**：不使用 `accountList` 表达非 List 类型。
- **有意义的区分**：`Product` 与 `ProductInfo` 无区别，避免冗余。
- **读得出来**：使用完整单词，避免自造缩写。
- **可搜索的名称**：名称长度与其作用域大小成正比。
- **每个概念一个词**：同一抽象层次统一用词（如 `fetch`/`retrieve`/`get` 选其一）。
- **使用解决方案领域名称**：对程序员友好（如 `JobQueue`、`Parser`）。
- **添加有意义的语境**：用类、包或命名空间提供上下文。

## 函数

- **短小**：函数不超过 20 行，if/else/while 等块内只有一行调用。
- **只做一件事**：函数内所有操作处于同一抽象层级。
- **使用描述性名称**：名称表达行为，如 `isEligibleForBonus`。
- **参数越少越好**：0 个最佳，避免 3 个及以上。多个参数封装为对象。
- **无副作用**：函数名承诺什么就做什么。
- **分隔指令与询问**：`if (set("username", "张三"))` 是坏味道，应改为先 `exists` 再 `set`。
- **用异常代替错误码**：错误码导致调用者必须立即处理，异常可分离错误处理逻辑。
- **抽离 try/catch**：将 `try-catch` 块单独提取为函数。
- **DRY**：消除重复。

## 注释

- **注释不能美化糟糕的代码**：优先重写代码，而非补注释。
- **好注释**：法律信息、意图解释、警告、TODO。
- **坏注释**：多余的、误导的、注释掉的代码。
- **能用代码表达的不用注释**：如用 `assert` 或枚举替代 `// 状态只能是 1,2,3`。
- **接口**：所有的接口和公有方法都要注释，接口需要说明设计的意图、解决的问题、使用方式建议等。
- **文档注释**：数据传输模型（DTO）均需要注释。

## 格式

- **垂直格式**：文件不宜超过 500 行；相关概念就近放，变量声明在使用位置附近。
- **水平格式**：一行不超过 120 字符；缩进用 4 空格。
- **团队统一**：使用 `.editorconfig` 或 IDEA Code Style。

## 对象与数据结构

- **数据抽象**：暴露抽象接口而非内部变量（如 `double area();` 而非 `getWidth(); getHeight();`）。
- **得墨忒耳律**：只与直接朋友交谈，避免 `a.getB().getC().doSomething()`。
- **隐藏结构**：封装多级调用。
- **DTO**：纯数据载体，不含行为（贫血模型）。
- **数据结构与算法分离**：数据结构与算法应分离，算法应独立测试。

## 错误处理

- **使用异常而非返回码**。
- **断言优先**：优先使用断言（例如：AssertUtils），而不是手动抛出。
- **使用非受检异常**：受检异常破坏封装。
- **提供环境信息**：异常消息中包含操作描述和失败数据。
- **按调用者需求定义异常类**：包装第三方异常，统一抛出业务异常。
- **别返回 null**：用空集合或 `Optional` 替代。
- **别传递 null**：方法参数禁止预期为 null，用 `@NotNull` 或 `Optional` 参数。

## 单元测试

- **TDD 三定律**：先写失败的测试，只写刚好失败的测试，只写刚好让测试通过的代码。
- **测试整洁**：测试代码和生产代码同等重要，保持可读性。
- **每个测试一个断言**：一个测试方法只测试一个逻辑分支。
- **FIRST**：Fast、Independent、Repeatable、Self-Validating、Timely。
- **测试即文档**：好的测试用例展示方法的使用方式和边界行为。

## 类

- **短小**：类应小于 200 行。
- **单一职责原则（SRP）**：一个类只有一个修改理由。
- **内聚**：类的实例变量被所有方法使用。若方法只操作部分变量，应拆分类。
- **依赖抽象**：依赖接口或抽象类，不依赖具体实现（DIP）。

## 系统

- **构造与使用分离**：使用依赖注入容器（如 Spring）。
- **增量演进**：先实现今天的故事，然后重构，再扩展。
- **横切关注点**：用 AOP（如 Spring AOP、AspectJ）处理日志、事务等。
- **最简单方案**：在满足需求的前提下，选择最简洁的设计。

## 并发编程

- **分离并发代码**：将多线程逻辑与业务逻辑分离。
- **限制共享数据作用域**：用副本或独立子集避免锁竞争。
- **保持同步区域微小**：只锁必要代码。
- **测试并发**：在不同平台、不同负载下跑测试，用 JCStress 等工具。

## 日志
- 统一使用 slf4j 进行日志输出（建议结合 lombok 使用）
