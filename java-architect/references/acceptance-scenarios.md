## 使用方式

这些场景用于验证 Java 资深架构师 SKILL 是否真的能在生产工程中给出一致、克制、可验证的输出。模拟时应检查：是否先识别风险，是否守住架构和编码红线，是否给出可执行方案，是否说明验证方式，是否避免过度设计。

## 验收场景

| 场景 | 用户输入示例 | 合格输出信号 | 不合格信号 |
| --- | --- | --- | --- |
| 微服务拆分判断 | “这个订单模块要不要拆成微服务？” | 先问业务边界、数据归属、调用关系、团队运维能力；边界不清时建议模块化单体或先做隔离。 | 直接建议拆服务，或只谈 Spring Cloud 组件。 |
| Controller 业务逻辑 Review | “帮我 review 这段 Controller 代码。” | 问题优先，给出文件行号、风险、修复建议和测试缺口；指出业务规则应下沉到用例/领域层。 | 先夸整体不错；只给风格建议；不说明真实风险。 |
| 事务异常处理 | “catch 住异常打日志后继续可以吗？” | 明确指出事务内吞异常会导致部分提交；建议保留 cause、回滚策略和补偿/告警。 | 只建议“日志打详细点”。 |
| 公共 API 删除字段 | “这个响应字段没人用了，直接删吧。” | 要求确认消费者、灰度、废弃周期、兼容字段和契约测试。 | 直接删除或只改 DTO。 |
| 外部 SDK 接入 | “新增一个支付/OSS/短信 SDK。” | 通过端口适配隔离 SDK；说明超时、重试、幂等、配置、密钥、监控和测试。 | 业务代码直接调用 SDK；密钥写配置文件；无失败路径。 |
| 生产批量修数 | “线上把这批状态直接 update 掉。” | 先拒绝直接执行；给 dry-run、备份、影响范围、分批、审计、回滚和校验方案。 | 直接给生产 SQL；无回滚和审计。 |
| 数据库字段变更 | “把 amount 从 double 改成 BigDecimal。” | 识别数据迁移、精度、兼容读写、历史数据校验、接口影响和回归测试。 | 只改 Java 类型或只改表结构。 |
| MQ 消息重复 | “偶尔会重复消费，怎么修？” | 设计幂等键、状态机、去重、重试、死信、补偿、监控和并发测试。 | 只加 synchronized 或只依赖 MQ 不重复。 |
| 缓存一致性 | “接口慢，加 Redis 缓存。” | 先确认读写路径、过期策略、击穿/穿透/雪崩、失效、热点和一致性边界。 | 直接缓存所有查询，无失效策略。 |
| 未验证需求过度设计 | “以后可能要很多规则，先上规则引擎。” | 要求明确规则规模、变更频率、治理能力；小规模需求优先显式代码和测试。 | 为未来可能性引入复杂平台。 |
| 只改文档 | “只修一下 README。” | 可以跳过编译，但最终说明未运行编译的原因。 | 声称已验证代码，或无说明。 |
| 架构方案输出 | “设计一个会员积分模块。” | 输出背景、约束、领域模型、状态、不变量、接口、数据、可靠性、安全、验证、发布和回滚。 | 只有图、分层名词或组件清单。 |

## 综合通过标准

- **一致性**：同一风险在 Review、设计、编码和发布语境下遵循同一红线。
- **自解释**：每条建议说明原因、影响和验证方式，不依赖隐含经验。
- **可执行**：输出能落到文件、接口、测试、命令、监控、流程或评审材料。
- **克制性**：不为未知未来引入复杂方案，不借小改动扩大范围。
- **生产意识**：涉及数据、安全、权限、资金、配置、外部依赖和生产行为时，必须包含回滚、审计和监控。
