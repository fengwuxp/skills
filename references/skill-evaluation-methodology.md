# Skill Eval 方法论

本文定义本仓库评估 Skill 质量的可复用方法。它补充 `scripts/evaluate-skills.py` 的结构指标，不替代真实任务验证、人工评审和用户反馈。

## 使用时机

- 新建、重构或吸收外部方法后，需要判断 Skill 是否能泛化到真实用户请求。
- 修改 `SKILL.md` frontmatter、场景路由、reference 索引、脚本或 fixture 后，需要做触发和可用性回归。
- 提交或同步 Codex 前，需要说明本轮变更是否只通过结构检查，还是也通过真实 prompt 检查。

## 不适用场景

- 不用于证明某个产品、架构、支付、合规或代码结论本身正确。
- 不把一次成功演示当作稳定能力；复杂 Skill 至少要看正例、难负例和失败路径。
- 不复制外部文章原文、代码块或作者表达；只吸收可迁移的评估维度和检查方式。
- 不用模型记忆、标题或未读取正文的来源作为吸收依据。

## 读取后必须产出

- 评估目标：本次评估的是触发、输出质量、效率、稳定性还是特定脚本。
- Prompt 矩阵：每个相关 Skill 至少包含应该触发、应该让路、边缘和难负例。
- 检查方式：哪些可以脚本断言，哪些必须人工评审，哪些需要对照实验。
- 改进建议：应泛化为路由、reference、script、fixture，还是只保留为当前任务经验。

## 需要继续读取的 reference

- 新建或更新 Skill 读系统 `skill-creator`。
- 仓库级规则、安全边界和外部文章读取约规读 `AGENTS.md`。
- Prompt 样例读 `fixtures/skill-eval/prompt-cases.json`。
- 确定性结构评分读 `scripts/evaluate-skills.py`。
- Prompt fixture 完整性检查读 `scripts/audit-skill-eval-fixtures.py`。

## 评估维度

### 触发准确率

触发评估必须同时覆盖正例和负例：

- 正例要包含正式、口语、简略和未点名 Skill 的表达。
- 负例要选择“差一点会误触发”的请求，而不是明显无关的简单题。
- 与其他 Skill 有竞争关系时，负例必须写明应让路给谁或应如何处理。
- 修改 description 后，应优先回放 `fixtures/skill-eval/prompt-cases.json` 中同类样例。

### 输出质量

输出质量分为脚本可断言和人工评审两层：

- 可断言项包括文件是否存在、格式是否合法、关键字段是否齐全、脚本是否返回期望退出码。
- 人工评审项包括内容是否切题、风险是否说明、风格是否符合场景、细节是否可执行。
- 不能把可断言通过误写成业务质量通过；脚本只能证明结构完整或规则命中。

### 效率指标

效率指标用于发现 Skill 是否拖慢 Agent：

- 重复、脆弱、容易写错的动作应进入 `scripts/`。
- 大段领域知识应进入 `references/`，并提供按任务读取索引。
- 如果每次都需要扫描长文档或重复推断参数，应考虑新增 fixture、schema 或脚本入口。

### 对照实验

评估重要变更时，尽量比较新旧行为：

- 新 Skill 与无 Skill 的差异。
- 新版 `description` 与旧版 `description` 的触发差异。
- 新增脚本前后，步骤数、人工判断点和错误率是否下降。

没有对照实验时，结论只能写成“通过当前结构与 fixture 检查”，不能写成“能力已全面证明”。

### 方差检查

复杂 Skill 的同一 prompt 可能多次运行结果不同：

- 高风险或高价值 prompt 应保留多次运行记录或人工复核结论。
- 如果不同运行路径差异很大，应先检查路由、默认值、上下文泄漏和输出约束。
- 不要为了让某个样例通过而堆硬规则；应找出可迁移的缺口。

## Prompt 矩阵规则

Prompt fixture 应满足：

- 每个 Skill 至少有两个应该触发的真实请求。
- 每个 Skill 至少有两个不应该触发的难负例。
- 每个 Skill 至少有一个正例不直接点名 Skill。
- 每个负例必须说明误触发风险、推荐路由或正确处理方式。
- 样例应带上下文、目标产物、边界或风险，不使用过于显然的玩具 prompt。

## 改进归位

评估发现的问题按层归位：

- 触发太宽或太窄：优先调整 `SKILL.md` frontmatter description，并补充 prompt fixture。
- 已触发但读错资料：调整场景 routing reference，不把长规则塞回 `SKILL.md`。
- 输出结构缺字段：补充 deterministic checker、schema 或 deliverable fixture。
- 反复手写相同逻辑：沉淀到 `scripts/`。
- 属于风格、语义或业务判断：进入 reference 的清单、模板或人工评审项。
