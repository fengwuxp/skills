# 测试驱动设计与测试资产治理

测试不是编码完成后的补丁，而是架构设计、行为约束和长期演进的安全网。架构师必须把测试视为生产级资产：可读、可改、可运行、可诊断、可持续维护。

本文是测试总纲。Java/Spring 具体实践、复杂业务测试手段、资金域专项和非 Java 适配细节见 `testing-practices.md`。

## 1. 核心定位

- **测试驱动设计**：TDD 用测试先描述行为、边界和不变量，再反推接口、模型、依赖方向和实现。
- **测试即活文档**：测试应说明业务场景、输入条件、触发行为、期望输出和边界含义。
- **测试先表达业务行为**：每个测试先回答场景、输入、行为、输出和红线，不能只为覆盖代码分支。
- **测试默认使用真实代码**：除外部依赖、不可控环境、端口失败路径或明确的边界捕获外，原则上使用真实代码、真实内部组件和真实依赖装配。
- **测试保护演进**：测试套件用于支撑重构、依赖升级、模块拆分、接口调整、数据库迁移和缺陷修复。
- **测试代码等同生产代码**：测试命名、结构、数据、断言、辅助方法和测试基建都需要长期维护。
- **测试成本必须可控**：测试应快速、独立、可重复、自验证、及时，避免慢、脆、难定位。

## 2. 测试选择速查

写测试前先判断测试保护什么，再选择最小有效测试形态。

| 保护对象 | 推荐测试 | 真实链路 | 替身边界 |
| --- | --- | --- | --- |
| 纯规则、值对象、工具逻辑 | 单元测试 | 直接 new，被测逻辑真实执行 | 时间、ID、随机数、外部端口。 |
| 请求、命令、DTO、枚举、金额和字段映射 | 命令/转换测试 | 转换器、校验和默认值真实执行 | 可用 Recording 捕获输出。 |
| 用例流程、事务、状态流转、幂等 | 应用服务/用例流程测试 | 应用服务、领域服务、策略、Repository、内部编排真实装配 | 只替换第三方、MQ、Redis、远程 HTTP、时间、ID。 |
| SQL、Mapper、Repository、事务和约束 | 数据库/事务测试 | DAO/Mapper、SQL、事务真实执行 | 不 mock DAO/Mapper；必要时用 H2/Testcontainers。 |
| API、DTO、枚举、消息、SDK、错误码 | 契约测试 | 公开契约真实校验 | 不验证私有实现。 |
| 端口协议、外部失败路径、重试和幂等短路 | 端口/适配器测试 | 协议转换、错误处理、重试策略真实执行 | 可用 Fake/WireMock/Recording 替换外部系统。 |
| 依赖方向、包结构、分层规则 | 架构测试 | 静态规则真实扫描 | 不适用。 |
| 关键用户路径和跨系统链路 | E2E/验收测试 | 保留关键业务链路 | 替换不可控外部系统。 |
| 资金、权限、审计、数据一致性 | 高风险专项测试组合 | 业务事实、状态、持久化、审计链路真实断言 | 不用 mock 掩盖红线事实。 |

## 3. 任务到测试组合

不同变更需要不同测试组合，不能用单一覆盖率思维处理。

| 任务类型 | 必要测试动作 |
| --- | --- |
| 新功能 | 先写验收场景，再补领域规则、用例流程、契约和关键异常测试。 |
| Bug 修复 | 先写能复现失败的回归测试，再修复，再确认旧缺陷不会复发。 |
| 重构 | 先补 characterization/regression tests 固化现有行为，再改结构。 |
| 接口或 SDK 变更 | 补契约测试、兼容性测试和废弃/迁移路径验证。 |
| 数据库或迁移变更 | 补 SQL、Mapper、迁移、回滚、兼容字段和历史数据测试。 |
| 状态机、幂等、并发改动 | 补状态流、重复请求、并发提交、锁、唯一约束和回滚测试。 |
| 资金、权限、审计改动 | 补资金事实、权限边界、审计记录、幂等、失败回滚和追溯测试。 |

## 4. TDD 设计闭环

TDD 的价值不在“先写测试”这个动作，而在用可执行行为约束设计。

```text
业务行为/验收语义
-> 失败测试
-> 最小实现
-> 重构设计
-> 边界固化
-> 回归保护
```

实践原则：

- 先写失败测试，确认行为尚未被实现。
- 只写刚好让测试通过的实现，避免提前设计。
- 通过重构改善命名、结构、边界和依赖，不改变行为。
- 用测试表达业务不变量、异常语义、权限、幂等、状态流转和边界条件。
- 用可测试性反推架构边界：核心规则不依赖容器、数据库、远程服务和系统时间。

适用场景：

- 复杂业务规则、金额、库存、权限、状态机、幂等、一致性。
- 缺陷修复、重构前保护、核心接口和领域模型设计。

不适合机械套用的场景：

- 一次性脚本、纯配置、低风险样板代码。
- UI 视觉细节或交互体验探索阶段。
- 外部依赖行为尚不明确且缺少可控替身时。

## 5. 测试用例落地流程

写测试前先完成用例定性，不允许直接从“覆盖这个方法”开始。

每个测试先回答 5 个问题：

| 问题 | 要说明什么 |
| --- | --- |
| 场景 | 这个用例代表哪个真实业务动作、用户动作、系统事件或历史缺陷？ |
| 输入 | 前置状态、请求参数、账号、金额、币种、状态、权限、外部事件是什么？ |
| 行为 | 触发哪个服务、接口、命令、流程、编排器或端口？ |
| 输出 | 返回值、异常、状态变化、副作用、写库事实、消息、日志或审计是什么？ |
| 红线 | 哪些事情绝不能发生，例如重复入账、越权、状态倒退、半成功事实、错误通知？ |

测试设计顺序：

1. **定保护对象**：业务行为、接口契约、流程编排、持久化事务、端口协议、工具函数还是历史缺陷。
2. **定风险来源**：业务规则、模块装配、数据库、外部系统、并发、幂等、权限、安全还是数据一致性。
3. **定测试层级**：单元、命令/转换、流程、端口、数据库、契约、E2E 或专项测试。
4. **定真实链路**：确定哪些领域模型、应用服务、领域服务、Repository、转换器、策略、编排器和适配器应该真实执行。
5. **定替身边界**：只替换真正外部依赖、不可控环境、端口失败路径或明确的边界捕获。
6. **定断言事实**：断言业务事实、契约事实、状态事实、持久化事实和副作用，而不是只断言调用次数或不抛异常。
7. **定可诊断性**：测试失败时应能直接看出是输入、业务语义、装配、数据库、外部端口还是断言错误。

## 6. DDD/分层架构测试流程

在 DDD、整洁架构或六边形架构中，测试应沿业务行为从内到外逐层验证，每一层都尽量使用真实代码，只在系统边界替换外部依赖。

| 层次 | 测试目标 | 推荐方式 | 替身边界 |
| --- | --- | --- | --- |
| Domain Model / Value Object | 业务不变量、计算规则、状态合法性 | 直接 new，纯单元测试 | 通常不需要替身；时间/ID 可注入可控实现。 |
| Domain Service / Policy / Specification | 跨对象规则、策略选择、领域约束 | 直接 new 或轻量注入真实协作者 | 只替换外部端口和不可控环境。 |
| Application Service / Use Case | 用例编排、事务、权限、幂等、状态流转 | 最小上下文，注入真实内部组件 | 只替换第三方系统、远程服务、MQ、Redis、时间、ID。 |
| Repository / DAO / Mapper | 持久化事实、查询语义、事务和约束 | H2/Testcontainers 或等价数据库测试 | 不 mock DAO/Mapper。 |
| Adapter / Port | 外部协议映射、失败路径、重试、幂等短路 | Recording/Fake/WireMock/Testcontainers | 可替换外部系统，但真实执行适配逻辑。 |
| Contract | API、DTO、消息、枚举、错误码和跨模块边界 | 契约测试、schema、OpenAPI、兼容性测试 | 不验证私有实现。 |

通用判断：

- 如果测试目标是领域规则，就不要启动框架容器，也不要 mock 被测领域逻辑。
- 如果测试目标是应用流程，就应该真实装配领域服务、应用服务、转换器、策略、Repository 和内部编排组件。
- 如果测试目标是端口协议或失败路径，可以使用 Recording/Fake 捕获边界输出，但内部协议转换和错误处理要真实执行。
- 如果测试目标是数据库事务或查询语义，就不要 mock DAO/Mapper。
- 如果测试目标是跨模块契约，就不要依赖私有实现细节，只验证公开契约。

## 7. 真实代码优先与替身边界

真实代码优先不等于全量启动。正确方向是“最小真实链路 + 明确替身边界”。

- **真实代码优先**：核心业务链路、内部协作者、领域规则、转换逻辑、持久化语义和契约校验应尽量真实执行。
- **最小上下文优先**：只启动或装配测试目标需要的 Bean、模块、配置和数据库，不默认全量容器或完整应用。
- **外部边界可替换**：第三方通道、银行、MQ、Redis、远程 HTTP、对象存储、KMS、时间、随机数、ID 生成器和线程调度可以替换。
- **Recording 只捕获边界输出**：适合捕获命令、事件、端口调用或外部请求，不适合替代本层内部真实链路。
- **Mock 不验证业务正确性**：只验证 mock 被调用，不能证明业务事实、状态、副作用、持久化或外部端口输出正确。

不合理的替身：

- 流程测试 mock 掉内部策略、resolver、application service、domain service 或 Repository。
- 流程测试 mock 掉 orchestrator、posting assembler 或内部 lifecycle 后声称验证了真实流程。
- 只验证调用次数，不验证业务结果、状态、副作用、持久化事实或外部端口输出。

## 8. 测试完成定义

一个测试用例只有满足以下条件，才算真实有效：

- 能说明业务场景，而不是只覆盖某个方法或分支。
- 输入、触发行为、期望输出和红线清楚。
- 至少有一个能失败的业务断言或契约断言。
- 能验证状态、持久化、副作用、外部输出或公开契约中的关键事实。
- Mock/Fake/Recording 的边界能解释清楚，且没有替代核心业务链路。
- 失败时能定位业务语义差异、装配问题、数据库问题或外部边界问题。
- 涉及资金、权限、幂等、审计、状态机时，有对应红线断言。
- 新增缺陷或线上问题已经沉淀为回归测试。

## 9. 测试代码整洁

- 测试类、测试方法和变量名应表达业务意图，不使用空泛命名。
- 需要补充说明的测试注释写在测试方法上方，优先使用 Javadoc 或方法级单行注释；不要把场景说明塞进方法体内部。
- 优先使用 Given/When/Then 或 Arrange/Act/Assert。
- 一个测试聚焦一个行为；可以有多个必要断言，但应服务同一个场景。
- 测试数据要有语义，金额、状态、时间、权限等数据应服务测试意图。
- 测试 fixture、builder、factory 应降低理解成本，而不是隐藏关键条件。
- 避免只断言不为 null、没有异常或 mock 被调用。

推荐注释结构：

```java
/**
 * 场景：余额不足时支付失败。
 * 输入：账户余额 100.00，支付金额 150.00。
 * 输出：抛出余额不足业务异常，账户余额不变。
 * 预期：错误码为 INSUFFICIENT_BALANCE。
 */
@Test
void testPayOrderWithInsufficientBalance() {
    // given / when / then
}
```

## 10. 测试资产治理

常见坏味道：

- 慢测试过多：所有场景都启动完整上下文或真实外部依赖。
- 脆弱测试：依赖执行顺序、系统时间、随机数、环境状态或数据库残留数据。
- 断言过宽：只断言不报错、不为空，不验证业务结果。
- Mock 过度：把实现细节都 mock 掉，测试只验证 mock 行为。
- 测试数据混乱：大量复制粘贴 fixture，字段含义不明。
- 测试耦合实现：过度验证私有方法、调用次数和内部步骤。

治理原则：

- 定期清理重复测试、无效测试、长期跳过测试和随机失败测试。
- 对慢测试分层管理，区分快速单测、集成测试和 E2E 测试。
- 把公共测试数据构造沉淀为清晰的 builder 或 fixture。
- 对外部依赖建立稳定替身，如 fake、stub、WireMock、Testcontainers。
- 测试基类、测试容器、测试配置也要被视为基础设施资产维护。

## 11. Review 清单

评审测试时优先看：

- 测试是否先说明场景、输入、行为、输出和红线？
- 测试保护的是业务行为、接口契约、流程编排、持久化事务、端口协议还是工具函数？
- 测试层级是否正确，是否把命令/转换测试、流程测试、端口测试、数据库测试混在一起？
- 是否覆盖正常、异常、边界、权限、幂等、并发和兼容场景？
- 断言是否验证业务结果，而不是只验证代码跑过？
- Mock/Fake/Recording 是否只用于外部依赖、不可控环境或明确的边界捕获？
- 流程测试是否使用真实内部组件，而不是手工 new 或 mock 内部链路？
- 涉及资金时，是否断言 route、posting plan、ledger transaction、ledger entry、余额桶、幂等、回滚和可追溯关系？
- 是否存在过度 mock、慢测试、脆弱测试、重复测试数据或失败难诊断？
- 架构规则是否可以用 ArchUnit、静态扫描或脚本自动验证？

## 12. 何时读取 testing-practices.md

遇到以下任务时读取 `testing-practices.md`：

- 需要给 Java/Spring/Wind 项目写测试、改测试或评审测试。
- 需要选择 JUnit、Spring Test、MockMvc、H2、Testcontainers、WireMock、ArchUnit 等工具。
- 需要测试复杂业务规则、状态机、幂等、并发、遗留系统重构或属性型输入空间。
- 涉及资金、账本、余额、原交易回放、审计追踪等高风险业务。
- 需要把本测试原则迁移到 Go、Node.js、Python、Rust、前端或数据工程生态。
