# 测试驱动设计与测试资产治理

测试不是编码完成后的补丁，而是架构设计、行为约束和长期演进的安全网。架构师必须把测试视为生产级资产：可读、可改、可运行、可诊断、可持续维护。

本文是测试总纲。Java/Spring 具体实践、复杂业务测试手段、资金域专项和非 Java 适配细节见 `testing-practices.md`。

## 1. 核心定位

- **测试驱动设计**：TDD 先用测试描述行为、边界和不变量，再反推接口、模型、依赖方向和实现。
- **测试即活文档**：测试必须说明场景、输入、行为、输出和红线，不能只为覆盖分支。
- **真实代码优先**：除外部依赖、不可控环境、端口失败路径或边界捕获外，原则上使用真实代码和真实内部组件。
- **保护长期演进**：测试支撑重构、升级、拆分、迁移、缺陷修复和公共契约演进。
- **测试也是代码**：测试命名、结构、数据、断言、辅助方法和基建都要快速、独立、可重复、可诊断。

## 2. 测试选择速查

当用户说“写些测试”“补测试”“加测试”“这个类怎么测”“按 TDD 推进”“先写失败测试”“红绿重构”“给一组测试用例”时，先进入本节。先判断保护对象，再选择最小有效测试形态；TDD 任务先写能失败的行为测试，再最小实现、重构和补边界。

快速识别：

| 用户表达 | 优先判断 | 下一步 |
| --- | --- | --- |
| 写些测试、补测试、加测试、给一组测试用例 | 保护对象和风险来源 | 用下表选形态，再按第 5 节落地。 |
| 按 TDD 推进、先写失败测试、红绿重构 | 业务行为和验收语义 | 读第 4 节，再选择第一批失败测试。 |
| 这个类/方法怎么测 | 类型：纯规则、转换、用例、DAO、Controller、端口 | 用下表选层级；命中第 6/12 节条件时再读 `testing-practices.md`。 |
| DDD/分层架构怎么测 | 领域事实、用例流程、持久化事实、契约 | 本节选形态，再读第 6 节。 |
| 测试跑得慢、mock 太多、测试不稳定 | 上下文过重、替身越界、断言不稳 | 读第 7、10、11 节；Java/Spring 再读实践 1.5。 |

| 保护对象 | 推荐测试 | 真实链路 | 替身边界 |
| --- | --- | --- | --- |
| 纯规则、值对象、工具逻辑 | 单元测试 | 直接 new，被测逻辑真实执行 | 时间、ID、随机数、外部端口。 |
| 请求、命令、DTO、枚举、金额和字段映射 | 命令/转换测试 | 转换器、校验和默认值真实执行 | 可用 Recording 捕获输出。 |
| 用例流程、事务、状态流转、幂等 | 应用服务/用例流程测试 | 应用服务、领域服务、策略、Repository、内部编排真实装配 | 只替换第三方、MQ、Redis、远程 HTTP、时间、ID。 |
| SQL、Mapper、Repository、事务和约束 | 数据库/事务测试 | DAO/Mapper、SQL、事务真实执行 | 不 mock DAO/Mapper；必要时用 H2/Testcontainers。 |
| API、DTO、枚举、消息、SDK、错误码 | 契约测试 | 公开契约真实校验 | 不验证私有实现。 |
| 端口协议、外部失败路径、重试和幂等短路 | 端口/适配器测试 | 协议转换、错误处理、重试策略真实执行 | 可用 Fake/WireMock/Recording 替换外部系统。 |
| 依赖方向、包结构、分层规则 | 架构测试 | 静态规则真实扫描 | 不适用。 |
| 关键用户路径和跨系统链路 | E2E/验收测试 | 保留关键业务链路 | 替换不可控外部系统。 |
| 资金、权限、审计、数据一致性 | 高风险专项测试组合 | 业务事实、状态、持久化、审计链路真实断言 | 不用 mock 掩盖红线事实。 |

## 3. 任务到测试组合

不同变更需要不同测试组合，不能用单一覆盖率思维处理。

| 任务类型 | 必要测试动作 |
| --- | --- |
| 新功能 | 先写验收场景，再补领域规则、用例流程、契约和关键异常测试。 |
| Bug 修复 | 先写能复现失败的回归测试，再修复，再确认旧缺陷不会复发。 |
| 重构 | 先补 characterization/regression tests 固化现有行为，再改结构。 |
| 接口或 SDK 变更 | 补契约测试、兼容性测试和废弃/迁移路径验证。 |
| 数据库或迁移变更 | 补 SQL、Mapper、迁移、回滚、兼容字段和历史数据测试。 |
| 状态机、幂等、并发改动 | 补状态流、重复请求、并发提交、锁、唯一约束和回滚测试。 |
| 资金、权限、审计改动 | 补资金事实、权限边界、审计记录、幂等、失败回滚和追溯测试。 |

## 4. TDD 设计闭环

TDD 的价值不在“先写测试”这个动作，而在用可执行行为约束设计。这里同时覆盖设计前的“测试驱动设计”和编码阶段的 red-green-refactor，但两者不能混同：设计前先构造用户用例、测试用例、边界条件、异常路径和验收标准；编码阶段再用失败测试约束公共接口和行为。

```text
业务行为/验收语义
-> 失败测试
-> 最小实现
-> 重构设计
-> 边界固化
-> 回归保护
```

- 先从 public interface 或对外可观察行为写测试，不优先验证私有方法和实现步骤。
- 先写失败测试，确认行为尚未被实现。
- 只写刚好让测试通过的实现，避免提前设计。
- 通过重构改善命名、结构、边界和依赖，不改变行为。
- 用测试表达业务不变量、异常语义、权限、幂等、状态流转和边界条件。
- 用可测试性反推架构边界：核心规则不依赖容器、数据库、远程服务和系统时间。
- 对复杂功能按 vertical slice 推进：每一片都包含场景、契约、最小实现和验证，而不是先铺完整框架再补行为。

适用于复杂业务规则、金额、库存、权限、状态机、幂等、一致性、缺陷修复、重构前保护、核心接口和领域模型设计；一次性脚本、纯配置、低风险样板代码、UI 探索或外部依赖尚不明确时，不机械套用。

Bug 修复时，先读 `debugging-diagnosis.md` 建立稳定失败反馈环；本节负责把失败反馈环转化成合适层级的回归测试。

## 5. 测试用例落地流程

写测试前先完成用例定性，不允许直接从“覆盖这个方法”开始。

每个测试先回答 5 个问题：

| 问题 | 要说明什么 |
| --- | --- |
| 场景 | 这个用例代表哪个真实业务动作、用户动作、系统事件或历史缺陷？ |
| 输入 | 前置状态、请求参数、账号、金额、币种、状态、权限、外部事件是什么？ |
| 行为 | 触发哪个服务、接口、命令、流程、编排器或端口？ |
| 输出 | 返回值、异常、状态变化、副作用、写库事实、消息、日志或审计是什么？ |
| 红线 | 哪些事情绝不能发生，例如重复入账、越权、状态倒退、半成功事实、错误通知？ |

测试设计顺序：

1. 定保护对象：业务行为、接口契约、流程编排、持久化事务、端口协议、工具函数或历史缺陷。
2. 定风险来源：业务规则、模块装配、数据库、外部系统、并发、幂等、权限、安全或数据一致性。
3. 定测试层级：单元、命令/转换、流程、端口、数据库、契约、E2E 或专项测试。
4. 定真实链路：哪些领域模型、应用服务、Repository、转换器、策略、编排器和适配器必须真实执行。
5. 定替身边界：只替换外部依赖、不可控环境、端口失败路径或边界捕获。
6. 定断言事实：断言业务、契约、状态、持久化和副作用，不只断言调用次数或不抛异常。
7. 定可诊断性：失败时能定位输入、业务语义、装配、数据库、外部端口或断言问题。

## 6. DDD/分层架构测试流程

本节是 DDD、整洁架构、六边形架构和常见分层架构的测试路由，只判断“在哪一层测、是否需要进入测试专项实践”。只有 Domain Service / Policy / Specification、Application Service / Use Case、Repository / DAO / Mapper 这三类层次优先路由到 `testing-practices.md`。

如果被测对象属于上述三类，且依赖 Spring 装配、数据库、事务、Mapper/Repository、AOP、缓存、锁、消息、远程端口，或它是多协作者复杂类，先读 `testing-practices.md` 选择专项实践；不要把它按普通类单测处理。Domain Model / Value Object、Adapter / Port、Controller / API Contract 和 Cross-layer High Risk 按本节保护事实选择测试形态，不因“复杂”自动进入层次路由。

分层测试不是每层都写一遍，而是选择能保护关键事实的最小有效层级：

1. 先定业务事实：领域不变量、用例流程、持久化事实、端口协议、公开契约或跨层高风险链路。
2. 先测最内层有效点：领域规则能用普通单测证明时不启动 Spring；数据库事实必须落库证明时不用 mock。
3. 流程测试使用真实内部链路：真实装配领域服务、转换器、策略、Repository、内部编排和事务边界。
4. 外部边界才替换：第三方系统、远程 HTTP、MQ、Redis、时间、ID、随机数和不可控并发可替换。
5. 高风险组合验证：资金、权限、审计、状态机、幂等、并发和一致性通常需要多层组合。
6. 非 Java 项目迁移测试意图和边界判断，不强套 Spring 注解或 H2。

| 层次 | 保护事实 | 默认测试形态 | 专项实践路由 |
| --- | --- | --- | --- |
| Domain Model / Value Object | 业务不变量、金额/时间/状态计算、对象合法性 | 直接 new，纯单元测试，真实执行领域逻辑 | 不作为专项实践层次路由；输入空间复杂时在本层补充等价类、边界值和属性化测试思路。 |
| Domain Service / Policy / Specification | 跨对象规则、策略选择、领域约束、规则组合 | 简单规则可直接 new；多协作者复杂类先读专项实践 | 读 `testing-practices.md` 2.1/2.2/2.3；依赖 Spring/DB/事务时读 1.2/1.4。 |
| Application Service / Use Case | 用例编排、事务、权限、幂等、状态流转和副作用 | 默认按复杂类处理，优先进入专项实践；需要最小 Spring 上下文或等价 DI | 读 `testing-practices.md` 1.4；需要 DB/事务读 1.2；上下文过重读 1.5。 |
| Repository / DAO / Mapper | SQL、查询语义、唯一约束、事务、锁和持久化事实 | H2/Testcontainers 或等价数据库测试 | 读 `testing-practices.md` 1.2；锁/并发读 2.3。 |
| Adapter / Port | 外部协议映射、失败路径、重试、降级、幂等短路 | Fake/Recording/WireMock/Testcontainers，真实执行适配逻辑 | 不作为专项实践层次路由；按外部协议契约、失败映射、重试降级和幂等短路设计断言。 |
| Controller / API Contract | 参数绑定、校验、认证授权、DTO、错误码、兼容性 | MockMvc、契约测试、schema/OpenAPI、兼容性测试 | 不作为专项实践层次路由；按公开契约、HTTP 语义、校验、认证授权和兼容性设计断言。 |
| Cross-layer High Risk | 资金、权限、审计、账务、状态机、数据一致性和追溯 | 组合测试：领域 + 流程 + DB + 契约/端口 | 不作为单一层次路由；先拆分为领域、用例、持久化、契约和端口事实，再按对应层次组合验证。 |

升级和降级判断：

- 当被测对象属于 Domain Service / Policy / Specification、Application Service / Use Case、Repository / DAO / Mapper，且依赖数据库、事务、Mapper/Repository、AOP、Spring Bean 生命周期或多个内部协作者时，先读 `testing-practices.md`，不要停留在普通类单测判断。
- 需要验证事务、AOP、Bean 装配、状态流转或内部协作者协同时，升级为 Application Service / Use Case 流程测试。
- 需要证明 SQL、约束、事务、锁或查询语义时，升级为数据库/事务测试。
- 能用领域单测证明的，不升级为 Spring 流程测试。
- 只验证端口命令、外部请求或失败映射时，可以使用 Recording/Fake，但不得声称验证完整业务流程。
- 流程测试 mock 掉内部 resolver、orchestrator、assembler、domain service 或 Repository，只能算边界/命令测试。
- 非 Java 项目按同样层次选择本地生态工具，再读 `testing-practices.md` 第 4 节。

## 7. 真实代码优先与替身边界

真实代码优先不等于全量启动。正确方向是“最小真实链路 + 明确替身边界”。

- **真实代码优先**：核心业务链路、内部协作者、领域规则、转换逻辑、持久化语义和契约校验应尽量真实执行。
- **最小上下文优先**：只启动或装配测试目标需要的 Bean、模块、配置和数据库，不默认全量容器或完整应用。
- **外部边界可替换**：第三方通道、银行、MQ、Redis、远程 HTTP、对象存储、KMS、时间、随机数、ID 生成器和线程调度可以替换。
- **Recording 只捕获边界输出**：适合捕获命令、事件、端口调用或外部请求，不适合替代本层内部真实链路。
- **Mock 不验证业务正确性**：只验证 mock 被调用，不能证明业务事实、状态、副作用、持久化或外部端口输出正确。

不合理的替身：

- 流程测试 mock 掉内部策略、resolver、application service、domain service 或 Repository。
- 流程测试 mock 掉 orchestrator、posting assembler 或内部 lifecycle 后声称验证了真实流程。
- 只验证调用次数，不验证业务结果、状态、副作用、持久化事实或外部端口输出。

## 8. 测试完成定义

一个测试用例只有满足以下条件，才算真实有效：

- 能说明业务场景，而不是只覆盖某个方法或分支。
- 输入、触发行为、期望输出和红线清楚。
- 至少有一个能失败的业务断言或契约断言。
- 能验证状态、持久化、副作用、外部输出或公开契约中的关键事实。
- Mock/Fake/Recording 的边界能解释清楚，且没有替代核心业务链路。
- 失败时能定位业务语义、装配、数据库或外部边界问题。
- 资金、权限、幂等、审计、状态机有对应红线断言。
- 新增缺陷或线上问题已经沉淀为回归测试。

## 9. 测试代码整洁

- 测试类、测试方法和变量名应表达业务意图，不使用空泛命名。
- 需要补充说明的测试注释写在测试方法上方，优先使用 Javadoc 或方法级单行注释；不要把场景说明塞进方法体内部。
- 优先使用 Given/When/Then 或 Arrange/Act/Assert。
- 一个测试聚焦一个行为；可以有多个必要断言，但应服务同一个场景。
- 测试数据要有语义，金额、状态、时间、权限等数据应服务测试意图。
- 测试 fixture、builder、factory 应降低理解成本，而不是隐藏关键条件。
- 避免只断言不为 null、没有异常或 mock 被调用。

推荐注释结构：

```java
/**
 * 场景：余额不足时支付失败。
 * 输入：账户余额 100.00，支付金额 150.00。
 * 输出：抛出余额不足业务异常，账户余额不变。
 * 预期：错误码为 INSUFFICIENT_BALANCE。
 */
@Test
void testPayOrderWithInsufficientBalance() {
    // given / when / then
}
```

## 10. 测试资产治理

- 慢测试过多：所有场景都启动完整上下文或真实外部依赖。
- 脆弱测试：依赖执行顺序、系统时间、随机数、环境状态或数据库残留数据。
- 断言过宽：只断言不报错、不为空，不验证业务结果。
- Mock 过度：把实现细节都 mock 掉，测试只验证 mock 行为。
- 测试数据混乱：大量复制粘贴 fixture，字段含义不明。
- 测试耦合实现：过度验证私有方法、调用次数和内部步骤。

- 定期清理重复测试、无效测试、长期跳过测试和随机失败测试。
- 对慢测试分层管理，区分快速单测、集成测试和 E2E 测试。
- 把公共测试数据构造沉淀为清晰的 builder 或 fixture。
- 对外部依赖建立稳定替身，如 fake、stub、WireMock、Testcontainers。
- 测试基类、测试容器、测试配置也要被视为基础设施资产维护。

## 11. Review 清单

评审测试时优先看：

- 测试是否先说明场景、输入、行为、输出和红线？
- 测试保护的是业务行为、接口契约、流程编排、持久化事务、端口协议还是工具函数？
- 测试层级是否正确，是否把命令/转换测试、流程测试、端口测试、数据库测试混在一起？
- 是否覆盖正常、异常、边界、权限、幂等、并发和兼容场景？
- 断言是否验证业务结果，而不是只验证代码跑过？
- Mock/Fake/Recording 是否只用于外部依赖、不可控环境或明确的边界捕获？
- 流程测试是否使用真实内部组件，而不是手工 new 或 mock 内部链路？
- 涉及资金时，是否断言 route、posting plan、ledger transaction、ledger entry、余额桶、幂等、回滚和可追溯关系？
- 是否存在过度 mock、慢测试、脆弱测试、重复测试数据或失败难诊断？
- 架构规则是否可以用 ArchUnit、静态扫描或脚本自动验证？

## 12. 何时读取 testing-practices.md

遇到以下任务时读取 `testing-practices.md`：

- 需要给 Java/Spring/Wind 项目中依赖 Spring 装配、数据库、Controller/HTTP、外部端口或测试基础设施的对象写测试、改测试或评审测试。
- 需要为 Domain Service / Policy / Specification、Application Service / Use Case、Repository / DAO / Mapper 选择专项测试实践、拆分测试任务或制定测试计划。
- 需要选择 JUnit、Spring Test、MockMvc、H2、Testcontainers、WireMock、ArchUnit 等工具。
- 需要测试复杂业务规则、状态机、幂等、并发、遗留系统重构或属性型输入空间。
- 涉及资金、账本、余额、原交易回放、审计追踪等高风险业务。
- 需要把本测试原则迁移到 Go、Node.js、Python、Rust、前端或数据工程生态。
