# ADR 与技术取舍

本文用于记录架构决策、技术选型、重大重构、服务拆分、数据模型调整、公共契约变更和生产高风险方案。ADR 不是形式文档，而是让团队知道为什么选择、放弃了什么、未来何时复审。

## 使用时机

以下情况建议或必须产出 ADR 风格结论：

- 数据库、缓存、MQ、搜索、网关、框架、云服务、SDK 等关键技术选型。
- 单体、模块化单体、微服务、事件驱动、CQRS、规则引擎等架构形态选择。
- 公共 API、消息体、数据模型、错误码、权限模型或状态机发生重要变化。
- 方案有多个可行选项，且会影响长期维护、成本、性能、安全或团队协作。
- 决策不可逆、回滚成本高，或需要跨团队达成共识。

## ADR 模板

```text
标题：一句话说明决策。
状态：提议中 / 已接受 / 已废弃 / 已替代。
日期：YYYY-MM-DD。
背景：业务目标、技术约束、问题现状和非目标。
决策：选择什么方案，不选择什么方案。
备选方案：
- 方案 A：优点、缺点、风险、成本。
- 方案 B：优点、缺点、风险、成本。
- 方案 C：优点、缺点、风险、成本。
选择理由：为什么当前阶段它最合适。
放弃理由：为什么其他方案暂不采用。
影响范围：模块、接口、数据、部署、运维、团队和下游影响。
复杂度位置：新增、删除或隐藏了哪些节点和通信边；复杂度从哪里转移到哪里；关键状态传播、调试入口和观测方式是什么。
风险与缓解：兼容性、性能、安全、数据、稳定性、成本和组织风险。
验证方式：PoC、测试、压测、灰度、监控、回滚演练或验收标准。
回滚/复审：什么时候回滚，什么时候复审，触发条件是什么。
```

## 探索期多框架分析

当架构问题还没有收敛，且过早推荐单一方案会锁死后续设计时，先做多框架分析，再进入 ADR 结论。适用场景包括：服务拆分与模块化单体、同步与异步、强一致与最终一致、自建与采购、平台化与场景化、立即重构与渐进治理、单库与分库分表等高影响取舍。

多框架分析要求保留 3-5 个真正不同的判断框架。每个框架至少说明：

- 它关注的工程问题是什么。
- 默认假设是什么。
- 能解释哪些当前现象、约束或目标。
- 容易忽略哪些成本、风险或组织能力。
- 与其他框架冲突在哪里。
- 什么条件下该框架失效，需要换路或复审。

框架差异必须真实存在，至少体现在关注对象、评估标准、时间尺度、系统边界或风险偏好中的两个维度。不要把同一个结论包装成多个框架，也不要在分析阶段偷偷把一个框架写成唯一正确答案。

探索期可以先输出问题地图、冲突轴和待验证信息；一旦用户要求决策、技术方案或 ADR，必须收敛到备选方案、选择理由、放弃理由、风险缓解、验证方式和复审条件。

## 取舍维度

| 维度 | 判断问题 |
| --- | --- |
| 业务匹配 | 是否直接服务当前业务目标？是否为不确定未来过度设计？ |
| 边界清晰 | 是否让业务、模块、数据、安全和运维边界更清楚？ |
| 复杂度 | 新增复杂度是否小于它解决的问题？复杂度是被删除、被暴露，还是转移到网络、配置、框架黑箱、隐式状态或运维排障？团队是否能长期维护？ |
| 可验证 | 是否能通过测试、压测、监控、灰度或运行指标验证？ |
| 兼容性 | 是否影响 API、DTO、消息、数据库、配置或外部依赖？ |
| 可回滚 | 失败后能否代码回滚、配置回滚、数据回滚或前滚修复？ |
| 安全合规 | 是否引入权限、租户、敏感数据、密钥、审计或合规风险？ |
| 成本 | 研发、运维、资源、许可证、迁移、培训和机会成本是否可接受？ |

## 常见决策建议

- **微服务拆分**：只有业务边界、数据归属、团队运维能力、故障隔离、发布节奏和服务间通信边清楚时才拆；否则优先模块化单体和边界治理。
- **规则引擎**：只有规则规模、变更频率、治理流程、测试方式和权限审计明确时才引入；小规模规则优先显式代码和测试。
- **新中间件/SDK**：必须说明必要性、替代方案、维护责任、许可证、安全风险和退出策略；如果它隐藏调用、上下文、状态传播或重试链路，还必须说明调试入口、观测信号和失败时的人工接管方式。
- **数据库变更**：优先向前兼容，采用先加后迁、双写/回填、验证后下线旧字段，避免一次性破坏。
- **缓存方案**：必须说明读写路径、失效策略、一致性窗口、热点、穿透、击穿、雪崩和回源保护。
- **异步化**：不能只为削峰；必须设计幂等、顺序、补偿、死信、重放、积压监控和人工兜底。
- **一致性方案**：涉及跨服务、跨库、MQ 或外部依赖时，必须比较本地事务、Outbox、Saga、TCC、事务消息、对账补偿等方案。
- **遗留系统迁移**：必须比较原地改造、防腐层、绞杀者模式、双写/回填/切流等路径，并说明旧能力下线条件。
- **安全方案**：认证、授权、租户隔离、密钥和敏感数据方案必须说明威胁、控制点、审计、测试和残余风险。

## 复审触发条件

- 业务规模、团队规模、流量、数据量或合规要求发生显著变化。
- 原决策依赖的外部服务、开源项目、云产品或许可证发生重大变化。
- 运行指标显示当前方案成为性能、成本、稳定性或交付瓶颈。
- 出现多次因同一架构边界导致的线上事故、重复缺陷或协作冲突。
- 出现多次因隐藏通信边、隐式状态传播或框架黑箱导致的排障困难、误判或回滚失败。
- 新方案的迁移成本显著下降，或原方案维护成本持续上升。

## 红线

1. 不得只给“推荐 X”而不说明备选方案和放弃理由。
2. 不得把流行度、个人偏好或团队熟悉度包装成唯一依据。
3. 不得忽略迁移成本、回滚条件、验证方式和长期维护责任。
4. 不得用 ADR 为既定结论背书；如果结论已定，必须如实说明约束来源。
5. 不得为未知未来引入平台化、插件化、规则引擎、微服务或复杂扩展点。
