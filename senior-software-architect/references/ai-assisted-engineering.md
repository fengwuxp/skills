# AI 编码协作与三层作业模型

本文定义资深架构师在 AI 时代的编码协作方式。AI 可以提升产出速度，但不能替代规格、纪律、验证和工程判断。架构师必须把 AI 编码纳入可解释、可审查、可测试、可回滚的工程系统中。

## 1. 核心定位

AI 编码不是“让工具直接写代码”，而是把需求、规范、测试、实现、审查、重构和交付串成受控闭环：

```text
OpenSpec 定标准，Superpowers 保纪律，Harness 管团队。
```

三层模型：

| 层级 | 名称 | 解决问题 | 类比 | 核心产物 |
| --- | --- | --- | --- | --- |
| 规范层 | OpenSpec | 做什么、做到什么程度 | 施工图纸 | 需求、接口、领域模型、验收标准、非功能约束、边界条件 |
| 纪律层 | Superpowers | 怎么做、如何保证质量 | 施工规范手册 | TDD、Review、Refactor、编码红线、测试门禁、AI 输出审查 |
| 协作层 | Harness | 谁来做、如何协同和集成 | 项目经理 + 工地管理 | 任务拆分、角色分工、上下文包、执行顺序、交接、验证矩阵 |

当用户希望进入受控自治开发时，应在三层模型外增加一层执行授权：

```text
OpenSpec 定目标
-> Harness 排计划
-> Superpowers 用 TDD 推进
-> Execution Grant 控制可自动执行的权限边界
```

## 2. 适用场景

- 使用 AI 进行代码生成、Bug 修复、重构、测试补充、迁移改造或文档到代码转换。
- 使用多个 AI Agent、子任务、人员或工作流并行完成同一个工程目标。
- 需求、接口、数据、权限、资金、状态机、审计、兼容性或生产行为存在风险，需要先把标准和验证方式讲清楚。
- 需要审查 AI 生成代码是否符合架构红线、编码约规、测试要求和交付标准。

不适合重流程化的场景：

- 低风险的一行文案、注释、格式修正或局部配置说明。
- 探索阶段的临时方案讨论，尚未进入实现承诺。
- 用户明确只要求概念解释、思路评估或只读审查。

## 3. OpenSpec：规范层

OpenSpec 负责把“用户想法”转为“可执行、可验证、可审查”的工程标准。它不追求长文档，而追求边界清楚。

最小结构：

```text
目标：要解决什么问题。
范围：本次包含哪些修改。
非目标：明确不做什么。
业务规则：核心行为、不变量、状态流转。
接口契约：入参、出参、异常、空值、分页、幂等、兼容性。
数据约束：金额、时间、ID、权限、租户、审计、敏感字段。
验收场景：正常、异常、边界、权限、幂等、并发、兼容。
技术约束：项目现有模式、依赖边界、编码约规、禁止行为。
验证方式：测试、编译、lint、静态检查、Review、人工验收。
```

OpenSpec 红线：

- 没有明确验收标准，不进入复杂编码。
- 涉及公共 API、DTO、枚举、错误码、配置项、消息体或数据库字段时，必须说明兼容策略、迁移路径和废弃周期。
- 涉及金额、权限、租户、审计、状态机、生产数据或安全边界时，必须说明业务不变量、失败路径和风险控制。
- 不允许把用户一句需求直接等同于可执行规格；不清楚时最多给 3 个澄清问题、选项或假设。
- 不允许把 OpenSpec 写成百科说明；它必须服务当前交付和验证。

## 4. Superpowers：纪律层

Superpowers 负责让 AI 编码服从工程纪律，核心是 TDD、Review、Refactor 和验证闭环。

标准闭环：

```text
场景先行
-> 失败测试或验收用例
-> 最小实现
-> 代码审查
-> 行为保持下重构
-> 再验证
```

纪律要求：

- 先用场景、测试或验收语义约束行为，再实现代码。
- 只写刚好满足当前 OpenSpec 的实现，不追加未来扩展。
- 重构必须发生在测试保护下，且不改变外部可观察行为。
- AI 生成代码必须接受架构边界、编码约规、安全、测试和交付红线审查。
- 重要规则必须有测试、静态检查、Review 清单或运行指标支撑。

AI 编码审查重点：

| 检查项 | 对应资深架构师约规 |
| --- | --- |
| 是否越过分层边界 | 核心业务不依赖 Controller、Mapper、ORM Entity、SDK、MQ、缓存或 HTTP Client |
| 是否破坏公共契约 | API、DTO、枚举、错误码、配置、消息体、数据库字段必须有兼容策略 |
| 是否错误处理空值 | 内部契约用 JSpecify，API 数据模型用 Bean Validation，业务断言用 AssertUtils |
| 是否吞异常或丢上下文 | 禁止空 catch、只打印日志后继续、事务异常不回滚、包装异常丢 cause |
| 是否泄露敏感信息 | 日志、异常、测试数据不得输出密码、token、密钥、身份证、银行卡、手机号等明文 |
| 是否机械复用 | 不为 DTO 构建、一行代码或偶发重复提取公有方法或工具类 |
| 是否参数过多 | 公有方法超过 5 个参数必须先和用户确认 |
| 是否过度设计 | 不为未知未来添加通用平台、规则引擎、插件体系、多租户、中台或复杂扩展点 |
| 是否缺少验证 | 未说明编译、测试、静态检查或无法执行原因，不得交付 |

Superpowers 红线：

- 不允许 AI 为了“更优雅”扩大重构范围。
- 不允许 AI 擅自删除测试、放宽断言、跳过失败用例或伪造验证结果。
- 不允许 AI 编造不存在的工具类、框架 API、项目约定或外部能力。
- 不允许 AI 引入新依赖、starter、中间件、代码生成器或运行时代理而不说明必要性、替代方案、版本风险和维护责任。
- 不允许把 TODO、伪代码、未验证半成品包装成完成代码。
- 不允许用“重构”掩盖行为变化。

## 5. Harness：协作层

Harness 负责让人、AI、子任务和工具协作可控。它不是组织审批，而是工程执行编排。

最小 Harness Plan：

```text
任务：实现、修复、重构或验证什么。
Owner：用户、助手、子任务 Agent 或团队角色。
写入范围：允许修改哪些文件、模块或目录。
只读范围：只允许参考哪些文件、规范或外部资料。
依赖关系：哪些任务必须先完成，哪些可以并行。
禁止事项：不得修改哪些模块、配置、契约或生产行为。
验证命令：完成后运行哪些编译、测试、lint 或检查。
交接说明：产物、验证结果、残余风险、待确认点。
```

协作模式：

- 单 Agent：适合小范围修改、局部 Review、测试补充。
- 多 Agent：适合互不重叠的模块、文档审查、验证任务或并行探索。
- 人机结对：适合高风险设计、核心业务规则、生产变更、复杂重构。
- Gantt / Kanban / Handoff：适合周期较长或跨人协作的任务计划、进度可视化和交接记录。

Harness 红线：

- 不允许多个 AI 或多人无边界地同时修改同一职责范围。
- 不允许把任务拆给 AI，却不给 OpenSpec、项目规范、上下文、测试和验收标准。
- 不允许子任务只返回“完成了”；必须返回修改内容、验证结果、风险和待确认点。
- 不允许协作层绕过用户确认执行 Git 写操作、部署、外部 API 调用或生产操作。
- 高风险任务必须设置人工确认点。

## 6. 风险分级与流程强度

| 风险级别 | 场景 | 流程要求 |
| --- | --- | --- |
| 低风险 | 文档、注释、局部样板、明确的小修 | 轻量 OpenSpec，可直接执行并说明验证情况 |
| 中风险 | 普通功能、Bug 修复、测试补充、局部重构 | OpenSpec + 相关测试 + 最小修改 + Review |
| 高风险 | 公共契约、数据模型、权限、资金、生产行为、依赖升级、跨模块重构 | OpenSpec 先确认 + Harness Plan + TDD/验收测试 + 验证矩阵 + 残余风险 |

高风险 AI 编码流程：

```text
识别任务和风险
-> 形成 OpenSpec 并确认关键假设
-> 提取验收场景和测试用例
-> 制定 Harness Plan
-> 小步实现
-> Superpowers 审查与重构
-> 编译、测试、静态检查
-> 输出交付结论和残余风险
```

## 7. 受控自治开发模式（CAD Mode）

受控自治开发模式（Controlled Autonomous Development Mode，简称 CAD Mode）用于中高复杂度编码任务：用户希望在明确规格、计划和授权边界下，自动分轮完成测试、实现、重构、验证和提交，直到目标完成、用户中断、发生严重错误或出现需要用户判断的不确定点。

该模式是默认 Git 约定的显式例外。只有用户明确表达“开启 CAD Mode”“开启受控自治开发模式”“允许自动提交”“按轮自动推进”等含义，并确认 Execution Grant 后，才允许自动 `git add` / `git commit`。未显式开启时，仍遵循 `workflow.md`：Git 写操作默认由用户执行。

启动条件：

- 用户明确授权自动推进和每轮提交。
- 已有需求、OpenSpec、系分、Issue、PR 描述或可校准的目标。
- 已识别写入范围、禁止修改范围、验证命令、提交粒度和停止条件。
- 已检查工作区状态，能区分用户已有改动和本次自动推进改动。
- 涉及公共契约、资金、权限、生产数据、数据库迁移或外部接口时，已设置人工确认点。

Round 0 必须先完成规格与计划校准：

```text
目标：本次自动推进要完成什么。
范围：允许修改哪些模块、文件或接口。
非目标：明确不做什么。
验收场景：正常、异常、边界、兼容、权限、幂等。
测试策略：先补哪些失败测试，跑哪些验证命令。
风险点：哪些地方必须停下确认。
提交策略：每轮提交粒度、提交信息格式、是否允许自动 commit。
```

每一轮按固定闭环推进：

```text
Pick：选择本轮最小目标
-> Red：先写失败测试、验收用例，或说明不能 TDD 的理由
-> Green：做刚好通过测试的最小实现
-> Review：检查架构边界、编码红线、测试质量和无关修改
-> Refactor：在测试保护下做必要重构
-> Verify：运行本轮相关验证命令
-> Commit：仅在显式授权下提交本轮变更
-> Pause：每轮结束默认等待 5 秒，给用户人工中断窗口
-> Next：无阻塞则进入下一轮
```

若用户在 5 秒中断窗口内提出暂停、修改方向、调整授权、补充约束或要求人工审查，必须停止自动推进并处理用户输入；若没有人工打断，且未触发停止条件，则自动进入下一轮。严重错误优先于 5 秒中断窗口：一旦发生，立即停止自动推进，不得自动提交或进入下一轮。

每轮自动提交前必须检查：

- `git status` 确认只提交本轮相关文件。
- `git diff` 确认没有无关格式化、误删、重排或越界修改。
- 相关测试、编译、lint 或替代验证已经执行并记录结果。
- 红线扫描未发现密钥、裸日志、敏感信息、无主依赖或公共契约破坏。
- 提交信息包含轮次、目标、验证结果和 AI 辅助标识。

自动停止条件：

- 发生 P0/P1 级严重错误，例如可能导致数据破坏、资金损失、权限绕过、敏感信息泄露、生产不可用、仓库状态不可控或不可逆操作。
- 验证命令出现无法解释的严重失败，且继续修改可能掩盖真实问题、扩大影响或破坏测试可信度。
- OpenSpec 与现有代码、测试或业务规则冲突。
- 公共 API、DTO、枚举、错误码、数据库结构需要破坏性变更。
- 公有方法可能超过 5 个参数，或需要引入新依赖。
- 涉及资金、权限、租户、审计、生产数据、密钥、外部接口或真实支付/资金通道。
- 测试失败原因不明确，继续修改可能扩大影响。
- 工作区混有用户改动，无法干净区分本轮变更。
- 需要选择业务口径、产品规则、兼容策略、迁移路径或风险取舍。

结束时必须做双重收口：

- **资深架构师审查**：确认是否符合 OpenSpec、架构红线、编码约规、测试要求、验证要求和交付边界。
- **支付产品与资金系统专家审查**：仅当任务涉及支付、账户、资金、清结算、对账、VCC、ACH、卡、风控、争议或金融数据时执行；确认资金流、账务口径、合规边界、审计和异常路径。若不涉及支付/资金，应明确“不适用”，只做金融红线空检。

## 8. Execution Grant：执行授权包

Execution Grant 是 CAD Mode 的权限边界。它只能约束当前任务链，任务完成、用户暂停或授权被撤销后失效；不得被理解为永久授权或跨任务授权。

最小授权结构：

```text
适用任务：本次 OpenSpec / Harness Plan。
有效期限：当前任务链。
写入范围：允许修改的目录、模块、文件类型。
验证范围：允许自动运行的测试、编译、lint、codegen。
Git 范围：是否允许自动 add/commit，是否允许创建分支。
清理范围：允许删除的临时目录、构建产物、生成产物。
外部访问：是否允许查官方文档、下载依赖、调用本地服务。
停止条件：哪些情况必须回到用户确认。
```

默认可纳入授权包的低风险操作：

- 读取仓库代码、配置、测试和文档。
- 修改 Harness Plan 中声明的写入范围。
- 新增或修改测试用例。
- 运行项目本地测试、编译、lint、格式化和代码生成。
- 清理本轮生成的临时文件、构建产物和测试输出目录。
- 查看 `git status`、`git diff`、`git log`。
- 在每轮验证通过后，按授权自动 `git add` 本轮相关文件并 `git commit`。

必须在授权包中显式列出的操作：

- 创建或切换工作分支。
- 安装依赖、更新 lock 文件或下载外部依赖。
- 启动 Docker、数据库、消息队列或本地服务。
- 访问公网文档或外部 API。
- 运行耗时较长的全量测试。
- 修改公共 API、DTO、枚举、错误码、数据库迁移脚本。
- 修改跨模块架构、权限、安全、资金相关逻辑。
- 写入仓库外目录。

即使已进入 CAD Mode，也不得静默执行：

- `git push`、创建 PR、merge、rebase、reset hard 或强制覆盖历史。
- 删除用户未授权目录或不属于本轮的文件。
- 操作生产环境、生产数据库、真实外部 API 或真实支付/资金通道。
- 读取、写入或提交密钥、token、生产配置、客户敏感数据。
- 引入新中间件、新框架、新运行时代理或大版本依赖升级。
- 改变资金、权限、审计、合规、风控、结算等业务规则口径。

平台权限边界优先于 Execution Grant。若 Codex/工具层要求额外授权，应按工具规则请求；不得通过脚本绕过授权、沙箱或安全限制。

## 9. 与现有规范的关系

- `coding-standards.md` 是 Java/Spring 编码行为的主规范；AI 生成 Java 代码必须遵守。
- `testing.md` 是 TDD、测试分层、测试代码整洁和测试资产治理的主规范；AI 编码必须把测试作为安全网。
- `workflow.md` 是修改前后验证、PR、Git 和交付要求的主规范；AI 编码不得降低验证标准。
- `negative-constraints.md` 是禁止行为和权限边界的主规范；AI 协作不得扩大权限。
- `scenario-routing.md` 负责判断什么时候加载本文，以及是否需要补充安全、生产、分布式一致性、演进式架构等专项规范。

## 10. Review 清单

审查 AI 编码产物时至少检查：

- OpenSpec 是否明确目标、范围、非目标、验收和验证方式。
- 修改范围是否符合 Harness Plan，有无无关重构、格式化或依赖变更。
- 若启用 CAD Mode，Execution Grant 是否明确且本轮操作未越权。
- 代码是否符合现有架构边界和项目风格。
- 业务规则、权限、租户、幂等、金额、状态机和审计是否被测试覆盖。
- API、DTO、错误码、配置、消息体和数据库字段是否保持兼容或给出迁移策略。
- 异常、日志、敏感信息、事务和外部调用是否符合红线。
- 重构是否行为保持，有测试保护。
- 验证命令是否真实运行，失败或无法运行是否说明原因和替代验证。
