# 安全架构专项

本文用于身份认证、授权、租户隔离、敏感数据、密钥、审计、供应链安全和常见 Web/API 风险的设计与评审。安全不是上线前扫描，而是需求、模型、接口、数据、运行和运维全过程的边界约束。

## 使用时机

- 涉及登录、认证、授权、角色、租户、组织、审批、后台管理或开放 API。
- 涉及个人信息、资金数据、商业敏感数据、密钥、证书、token、文件上传或导出。
- 涉及第三方 SDK、外部回调、Webhook、插件、脚本、表达式、反序列化或动态执行。
- 涉及越权、数据泄露、审计、合规、供应链漏洞或生产高危操作。

## 不适用场景

- 普通代码可读性、命名、分层和测试问题优先读 `coding-review-deep-dive.md` 或 `testing.md`。
- 生产发布和数据变更的完整上线方案还需要读 `production-readiness.md`。
- 本文不替代安全团队、合规、法务或持牌机构的最终确认。

## 读取后必须产出

- 资产、主体、边界、威胁、控制落地、测试验收和剩余风险。
- 涉及 Spring Boot 时追加 Spring Security、数据权限、CSRF/CORS、代理头、限流和错误响应检查。
- 对高危操作必须说明审计链路、权限边界、敏感数据保护和人工确认点。

## 安全设计顺序

1. **资产识别**：保护什么数据、能力、密钥、配置、接口和操作。
2. **主体识别**：用户、商户、员工、系统、第三方、任务、AI 代理分别是谁。
3. **边界识别**：公网、内网、租户、组织、角色、数据域、服务和运维边界。
4. **威胁建模**：越权、伪造、篡改、重放、泄露、注入、滥用、供应链和配置错误。
5. **控制落地**：认证、授权、校验、加密、脱敏、审计、限流、告警和测试。

## 认证与授权

- 认证回答“是谁”，授权回答“能做什么、能访问哪些数据、能操作到什么粒度”。
- 权限模型应明确 RBAC、ABAC、ACL、数据权限、字段权限和操作权限的适用范围。
- 后台高危操作必须具备二次确认、审批、MFA 或等价保护，并记录审计。
- 服务间调用必须明确身份来源、凭证生命周期、最小权限和撤销机制。
- 不得只依赖前端隐藏按钮、Header 标识或客户端参数作为权限判断。

## 租户与数据隔离

- 租户 ID、组织 ID、商户 ID、用户 ID 等归属字段必须在查询、写入、导出、缓存和消息中贯穿。
- 跨租户访问必须有明确授权场景、审计和测试覆盖。
- 缓存 Key、消息 Key、对象存储路径、搜索索引和报表任务都要考虑租户隔离。
- 管理后台、客服工具、运营导出和批处理脚本是越权高发区，必须单独评审。

## 敏感数据与密钥

- 敏感数据采集遵循最小化原则；不采集当前业务不需要的数据。
- 存储时区分哈希、加密、脱敏和掩码，不能用一种处理方式覆盖所有场景。
- 日志、异常、测试数据、埋点、导出文件和告警通知不得输出敏感信息原文。
- 密钥、token、证书、生产地址和个人凭证不得进入代码库、镜像、测试数据或日志。
- 密钥需要轮换、过期、访问控制、审计和泄露后的吊销方案。

## 常见风险检查

| 风险 | 设计检查 |
| --- | --- |
| SQL/NoSQL 注入 | 参数化查询、白名单排序字段、禁止拼接未校验表达式。 |
| XSS | 输出编码、富文本白名单、CSP、避免把用户输入直接渲染为 HTML。 |
| CSRF | SameSite、CSRF Token、重要操作二次确认。 |
| SSRF | URL 白名单、内网地址拦截、协议限制、重定向控制。 |
| 反序列化 | 禁止反序列化不可信输入，限制类型白名单，避免危险 gadget。 |
| 文件上传 | 类型校验、大小限制、病毒扫描、隔离存储、随机文件名和访问控制。 |
| Webhook/回调 | 签名校验、时间戳、防重放、幂等、来源限制和告警。 |
| 供应链 | 依赖来源、许可证、漏洞扫描、锁定版本和升级责任人。 |

## Spring Boot 安全落地检查

识别到 Spring Boot、Spring Security、Servlet/WebFlux、OAuth2 Resource Server、网关或后台管理系统时，在通用安全检查后追加 Spring 专项检查。专项检查只做落地约束，不替代项目已有安全框架。

| 检查项 | 要点 |
| --- | --- |
| SecurityFilterChain | 明确哪些路径公开、哪些路径认证、哪些路径需要更高权限；默认拒绝未知路径。 |
| 认证方式 | JWT、Opaque Token、Session Cookie、API Key、mTLS 或服务间凭证必须说明来源、过期、撤销和刷新策略。 |
| 方法级授权 | 关键用例、后台高危操作和数据导出优先用服务层或方法级授权兜底，不只依赖 URL 匹配。 |
| 数据权限 | Controller 入参中的租户、组织、商户、用户 ID 不可信，必须在用例或查询层校验归属。 |
| CSRF | Session Cookie 表单/后台场景默认保留 CSRF；纯 Bearer Token API 可按项目安全策略关闭但需说明理由。 |
| CORS | 不使用 `*` 放开携带凭证请求；明确允许来源、方法、Header 和暴露 Header。 |
| 密码与凭证 | 密码使用 BCrypt、Argon2 或项目认可方案；密钥从环境、密钥系统或配置中心注入，不写入仓库。 |
| 请求校验 | Bean Validation、枚举白名单、排序字段白名单、分页上限和文件上传限制必须在入口或用例层体现。 |
| 安全响应头 | 按场景配置 HSTS、CSP、X-Frame-Options、X-Content-Type-Options、Referrer-Policy 等。 |
| 代理头 | `X-Forwarded-For`、`Forwarded`、真实 IP 只在可信代理链和框架策略配置正确时可用于审计或限流。 |
| 限流与滥用 | 登录、验证码、导出、搜索、批量操作、Webhook 和开放 API 必须考虑限流、熔断和告警。 |
| 错误响应 | 认证失败、权限不足、资源不存在和校验失败的错误码语义清晰，同时避免泄露内部实现。 |

Spring 安全 Review 输出建议：

```text
资产与主体：保护的数据、接口、后台能力和调用方。
认证授权链路：过滤器、方法级授权、数据权限和默认拒绝策略。
高风险接口：导出、批量、配置、审批、回调、文件和敏感数据。
测试缺口：未登录、低权限、跨租户、越权 ID、CSRF/CORS、重放和限流。
剩余风险：需要安全、SRE、合规或业务负责人确认的事项。
```

## 审计与可追溯

- 涉及资金、权限、配置、审批、敏感数据和后台高危操作时必须审计。
- 审计至少包含操作者、主体身份、对象、动作、前后值、时间、来源、结果和 traceId。
- 审计日志应防篡改、可检索、可导出、可关联业务证据。
- 审计数据本身也是敏感数据，访问和导出同样需要权限控制。

## 安全测试与验收

- 权限测试覆盖正向授权、未登录、低权限、跨租户、越权 ID、批量导出和后台操作。
- 敏感数据测试覆盖日志、异常、响应、导出、缓存、消息和埋点。
- 外部回调测试覆盖签名错误、重放、乱序、重复、超时和伪造来源。
- 依赖和镜像需要漏洞扫描；高危漏洞必须有修复、缓解或风险接受记录。
- 安全缺陷修复后必须补回归测试或检查清单，避免同类问题复发。

## 红线

1. 不得把权限判断只放在前端、网关提示或页面按钮上。
2. 不得在缺少租户/数据归属校验时开放查询、导出、更新或删除。
3. 不得把密码、token、密钥、证书、身份证、银行卡、手机号等敏感信息写入日志或版本库。
4. 不得反序列化不可信输入或执行不可信脚本、表达式和模板。
5. 不得在无审计链路的情况下提供后台高危操作、生产配置修改或敏感数据导出能力。
