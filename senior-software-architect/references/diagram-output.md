# 架构图形化交付

本文用于把架构设计、系统分析设计、技术方案、代码 Review、生产变更和 AI 协作中的关键结构转化为可评审、可维护、可验证的图形化产物。图是工程判断的表达面，不是架构本身；任何图形结论都必须能回到模块边界、接口契约、数据模型、测试、监控或发布验证。

## 使用时机

- 用户明确要求画系统架构图、模块图、流程图、时序图、状态机、ER 图、类图、部署图、依赖图、故障链路图或迁移路线图。
- 系分、技术方案、ADR、Review 或生产变更涉及多模块、多系统、多数据源、多状态、多消息链路或多团队协作。
- 需要把文字设计转化为默认 SVG 图形资产，并在交付时说明验证结论；Mermaid/Markdown 草图、PNG/PDF/截图等其他格式仅在使用者明确提出时处理。
- AI 编码协作、OpenSpec 或 Harness Plan 需要把上下文边界、写入范围、依赖关系和验证闭环可视化。

## 不适用场景

- 需求边界、数据归属、接口契约、部署环境或生产风险还不清楚时，不要用图包装不确定架构；先回到 `scenario-routing.md`、`architecture.md` 或专项 reference。
- 用户只需要代码修改、测试修复或问题 Review 时，不强行生成图片文件。
- 不把图当作设计验收；缺少接口契约、测试、静态检查、监控、回滚或数据校验时，图再完整也不能写成已完成。
- 不复制外部项目的脚本、模板、Logo、图形资产或未审查依赖；只吸收可公开复用的方法思想。

## 读取后必须产出

- 图形目标：服务于设计、Review、调试、迁移、上线、沟通还是 AI 协作。
- 图形类型：上下文图、容器/模块图、组件图、时序图、状态机、ER 图、类图、部署图、依赖图、数据流图、故障链路图或迁移路线图。
- 输出形态：正式图形化交付默认只生成 SVG 并验证；Mermaid/Markdown 草图、PNG/PDF/截图等其他格式只有在使用者明确提出时才额外生成。
- 工程落点：图中节点对应的模块、接口、数据、配置、消息、测试、监控或发布动作。
- 验证结论：说明是否已检查语法、文件可读性、远程资源、文本溢出、关键节点和与正文口径的一致性。

## 图形类型路由

| 场景 | 首选图形 | 适合表达 | 注意事项 |
| --- | --- | --- | --- |
| 架构方案 | 上下文图、容器/模块图、组件图 | 系统边界、模块职责、调用关系、依赖方向和外部系统。 | 必须标清核心业务、应用层、基础设施和外部依赖。 |
| 系分设计 | 模块图、流程图、时序图、状态机、ER 图 | 概要设计、详细设计、业务流程、接口交互、对象生命周期和数据模型。 | 图必须对应系分正文，不新增正文没有解释的节点。 |
| 技术选型 / ADR | 备选方案结构图、取舍矩阵附图 | 备选方案边界、迁移路径、代价和风险。 | 结论仍以 ADR 文本为准，图只辅助比较。 |
| 代码 Review | 依赖图、调用链图、状态流转图 | 循环依赖、越层访问、契约破坏、异常路径和测试缺口。 | 发现问题时仍按 Review 格式输出文件行号和风险。 |
| 分布式一致性 | 事务边界图、消息时序图、对账补偿图 | Outbox、Saga、TCC、幂等、去重、补偿、对账和人工兜底。 | 必须标清不变量、一致性窗口、失败路径和补偿触发。 |
| 遗留系统迁移 | 绞杀者路线图、防腐层图、双写/回填/切流图 | 新旧系统边界、迁移批次、灰度、回滚和下线条件。 | 不用图掩盖一次性替换核心链路的风险。 |
| 安全架构 | 信任边界图、权限链路图、数据流图 | 主体、资源、权限、租户隔离、敏感数据和审计。 | 密钥、token、生产地址和敏感信息不得进入图。 |
| 生产变更 | 部署图、发布路径图、故障处置图 | 灰度、开关、监控、告警、回滚、Runbook 和人工处理。 | 生产图必须和回滚、监控、验收标准绑定。 |
| AI 编码协作 | 上下文边界图、任务依赖图、验证闭环图 | OpenSpec、Harness Plan、写入范围、分工边界和验证命令。 | 不用图替代 Execution Grant、测试和人工确认点。 |

## 卓越图形能力要求

高质量架构图不能只是把系统名放进方框。输出前必须同时完成六个判断：

1. 图形意图：用于架构评审、系分设计、代码 Review、调试诊断、迁移计划、生产变更还是 AI 协作。
2. 图形范围：一张图只解决一个工程问题；超过 15 个核心节点时优先拆成上下文图 + 专项图。
3. 视觉风格：选择适合工程语境的风格，不用装饰性视觉替代边界、契约和验证。
4. 语义形状：同类技术对象使用同类形状和命名，例如入口、服务、领域、数据、消息、外部依赖、模型、工具和观测。
5. 语义箭头：箭头必须表达同步调用、异步事件、数据读写、控制策略、回调、补偿、降级或监控。
6. 验证闭环：图形必须能回到模块结构、接口契约、数据模型、测试、监控、发布或回滚方案。

## 技术架构图三级视图

技术架构图不要退化成技术组件堆叠图。它应表达逻辑组件、技术平台、职责边界、依赖关系和质量属性，并能回答：模块负责什么、模块之间是什么关系、数据如何流动、同步还是异步、关键容量或瓶颈在哪里、设计原则如何落到实现。

按读者和粒度选择视图：

| 层级 | 目标读者 | 表达重点 | 不应出现 |
| --- | --- | --- | --- |
| 整体技术架构图 | CTO、技术负责人、产品负责人、新成员 | 公司级或系统级宏观结构，业务服务、平台服务、基础设施、外部系统和核心依赖。 | Controller、ORM Entity、数据库表字段、框架注解、函数调用。 |
| 子领域技术架构图 | 领域架构师、核心开发、跨团队协作者 | 单个业务域内的服务、应用层、领域层、基础设施、事件、数据和自治边界。 | 和本领域无关的全局细节、跨域全量拓扑。 |
| 应用粒度技术架构图 | 开发、测试、维护人员 | 单个应用内部组件、端口适配器、关键类簇、数据流、异常路径、技术栈和测试落点。 | 管理层无法消费的全量代码结构，或没有工程落点的抽象口号。 |

给管理层看整体，给架构师看领域，给开发看应用内部。不要用一张图同时承载所有粒度；复杂系统应拆成整体图、领域图和应用详图。

## 从业务到技术的六步推导

画技术架构图时，先从业务和用例推导，再放置技术组件：

1. 从业务架构推演技术骨架：业务能力映射为服务、模块或应用，并补充网关、鉴权、链路追踪、配置中心等纯技术模块。
2. 从用例和验收场景推导非功能模块：高并发、异步处理、一致性、幂等、限流、对账、安全和审计都应映射到具体技术落点。
3. 提炼横向分层：接入层、应用层、领域层、基础设施层、数据层和观测支撑层要分层清楚，核心层不依赖外围细节。
4. 提炼纵向复用：认证、消息、ID、规则、文件、通知、网关等复用能力下沉或侧置，减少主链路交叉依赖。
5. 添加生产支撑模块：可观测性、熔断、限流、降级、健康检查、配置、调度、加密、审计和 WAF 等不能缺席。
6. 形成分层分域结构：按职责分层，按业务域分片；当连线过密或交叉过多时，优先重新审视边界、分层和中介者。

技术架构图必须能从业务架构找到对应技术方块，也必须能从技术方块回到职责、接口、数据、测试和运行指标。

### 通信复杂度图形检查

架构图应暴露系统复杂度在哪些节点和边上，而不是用更多方框掩盖问题。画图和评审时额外检查：

- 节点是否代表清晰责任，例如服务、模块、领域对象、数据存储、批任务、Agent、工具、外部系统或观测设施。
- 边是否标明同步调用、异步事件、数据读写、状态传播、控制策略、补偿、降级或监控链路。
- 边过密、交叉过多、跨层连线过多或出现依赖环时，优先重新审视边界、分层、聚合点和中介者，而不是继续美化布局。
- 抽象层、SDK、框架、工作流和 Agent 编排如果隐藏了关键调用、上下文、状态或失败路径，图中必须补出可观测边或调试入口。
- 如果一张图无法同时表达宏观边界和细节边，应拆成上下文图、关键链路图、状态传播图或故障传播图。

## 视觉风格路由

| 风格 | 使用场景 | 视觉要求 |
| --- | --- | --- |
| Architecture Doc Clean | 系分、技术方案、模块边界和跨团队评审。 | 白底、清晰分层、少颜色、强调边界、契约和依赖方向。 |
| Blueprint Engineering | 基础设施、微服务、部署、数据管道和迁移路线。 | 工程蓝图感、强分区、编号层级、适合表达拓扑和路径。 |
| Dark Ops | 生产故障、监控、告警、Runbook、调试诊断和应急路径。 | 深色底、强调关键链路、告警、回滚点和人工处理。 |
| Risk Review | 安全、权限、租户、敏感数据、供应链和生产变更。 | 突出信任边界、风险等级、人工确认和审计链路。 |
| AI Agent Systems | LLM、Agent、工具调用、RAG、记忆、策略护栏和多 Agent 协作。 | 明确模型、Agent、工具、存储、策略和观测，箭头区分读写、调用和控制。 |

## 语义形状和箭头

正式交付生成 SVG 时使用更丰富的形状和图例表达语义；只有用户明确要求 Mermaid/Markdown 草图时，才用稳定节点名、`subgraph` 分层和箭头标签表达语义。PNG/PDF/截图等派生格式不默认生成。

| 语义对象 | Mermaid 表达 | SVG 表达 |
| --- | --- | --- |
| 用户、调用方、外部系统 | 入口节点或外部泳道 | 外部边界卡片，避免和内部模块混淆。 |
| API Gateway、Controller、入口适配器 | 入口层节点 | 网关/边界形态，标注协议和认证点。 |
| Application Service、Use Case | 应用层节点 | 应用服务卡片，强调事务、幂等和权限边界。 |
| Domain、Policy、规则、聚合 | 核心层节点 | 核心业务块，避免依赖基础设施。 |
| Repository、Mapper、Client、MQ、Cache | 基础设施节点 | 基础设施分组，标注外部细节和 owner。 |
| Database、Warehouse、Object Storage | 数据节点 | 圆柱/存储形态，标注读写模型和一致性边界。 |
| Queue、Topic、Event Bus | 消息节点 | 队列/事件总线形态，标注事件名、重试和 DLQ。 |
| LLM、Agent、Tool、Memory、Guardrail | AI 专项节点 | 模型、Agent、工具、记忆、策略护栏分形态表达。 |
| Observability、Audit、Feature Flag | 控制与观测节点 | 辅助边界，不混入核心业务路径。 |

箭头语义保持一致：

- 实线箭头：同步调用、事务内调用或主路径。
- 虚线箭头：异步事件、回调、通知、观测或非主路径。
- 粗箭头或明确标签：数据写入、资金/账务、核心状态迁移或高风险操作。
- 回环箭头：重试、补偿、降级、回滚、人工兜底或 Agent 循环。
- 反向箭头必须谨慎使用；如果表示依赖反转、回调或控制流，必须标注原因。

布局语义保持稳定：

- 上下：越上层越接近用户或入口，越下层越接近数据、基础设施和外部细节。
- 左右：默认左进右出，左侧放上游、入口或触发方，右侧放下游、输出或后续处理。
- 内外：外层表达网关、认证、外部系统和信任边界，内核表达核心业务和领域规则。
- 聚类：相关模块使用分组边界表达同一业务域、平台能力或支撑平面；颜色只辅助识别，不能承担唯一语义。
- 稳定性：高频变化模块放边缘，稳定核心放中心或核心层，减少改图成本和误读。

## 稳定提示配方

当用户要求直接画图但上下文足够时，可用这些配方组织输出：

- 分层系统架构：按“Client / API / Application / Domain / Infrastructure / Observability”分层；标出依赖方向和禁止反向依赖。
- 分布式一致性：按“业务命令、事务边界、Outbox/事件、消费者、幂等、补偿、对账、人工兜底”组织。
- 生产发布：按“入口流量、灰度开关、版本实例、监控告警、回滚路径、人工确认”组织。
- 遗留迁移：按“旧系统、防腐层、新能力、双写/回填、校验、切流、下线”组织。
- AI Agent 运行时：按“用户输入、规划器、模型运行时、工具调用、记忆/知识库、策略护栏、观测、输出格式化”组织。
- Review 依赖图：只画和问题相关的调用链、依赖方向和破坏点；仍必须配合文件行号和修复建议。

## 输出形态选择

默认使用 SVG 作为正式图形化交付物。Mermaid 仅在用户明确要求 Markdown 内嵌、文档草图、便于 diff 的文本图，或项目已有 Mermaid 维护约定时使用。

- `flowchart`：上下文图、模块图、部署关系、数据流、迁移路径、故障链路和验证闭环。
- `sequenceDiagram`：接口调用、异步消息、外部回调、重试补偿和人工介入。
- `stateDiagram-v2`：状态机、审批流、订单/任务/账户生命周期和异常回滚。
- `classDiagram`：领域类、接口和关键抽象关系；不要把完整源码机械转成类图。
- `erDiagram`：核心实体、聚合、读写模型和表关系草图；生产 DDL 仍需工程/DBA 确认。
- Markdown 表格：取舍矩阵、风险矩阵、验证矩阵；不要把复杂取舍强行塞进图。

正式图形化交付默认只生成 SVG。Mermaid/Markdown 草图、PNG/PDF/截图等其他格式必须由使用者明确提出，或在已有 SVG 无法满足其指定消费场景时，经说明后再处理。生成文件时应优先使用本地、可审查、无网络依赖的脚本或工具；新增脚本必须说明输入、输出、写入范围、失败行为和是否访问网络。

## 图形内容规则

- 每张图只表达一个工程问题：边界、依赖、交互、状态、数据、部署、迁移、风险或验证。
- 节点命名使用真实模块、接口、领域对象、数据源、队列、任务、配置或外部系统名；不要用泛泛的“业务服务”“公共服务”替代边界。
- 箭头必须标注语义：调用、事件、消息、数据同步、事务、补偿、回调、依赖、部署或监控。
- 关键模块必须标注职责；关键连线必须标注协议、事件、数据流、同步/异步语义、容量指标或一致性语义。
- 颜色应有统一语义，建议不超过 5 类；不要每个模块一个颜色，也不要让颜色成为理解图的唯一入口。
- 高风险节点用短标签标注，例如 P0/P1、破坏性变更、生产数据、外部依赖、人工确认、回滚点、测试缺口。
- 图形不得和正文、代码、OpenSpec、Harness Plan、测试或发布方案冲突；冲突时先修正权威口径。
- Review 场景不要只给图；必须继续输出问题、证据、影响、建议和验证。

## 常见反模式

- 组件堆叠图：只罗列 Redis、Kafka、Gateway、Service、DB，却没有职责、关系、容量、瓶颈和质量属性。
- 粒度混乱：整体架构图里塞类、注解、表字段，或应用内部图里混入无关全局系统。
- 一图讲完全部：边界、时序、状态、部署、数据和异常都塞进一张图，导致无法评审。
- 线条蜘蛛网：模块互相全连接，说明抽象、分层、中介者或事件边界需要重审。
- 只有图没有解释：节点没有职责，连线没有语义，读者无法判断哪里能改、哪里不能动。
- 画完就丢：图不进版本控制，不随代码、接口、监控和部署演进，三个月后变成误导材料。
- 忽视读者：给管理层看类图，给开发看只有战略框的大图。

## 验证清单

- SVG：文件能打开，SVG XML 可解析，包含 `viewBox` 或稳定宽高，不引用远程资源，不嵌入敏感信息。
- Mermaid/Markdown 草图：只有用户明确要求时才生成；生成后检查代码块类型、节点 ID、箭头和分组语义。
- 派生格式：只有用户明确要求 PNG/PDF/截图等其他格式时才生成；生成后必须检查尺寸、裁切、文本截断、图例和关键路径可读性。
- 可读性：节点数量不过载，长文本拆行，关键路径从左到右或从上到下清晰，颜色和强调不承担唯一语义。
- 粒度：明确当前图属于整体、子领域还是应用粒度，并与目标读者匹配。
- 可决策：能看出职责、依赖、同步/异步、容量瓶颈、风险点、可改区域和不可破坏边界。
- 活文档：图形源文件或生成脚本进入版本控制；关键架构变化、接口变化、部署变化后同步更新。
- 工程一致性：图中模块、接口、数据、状态、消息、依赖和风险与代码、设计文档、测试、监控、发布方案保持一致。
- 可追溯：交付时报告生成或修改的文件路径、图形用途、关键假设、验证动作和剩余风险。

正式、完整、可评审、提交前、CR 或触发验证场景中，提交图形 brief 给人工或 `$fireworks-tech-graph` 前必须运行 `scripts/check_architecture_deliverable.py --kind diagram-brief` 检查图形目标、目标读者、图形类型、工程落点、节点/分组、箭头语义、假设/待确认项和输出格式是否齐全。该脚本只检查本地文本结构完整性，不判断架构质量、视觉美观度或渲染质量；无法运行时必须说明原因、人工检查结果和残余风险。

## 可选续作：fireworks-tech-graph

当用户明确要求复杂、精美、可交付的系统架构图、模块图、能力图、流程图、时序图、状态机、ER 图、类图、部署图、迁移图，或要求继续生成 SVG 等视觉资产时，可以在完成本技能的架构边界、工程落点、风险、验证和生产约束判断后，调用 `$fireworks-tech-graph` 继续完成图形化表达；PNG/PDF/截图等其他格式仍需使用者明确提出。

调用边界：

- 只有 `fireworks-tech-graph` 技能已安装、可用，或用户明确要求使用它时才调用；不得假定外部技能一定存在。
- 本技能负责架构语义、模块边界、接口契约、数据模型、测试、监控、发布和回滚约束；`fireworks-tech-graph` 负责复杂视觉呈现、图形模板、风格系统、SVG 导出和渲染校验。
- 交给 `$fireworks-tech-graph` 前，必须提供图形目标、目标读者、图形类型、节点/分组/箭头语义、风险节点、工程落点、验证标准和输出格式；未指定输出格式时按 SVG 处理。
- 不把外部技能输出当作架构验收；图形结果仍需回到代码、接口契约、测试、监控、发布方案和风险清单。
- 不复制外部技能脚本或模板进本仓库；需要引入可执行资产时，先按仓库 `AGENTS.md` 做供应链安全审查。
