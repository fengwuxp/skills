# 系统分析设计模板

本文承载可复制系统分析设计模板。主规则、写作要求、评审清单和触发边界见 `system-analysis-design.md`；只有当用户要求“写系分”“系统分析设计”“详细设计”“技术详细方案”或“按团队系分模板输出”时才读取本文。

本文把产品事实推进为可编码、可测试、可发布的系统约束。内部按“名 -> 事 ->（图）-> 法 -> 器 -> 验”检查贯通：先定义系统职责与边界，再用运行时场景展开真实行为；复杂关系用最小设计视图暴露边界和冲突，再用工程规则约束状态和失败，最后落到模块、契约、数据与验证证据。正式系分使用工程章节名，不展开文化化推演过程。

模板占位符统一使用 `〈...〉`；正式文档不得保留未替换占位符。

## 文档标识与文件命名

创建正式工程文档前，先确定规范主题、文档类型和目标路径，优先级为：用户明确指定 > 项目 `AGENTS.md` / `README.md` > 仓库已一致执行的约定 > 本模板回退规则。仓库现有文件名相互冲突时，不把单个历史文件当成规范；将冲突列为待确认并询问 owner。只有用户明确授权项目约规治理时才修改 `AGENTS.md` / `README.md`；未授权时，更新既有文档保持原路径，新建文档使用本模板回退规则并说明。

- 系统分析设计和系分统一使用 `〈主题〉-系分设计.md`，一级标题使用 `# 〈主题〉系统分析设计`。
- 架构方案、技术方案及附属设计材料使用 `〈主题〉-架构方案.md`、`〈主题〉-技术方案.md` 或 `〈主题〉-〈明确产物类型〉.md`；附属类型可以是 `代码知识图谱`、`完整场景手册` 等，不得只写模糊的“设计”或“说明”。
- 消费 Product Context Card 或产品文档时必须复用其规范主题。系分范围更窄时使用 `〈主题〉-〈子主题〉-系分设计.md`；范围扩大或业务概念变化时先退回产品 owner 确认，不自行改名吸收。
- 主题和文档类型之间只使用一个 ASCII 连字符 `-`；主题内部保留稳定领域术语和 `API`、`DSL` 等既有缩写，不用空格、下划线或中文破折号做结构分隔。
- 文件名默认不包含序号、日期、版本、`草稿`、`最终版`、`新版` 等状态；顺序、版本和状态进入索引、文档头与 Git 历史。
- 更新既有文档默认保持原路径。只有用户明确进入命名迁移任务时才改名，并同步 Markdown 链接、任务材料、脚本、测试和其他硬编码路径。

## 使用时机

- 需要直接生成完整系统分析设计文档。
- 需要按团队系分模板补齐章节。
- 需要把需求转成可评审、可编码、可验收的设计文档骨架。
- 需要承接 PRD/产品方案中的业务目标、验收种子和质量属性种子，继续推导系统边界、模块、接口、数据、测试、监控和发布门禁。

## 不适用场景

- 只做系统设计 CR、结构检查或判断是否需要系分时，先读 `system-analysis-design.md`。
- 只做架构图时，读 `diagram-output.md`。
- 只做生产上线检查时，读 `production-readiness.md`。

## 读取后必须产出

- 一份经过裁剪的系统分析设计文档；不适用章节必须写明“不适用原因”，不要直接删除高风险章节。

## 需要继续读取的 reference

- 系分写作要求和评审清单读 `system-analysis-design.md`。
- 图形化表达读 `diagram-output.md`。
- 生产风险读 `production-readiness.md`。
- 一致性读 `distributed-consistency.md`。
- 安全读 `security-architecture.md`。
