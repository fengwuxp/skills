## 使用时机

当用户要求代码审查、架构设计、技术方案、生产变更、上线评审、PR 说明或重构计划时，优先使用本文件的输出结构。模板不是固定话术，而是最低信息完整性要求；可以删减不相关项，但不得省略风险、约束、验证和回滚。

代码 Review 需要进一步判断业务语义、分层边界、契约完整性、失败路径和工程一致性时，结合 `coding-review-deep-dive.md`。

## 输出格式约定

- 文档结构优先保证可读、可扫读、可评审：先给结论或摘要，再给依据、方案、风险和待确认项。
- 段落不宜过长；复杂内容优先拆成小节、列表、流程、矩阵或清单。
- 表格只用于对比、矩阵、字段、规则、状态、验收等结构化内容；表格过宽或单元格过长时，应拆成多个表或改为分节描述。
- 正文段落和表格单元格默认使用纯文本；一般不对字段名、状态值、枚举值、示例值、角色、模块名、风险项加粗或使用代码背景。
- 只有代码标识符、命令、配置键、协议字段、文件路径、风险等级或需要强调的结论，才使用加粗或代码背景。
- 标题层级不受此规则影响，继续使用 Markdown 标题表达文档结构。
- 不为了“显得正式”滥用加粗、代码背景、复杂表格和多层嵌套列表。

## 代码 Review 输出模板

Review 默认采用问题优先，而不是总结优先。

```text
发现
- [P0/P1/P2/P3] 文件:行号 标题
  风险：说明会导致的真实业务、数据、稳定性或安全后果。
  证据：引用代码路径、调用链、状态流转或测试缺口。
  建议：给出可落地修复方向，避免只说“优化一下”。
  验证：说明应补充或已执行的测试/检查。

开放问题
- 仅列影响判断的未知项，不超过 3 个。

变更摘要
- 简要说明本次变更意图、影响范围和已验证内容。
```

严重级别定义：

| 级别 | 判定标准 |
| --- | --- |
| P0 | 会造成资金损失、数据破坏、权限绕过、生产不可用或不可逆事故。 |
| P1 | 会造成主要业务流程错误、兼容性破坏、明显安全风险或稳定性风险。 |
| P2 | 会造成边界场景错误、可维护性明显下降、测试缺口或可观测性缺失。 |
| P3 | 局部可读性、命名、格式、文档或低风险工程一致性问题。 |

## 架构方案输出模板

```text
背景与目标：要解决的业务问题、成功标准、非目标。
现状诊断：当前模块、数据、接口、运维和组织协作中的真实约束。
核心决策：选型、边界、职责、数据归属和一致性模型。
决策依据：备选方案、选择理由、放弃理由和复审条件；重大取舍见 `adr-and-tradeoff.md`。
领域模型：核心概念、状态、业务不变量、权限和审计要求。
模块边界：应用层、领域层、基础设施层、接口层的依赖方向。
接口契约：入参、出参、错误码、幂等、分页、鉴权、兼容策略。
数据方案：表结构、索引、迁移、归档、修复、对账或补偿。
一致性方案：事务边界、幂等、消息、补偿、对账、一致性窗口和人工兜底；复杂场景见 `distributed-consistency.md`。
演进方案：遗留系统、防腐层、双写/回填/切流、契约测试、旧能力下线；复杂场景见 `evolutionary-architecture.md`。
可靠性：超时、重试、限流、降级、熔断、幂等、告警和人工兜底。
安全合规：敏感信息、密钥、权限、租户隔离、审计留痕；复杂场景见 `security-architecture.md`。
可观测性：日志、指标、链路追踪、业务埋点和告警阈值。
验证方案：单测、集成测试、契约测试、压测、回归范围。
发布方案：灰度、开关、数据迁移、回滚、监控和应急预案。
备选方案：至少说明 1 个被放弃方案及原因。
```

涉及生产发布、数据变更、外部依赖、性能容量或高风险链路时，必须同时使用 `production-readiness.md` 检查 SLO、容量、灰度、监控、Runbook 和回滚。

## 实施计划输出模板

```text
范围：本次做什么、不做什么、受影响模块。
步骤：按可提交、可验证的粒度拆分。
风险：列出兼容性、数据、安全、性能和运维风险。
验证：每一步对应的编译、测试、静态检查或人工验收。
交付：代码、文档、迁移脚本、配置、监控、回滚材料。
```

## 生产变更 Checklist

- 是否存在破坏性 API、DTO、枚举、错误码、消息体、数据库字段或配置变更。
- 是否有明确 SLO、容量基线、灰度观察指标和回滚阈值。
- 是否能灰度发布，是否有开关，开关默认值是否安全。
- 数据库 DDL 是否向前兼容，是否避免长事务、大锁表和不可逆变更。
- 数据修复是否有 dry-run、备份、分批、审计、回滚和结果校验。
- 外部依赖是否配置超时、重试、熔断、降级和告警。
- MQ、任务和批处理是否具备幂等、去重、补偿和人工兜底。
- 跨服务或异步一致性是否有不变量、事务边界、一致性窗口、对账和差异处理。
- 权限、租户隔离、敏感数据、密钥和审计是否覆盖高风险路径。
- 遗留系统迁移是否有防腐层、契约测试、灰度切流、回退和旧能力下线计划。
- 缓存是否有失效策略、预热策略和一致性处理。
- 监控是否覆盖成功率、耗时、错误码、积压、重试、资金或核心业务指标。
- 回滚是否覆盖代码、配置、数据、任务、缓存和消息。
- 是否准备 Runbook，包含现象、影响、定位、止血、恢复、回滚、验收和复盘材料。

## 兼容性治理规则

| 契约类型 | 禁止直接做 | 推荐做法 |
| --- | --- | --- |
| REST/RPC API | 删除字段、改语义、改错误码、改权限默认值 | 新增字段保持可选；旧字段标记废弃；提供迁移窗口。 |
| DTO/消息体 | 改字段类型、改枚举含义、复用字段表达新语义 | 使用新字段或新版本消息；消费者先兼容，生产者后切换。 |
| 数据库 | 直接删列、改列类型、重命名表列 | 先加后迁、双写/回填、验证后下线旧字段。 |
| 配置项 | 改默认值导致生产行为突变 | 新增配置项并给安全默认值，灰度验证后再替换。 |
| 错误码 | 复用旧错误码表达新错误 | 新增错误码并补充文档、监控和下游处理。 |

## 常见反模式

- Controller 直接写复杂业务规则，导致领域规则不可测试、不可复用。
- 领域层直接依赖 ORM Entity、Mapper、HTTP Client、MQ Client 或第三方 SDK。
- 为一个未验证需求引入规则引擎、插件体系、微服务或多租户平台。
- 多服务共享数据库表，绕过服务边界直接读写他人数据。
- 用“最终一致”掩盖无重试、无补偿、无对账、无告警。
- 事务内吞异常、异步失败不暴露、日志只记录“操作失败”。
- 生产修数脚本无 dry-run、无备份、无分批、无审计、无回滚。
