# 架构师核心原则与职责边界

本文定义架构师的核心抽象、职责边界、决策原则和常见反模式。服务分层、方法命名、包结构等落地细则见 `project-governance-standards.md`。

## 使用时机

- 用户需要通用架构设计、架构评审、架构职责边界、质量属性、反模式或架构落地原则。
- 任务尚未进入具体 Java 约规、生产发布、安全专项或系统分析模板，需要先确立架构判断主轴。
- 需要判断某个方案是否符合简洁、边界、演进、可靠性、可验证和团队落地原则。
- 用户面对无标准答案、跨模块跨团队、遗留系统、线上约束、AI 编码失控或复杂技术债问题，需要先把问题机制和验证路径建模。

## 不适用场景

- 用户要求完整系分、详细设计或可复制模板时，优先读 `system-analysis-design.md` 与 `system-analysis-template.md`。
- 用户要求代码 Review、Java/Spring 约规、Bug 修复、测试或生产变更时，先回到 `scenario-routing.md` 选择专项 reference。
- 用户要求 PRD、产品架构、支付资金方案或业务规则矩阵时，优先交给 `产品架构专家`。

## 读取后必须产出

- 架构问题定义、边界判断、核心取舍、质量属性优先级和验证落点。
- 说明当前方案适合什么业务阶段、组织能力和技术约束，以及哪些结论需要 ADR、系分、测试或生产验证承接。
- 对反模式给出真实风险和最小治理动作，不把架构原则写成口号。

## 1. 架构师的定位

架构师不是“画图的人”，也不是“替团队做所有技术决定的人”。架构师对系统长期结构质量负责：识别关键问题，定义边界和约束，做出可解释的取舍，并推动设计在代码、测试、部署、监控和团队协作中落地。

在代码生成和 AI 协作场景下，架构师的稀缺价值更不是多写代码，而是定义问题、控制复杂度、分配团队注意力。实现成本下降会放大无效尝试、浅层抽象和后验修补倾向；架构师必须用前置约束让系统保持可理解、可验证和可演进。

架构师的价值在于让团队更容易做正确的事：系统边界更清楚，变化影响更局部，复杂问题可拆解，技术取舍可解释，质量要求可验证，团队协作可持续，生成和实现受约束，不把低成本代码变成高成本理解和维护。

### 1.1 架构师底层思维

架构师的判断力来自可迁移的底层思维，而不是掌握了多少语法、框架或流行术语。处理架构设计、代码 Review、复杂缺陷和生产风险时，至少显式使用七类思维：

- 抽象思维：识别核心概念、领域模型、不变量和稳定边界，避免用 `Object`、`Map`、`String` 等过度泛化类型掩盖真实语义。
- 逻辑思维与结构化思维：基于事实、规则、因果链、排除法、假设验证和反例检查推导结论，再把问题拆成目标、边界、对象、流程、异常、验证和行动项。
- 批判思维与成长型思维：对新技术、平台化、微服务和 AI 生成保持“质疑、验证、适配”，把失败、Review 评论和生产事故转成可练习能力。
- 复盘思维与数据思维：从选择、结果、证据和代价中提炼行动项，用指标、日志、压测、用户行为、收益/成本和运行数据支撑架构取舍。

### 1.2 问题核心诊断

架构师处理复杂问题时，先判断病机，再决定药方；这里的病机不是医学判断，而是工程问题真正起作用的核心机制。可用三层诊断：**病** 是整体问题类型，例如目标不清、边界错误、状态机缺失、可靠性不足、数据一致性破坏、权限安全风险、交付流程失控或 AI 生成失控；**证** 是结构性证据，例如调用链绕边界、数据不变量被破坏、异常无补偿、测试只测实现、CR 无法理解影响范围、监控无法证明行为；**症** 是可见症状，例如报错、超时、状态卡死、测试失败、返工、上线回滚或评审争议。先从症状收集事实，再从证据定位结构问题，最后确认整体病机并决定补需求、改边界、重构模块、补测试、改发布门禁还是回到 owner；AI 总结、可视化图、静态扫描或外部文章只能给线索，不能替代源码、测试和生产证据。

### 1.3 非标工程问题

非标工程问题通常没有现成 SOP，也不能靠“多写点代码”解决。它常见于跨模块跨团队、遗留系统演进、线上行为不稳定、公共契约变更、数据一致性、性能容量、安全权限、AI 编码失控或目标与约束同时不清的场景。架构师要先把问题建模成可证伪方案，而不是直接拆成执行清单。

非标工程问题卡：

```text
问题背景:
可见症状:
结构证据:
核心机制:
影响面:
关键不确定性:
候选方案:
最小可逆实验:
验证命令 / 观测指标:
决策标准:
写入边界:
回滚 / 降级:
owner / 下一步:
停止条件:
```

处理规则：

- 先只读侦察源码、测试、日志、运行配置和现有文档；缺证据时不直接建议重构、拆服务、换框架或开启 CAD。
- 候选方案至少说明代价和放弃理由；高风险取舍进入 ADR 或系分，不在聊天结论里一笔带过。
- 最小可逆实验优先是失败测试、契约测试、只读数据校验、灰度开关、影子读、压测片段、日志指标或小范围重构切片。
- 进入编码前必须有写入边界、验证命令和回滚方式；进入实际项目编码 Loop 时还要有状态回写位置和独立 Checker。
- 若问题本质是产品目标、业务对象、验收或 owner 不清，退回产品专家或 AI Native，不由架构师用工程假设补业务事实。

## 2. 架构师应该做的事

架构师应该围绕四件事工作：理解问题，先明确业务目标、用户价值、约束、非目标、核心复杂度和当前阶段最重要的质量属性；定义边界，划清业务、模块、服务、数据、安全、对外契约和内部实现细节，并约束状态、接口和依赖方向；做出取舍，在多个可行方案中选择当前最合适的方案，说明放弃什么、代价是什么、未来如何演进，并用 ADR、技术方案或评审记录沉淀关键决策；推动落地，把架构决策落实为模块结构、接口契约、编码规范、测试策略、发布方案、检查清单、自动化门禁和运行指标。

## 3. 架构师不应该做的事

架构师不应把架构变成脱离代码和团队的 PPT，不为炫技引入团队无法掌控的复杂技术，不在需求不清楚时替业务做过度假设，不绕过团队共识强推方案，不把所有问题都抽象成框架、平台或中台，不借“架构治理”做无关重构和大范围改动，不只关注技术正确性而忽视成本、风险、组织能力和交付节奏，也不替团队写完所有关键代码；更重要的是让团队具备持续正确实现的能力。

## 4. 架构关注的核心抽象

架构关注三类抽象：业务抽象，回答系统为什么存在、用户或系统如何完成目标、什么可以发生、什么不可以发生、哪些业务不变量必须成立、团队如何统一领域语言；结构抽象，回答模块职责、边界、接口契约、数据事实/状态/历史、同步/异步/补偿/人工介入/异常恢复路径；运行抽象，回答容量、稳定性、安全性、可观测性、可运维性，以及系统处在单体、模块化单体、微服务、服务网格、无服务或云原生的哪个演进阶段。

## 5. 架构决策原则

### 5.1 业务牵引

技术方案必须服务业务目标。不能只因为技术先进、流行或优雅就引入它。技术架构本质上是业务架构的支撑结构：它的价值不在于自己像产品，而在于降低某类业务能力的落地、复制、运维和演进成本。评审平台化、中台化、复用框架、规则引擎、多租户或架构升级时，必须先说明上层业务能力、目标用户、核心流程、验收口径和成本函数；如果赚钱业务无法在该技术架构上运行，优先审查架构适配性，而不是要求业务迁就技术洁癖。

### 5.1.1 业务同质性和复制成本

架构复用的前提是业务同质性，不是技术栈相似。复用设计前先确认：要复制的业务能力块和具体主体 / 问题；对象、状态、规则、流程、权限、数据口径和验收是否共享；技术架构降低的是开发、交付、实施、运营、配置、合规、测试、迁移、排障还是变更成本；哪些非标、高耦合、结构特殊部分应进入业务裁剪、配置边界、扩展点、人工流程或项目化交付；复用收益是否大于新增抽象、治理、数据迁移、排障和维护成本。若业务同质性不足，只能输出业务差异、成本风险和回退方案；不得用“未来可复用”“架构升级”“技术平台产品化”掩盖当前无法验收、无法复制或无法证明收益的问题。

### 5.1.2 合适优先

架构不是行业领先方案的搬运，而是当前业务规模、团队能力、技术积累、运行约束和验证成本下的最优解。评审任何方案时，先问它是否能被当前团队实现、维护、排障和演进，再问它是否先进、完整或漂亮。

需要警惕三类失配：

- 人力失配：方案需要的研发、测试、运维、SRE、DBA、安全或业务协作能力超过当前团队可持续承载范围。
- 积累失配：直接照搬大厂多年演进后的平台、治理和中间件形态，却缺少对应业务规模、历史问题和运行经验。
- 场景失配：为了不存在或尚未到来的业务压力，引入过早分布式化、平台化、规则引擎、多租户或复杂扩展点。

### 5.2 简洁优先

能用模块化单体解决的问题，不急于微服务化；能用明确代码解决的问题，不急于框架化；能用规则和测试约束的问题，不急于平台化。

简单不是简陋，而是降低结构复杂度和逻辑复杂度。结构复杂度来自更多组件、更多依赖、更多失败点和更长排障链路；逻辑复杂度来自单个组件承担过多职责、状态流转和规则判断。方案评审时必须说明新增组件、服务、队列、缓存、规则层或抽象层带来的故障面、协作成本和验证成本。

### 5.2.1 通信复杂度视角

架构复杂度应优先落到节点、边、状态传播和可观测性上，而不是只数服务、模块或代码行。节点是服务、模块、函数、批任务、数据存储、Agent、工具或外部系统；边是同步调用、异步事件、数据读写、配置依赖、状态传播、控制策略或观测链路。

技术选型通常不会消灭复杂度，只会改变复杂度的位置和可见性。单体拆成微服务，复杂度可能从调用栈转移到网络、拓扑、发布和观测；引入框架、SDK、工作流或 Agent 编排，复杂度可能从业务代码转移到配置、上下文、隐式状态和调试工具。评审时必须说明复杂度从哪里移到哪里，以及新位置是否更容易理解、验证、观测和恢复。

通信复杂度治理可以用三类动作检查：

- 解耦是降低边密度，让一个节点变化时不需要修改大量既有边。
- 抽象是把子图封装成复合节点，降低认知负担，但必须保留可观测、可追踪、可重放和可替换的边界。
- 治理是减少环路、跨层依赖和隐式状态传播，让依赖方向、状态扩散和故障传播更可预测。

真正降低复杂度往往来自删除节点或边：明确某个前提、收缩一个能力、取消一个适配层、延后一个扩展点或拒绝一个不成熟场景。删除必须记录假设、收益、关闭的选项和复审触发条件，避免把当前简化变成未来债务黑箱。

与业务驱动架构和 TDD 联动时，业务目标、用例和验收示例是判断节点和边是否必要的依据；TDD、监控和评审门禁是让关键边上的业务事实、状态传播和失败语义变成反馈的手段。无法回指业务 driver、质量属性或验收路径的节点和边，应作为删除、合并、延后或降级候选。

### 5.3 边界清晰

复杂度的核心治理手段是边界。边界不清，任何设计模式、框架和规范都会退化。

### 5.3.1 约束优先

代码越容易生成，越要先约束系统。架构师必须回答：什么问题值得解决，哪些代码不该写，哪些状态不该暴露，哪些依赖不该成立，哪些实现必须删除、收缩或延后。复杂度治理优先依靠边界、接口、状态机、测试和审查，而不是靠后续修补。系统 DNA 先于实现：复杂系统不能按“需求 -> 功能 -> 事故 -> 补规则”的顺序生长，进入实现前先定义核心不变量、状态流转、边界、演化规则和验证方式，并让它们进入 AC、代码契约、测试、监控、CR 和发布门禁。

### 5.4 可演进

架构应允许系统在业务、团队和技术变化中逐步演进。为已知变化设计扩展点，为未知变化保留简单性。

演进式设计优于一步到位。架构师应定义当前阶段的最小稳定结构、可观测信号、复审触发条件和下一阶段演进路径，而不是试图一次性预测所有未来。只有当业务规模、变化频率、故障证据、团队能力或监管/安全要求真实触发时，才升级到更复杂的架构形态。

### 5.5 可变性显式化

能用不可变表达的事实、契约和值对象，应优先不可变；真正需要状态变化的地方必须显式建模、集中管理、可审计、可测试。

适合不可变的对象包括值对象、配置、规则参数、领域事件、消息、审计快照、API 输出模型、跨线程传递数据和缓存 Key。需要状态流转的聚合根、状态机对象、ORM Entity、批量填充模型或框架强依赖可变结构的对象，不应机械追求不可变；它们的可变性必须有清晰生命周期和边界。

不可变对象一旦跨进程、跨服务、落库、缓存或进入 MQ，就必须同时评估序列化和反序列化能力，例如 Jackson、Kryo、JDK Serialization、Protobuf 或 JSON-B 对构造器、record、final 字段、默认值、版本兼容和集合类型的支持。

### 5.6 凤凰式可靠性

可靠系统不是假设组件永不失败，而是承认代码、节点、网络、依赖和人员都会出错，并通过隔离、熔断、限流、降级、重试、补偿、淘汰、重建和自动化发布恢复整体服务能力。

架构师必须区分“让系统活得更好”和“让系统活下去”：性能、异构、伸缩是锦上添花；失败可控、故障可恢复、发布可回滚、问题可观测是系统生存底线。生命化复杂系统治理要求分化有边界、协作可追踪、失败可生存、感知可定位、自治与协作平衡；微服务、事件驱动、自动化运维或 AI Agent 只能在降低协调成本时使用。

### 5.7 可验证

架构决策必须能被验证：通过测试、静态检查、压测、监控、故障演练、评审清单或运行指标验证。

### 5.8 测试驱动设计

TDD 不只是“先写测试”，而是用测试驱动设计边界、接口和行为。架构师应能通过先定义可验证行为，倒逼领域模型、应用服务、端口适配、异常语义和依赖方向保持清晰。

TDD 对架构的价值：用测试表达业务不变量、边界条件和验收语义，用可测试性反推低耦合设计，避免核心业务依赖框架、数据库和远程服务；用红-绿-重构节奏控制复杂度，防止一次性大设计和不可验证抽象；用测试保护重构，并作为活文档降低团队交接、Review 和维护成本。

### 5.9 局部最优服从整体最优

单个模块的便利不能破坏系统整体边界；短期交付效率不能长期透支系统健康。

### 5.10 架构风格取舍框架

架构风格不是技术名词菜单，而是约束下的收益和代价选择。评估 DDD、Clean/Hexagonal/Onion、微服务、SOA、CQRS、Event Sourcing、12-Factor 或 Reactive 时，先回答四个问题：

- 它解决的主要问题是什么：边界、依赖方向、独立部署、读写优化、审计追溯、云原生运行还是 UI 状态分离。
- 它牺牲什么换取什么：建模成本、间接层、运维复杂度、最终一致性、消息复杂度、组织调整或运行约束。
- 它要求哪些前置条件：业务语言是否稳定、限界上下文是否清楚、团队是否具备运维和观测能力、测试与发布门禁是否跟得上。
- 它是否只在局部使用：CQRS、Event Sourcing、Reactive、微服务拆分不应全系统默认铺开，应先绑定到具体业务 driver、质量属性和失败路径。

| 架构族 | 解决什么 | 主要代价 | 评审提醒 |
| --- | --- | --- | --- |
| DDD / 领域架构族 | 用通用语言、限界上下文和上下文映射明确业务边界。 | 建模、业务协作和术语校准成本。 | DDD 不是分层架构；战略设计先定语言和边界，战术设计再用聚合、实体、值对象、领域事件、仓储和应用服务组织代码。 |
| Clean / Hexagonal / Onion | 让依赖方向指向核心，隔离 Web、DB、框架和外部系统。 | 多端口、多适配器和模型翻译成本。 | 三者名称不同但目标接近；不要机械套四层或同心圆，重点看依赖规则和可测试性。 |
| Microservices / SOA | 提供独立部署、独立演进、企业集成或跨团队协作能力。 | 网络、数据一致性、观测、发布和组织协作复杂度。 | 边界不清时不要拆；Microservices 倾向先拆再连，SOA 倾向先连再拆，集成代价不同。 |
| CQRS / Event Sourcing | 分离读写优化空间，或用事件序列提供审计、回放和状态重建能力。 | 最终一致窗口、事件版本、读模型同步和排障成本。 | CQRS 与 Event Sourcing 可组合但不绑定；只在读写差异、审计追溯或复杂状态流有明确收益时使用。 |
| PACELC / BASE / FLP 相关分布式权衡 | 在分区、延迟、一致性和可用性之间做可解释取舍。 | 对业务一致性窗口、补偿和降级策略提出更高要求。 | CAP 不覆盖正常运行时延迟/一致性账单；最终一致不是不用补偿和对账。 |
| 12-Factor / Reactive / 运行架构 | 约束部署、配置、进程、消息边界和弹性恢复。 | 运行平台、消息治理、配置治理和无状态约束成本。 | 运行约束必须回到发布、伸缩、故障隔离和观测，不是口号式云原生。 |

### 5.11 四定变化治理

变化治理不能从“先加扩展点”开始。先定方向，再定性质，再定位置，最后定量；缺少前三项时，容量、排期、覆盖率和自动化都可能变成伪精确。定向看业务目标、质量属性或风险边界；定性看新业务能力、规则、契约、数据、容量、安全、协作或技术债；定位看业务域、模块、接口、表、消息、配置、测试、部署或 owner；定量看容量、SLO、成本、排期、覆盖率、缺陷率、返工率或风险阈值。架构评审先问定向和定性，再看定位和定量；GSD/CAD 任务拆解必须能说明每个 Atomic Task 的定向、定性、定位和验证方式；只有定量没有方向的任务不进入执行。

DDD 使用时必须区分两层：战略设计：通用语言、限界上下文、上下文映射和防腐层，回答“业务语言是什么、边界在哪里、上下文之间怎么对话”；战术设计：聚合、实体、值对象、领域事件、领域服务、仓储和应用服务，回答“边界内如何保护不变量、表达行为和组织用例”。

常见误读：把 DDD 当成分层、富领域模型或复杂系统专属方法；把 Repository 当 DAO，Repository 面向聚合根和领域集合，DAO/Mapper 面向表和持久化细节；把 ApplicationService 当业务规则容器，它应编排用例、事务和端口协作；把微服务、CQRS、Event Sourcing 当成熟架构标配，没有业务 driver、质量属性和运行能力支撑时只是把复杂度搬家。

## 6. 架构质量属性

架构设计必须明确优先级。所有质量属性都重要，等于没有优先级。

| 质量属性 | 关注点 | 常见落地方式 |
|----------|--------|--------------|
| 可维护性 | 易读、易改、影响范围可控 | 分层、模块化、命名、测试、代码评审 |
| 可扩展性 | 面对变化低成本调整 | 接口隔离、策略模式、事件、插件化 |
| 可靠性 | 失败可控、可隔离、可恢复 | 超时、重试、熔断、降级、补偿、容灾、故障演练 |
| 性能 | 吞吐、延迟、资源效率 | 缓存、索引、异步、批处理、容量评估 |
| 安全性 | 身份、权限、数据保护 | 认证授权、加密、脱敏、审计、最小权限 |
| 可观测性 | 问题可发现、可定位 | 日志、指标、链路、告警、仪表盘 |
| 可测试性 | 低成本验证正确性，并通过 TDD 反推清晰边界 | 依赖注入、端口适配、单测、集成测试、契约测试 |
| 可运维性 | 易发布、易回滚、易扩容、易重建 | CI/CD、配置管理、灰度、K8s、Runbook、不可变基础设施 |

### 6.1 质量属性场景化

质量属性不能只写成“高可用、高性能、可扩展”。架构评审必须把业务目标转成可验证场景：

| 场景要素 | 说明 |
| --- | --- |
| 业务 driver | 哪个业务目标、风险、法规、SLO、用户体验或运营成本触发该质量要求。 |
| 刺激源和触发条件 | 谁在什么情况下触发，例如高峰流量、外部通道超时、规则变更、权限越权、批处理延迟。 |
| 受影响资产 | 哪个用例、模块、数据、接口、队列、任务、报表或运营动作受影响。 |
| 期望响应 | 系统应该隔离、降级、重试、补偿、拒绝、审计、告警还是人工接管。 |
| 度量和验收 | RT、错误率、一致性窗口、恢复时间、审计完整性、人工处理时限、监控指标和测试方式。 |
| 架构取舍 | 为满足该场景牺牲了什么，例如一致性、成本、复杂度、交付速度或运维负担。 |

如果质量属性无法追溯到业务 driver，或无法落到测试、压测、演练、监控、Runbook、ADR 或发布门禁，就不能作为架构定版依据。

## 7. 架构评审关注点

架构评审不只是问“能不能实现”，而是确认方案是否适合当前业务、团队和系统阶段。

- 问题是否定义清楚？目标和非目标是否明确？
- 业务边界、模块边界、数据边界是否清晰？
- 领域语言是否能在代码、文档、测试和业务对话中一致出现；业务人员说的关键术语是否能在代码中找到对应？
- 限界上下文是否先于服务拆分、数据库拆分和包结构被定义；上下文之间是否有明确映射、防腐层或开放接口？
- 核心业务用例、主干流程、验收场景和系统 DNA 是否先于技术栈、分层、服务拆分和扩展点被定义；核心不变量、状态流转、边界、演化规则和验证方式是否能被 PRD、系分、代码、测试和监控追踪？
- 服务、模块或团队边界是否来自业务能力、限界上下文、领域语言和变化频率，而不是 UI、数据库、中间件或组织惯性？
- 质量属性是否已场景化，并能说明业务 driver、触发条件、期望响应、度量、验收和取舍？
- 方案是否匹配当前人力、技术积累、业务规模、运行约束和验证成本，而不是照搬行业领先或大厂最终形态？
- 新增组件、服务、队列、缓存、规则层或抽象层是否带来可解释收益，并已评估结构复杂度、逻辑复杂度、故障面和排障成本？
- 关键节点和通信边是否已列出，边的同步/异步、数据读写、状态传播、控制策略、失败影响和观测方式是否清楚？
- 技术选型、服务拆分、框架化或平台化是否只是把复杂度搬到网络、配置、框架黑箱、隐式状态或运维排障上？
- 选用 DDD、Clean、微服务、CQRS、Event Sourcing、12-Factor 或 Reactive 时，是否说明它牺牲什么、换取什么，以及满足哪些前置条件？
- 新增抽象是否保留可观测、可追踪、可重放和可替换的边界，而不是隐藏关键调用、状态和失败路径？
- 是否存在可以通过明确假设删除节点或边的机会；若删除，关闭了哪些未来选项，何时需要复审？
- 是否定义了当前阶段的最小稳定结构、复审触发条件和演进路径，而不是追求一步到位的完美架构？
- 依赖方向是否符合高层策略不依赖低层细节的原则？
- 是否只画了分层图却没有约束源代码依赖方向、端口位置和外层 adapter 边界？
- 核心业务规则和用例流程能否脱离数据库、Web 容器、消息中间件和真实 SDK 被测试？
- 数据模型是否能保护业务不变量？
- 接口契约是否稳定、可演进、可测试？
- 是否因为实现成本低而引入了无效尝试、过量代码或过早抽象？
- 方案是否降低理解、验证和调试成本，而不是把注意力消耗转移给后续维护者？
- 是否能通过 TDD 或先行验收测试描述核心行为和边界条件？
- 异常、超时、重试、幂等、补偿和回滚是否设计完整？
- 故障是否能被隔离、降级、熔断、淘汰、重建和观测；复杂系统是否具备分化、事件驱动协作、可恢复、可观测和自治/协作平衡，而不是只把系统拆得更细？
- 当前架构形态是否匹配业务规模、团队能力和运维成熟度？
- 安全、权限、审计和数据隔离是否覆盖关键路径？
- 日志、指标、链路追踪和告警是否能支撑排障？
- 发布方案、灰度方案和回滚方案是否可执行？
- 团队是否具备实现、维护和演进该方案的能力？

### 7.1 代码库架构深化检查

AI 生成代码、快速迭代和局部修补容易让代码库变浅：模块数量变多，但每个模块缺少稳定语义；接口很多，但没有隔离真实变化；公共工具变多，但业务表达变弱。评审时应额外检查：

- 模块是否有明确业务责任、数据归属和变化边界，还是只按技术层或文件类型堆放。
- 接口是否表达稳定契约、隔离外部变化或保护核心规则，还是只为“方便 mock”或一对一转发而存在。
- 适配器是否把框架、SDK、MQ、HTTP、数据库和缓存细节挡在核心业务之外。
- 变更是否具备局部性：修改一个规则时，影响范围是否集中、可测试、可回滚。
- 抽象是否有真实杠杆：能减少有意义的重复、保护不变量或降低变化成本，而不是增加跳转和认知负担。
- AI 生成代码是否引入浅模块、无主依赖、重复 DTO、幻觉工具类、宽泛命名和无测试保护的扩展点。

不要强行套用某套固定术语；用项目已有领域语言描述问题，并把建议落到模块边界、接口契约、测试、静态检查或重构计划。

## 8. 架构落地方式

架构必须落到工程资产中，而不是停留在口头共识中。

- **文档化**：技术方案、ADR、架构图、接口文档、数据字典、Runbook。
- **代码化**：样板代码、基础类、starter、SDK、脚手架、测试基类。
- **自动化**：静态检查、ArchUnit、CI 门禁、依赖扫描、规范扫描。
- **流程化**：设计评审、代码评审、发布检查、故障复盘、技术债治理。
- **指标化**：用质量指标和运行指标观察架构是否真的有效。

## 9. 常见架构反模式

- **PPT 架构**：图很好看，但没有代码、测试、部署和监控支撑。
- **技术先行架构**：先定技术栈、分层、微服务和中间件，再把业务用例塞进去；组件、依赖和扩展点无法回到核心业务闭环、变化频率和验收场景。
- **质量属性口号化**：只写高可用、高性能、可扩展，不说明业务 driver、触发条件、期望响应、度量和验证方式。
- **事故沉淀式架构**：先堆功能，出问题后不断补 if、补表字段、补配置、补规则和补脚本；核心不变量、状态流转、边界和演化规则长期缺位，导致系统靠历史事故记忆运行。
- **行业领先搬运**：把大厂多年演进后的平台、中间件、组织分工或复杂架构直接搬到当前团队，缺少匹配的人力、业务规模、历史问题和运行积累。
- **过度抽象**：为不确定未来增加接口、配置、扩展点和层级。
- **贫血分层**：所有业务规则堆在 ApplicationService 或 Controller 中。
- **公共模块垃圾桶**：所有“不知道放哪”的代码都进入 common。
- **微服务幻觉**：边界不清、数据不清、团队不清时强拆服务。
- **通信复杂度搬家**：技术选型、服务拆分或平台化没有减少复杂度，只是把复杂度搬到网络拓扑、配置、框架隐式状态、发布协调或观测排障里。
- **隐藏边抽象**：抽象层、SDK、工作流、Agent 编排或平台能力隐藏关键调用、状态传播和失败路径，导致调试只能猜上下文、猜顺序、猜隐式副作用。
- **一步到位幻觉**：试图一次性设计永不变化的完美架构，缺少阶段目标、复审触发条件、迁移路径和运行反馈。
- **生成噪声膨胀**：因为代码或原型生成便宜，就不断叠加功能、接口、适配器和抽象，最后把成本转移到理解、验证、调试和维护。
- **完美运行幻觉**：假设网络、节点、依赖和人员永不出错，缺少隔离、恢复和回滚设计。
- **分布式裸奔**：拆成服务后缺少服务发现、流量治理、可靠通信、安全和可观测性。
- **框架绑架业务**：业务规则依赖框架细节，导致核心逻辑难测试、难迁移。
- **只治标不治本**：只修线上现象，不处理设计缺陷和系统体质。
- **规范不可执行**：规范只存在文档里，没有检查清单、工具和评审机制。
