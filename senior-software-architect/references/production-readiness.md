# 生产就绪与非功能专项

本文用于架构方案、系统分析设计、上线评审、生产变更和高风险代码修改的生产可用性检查。生产就绪不是最后的上线清单，而是设计阶段就要进入的约束。

## 使用时机

以下场景必须使用本文件：

- 涉及生产发布、灰度、回滚、开关、监控、告警或应急。
- 涉及数据库迁移、数据修复、回填、重放消息、清理缓存或重建索引。
- 涉及资金、权限、租户、敏感数据、公共契约、外部依赖、MQ、批处理或定时任务。
- 涉及分布式一致性、遗留系统迁移、双写/回填/切流、认证授权或租户隔离。
- 涉及性能、容量、稳定性、安全、可观测性或运维能力评审。

## 不适用场景

- 只做低风险本地代码整理、文档修改或概念解释，且不影响生产行为、数据、契约或外部依赖。
- 生产事故诊断先读 `debugging-diagnosis.md` 建立事实时间线；本文负责上线、变更和生产可用性检查。
- 安全、迁移、一致性等专项问题需要继续读取对应 reference，不在本文展开全部实现细节。

## 读取后必须产出

- 生产影响范围、SLO/指标、容量、可靠性、数据安全、一致性、安全、兼容性、可观测性、发布回滚和 Runbook 检查结论。
- 对数据变更、外部依赖、异步链路和不可逆操作给出 dry-run、备份、分批、审计、回滚和人工确认点。
- 无法给生产就绪结论时，明确缺失证据和残余风险。

## 需要继续读取的 reference

- 生产现象和根因诊断读 `debugging-diagnosis.md`。
- 数据一致性和 MQ 补偿读 `distributed-consistency.md`。
- 遗留迁移读 `evolutionary-architecture.md`；安全读 `security-architecture.md`。
- 交付输出读 `review-and-output-templates.md`；验证流程读 `workflow.md`。

## 生产就绪维度

| 维度 | 必须回答的问题 | 常见产物 |
| --- | --- | --- |
| 目标与 SLO | 成功标准是什么？核心接口、任务或链路的可用性、RT、错误率、吞吐要求是什么？ | SLO/SLA、验收指标、错误预算。 |
| 容量与性能 | 峰值 QPS、并发、数据量、消息积压、线程池、连接池、缓存和数据库压力是否有基线？ | 容量评估、压测报告、资源水位。 |
| 可靠性 | 失败路径是什么？如何超时、重试、限流、熔断、降级、补偿、去重和人工兜底？ | 失败路径设计、补偿方案、故障演练。 |
| 数据安全 | 数据是否可回滚、可校验、可追溯？敏感数据是否脱敏、加密和最小化访问？ | 数据迁移方案、审计日志、脱敏策略。 |
| 一致性 | 跨服务、跨库、异步和外部依赖是否定义不变量、幂等、补偿、对账和一致性窗口？ | 一致性方案、对账任务、差异处理。 |
| 安全 | 认证、授权、租户隔离、密钥、敏感数据和高危操作是否可控可审计？ | 安全设计、权限测试、审计方案。 |
| 兼容性 | API、DTO、消息、数据库、配置、新旧版本和外部依赖是否兼容？ | 兼容矩阵、废弃周期、契约测试。 |
| 可观测性 | 关键路径是否有日志、指标、trace、业务埋点和告警阈值？ | Dashboard、告警规则、排障手册。 |
| 发布与回滚 | 是否支持灰度、开关、分批、暂停、回滚和新旧版本共存？ | 发布计划、回滚预案、开关清单。 |
| 运维与应急 | 故障发生后谁处理、怎么定位、怎么止血、怎么恢复、怎么复盘？ | Runbook、值班说明、应急预案。 |

## SLO 与指标建议

- **接口链路**：QPS、P50/P95/P99 RT、成功率、错误率、超时率、限流数、降级数。
- **任务链路**：调度成功率、处理耗时、积压量、重试次数、死信数、补偿成功率。
- **数据链路**：迁移进度、回填行数、失败行数、校验差异、重复处理数、对账差异。
- **资源水位**：CPU、内存、GC、线程池、连接池、磁盘、网络、Pod 重启、OOM、CPU throttling。
- **业务指标**：订单成功率、支付成功率、库存扣减成功率、审批成功率、资金或核心业务差异。

指标必须能支撑决策：灰度继续、暂停、回滚、扩容、降级或人工介入。

## 发布检查

- 是否明确本次发布的范围、非目标、影响模块、影响用户和外部依赖。
- 是否存在破坏性公共契约变更；如果存在，是否有兼容策略和迁移窗口。
- 是否有安全默认值的配置开关；开关是否支持按租户、用户、地域、比例或实例灰度。
- 是否能分批发布、暂停发布、回滚代码、回滚配置、回滚数据或停止任务。
- 是否已准备监控面板、告警规则、日志查询语句和关键业务核对方式。
- 是否明确上线负责人、验证负责人、回滚负责人和应急联系人。
- 是否说明无法回滚或只能前滚的部分，并给出风险接受人。

## 数据变更检查

- 先 dry-run，输出影响范围、样本、预估耗时和异常样例。
- 先备份或保留可恢复材料，不能只依赖“脚本应该没问题”。
- 分批执行，设置批次大小、间隔、超时、失败暂停和人工确认点。
- 每批执行后校验影响行数、业务状态、对账差异和下游副作用。
- 保留操作者、时间、输入、输出、批次、失败原因和回滚动作的审计记录。
- 对不可逆变更必须明确风险接受、替代兜底和事后核验方式。

## 外部依赖与异步链路检查

- 外部调用必须有连接超时、读取超时、重试上限、退避策略、熔断、限流和降级。
- 重试必须结合幂等键和去重策略，不能用重试放大资金、库存、订单或消息副作用。
- MQ、任务、批处理必须说明重复消息、乱序、积压、死信、重放和人工补偿。
- 第三方 SDK、接口、证书、密钥、额度、配额和限频规则必须纳入发布前检查。
- 外部规则、依赖版本和接口行为可能变化时，必须保留时效性说明和验证日期。

## 安全与迁移专项入口

- 分布式一致性、MQ、幂等、补偿和对账：见 `distributed-consistency.md`。
- 遗留系统迁移、双写、回填、切流、契约测试和旧能力下线：见 `evolutionary-architecture.md`。
- 认证授权、租户隔离、敏感数据、密钥、审计和安全测试：见 `security-architecture.md`。

## Runbook 最低要求

```text
现象：如何发现问题，告警名称和触发条件是什么。
影响：影响哪些用户、接口、任务、数据或外部系统。
定位：看哪些日志、指标、trace、表、队列和配置。
止血：如何关闭开关、降级、限流、暂停任务或切回旧链路。
恢复：如何补偿、重试、回填、重放、清理脏数据或通知下游。
回滚：代码、配置、数据、任务、缓存、消息分别如何处理。
验收：恢复后如何验证业务正确性和数据一致性。
复盘：需要保留哪些证据、指标和时间线。
```

## 红线

1. 没有监控和回滚，不得给生产就绪结论。
2. 没有 dry-run、备份、分批、审计和校验，不得执行生产修数或迁移。
3. 没有幂等、去重、补偿和告警，不得把异步失败交给“最终一致”兜底。
4. 没有兼容策略，不得直接删除或改义公共 API、DTO、消息、配置或数据库字段。
5. 没有权限、租户、敏感数据和审计设计，不得发布高风险后台、开放 API 或数据导出能力。
6. 没有契约测试、灰度切流和回退路径，不得替换遗留系统核心链路。
7. 没有负责人、阈值和应急动作的告警，不算有效告警。
8. 无法回滚的变更必须明确风险接受人、验证方案和前滚预案。
