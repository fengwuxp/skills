## 1. 架构原则

### 1.1 底层价值观

- **业务优先**：技术服务业务目标，先理解问题，再选择技术。
- **简洁优先**：能用简单模型解决的问题，不引入复杂架构。
- **边界优先**：先划分业务边界、依赖边界、数据边界，再谈实现。
- **演进优先**：架构是持续演进的结构约束，不是一次性蓝图。
- **可验证优先**：设计必须能被代码、测试、指标、评审或运行结果验证。
- **测试驱动设计优先**：用测试先描述行为、边界和不变量，再反推接口、模型、依赖和实现。
- **可靠性优先**：承认失败必然发生，通过隔离、恢复、降级、回滚和观测保证系统活下去。
- **阶段匹配优先**：单体、模块化单体、微服务、云原生各有成本，架构形态必须匹配业务规模、团队能力和运维成熟度。
- **通信复杂度显式化**：把系统看成节点、边和状态传播的图，评审复杂度是否被删除、暴露或转移到隐藏链路。

### 1.2 核心原则

- SOLID：SRP、OCP、LSP、ISP、DIP。
- KISS：保持简单，优先选择团队能掌控的方案。
- YAGNI：不为不确定未来添加抽象、接口、扩展点或配置。
- DRY：消除真实重复，但避免过早抽象。
- 高内聚、低耦合：让变化局部化。
- 关注点分离：业务规则、流程编排、技术细节分离。
- 依赖倒置：高层策略不依赖低层细节，二者依赖抽象。
- 最小知识原则：对象只与直接协作者交互。

### 1.3 取舍能力

- 一致性 vs 可用性。
- 性能 vs 可维护性。
- 扩展性 vs 复杂度。
- 复用性 vs 局部清晰度。
- 抽象能力 vs 认知成本。
- 解耦收益 vs 新增通信边和观测成本。
- 短期交付 vs 长期演进。
- 技术先进性 vs 团队掌控力。

## 2. 产品语义校准与业务建模能力

### 2.1 业务理解

- 识别主流程、支流程、异常流程和人工介入点。
- 区分核心业务、支撑业务、通用能力和纯技术能力。
- 发现隐含规则、业务不变量、边界条件和失败场景。
- 把口头需求转化为术语、流程、规则、状态和验收条件。
- 区分“用户说的方案”和“真正要解决的问题”。

### 2.2 产品语义与工程映射

工程侧产品语义校准见 `product-design.md`；完整 PRD、产品架构、规则矩阵、运营后台、数据指标和金融/支付产品方案优先交由 `产品架构专家`。

- 能检查需求材料是否说明背景、目标、非目标、角色、流程、规则、数据、验收和风险。
- 能把产品语义映射为模块边界、接口契约、数据模型、权限边界、状态机和测试场景。
- 能用用例图、流程图、时序图、状态机、上下文图等方式表达工程需要理解的业务路径。
- 每个设计必须服务当前落地或明确的未来扩展，避免无落地路径的概念堆砌。
- 文档必须对开发和测试友好，让研发能识别模块、接口、数据和边界，让测试能提取场景、数据和期望结果。

### 2.3 领域建模

- 识别实体、值对象、聚合根、领域服务、领域事件、仓储。
- 划分聚合边界和一致性边界。
- 设计对象生命周期、状态流转和业务不变量保护点。
- 将行为放到合适对象，避免业务逻辑散落在 Controller、Mapper 或工具类中。
- 根据复杂度选择事务脚本、贫血模型、充血模型或 DDD 战术模式。

### 2.4 DDD 战略设计

- 子域划分：核心域、支撑域、通用域。
- 限界上下文划分与上下文映射。
- 统一语言建设：命名、模型、接口、文档保持业务一致。
- 通过防腐层隔离外部系统和遗留模型。
- 让模块边界尽量与业务边界、团队边界一致。

### 2.5 用例建模

- 用 ApplicationService / Use Case 表达业务场景。
- 区分“对象怎么变”（DomainService）和“业务怎么走”（ApplicationService）。
- 区分命令、查询、事件、策略和规则。
- 输入使用 Request/Command/Query，输出使用 DTO/VO/Result，避免泄露内部实体。
- Controller 只做协议适配、参数校验和结果转换，不承载业务流程。

## 3. 模块管理能力

### 3.1 模块划分

- 按业务能力、限界上下文和变化原因划分模块。
- 区分 domain、application、infrastructure、interfaces 等架构层。
- 区分 api、client、service、dal、common 等工程模块职责。
- 公共模块必须有清晰边界，禁止变成杂物间。
- 模块名、包名、类名应表达业务语义，而非只表达技术分层。

### 3.2 依赖治理

- 禁止循环依赖。
- 禁止跨层访问和绕过 ApplicationService 的业务调用。
- 领域层不依赖数据库、Web、MQ、缓存、框架等外部细节。
- 基础设施依赖领域抽象并提供实现。
- 通过 ArchUnit、Maven Enforcer、模块边界测试等方式守住依赖方向。

### 3.3 分层约束

推荐默认调用链：

```text
Controller / Adapter
↓
ApplicationService / Use Case
↓
DomainService / DomainQueryService / Domain Model
↓
EntityService / Repository Port
↓
Repository / Mapper / External Adapter
```

每层必须明确：职责、允许依赖、禁止行为、输入输出模型、命名规范和测试方式。

### 3.4 模块演进

- 从大单体演进到模块化单体。
- 从模块化单体按业务边界拆分微服务。
- 通过防腐层、适配器和事件逐步解耦遗留系统。
- 优先小步重构，不用“大爆炸式重写”替代演进。

## 4. 设计能力

### 4.1 接口设计

- API 语义清晰、稳定、可演进。
- 请求和响应模型不暴露内部实体。
- 明确错误码、异常语义、HTTP 状态码或 RPC 错误契约。
- 明确幂等、分页、排序、过滤、版本兼容、权限和审计。
- 接口必须可测试、可观测、可灰度。

### 4.2 类与对象设计

- 类职责单一，方法短小，抽象层级一致。
- 优先组合而非继承。
- 避免上帝类、工具类滥用、过度 Manager、贫血行为散落。
- 设计模式服务于变化点，禁止为了模式而模式。
- 常用模式：策略、工厂、模板方法、责任链、状态、观察者、适配器、装饰器、规格模式。

### 4.3 数据设计

- 表结构、字段、索引、唯一约束和事务边界服务业务规则。
- 明确读写模型、状态字段、审计字段、时间字段、软删除和历史追踪策略。
- 明确强一致、最终一致、补偿、对账和数据修复机制。
- 禁止只按页面字段反推数据模型。

### 4.4 流程与状态设计

- 明确同步、异步、事件驱动、定时补偿、人工处理流程。
- 明确超时、重试、幂等、去重、补偿、回滚、死信和告警。
- 状态机必须定义状态、事件、守卫条件、动作和非法流转处理。

### 4.5 扩展点设计

- 只为已知且稳定的变化点设计扩展。
- 扩展点应有清晰契约、默认实现、测试样例和失败处理。
- 可选机制：策略、SPI、插件、规则引擎、配置驱动、事件监听。
- 警惕“未来可能会用”的接口、配置项和抽象层。
