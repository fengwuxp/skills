# 项目治理：数据、安全与质量

本文只承载数据责任、安全边界、质量策略和验证证据的架构裁决，不保存数据库字段、Java 日志、测试命名或 Wind 框架细则。

## 使用时机

- 需要评审数据所有权、一致性、迁移、权限、审计、安全或测试治理。
- 需要把规则偏差映射成生产风险、验证矩阵或整改优先级。

## 不适用场景

- 纯 Java/Wind 约规检查交给 `wind-coding-conventions`。
- 测试实现细节读 `testing.md` 和 `testing-practices.md`。
- 安全专项设计读 `security-architecture.md`。

## 读取后必须产出

- 数据、安全与质量风险，责任 owner，最小控制措施和可执行验证方式。

## 需要继续读取的 reference

- 复杂安全设计读 `security-architecture.md`。
- 测试策略读 `testing.md`。
- Java/Wind 具体条目读取 `wind-coding-conventions` 的通用层与按证据启用的专项层。

## 数据责任裁决

先确认业务对象、数据 owner、一致性边界、生命周期和删除责任，再评审表、索引、缓存、消息或搜索方案。

评审时回答：

1. 哪个业务不变量必须由数据库、事务、状态条件或其他确定性机制保护。
2. 写入、读取、派生数据和缓存分别由谁负责，允许多长不一致窗口。
3. 迁移、回填、修数和删除如何 dry-run、校验、回滚与审计。
4. 查询和索引是否服务真实访问路径，并有容量或性能证据。
5. 规则 Skill 给出的字段、唯一性或访问约规是否与当前业务语义一致。

## 安全边界裁决

安全评审围绕资产、主体、权限、信任边界、敏感数据、密钥、审计和恢复，不围绕某个框架注解清单。

重点确认：

- 谁能以什么身份访问哪个资源，授权失败如何处理和审计。
- 敏感数据在采集、传输、存储、日志、导出和删除各阶段如何保护。
- 外部输入、文件、回调、消息和管理操作的信任边界在哪里。
- 高风险操作是否有双重确认、最小权限、追踪和恢复路径。
- 安全控制是否有自动化测试、扫描、日志证据或人工验收。

## 质量与验证裁决

测试和质量策略由业务风险决定，不由固定目录、类名或方法名前缀决定。架构师负责选择能证明行为的最小验证组合：单元、契约、集成、端到端、架构守卫、静态检查、性能、安全或生产观测。

每个准出结论必须说明：

- 被保护的业务事实、不变量或公共契约。
- 验证层级、真实链路与替身边界。
- 已执行命令、结果和未覆盖风险。
- 失败后的停止、回滚或人工接管方式。

## 规则结果消费

项目本地规范和 `wind-coding-conventions` 负责具体编码条目。架构师只判断规则是否适用、偏差造成什么真实风险、应由谁修复以及什么证据足以准出；不得把规则违反自动等同于高严重级别。

## 评审清单

- 数据所有权、一致性、迁移和删除责任是否清楚。
- 权限、敏感数据、审计和恢复边界是否可验证。
- 测试是否证明外显行为，而非当前实现步骤。
- 规则偏差是否已映射到真实风险、owner、最小修复和验证。
- 无法验证的结论是否明确标为待确认或残余风险。
