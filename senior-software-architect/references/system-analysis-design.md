# 系统分析设计约规

本文基于团队《系分模板》整理，用于架构师输出系统分析设计、设计评审和可编码方案。系分文档不是形式材料，而是把产品语义、目标、边界、运行时场景、工程规则、实现载体和验证证据统一到一份可执行设计中。

## 1. 使用时机

以下场景必须优先使用本约规：

- 新增核心业务能力、跨模块能力、公共平台能力或中间件/SDK 产品能力。
- 涉及服务边界、模块边界、数据模型、状态机、权限、安全、资金、库存、账务、审计等高风险设计。
- 涉及性能、容量、可用性、兼容性、灰度、回滚、应急或生产运维风险。
- 需求需要跨产品、研发、测试、运维、架构或外部系统协同时。

仅小修文案、局部样式、低风险配置或单点 Bug 修复可不写完整系分，但仍需在 PR 中说明背景、影响、验证和回滚。

正式、完整、可评审、提交前、CR 或触发验证场景中，系统分析设计必须运行 `scripts/check_architecture_deliverable.py --kind system-design` 检查本地文本或显式传入的本地文件。该脚本只检查背景问题、目标范围、设计依据、概要设计、详细设计、关键流程、业务规则、接口抽象、非功能风险、验收摘要和执行计划引用是否齐全，不联网、不写文件、不替代架构评审或工程验证；无法运行时必须说明原因、人工检查结果和残余风险。

## 2. 文档结构

```text
文档状态与版本信息
一、需求背景
二、目标、非目标、定性与范围
三、概要设计
四、详细设计
五、非功能设计
六、风险、验收摘要与执行计划引用
七、参考资料
```

推荐微调：

- 在“目标”中补充“非目标”，避免需求膨胀。
- 在“概要设计”中明确系统边界、依赖边界和核心取舍。
- 在“详细设计”中先说明关键流程、模块职责和接口抽象，再补齐契约、数据、状态、异常、幂等和事务。
- “非功能设计”中技术风险应视为必填，尤其是可监控、可灰度、可应急。
- 系分正文用短段落解释关键业务规则如何落到模块、守卫、契约和数据约束；多规则需要横向核对时再使用规则落地表。
- 系分正文只保留验收摘要、关键验证方向和执行计划链接；详细验收矩阵、AC 与测试映射、第一实施切片、写入范围、验证命令、owner、停止条件、排期和任务状态进入执行计划。
- PRD/产品方案到系分 / 业务驱动系统设计时，先读取 `product-design.md`，再把业务目标、参与方、核心行为、规则、验收和质量属性追踪到工程资产。

## 3. 章节写作要求

### 3.0 可复制系分模板

当用户要求“写系分”“系统分析设计”“详细设计”“技术详细方案”或“按团队系分模板输出”时，读取 `system-analysis-template.md` 获取可复制模板，再按项目实际裁剪。不适用章节必须写明“不适用原因”，不要直接删除高风险章节。

### 3.0A 系分前需求分析门禁
进入系分正文前，先确认需求分析已经形成可工程化结论：原始诉求和根源需求、产品定义、产品边界、稳定点 / 变化点、边界坐标和上下游分工。缺少这些信息时，不要急着画架构图、选技术栈、拆服务或生成代码；缺口可以进入待确认项和架构假设，但影响系统边界或高风险设计时必须回退产品专家或业务 owner。
系统边界、模块拆分、接口契约、数据模型和测试矩阵必须能回指产品定义、产品边界、稳定点 / 变化点或业务不变量。扩展点、抽象层、规则引擎、平台化、缓存、MQ 或微服务拆分，必须说明对应的真实变化轴、owner 和验收方式；否则列为过度设计风险。用户原始功能请求、AI 原型或页面愿望只能作为输入证据，不能直接作为系统能力分配依据。
若系统需求、产品需求或外部规则未确认，不进入详细设计、编码或测试设计；先记录缺口、影响、确认 owner 和未确认期间默认处理。需求条目应表达外部可见行为，具备可验证性和可追踪性；衍生需求、异常恢复、安全监控、状态异常处理和鲁棒性要求必须标明来源、owner、验收方式和是否回写上游，不能由架构师或 AI 在设计阶段私自补齐。复杂业务、核心链路、状态机、资金/库存/权限/审计、跨系统协作或 AI/GSD/CAD 任务，还必须补齐系统 DNA：核心不变量、状态流转、责任边界、演化规则和验证方式；缺失时不得拆模块、生成任务或进入详细设计，只能写成待确认、ADR 候选或退回产品专家补产品 DNA。

### 3.1 文档状态与版本信息

正式系分以最终标准版本为主，记录当前版本号、状态、owner、发布日期、评审结论摘要、影响范围和过程记录链接；完整修订流水、评审争论、AI 推理轨迹、迭代草稿和被拒方案进入评审报告、ADR、Decision Log、任务计划或中间任务文档，不复制进系分正文。允许在系分中保留“当前有效取舍”和“待确认项”，但不得把旧版本方案、历史讨论和多轮草稿与当前设计结论混写；需要追溯时用链接指向过程资产，避免读者分不清当前有效设计。

### 3.2 需求背景

必须回答“为什么做这件事”：

- 当前业务、客户或系统问题是什么；用户原始诉求已经如何被转换成根源需求，如果仍是功能请求或二手转述，说明证据缺口和确认方。
- 产品定义、产品边界、稳定点 / 变化点和边界坐标是什么，以及它们如何影响本次系统边界。
- 涉及哪些用户、角色、系统、模块和外部依赖。
- 不做会带来什么业务、技术、数据、安全或运维风险。
- 已知约束是什么，例如合规、成本、时间、团队能力、历史包袱；上游产品方案是否给出目标、非目标、参与方、核心行为、对象状态、规则矩阵、验收示例和质量属性种子。
- 如果产品语义缺失，不直接补技术组件；先退回产品语义补齐，或把缺口写成待确认项和架构假设。

### 3.3 目标

目标必须能约束设计和编码，不只是复述需求。

- **业务目标 / 技术目标**：解决什么问题、带来什么业务价值、影响哪些角色，以及容量、吞吐、RT、CPU、内存、成功率、错误率、资源成本等可量化指标。
- **系统目标 / 非目标**：建设什么系统能力，例如幂等、对账、补偿、审计、灰度、扩展点；同时明确本次不做什么、哪些能力留到后续版本、哪些边界不扩张。

### 3.4 定性与范围

- 先判断这是新增能力、既有能力增强、架构治理、迁移重构还是可靠性修复，再决定设计深度。
- 明确系统必须承担和明确不承担的责任、稳定点与变化点、影响模块与外部依赖，以及当前设计方向的事实依据。
- 事实不足时标为架构假设或待确认，不用组件、接口或数据表替代定性判断。

### 3.5 概要设计

概要设计负责说明主体思路、系统边界和关键取舍。

- 用上下文图、用例图、组件图或流程图说明系统与用户、外部系统、内部模块的关系，并明确业务边界、模块边界、服务边界、数据边界、安全边界和运维边界。
- 说明核心流程、关键依赖、同步/异步关系、读写路径和失败路径。
- 对至少一个备选方案做取舍说明，解释为什么采用当前方案。
- 明确当前方案是否需要兼容历史数据、历史接口、历史客户端或新旧版本共存。
- 用正文说明目标、参与方、核心行为和验收摘要如何推导系统边界、模块、接口、数据与质量属性；详细业务驱动追踪表进入执行计划或按需附录。
- 说明功能分配关系：功能不是头脑风暴得到的功能点，而是对象对外提供的可见、有价值交互行为；下层系统、模块或 CSCI 的功能应来自上层对象目标和功能分解，并保留正向分解和逆向追溯。

### 3.6 详细设计

详细设计必须达到可编码状态。不能只停留在“这里调用某服务”“这里入库”。

- **模块/类设计**：职责、依赖方向、扩展点、禁止依赖、主要类和方法边界。
- **接口抽象与契约**：先说明服务对象、职责边界、输入前提、业务输出、副作用和失败语义，再展开 API、RPC、MQ、回调、任务、导入导出的入参、出参、错误码、幂等和权限。
- **数据设计**：表结构、字段语义、索引、唯一约束、状态字段、审计字段、迁移和归档。
- **状态机设计**：状态、事件、守卫条件、动作、终态、非法流转和回滚流转。
- **流程/时序设计**：主流程、异常流程、补偿流程、人工介入流程和超时处理。
- **一致性设计**：事务边界、幂等键、重试、去重、补偿、对账、最终一致性窗口；复杂场景见 `distributed-consistency.md`。
- **演进设计**：涉及遗留系统、历史接口、历史数据或核心链路替换时，说明防腐层、契约测试、双写/回填/切流、灰度和旧能力下线标准；详细检查见 `evolutionary-architecture.md`。
- **重构设计准入**：跨模块、公共契约、数据迁移、核心链路替换、双轨切流或旧能力退役时，读取 `refactoring-design-template.md` 设计从当前状态到目标状态的迁移；局部、行为保持且可测试的重构不创建独立重构设计文档。
- **安全设计**：鉴权、授权、租户隔离、敏感信息脱敏、密钥管理和审计留痕；复杂场景见 `security-architecture.md`。
- **验证设计摘要**：说明哪些规则、不变量、契约和质量目标需要由测试、观测或人工评审保护；详细测试矩阵、`AC-xxx` / `QA-xxx` 映射、失败测试、命令和 owner 进入执行计划。
- **需求分析与设计分工**：需求分析站在对象外部，规格化已分配功能的可见行为、输入输出、约束和验收；设计站在对象内部，说明结构、模块、接口、数据和运行机制如何更经济、更可靠地实现这些功能。
- **高层 / 详细设计分工**：高层设计说明系统对外做什么、边界和分配关系是什么；详细设计说明内部如何实现。两层都必须能追踪到需求、测试和发布证据，不把内部实现细节反写成需求。

#### 3.6.1 数据表结构设计模板

详细设计涉及新增或调整数据表时，按 `system-analysis-template.md` 第 4 节给出可编码、可评审、可生成 DDL 的表结构，不要只写“新增订单表”或“字段见代码”。普通项目先继承本地数据库规范；项目声明、依赖或源码上下文命中 Wind/Nobe 专项时执行 Wind 通用字段和新增字段默认值约规。评审只锁定以下工程事实：

- 表用途、数据归属、约规来源、主键策略、字段语义、必填性、默认值、类型精度、状态/金额口径和审计字段。
- 主键、数据库业务唯一键、查询索引及字段顺序必须回指业务规则和访问路径。
- 业务唯一约束必须使用数据库唯一键保证；不存在业务唯一性时明确说明，不用外部 `requestSn` 等请求标识冒充业务唯一键。
- Wind 表必须包含 `id`、`gmt_create`、`gmt_modified`；`creator`、`modifier`、`order_index` 按项目需要选择。
- 新增必填字段必须给兼容迁移方案；默认值、分阶段回填或暂时允许为空按真实业务语义和数据库变更能力选择，不得为了通过 DDL 虚构业务默认值。改名、改类型和删除字段必须有新旧版本兼容、回填、校验和回滚路径。
- 逻辑关联、数据所有权、一致性维护、归档清理、大字段风险和敏感数据边界必须明确。
- 表结构应能反向支撑 Entity、DTO、Query、Mapper、迁移脚本、测试数据和代码生成。

### 3.7 非功能设计

非功能设计默认必填；确实不适用时需要说明原因。

- **性能考虑**：QPS、RT、慢路径、热点、缓存、索引、批量处理、线程池和连接池。
- **容量考虑**：并发、CPU、内存、带宽、磁盘、数据库连接、消息积压和数据增长。
- **可用性考虑**：哪些情况会失败，如何发现，如何隔离，如何恢复，自动恢复还是人工恢复。
- **兼容性考虑**：API、DTO、消息、数据库、配置、新旧版本、老数据和外部依赖兼容。
- **生产就绪**：SLO、容量基线、灰度指标、回滚阈值、Runbook 和应急负责人；详细检查见 `production-readiness.md`。
- **技术风险**：可监控、可灰度、可应急必须说明；高风险项给出负责人和验证方式。
- **可运维性**：安装、部署、升级、排障、开关、日志、指标、链路、告警和 Runbook。
- **质量属性场景**：每项非功能都写明业务 driver、触发条件、受影响资产、期望响应、度量验收和取舍。

### 3.8 风险、验收摘要与执行交接

正式系分在末尾只保留人类评审需要确认的稳定结论：

- 本设计必须保护的业务结果、不变量、公共契约和质量目标。
- 关键边界、异常、红线、残余风险和待确认 owner。
- 实现 Spec / 执行计划的权威路径及适用版本。
- 详细验收矩阵、AC 与测试映射、失败测试、验证命令、执行 / 验证 owner、灰度、回滚、停止条件、排期和任务状态进入执行计划，不复制进正式系分。

### 3.9 参考资料

列出 PRD、接口文档、架构图、历史方案、ADR、外部规范、开源文档和相关工单。引用材料应能支撑方案判断，而不是堆链接。

## 4. 设计图要求

- 用例图：说明用户、角色、系统边界和可见能力。
- 上下文图：说明本系统与外部系统、内部平台、第三方依赖的关系。
- 组件图：说明模块职责、依赖方向和可替换边界。
- 流程图：说明主流程、异常流程、人工介入和回滚路径。
- 时序图：说明跨系统、消息、回调、任务和事务边界。
- 状态机图：说明状态、事件、守卫条件、动作和非法流转。

每张图必须有图前问题和图后结论。图中术语必须与代码、接口、数据库、测试和业务文档一致。

## 5. 评审清单

- 背景是否说明为什么做、影响谁、不做的风险是什么。
- 目标是否覆盖业务目标、技术目标、系统目标和非目标。
- 是否能从业务目标、参与方、核心行为和验收示例追踪到系统边界、模块、接口、数据、质量属性和测试。
- 系统需求、产品需求和外部规则是否形成可验证基线；未确认项、衍生需求和鲁棒性要求是否有 owner、影响和停止条件。
- 系统 DNA 是否可追踪：核心不变量、状态流转、责任边界、演化规则和验证方式是否已经从产品 DNA / PRD 映射到系分、测试、监控和 CR。
- 功能是否能追溯到上层目标或父对象功能分配，而不是头脑风暴功能点；需求分析是否只规格化外显行为，设计是否再解释内部结构和实现取舍。
- 概要设计是否明确边界、依赖、核心流程和关键取舍。
- 关键取舍是否有备选方案、放弃理由和复审条件；重大决策是否需要 ADR。
- 详细设计是否达到可编码状态，接口抽象与契约、数据、状态、异常、幂等和验证方向是否完整。
- 关键业务规则是否已说明工程承接和失败语义；多规则或高风险场景是否按需使用规则落地表。
- 数据表结构是否包含表中文名、表名、业务用途、数据归属、约规来源、主键策略、字段清单、必填性、默认值、变更类型、字段语义、枚举、金额单位、索引、唯一约束、迁移兼容、回滚和归档策略。
- Wind 表是否包含 `id`、`gmt_create`、`gmt_modified`；新增必填字段是否有默认值，否则是否保持可空。
- 表设计是否遵循项目数据库约规：命名与审计字段、数据库业务唯一键、外键策略、必填字段兼容、DDL 兼容发布、索引字段顺序和查询/排序场景。
- 一致性设计是否覆盖业务不变量、事务边界、幂等、去重、补偿、对账、告警和人工兜底。
- 遗留系统或迁移方案是否覆盖防腐层、契约测试、双写/回填/切流、灰度、回滚和下线条件。
- 安全设计是否覆盖认证、授权、租户隔离、敏感数据、密钥、审计和安全测试。
- 非功能是否覆盖性能、容量、可用性、兼容性、监控、灰度、应急和运维。
- 生产就绪是否覆盖 SLO、容量、压测、开关、监控、Runbook、回滚阈值和残余风险。
- 是否明确新旧版本、老数据、老接口、老客户端和外部依赖兼容策略。
- 是否有灰度、切流、开关、回滚、告警和人工兜底。
- 系分末尾是否只保留验收摘要、风险和执行计划链接；第一实施切片、写入范围、验证命令、执行 / 验证 owner、停止条件和易漂移排期是否进入执行计划。
- 参考资料是否足以支撑设计判断和后续追溯。
- 正式系分是否只保留当前有效设计、取舍、风险、待确认和验证；讨论过程、迭代草稿、AI 推理轨迹和被拒方案是否已经外置到评审报告、ADR、Decision Log、任务计划或中间任务文档。

## 6. 常见反模式

- 背景只写需求名称，不说明问题、影响和风险。
- 目标不可衡量，导致设计和编码无法判断是否偏航。
- 把功能当页面按钮、技术组件或头脑风暴清单；或在需求分析阶段过早展开内部结构，导致外部可见行为、主执行者和追溯关系不清。
- 概要设计只有组件名，没有边界、依赖和取舍；或没有系统 DNA 就进入详细设计，后续靠事故、补丁、配置和特殊判断沉淀规则。
- 详细设计没有接口、数据、状态机、错误码和测试场景。
- 表结构只给 DDL 或只给字段名，不说明字段语义、必填性、枚举、金额单位、索引依据和兼容迁移。
- 非功能设计写“无”，但实际涉及容量、兼容、上线、监控或回滚风险。
- 只写正常流程，不写异常流程、补偿、人工介入和应急开关。
- 把完整排期、第一实施切片、命令和任务状态复制进正式系分，挤占背景、目标、定性、方案、流程、规则和接口抽象的阅读主线。
- 把评审争论、迭代草稿、被拒方案和 AI 推理过程留在正式系分正文里，导致当前有效设计和历史过程混杂。

## 7. 与其他规范的关系

- `architecture.md`：定义架构职责、核心抽象、质量属性和架构评审关注点。
- `product-design.md`：定义产品目标、能力地图、业务规则和面向开发/测试的表达。
- `review-and-output-templates.md`：提供架构方案、实施计划、生产变更和兼容性治理模板。
- `production-readiness.md`：定义 SLO、容量、灰度、监控、数据变更、外部依赖、Runbook 和回滚要求。
- `adr-and-tradeoff.md`：定义重大决策和技术取舍的 ADR 结构。
- `distributed-consistency.md`：定义分布式一致性、消息、幂等、补偿和对账要求。
- `evolutionary-architecture.md`：定义遗留系统改造、迁移、切流和微服务拆分门槛。
- `refactoring-design-template.md`：定义独立重构设计准入、行为不变量、MIG 切片、迁移验证和旧能力退役。
- `security-architecture.md`：定义认证授权、租户隔离、敏感数据、密钥、审计和安全测试。
- `workflow.md`：定义代码修改、验证命令、PR 和 Git 流程。
- `testing.md`：定义如何把验收标准沉淀为测试资产。
