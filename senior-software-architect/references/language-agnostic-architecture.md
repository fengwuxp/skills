# 语言无关架构能力

本文定义资深架构师在任意语言、任意技术栈中都必须保持的判断框架。Java/Spring 是本技能的核心专长，但架构设计、系统分析、质量属性、工程治理和生产风险控制不应被单一语言限制。

## 1. 使用原则

- 用户未指定语言或仓库技术栈不明时，先使用本文件的语言无关原则，不强套 Java/Spring 规范。
- 识别到 Java、Spring、Maven、Gradle、MyBatis、Wind 项目族时，再加载 Java 专项规范。
- 识别到 Go、Node.js、Python、Rust、前端、数据工程等技术栈时，先尊重项目已有目录、构建、测试、lint、格式化和发布方式。
- 架构建议必须落到该技术栈能执行的模块边界、接口契约、测试命令、构建命令、监控指标和发布流程。

## 2. 技术栈识别

接手代码或方案时先识别：

- 语言与运行时：Java/JVM、Go、Node.js、Python、Rust、.NET、前端、数据平台等。
- 构建工具：Maven、Gradle、Go toolchain、npm/pnpm/yarn、Poetry、Cargo、Make、Bazel 等。
- 框架与边界：Web 框架、RPC 框架、ORM、消息、任务调度、配置、依赖注入、插件机制。
- 测试体系：单测、集成测试、契约测试、E2E、性能测试、架构测试。
- 质量工具：formatter、linter、static analysis、dependency scan、security scan。
- 运行形态：进程、容器、Serverless、Kubernetes、批处理、流式任务、边缘环境。

## 3. 架构设计共性

任何语言下，架构方案都必须说明：

- **目标与非目标**：解决什么问题，不解决什么问题。
- **边界**：业务边界、模块边界、服务边界、数据边界、安全边界和运维边界。
- **依赖方向**：核心规则不依赖外部细节，外部系统通过端口/接口/适配器隔离。
- **接口契约**：输入、输出、错误、权限、幂等、分页、兼容、版本和废弃策略。
- **数据模型**：核心实体、状态、业务不变量、唯一约束、审计字段和迁移策略。
- **流程与状态**：主流程、异常流程、补偿流程、人工介入和非法状态处理。
- **质量属性**：可靠性、性能、安全、可观测性、可测试性、可运维性和可演进性。
- **取舍**：至少说明一个备选方案及放弃原因。

## 4. 代码与模块治理共性

- 模块、包、目录、命名必须表达业务语义和变化边界。
- 公共 API、SDK、库、组件、消息和数据库结构必须有兼容策略。
- 业务规则不应散落在 Controller、Handler、Resolver、Job、Mapper、DAO、ORM Hook 或脚本里。
- 第三方 SDK、外部服务、数据库、缓存、消息、文件系统应通过适配层隔离。
- 新依赖必须说明必要性、替代方案、维护责任、许可证和安全风险。
- 复杂转换、状态流转和规则判断必须可测试，不依赖隐式约定。
- 重构必须小步、行为保持、可验证，不借修 Bug 做大范围无关改动。

## 5. 数据与一致性共性

- 金额、库存、账户、权限、状态机、审批、审计等核心数据必须定义不变量。
- 事务边界必须清楚：本地事务、分布式事务、最终一致、补偿、对账分别适用于什么场景。
- 最终一致不能成为无重试、无补偿、无对账、无告警的借口。
- 数据迁移和修数必须有 dry-run、备份、分批、审计、回滚和结果校验。
- 缓存必须说明读写路径、失效策略、一致性窗口、穿透/击穿/雪崩和热点处理。
- 消息必须说明幂等键、顺序性、重复消费、死信、重放、积压和人工兜底。

## 6. 可靠性、安全与可观测性

- 外部调用必须设置超时、重试、熔断、限流、降级和告警。
- 异步、批处理、定时任务必须考虑失败重试、幂等、补偿和可恢复性。
- 权限、租户隔离、敏感数据、密钥、审计和合规要求必须在设计和测试中体现。
- 日志、指标、链路追踪和告警必须覆盖关键业务路径、失败路径和容量风险。
- 发布必须考虑灰度、开关、回滚、兼容、新旧版本共存和数据回退限制。

## 7. 验证与交付

任何语言下，交付前都要按项目生态选择验证命令：

- 编译或类型检查。
- 与修改相关的单元测试。
- 相关集成测试、契约测试或 E2E。
- formatter、linter、static analysis。
- 依赖、许可证和漏洞扫描。
- 高风险场景的手工验收、压测、演练或灰度观察。

最终回复必须说明已执行的验证，或说明未执行原因和替代验证。

## 8. 跨语言适配表

| 技术栈 | 优先识别 | 常见验证 |
| --- | --- | --- |
| Java/Spring | `pom.xml`、`build.gradle`、Spring Boot、JVM、MyBatis | `mvn test`、`./gradlew test`、P3C/PMD/SpotBugs、ArchUnit |
| Go | `go.mod`、package 边界、interface、context、goroutine | `go test ./...`、`go vet ./...`、`gofmt`、`golangci-lint` |
| Node.js/TypeScript | `package.json`、tsconfig、framework、package manager | `npm/pnpm/yarn test`、`typecheck`、`lint`、`build` |
| Python | `pyproject.toml`、requirements、venv、framework | `pytest`、`ruff`、`mypy/pyright`、`build` |
| Rust | `Cargo.toml`、crate 边界、trait、ownership | `cargo test`、`cargo clippy`、`cargo fmt` |
| 前端 | Vite/Webpack/Next、组件边界、状态管理、API 契约 | `test`、`typecheck`、`lint`、`build`、E2E |
| 数据工程 | DAG、SQL、数据契约、质量规则、血缘 | SQL 校验、数据质量测试、回填 dry-run、任务重跑 |

表中命令只是示例，最终以项目已有脚本和 CI 为准。

## 9. 反模式

- 未识别项目技术栈，就把 Java/Spring/Maven 规范强套到其他语言。
- 只按文件夹或框架层级画架构，不说明业务边界和数据归属。
- 只给技术选型，不给失败路径、验证方案、发布回滚和运维策略。
- 以“语言特性很强”为理由忽视测试、边界、兼容和可观测性。
- 在不同语言项目中复用同一套命名、目录和工具规则，破坏本地生态习惯。
