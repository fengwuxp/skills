# 调试诊断与根因分析

本文用于 Bug 修复、复杂异常、测试失败、生产现象分析、交易链路排障和 AI 生成代码导致的异常定位。目标不是凭经验猜修复，而是先建立可重复反馈环，再用证据收敛根因，最后用回归测试保护修复。

## 使用时机

- 用户贴出异常、堆栈、日志、失败测试、错误截图或线上现象。
- 问题涉及 Java/Spring、事务、AOP、数据库、MQ、缓存、外部端口、异步任务、状态机、账务投影或路由回放。
- 代码“看起来能跑”，但行为与产品描述、测试预期、接口契约或生产事实不一致。
- AI 生成代码后出现幻觉 API、越界修改、缺失测试、过度 mock 或局部修复导致新问题。

低风险的一行文案、纯格式问题、用户只要求概念解释时，不必进入完整诊断流程。

## 不适用场景

- 没有错误现象、失败反馈或行为偏差，只是在做普通架构设计或代码风格 Review。
- 用户只要求知识解释，不需要定位、修复或回归保护。

## 读取后必须产出

- 现象界定、可重复反馈环、最小复现和可证伪假设。
- 证据采集方式、最小修复范围和回归测试形态。
- 无法自动化时的人工验证步骤、观察证据和残余风险。

## 需要继续读取的 reference

- 回归测试形态读 `testing.md`；Java/Spring 修复读 `coding-standards.md`。
- 生产现象、数据修复或上线风险读 `production-readiness.md` 和 `negative-constraints.md`。
- AI 参与修复读 `ai-assisted-engineering.md`。

## 诊断闭环

```text
现象界定
-> 可重复反馈环
-> 最小复现
-> 假设列表
-> 证据采集
-> 最小修复
-> 回归测试
-> 复盘沉淀
```

1. **现象界定**：说明实际现象、期望行为、影响范围、首次出现时间、最近变更和已知非目标。
2. **可重复反馈环**：优先找到能稳定失败的测试、命令、请求、脚本、日志查询或本地复现步骤。
3. **最小复现**：把输入、前置状态、触发动作和断言事实缩到最小，不一开始重跑全量系统。
4. **假设列表**：列出 2-4 个可验证假设，按可能性、风险和验证成本排序。
5. **证据采集**：用日志、断点、临时插桩、数据库事实、消息记录、调用链、测试断言或差异 diff 验证假设。
6. **最小修复**：只修改根因相关代码，不借修 Bug 做无关重构、格式化或依赖升级。
7. **回归测试**：先让复现测试失败，再修复通过；无法自动化时说明人工验证步骤和剩余风险。
8. **复盘沉淀**：高风险问题沉淀为测试、ADR、Runbook、监控告警或项目规范，而不是只写一句“已修复”。

## 反馈环优先级

| 问题类型 | 优先反馈环 | 证据重点 |
| --- | --- | --- |
| 纯函数、规则、转换错误 | 单元测试或转换测试 | 输入、输出、边界、异常语义。 |
| 应用服务、事务、状态流转 | 用例流程测试或最小 Spring 上下文 | 事务边界、状态变化、Repository 事实、副作用。 |
| SQL、Mapper、Repository | 数据库/事务测试 | SQL 语义、唯一约束、锁、索引、历史数据。 |
| Controller/API 契约 | MockMvc、契约测试或请求样例 | 参数绑定、校验、错误码、权限、兼容性。 |
| MQ、异步任务、回调 | 幂等/重试/补偿测试，消息记录 | 去重键、重试次数、死信、补偿、乱序。 |
| 外部端口或第三方 SDK | Fake/WireMock/Recording | 协议映射、超时、错误码、重试、降级。 |
| 生产现象 | 日志、指标、链路、数据库只读查询 | 时间线、影响面、异常分布、最近变更。 |

如果没有可运行反馈环，先构造最小 characterization test 或复现脚本；仍无法自动化时，必须明确“无法自动化的原因、人工复现步骤、观察证据和残余风险”。

## 假设与插桩规则

- 假设必须可证伪：能通过一个测试、日志、查询、断点或差异对比证明对错。
- 插桩要短命、可删除、低噪声；不得提交临时 `System.out.println`、敏感日志或大范围 debug 输出。
- 日志必须带关键业务上下文，例如订单号、脱敏或摘要后的账户标识、租户、状态、幂等键、消息 ID，但不得输出密码、token、证件、银行卡、账户号、密钥等敏感信息原文。
- 事务问题优先查看异常是否被吞、回滚规则、事务传播、AOP 代理是否生效、事务内是否有远程调用。
- 并发和幂等问题优先查看唯一约束、锁粒度、状态前置条件、重复消息、乱序回调和重试窗口。
- 资金、权限、审计问题必须同时检查业务事实、持久化事实、审计事实和回滚/补偿事实。

## 修复原则

1. **先复现再修复**：除非是编译错误或确定性 typo，否则先保留一个失败反馈环。
2. **根因优先**：修根因，不只绕过症状；临时止血必须标明后续永久修复。
3. **最小修改**：只改与根因、验证和必要防护直接相关的代码。
4. **保持契约**：不得为了让测试通过改变公共 API、DTO、错误码、枚举、数据库字段或消息语义，除非已有兼容方案。
5. **失败路径补齐**：涉及异常、事务、重试、补偿、幂等、权限和审计时，修复必须覆盖失败路径。
6. **验证可解释**：最终说明复现方式、根因、修改点、验证命令、未覆盖风险和是否需要监控/Runbook。

## 输出模板

只读诊断时：

```text
现象：实际行为、期望行为、影响范围。
最可能根因：按证据排序，不超过 3 个。
证据：日志、堆栈、代码路径、数据事实或测试结果。
建议验证：下一步最小复现或验证命令。
风险：如果判断错误或直接修复会带来的风险。
```

修复交付时：

```text
复现：使用了什么测试、命令、请求或日志确认问题。
根因：问题发生在哪个边界、契约、状态、事务或依赖上。
修复：修改了什么，为什么是最小修复。
回归：新增或调整了哪些测试，覆盖哪些失败路径。
验证：运行的命令和结果。
残余风险：未能验证的外部依赖、生产数据或人工确认项。
```

## 红线

- 不得跳过复现直接大范围重构。
- 不得把不稳定测试简单删除、放宽断言或标记忽略。
- 不得只根据单条日志或单次失败做确定性结论。
- 不得提交临时插桩、裸日志、敏感信息或伪造验证结果。
- 不得用 mock 掩盖内部业务链路、事务、Repository、状态机或账务事实。
- 不得把“这可能是缓存/并发/网络问题”当作根因，除非有证据闭环。
