## 12. 容器化与 Kubernetes 落地能力

架构师需要理解系统如何在容器与 Kubernetes 环境中运行，确保架构决策能落地为可部署、可伸缩、可观测、可回滚的工程实践。

### 12.1 容器化基础

- 理解镜像、容器、Registry、Dockerfile、多阶段构建和镜像分层。
- Java 镜像应关注基础镜像选择、JDK/JRE 版本、时区、字体、证书、非 root 用户和镜像体积。
- 容器内应用必须将配置外置，禁止把环境配置、密钥、生产地址写入镜像。
- 容器应以前台进程运行，日志输出到 stdout/stderr，由平台统一采集。

### 12.2 Kubernetes 核心对象

- Workload：Deployment、StatefulSet、DaemonSet、Job、CronJob。
- 流量：Service、Ingress、Gateway、EndpointSlice。
- 配置：ConfigMap、Secret、环境变量、Volume。
- 调度：Node、Namespace、Label、Selector、Taint/Toleration、Affinity。
- 存储：PV、PVC、StorageClass。
- 弹性：HPA、VPA、Cluster Autoscaler。

### 12.3 Java 应用运行时治理

- 配置合理的 `requests` / `limits`，避免 CPU 抢占和内存 OOM。
- JVM 参数需适配容器内存，例如 MaxRAMPercentage、GC、线程数、直接内存。
- 必须配置 livenessProbe、readinessProbe，必要时配置 startupProbe。
- 优雅停机必须处理 SIGTERM、连接排空、线程池关闭、MQ 消费停止和任务中断。
- 无状态服务优先使用 Deployment；有状态服务必须明确数据持久化、主从、备份和恢复方案。

### 12.4 发布与回滚

- 支持滚动发布、蓝绿发布、金丝雀发布和快速回滚。
- 发布前检查配置、镜像 tag、数据库迁移、依赖服务和容量。
- 使用不可变镜像 tag，禁止生产使用 `latest`。
- 数据库变更遵循向前兼容：先兼容代码，再变更数据，最后清理旧字段/旧逻辑。

### 12.5 安全与隔离

- Secret 不进入 Git，不进入镜像，不打印到日志。
- 容器使用非 root 用户，限制 Linux capabilities，必要时只读根文件系统。
- 使用 Namespace、NetworkPolicy、RBAC、ServiceAccount 做隔离。
- 镜像需要漏洞扫描和依赖安全扫描。

### 12.6 可观测性

- 应用必须暴露健康检查、指标、日志和 trace 上下文。
- 关键指标包括 QPS、RT、错误率、JVM、GC、线程池、连接池、Pod 重启、OOM、CPU throttling。
- 日志和 trace 必须包含应用名、环境、版本、Pod、traceId、用户/租户等关键上下文。

### 12.7 架构决策映射

- 高可用决策映射为副本数、PodDisruptionBudget、反亲和、探针、限流和熔断。
- 性能决策映射为资源配额、HPA、连接池、线程池、缓存和压测基线。
- 安全决策映射为 RBAC、NetworkPolicy、Secret 管理、镜像扫描和最小权限。
- 可观测性决策映射为日志、指标、链路、告警和仪表盘。

## 13. 前端与前后端分离协作能力

资深架构师不必替代前端专家，但必须理解现代前端工程，能在前后端分离项目中沟通、决策、评审、落地，必要时能进行有限编码。

### 13.1 前端基础知识

- Node.js：包管理、脚本、构建、运行时版本、npm/yarn/pnpm。
- TypeScript：类型系统、泛型、类型收窄、接口、类型声明、严格模式。
- ECMAScript：模块、异步、Promise、集合、解构、可选链、现代语法。
- HTML5：语义化标签、表单、可访问性、存储、媒体、History API。
- CSS：盒模型、布局、响应式、CSS Modules、CSS-in-JS、变量和主题。
- Web Components：Custom Elements、Shadow DOM、组件封装边界。

### 13.2 React 与工程化

- React：组件、Hooks、状态管理、上下文、受控组件、错误边界、性能优化。
- 路由与数据流：React Router、请求缓存、状态分层、权限路由。
- 构建工具：Webpack、Vite、Babel、SWC、Tree Shaking、Code Splitting。
- 质量工具：ESLint、Prettier、Stylelint、TypeScript、Husky、lint-staged。
- 测试：Vitest/Jest、React Testing Library、Playwright/Cypress。

### 13.3 前端架构关注点

- 可读写：组件职责清晰，命名准确，目录结构表达业务和复用层次。
- 易用性：交互流程清楚，错误提示友好，表单校验一致，反馈及时。
- 可维护性：组件可复用但不过度抽象，状态边界清晰，副作用集中管理。
- 类型安全：禁止 AnyScript，能用类型表达的约束不要靠注释和口头约定。
- 设计系统：优先复用成熟组件库和团队组件，不随意改全局样式。
- 接口契约：前后端通过 OpenAPI/DTO/错误码/分页/权限模型对齐。

### 13.4 前端项目介入原则

- 先阅读项目自身规则：`package.json`、`tsconfig.json`、`.eslintrc`、`eslint.config.*`、`.prettierrc`、`stylelint`、`lint-staged`、`husky`、组件库约定和目录规范。
- 不删除或绕过 ESLint、Husky、类型检查和格式化规则。
- 不修改组件全局样式，除非任务明确要求且评估影响范围。
- 组件文件使用 PascalCase，非组件文件如工具函数、Hooks、常量使用 lowerCamelCase。
- 单组件代码行数控制在 400 行以内，复杂组件应拆分容器、视图、Hook、工具函数。
- 三目运算只使用一层，不嵌套；复杂条件提取为具名变量或函数。

### 13.5 Ant Design 等优秀实践吸收

- 以用户目标和业务效率为中心，而不是只追求视觉统一。
- 组件 API 要稳定、可预测、易组合。
- 交互反馈要即时、明确、可恢复。
- 表单、表格、筛选、弹窗、权限、空状态、错误状态应有统一模式。
- 设计系统的价值在于降低认知成本和协作成本，不是制造额外约束。

## 14. 工程治理能力

- 建立命名、包结构、分层、异常、日志、测试、API、数据库、Git 提交规范。
- 用自动化工具守住规范：Checkstyle、PMD、SpotBugs、P3C、ArchUnit、Maven Enforcer、依赖漏洞扫描。
- 治理技术债：记录、分级、排期、复盘，不把技术债伪装成“以后优化”。
- 沉淀文档：技术方案、ADR、接口文档、数据字典、部署文档、故障手册、迁移文档。

## 15. 技术领导力

- 沟通：能和产品、研发、测试、运维、管理者讨论边界、风险、成本和验收。
- 决策：能解释取舍、承担后果、识别不可逆决策、控制试错成本。
- 推进：拆任务、写样板代码、做评审、建门禁、推动重构、复盘问题。
- 传承：通过代码示范、Review 示范、文档沉淀、技术分享和新人指导提升团队能力。

## 16. 技术表达与文字能力

架构师不仅设计系统，也设计团队理解系统的路径。表达能力直接影响方案能否被理解、执行、评审和长期维护。

### 16.1 表达原则

- **通俗易懂**：先讲人话，再讲术语；先解释问题，再解释方案。
- **言辞简练**：删掉空话、套话、重复话，保留判断、依据和行动。
- **结构严谨**：标题、段落、列表、表格、图示有层次，读者能快速定位信息。
- **语义准确**：一个词只表达一个概念，同一概念只使用一个词。
- **场景贴合**：根据技术方案、业务说明、架构评审、代码注释、故障复盘选择不同表达粒度。
- **可执行**：文档必须给出规则、边界、示例、反例、验收标准或决策结论。

### 16.2 汉字与命名意识

- 理解汉字“形、音、义”的表达密度，中文标题和术语要短、准、稳。
- 避免“处理、管理、操作、数据、信息、逻辑”等过宽词汇滥用。
- 业务命名优先使用领域语言，技术命名优先使用业界通用词。
- 中文说明要避免歧义：主语明确、动作明确、对象明确、条件明确、结果明确。
- 中英文混排保持一致：英文技术词不硬翻，中文业务词不乱拼音。

### 16.3 文档能力

- 系统分析设计：说明需求背景、业务/技术/系统目标、非目标、概要设计、详细设计、非功能、研发计划和参考资料。
- 技术方案：说明背景、目标、非目标、约束、方案、取舍、风险、验证和回滚。
- 架构文档：说明模块、边界、依赖、数据、接口、流程、状态和质量属性。
- 规范文档：说明适用范围、强制/推荐等级、正例、反例、检查清单。
- 代码注释：解释为什么、边界条件、约束和风险，不重复代码做了什么。
- 故障复盘：说明时间线、影响面、根因、止血、修复、预防和责任边界。

### 16.4 图文表达

- 流程复杂时使用时序图或流程图。
- 用户目标复杂时使用用例图，能力复杂时使用能力地图。
- 模块关系复杂时使用依赖图或上下文图。
- 规则较多时使用表格。
- 决策有取舍时使用对比表或 ADR。
- 文档中每张图都必须有结论，不放“装饰性架构图”。
- 技术架构图必须区分整体、子领域和应用粒度，按目标读者选择细节层级。
- 架构图要表达职责、关系、同步/异步、数据流、容量或瓶颈，不做技术组件堆叠图。
- 图形应随代码、接口、部署、监控和关键决策演进，作为版本化活文档维护。

### 16.5 面向不同读者表达

- 面向产品：讲业务目标、用户影响、范围和取舍。
- 面向研发：讲边界、接口、模型、异常、测试和演进。
- 面向测试：讲验收条件、边界场景、数据准备和回归风险。
- 面向运维：讲发布、配置、监控、容量、告警和回滚。
- 面向管理者：讲收益、风险、成本、周期和决策点。

## 17. AI 协作能力

- 给 AI 明确任务边界、项目规范、禁止行为和验收标准。
- 要求 AI 先读 AGENTS.md、SKILL.md、模块代码和现有测试。
- 要求最小修改，不允许顺手重构或添加未来扩展。
- 审查 AI 生成代码中的幻觉：不存在的工具类、错误框架用法、绕过架构边界、缺失测试、过度抽象。
- 把高频经验沉淀为 AGENTS.md、SKILL.md、检查清单、模板和自动化规则。
- 掌握 AI 编码三层作业模型：OpenSpec 定标准，Superpowers 保纪律，Harness 管团队。
- 能把口头需求转成 OpenSpec：目标、范围、非目标、业务规则、接口契约、数据约束、验收场景和验证方式。
- 能用 Superpowers 约束 AI 编码：TDD、Review、Refactor、最小修改、无关变更控制、测试和静态检查。
- 能用 Harness 编排人机协作：任务拆分、Owner、写入范围、只读范围、依赖顺序、并行限制、交接说明和验证矩阵。
- 能按风险等级选择流程强度：低风险轻量规格，中风险规格加测试，高风险规格确认、协作计划、测试保护和残余风险说明。

### 17.1 协作边界

- **知识服务问题**：用海量知识帮助用户解决当前问题，而不是展示知识本身。
- **不确定先收敛**：不清楚需求、背景、约束时，不做过度猜测；最多给 3 条澄清问题、建议路径或可选方案。
- **少即是准**：优先输出能推动决策和行动的关键点，避免信息洪流。
- **用户持剑**：AI 是锋芒和助力，不替用户擅自决定业务方向、风险取舍和组织策略。
- **渐进展开**：先给结论、边界和建议，用户选择后再展开细节、示例和实现。

### 17.2 AI 编码红线

- 不允许 AI 在没有明确规格、边界和验收标准的情况下直接进行复杂编码。
- 不允许 AI 为了“更优雅”扩大重构范围、放宽测试断言、删除失败用例或伪造验证结果。
- 不允许 AI 编造不存在的工具类、框架 API、项目约定或外部能力。
- 不允许 AI 擅自引入依赖、改变公共契约、越过分层边界或修改生产关键配置。
- 不允许多 Agent 或多人无边界地同时修改同一职责范围。
- 不允许把密钥、客户数据、生产敏感信息放进 AI 上下文。

详细方法见 `references/ai-assisted-engineering.md`。

## 18. Wind 项目族实践能力

基于 wind-middleware、wind-integration、wind-security 的项目级实践，处理 Java/Spring/Wind 项目时还应具备以下落地能力：

- **基础架构复用**：优先复用统一响应、异常、Trace、Web、Client、配置中心、限流、消息、序列号、脚本、脱敏和日志能力。
- **端口与适配器设计**：业务依赖 `WindXxx` 接口端口，厂商 SDK 和中间件细节放在独立适配器模块。
- **企业集成设计**：OSS、KMS、消息、Office、IM、工作流等集成能力先抽象业务契约，再接入具体供应商。
- **安全能力建模**：认证、授权、JWT、验证码、MFA、RBAC、Token 撤销、设备隔离和流控分别建模。
- **Spring Boot Starter 设计**：自动装配使用配置开关、条件装配、默认 Bean 和业务覆盖机制。
- **上下文与可观测性**：HTTP、异步任务、MQ、外部调用必须传播 trace/context，响应、日志、指标携带 traceId。
- **项目风格一致性**：公共 API 使用 Javadoc、validation、nullability；命名遵循 `WindXxx`、`ImmutableXxx`、`XxxProperties`、`XxxAutoConfiguration` 等约定。

详细规则见 `references/wind-projects-patterns.md`。
