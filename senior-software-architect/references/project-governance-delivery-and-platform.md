## 14. 前端项目协作规范

架构师介入前端项目时，目标是帮助前后端形成清晰契约、稳定工程边界和可维护实现，不用后端习惯替代前端实践。

介入前必读：

- `package.json`、Node 版本、包管理器和脚本。
- `tsconfig.json`、ESLint、Prettier、Stylelint、Husky/lint-staged。
- 项目 README、目录规范、组件库、路由、权限和状态管理约定。

强制规则：

- 不删除或绕过 ESLint、Husky、类型检查和格式化配置。
- 不用 `any` 逃避类型设计；确需使用必须说明原因并收敛作用域。
- 不在不了解影响范围时修改全局样式。
- API 响应、表单模型、路由参数、组件 Props 必须有类型。
- 必须处理 loading、empty、error、permission denied、timeout 等状态。
- 前后端字段、错误码、分页、排序、时间格式和 traceId 链路必须一致。

## 15. Git 协作与 PR

- 禁止 force push 到共享分支。
- 解决冲突时不得覆盖他人代码；不确定必须找相关开发一起处理。
- Commit 推荐 `<type>(<scope>): <subject>`，AI 参与时可追加 `Assisted-by`。
- PR 要小而完整，说明背景、改动点、验证方式、风险和回滚方案。
- 大量 DTO/生成代码、迁移、重命名或格式化必须单独说明。
- 合并前必须通过编译、相关测试和静态检查。

更细的提交、PR、验证和外部知识时效性要求读 `workflow.md`。

## 16. 代码评审与设计评审

设计评审看：

- 模块边界、依赖方向、数据模型、状态流转和审计是否清楚。
- API 是否暴露 Entity 或内部实现。
- 外部调用是否考虑超时、重试、幂等、限流和补偿。
- 日志、指标、traceId、测试和验收标准是否明确。
- 背景、目标、非目标、约束、取舍、风险和回滚是否可评审。

代码评审看：

- 是否有无关改动或顺手重构。
- `get/find/query` 是否误用，Service 是否直接访问 Mapper。
- Controller 是否写业务逻辑，API 是否暴露 Entity。
- 异常、日志、敏感信息和空值契约是否符合规则。
- 测试是否覆盖正常、异常、边界场景。
- 命名、注释和 Javadoc 是否准确表达业务语义。

深度代码 Review 读 `coding-review-deep-dive.md`。

## 17. 重构与演进

- 重构必须行为保持，不和功能变更混在一个 PR。
- 重构前先补测试或确认已有测试保护。
- 每次只改一个维度：命名、拆方法、拆类、移动包、解耦依赖。
- 大规模重构必须先写设计说明和迁移计划。
- 旧接口废弃必须使用 `@Deprecated` 并说明替代方案和删除版本。

推荐演进路线：

```text
单体 -> 分层单体 -> 模块化单体 -> 关键模块服务化 -> 平台化能力
```

不要一开始就微服务化；只有团队边界、发布边界、伸缩边界、数据边界都清晰时，才拆服务。

## 18. 容器化与 Kubernetes 落地规范

Kubernetes 规范用于保证架构决策能稳定运行，而不是只停留在设计文档中。

- 镜像必须不可变、可追溯、无密钥，生产镜像禁止使用 `latest`。
- 配置和 Secret 必须外置且可回滚；密钥禁止进入 Git、镜像和日志。
- 必须配置 CPU/Memory `requests` 与 `limits`，JVM 参数要适配容器资源限制。
- 探针要区分启动、就绪和存活；Java 启动慢时考虑 startupProbe。
- 优雅停机必须处理 SIGTERM、请求排空、线程池关闭、MQ 消费停止和定时任务中断。
- 发布时必须确认新旧版本对数据库、缓存、消息和外部接口兼容。
- 关键服务考虑副本数、PDB、反亲和、HPA、日志、指标、trace、告警和仪表盘。

生产发布、回滚、容量和 Runbook 读 `production-readiness.md`。

## 19. SOFAStack 参考实践吸收

SOFAStack 只作为模块化、分层插件化和类隔离思想参考，不作为默认技术选型。

- 先把模块边界、依赖方向、接口契约和测试门禁做好。
- 公共能力优先做成内部 starter 或可插拔模块。
- 厂商 SDK、外部系统、基础设施能力必须建立端口与适配器。
- 普通 Maven 模块无法解决依赖冲突或发布隔离时，再评估 SOFAArk 类隔离/插件化模型。

## 20. 最小落地门禁

每个项目至少具备：

- 编译通过。
- 相关单测或回归验证通过。
- 接口、实体、注解、重命名类变更后执行测试编译。
- P3C、PMD、SpotBugs、SonarLint 或项目等价静态检查至少一种。
- PR 说明改动、验证、风险和回滚。
- 禁止项扫描：`System.out.println`、`e.printStackTrace()`、密钥、token、`.env`。
- 高风险边界逐步引入 ArchUnit 或等价架构检查。

## 21. 系统分析设计门禁

新增核心业务能力、跨模块能力、公共平台能力、中间件/SDK 能力，或涉及数据模型、状态机、权限、安全、资金、审计、灰度、回滚等高风险变更时，必须先完成系统分析设计。

最低要求：

- 背景清楚：为什么做、影响谁、不做有什么风险。
- 目标清楚：区分业务目标、技术目标、系统目标和非目标。
- 概要设计清楚：系统边界、依赖关系、主体流程和关键取舍明确。
- 详细设计可编码：接口、数据、状态、异常、幂等、事务、安全和测试设计完整。
- 非功能可落地：性能、容量、可用、兼容、监控、灰度、应急和运维都有结论。
- 研发计划可协作：负责人、时间点、联调、提测、预发、生产发布、验收和回滚窗口明确。

详细写法见 `system-analysis-design.md`；低风险变更也必须在 PR 中说明背景、影响、验证和回滚。
