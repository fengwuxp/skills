# 项目治理：代码库、模块与依赖

本文只承载代码库类型、模块职责、依赖方向和演进边界的架构裁决，不保存语言、框架或 Wind 编码细则。

## 使用时机

- 需要判断代码库类型、治理强度、模块职责、依赖方向或第三方能力边界。
- 需要评审模块拆分、公共能力归属、循环依赖、跨层调用或新增依赖。

## 不适用场景

- 纯 Java/Wind 约规检查交给 `wind-coding-conventions`。
- 单点源码 CR 先读 `coding-review-deep-dive.md`；这里只在发现模块或依赖边界问题时补充读取。

## 读取后必须产出

- 当前代码库类型、模块职责、允许与禁止的依赖方向、变更理由和验证方式。

## 需要继续读取的 reference

- 服务、API 与模型边界读 `project-governance-service-api-modeling.md`。
- 数据、安全和质量边界读 `project-governance-data-security-quality.md`。
- 具体 Java/Wind 规则读取 `wind-coding-conventions` 的通用层与按证据启用的专项层。

## 代码库类型与治理强度

先判断代码库承担什么责任，再决定治理强度：

| 类型 | 首要目标 | 架构裁决重点 |
| --- | --- | --- |
| 基础设施 / 框架 | 稳定、兼容、低侵入 | 公共契约、默认行为、扩展边界、故障隔离 |
| 公共业务能力 | 语义稳定、边界清楚 | 领域归属、数据责任、跨团队兼容、演进成本 |
| 业务应用 | 正确交付、可维护、可发布 | 用例闭环、模块职责、数据边界、验证和回滚 |
| 实验 / 工具 | 快速验证、风险隔离 | 写入范围、数据安全、停用条件、不污染主工程 |

## 模块与依赖裁决

Clean Architecture 的落点是依赖规则，不是机械四层模板。层数可变，依赖方向不可逆：核心业务规则与用例定义稳定契约和端口，外层实现数据库、Web、消息、缓存与第三方适配。

评审时回答：

1. 模块是否对应稳定业务能力、用例责任或基础设施适配，而不是按表或技术名机械拆分。
2. 核心规则是否依赖外层实现细节；若是，端口应由哪一侧定义。
3. 数据所有权、事务责任、发布责任和故障责任是否落在同一可管理边界。
4. 拆分是否降低理解和变更成本，还是只增加调用、配置和发布节点。
5. 公共能力是否有真实调用方、兼容责任和维护 owner。

循环依赖、反向依赖、跨边界直接访问持久化实现或业务代码直连第三方实现，应按影响到的业务不变量、测试隔离和发布风险定级，不靠目录名机械裁决。

## 依赖准入

新增依赖前必须有当前需求、已有能力对照、版本与安全风险、替代路径、退出方式和维护 owner。架构师判断新增节点是否值得，具体语言依赖写法和编码条目回到项目本地规范与规则 Skill。

## 评审清单

- 模块名和职责是否表达业务边界或适配责任。
- 依赖方向是否让核心规则脱离外层细节独立测试。
- 数据、事务、权限、故障和发布责任是否一致。
- 新增模块或依赖是否降低整体复杂度，并有 owner、验证与退出路径。
- 结论是否有源码锚点、依赖图、测试或构建证据。
