# 代码整洁原则与评审启发

本文档用于沉淀 Clean Code 的核心判断原则，帮助架构师在阅读、实现、重构和评审代码时保持一致的审美与尺度。它不替代项目级强制规范；具体目录、框架、测试、日志和数据库细则以 `project-governance-standards.md`、`coding-standards.md` 为准。

## 使用时机

- 用户要求代码整洁、重构审美、可读性、命名、函数、类、模块或 Review 启发。
- 代码能运行但业务表达、边界、测试、错误处理或长期维护性存在疑虑。
- 需要在强规约之外，用 Clean Code / Clean Architecture 原则做解释性 Review。

## 不适用场景

- Java/Spring/Wind 强制编码规则、JSpecify、DTO 字段、MapStruct、查询和持久化约规优先读 `coding-standards.md`。
- 架构坏味、问题定级和 Review 输出优先读 `coding-review-deep-dive.md`。
- 生产风险、安全、测试专项或外部依赖变更先回到对应专项 reference。

## 读取后必须产出

- 可读性、职责、边界、错误处理、测试或演进问题的具体判断。
- 建议必须能落到命名、函数、类、模块、测试或小步重构，不输出空泛审美意见。
- 区分强制问题和启发式建议，避免把个人偏好包装成团队规约。

## 需要继续读取的 reference

- 编码强规约读 `coding-standards.md`。
- 深度 Review 和问题定级读 `coding-review-deep-dive.md`。
- 测试资产治理读 `testing.md`；架构原则读 `architecture.md`。

## 1. 基本立场

- **代码首先写给人看**：机器只需要代码能运行，人需要代码能理解、修改和演进。
- **清晰优先于炫技**：能直接表达业务意图的写法，优先于复杂、隐晦或过度抽象的写法。
- **简单不是简陋**：简单意味着职责明确、依赖可控、路径短；不是省略边界、校验和测试。
- **重构服务演进**：重构必须保持行为不变，并在测试、编译或评审保护下小步推进。
- **一致降低成本**：命名、分层、异常、日志、模型和测试风格保持一致，团队才能共享和接力维护。

## 2. 架构整洁之道

整洁代码关注局部表达，整洁架构关注系统边界。二者目标一致：让核心业务在变化中保持清晰、稳定和可测试。

- **业务核心独立**：实体、用例、领域规则不依赖 Web、数据库、消息、缓存、框架和第三方 SDK。
- **依赖方向向内**：外层可以依赖内层，内层不依赖外层；接口定义在更靠近业务意图的一侧。
- **框架是工具，不是中心**：Spring、ORM、MQ、Redis、K8s 等都是实现细节，不应主导领域模型。
- **用例表达业务流程**：ApplicationService 或 Use Case 负责组织一次业务动作，领域对象和领域服务负责规则本身。
- **边界通过模型表达**：跨层、跨模块、跨系统使用 DTO、Command、Query、Event 等契约模型，不泄露内部 Entity。
- **可测试性反推设计**：核心规则应能脱离数据库、容器和远程服务单独测试；难测试通常意味着边界不清。
- **插件化看待外部细节**：数据库、搜索、消息、文件、支付、短信等外部能力应通过 Gateway、Repository、Adapter 隔离。
- **架构服务演进**：不要为了形式套分层，也不要让短期便利穿透边界；所有边界都应服务长期可读、可改、可替换。

## 3. 命名

好的命名应让读者在不看实现的情况下理解“这是什么、做什么、为什么存在”。

- **名副其实**：名称直接表达业务或技术意图，避免靠注释解释真实含义。
- **避免误导**：不要用 `accountList` 表达非 List 类型，不要用 `manager`、`data`、`info` 掩盖职责。
- **同一概念一个词**：同一抽象层次统一术语，避免 `fetch`、`retrieve`、`query` 混用。
- **领域语言优先**：业务概念使用团队统一语言，技术概念使用业界通用词。
- **作用域越大，名称越完整**：局部变量可短，公共 API、领域对象、配置项必须清晰完整。

## 4. 函数

函数是最小的行为表达单元。函数越清晰，系统越容易被阅读、测试和重构。

- **只做一件事**：一个函数只处在一个抽象层级，不把校验、查询、转换、持久化、通知揉在一起。
- **名称表达行为**：用 `calculatePayableAmount`、`isEligibleForBonus` 这类名称表达动作和结果。
- **参数保持克制**：参数过多时优先抽取参数对象；公共方法尤其要关注调用方可读性。
- **减少副作用**：函数名承诺什么就做什么，不悄悄修改外部状态、触发远程调用或改变事务语义。
- **分离指令与询问**：查询方法不要顺手修改状态，命令方法不要伪装成普通 getter。
- **短小是结果，不是教条**：函数过长通常意味着抽象层级混杂；行数可作为预警，不作为机械指标。

## 5. 注释与表达

注释应解释代码无法直接表达的信息，而不是重复代码本身。

- **优先改代码，再补注释**：如果必须靠注释解释“这段代码在做什么”，先考虑重命名、拆函数或调整结构。
- **说明为什么**：复杂业务规则、兼容逻辑、风险边界、历史原因和临时方案需要说明背景。
- **公共契约要完整**：公共接口、扩展点、DTO/Request/Query、配置属性应说明用途、约束、返回语义和异常。
- **禁止误导性注释**：过期注释、注释掉的大段废弃代码、与实现不一致的说明应及时删除。
- **表达要短、准、稳**：技术文档和代码注释都要主语明确、动作明确、边界明确。

## 6. 对象与数据结构

对象隐藏行为和约束，数据结构承载数据和传输。两者混淆会导致职责不清。

- **对象暴露行为，不暴露内部结构**：优先让对象回答业务问题，而不是让外部拿字段拼逻辑。
- **数据模型保持纯粹**：DTO、Request、Query 作为边界数据载体，不承载复杂业务行为。
- **遵守得墨忒耳律**：避免 `a.getB().getC().doSomething()` 这类链式穿透，必要时封装为领域行为。
- **封装变化点**：第三方 SDK、数据库、缓存、MQ、外部 API 等细节应通过接口或适配器隔离。
- **数据结构与算法可分离**：复杂计算逻辑应独立测试，不散落在 Controller、Mapper 或转换器中。

## 7. 错误处理

错误处理是系统设计的一部分，不是主流程失败后的补丁。

- **异常优先于错误码**：业务失败、外部依赖失败和系统异常应有清晰分类。
- **异常携带上下文**：错误消息应包含操作、关键业务标识和必要诊断信息，但不得泄露敏感数据。
- **不要吞异常**：捕获异常后必须处理、包装、补充上下文或重新抛出。
- **空值是契约问题**：能返回空集合就不返回 null；可能不存在的对象要通过命名、注解或返回类型表达。
- **用户提示与诊断信息分层**：给用户看的消息要可理解，给研发看的日志要可定位。

## 8. 测试

测试是系统可演进的安全网，也是最具体的使用文档。TDD、测试分层和测试资产治理详见 `testing.md`；具体测试实践见 `testing-practices.md`。

- **测试代码等同生产代码**：测试命名、结构、断言、测试数据和辅助方法都要保持整洁，可读、可改、可维护。
- **测试说明业务意图**：每个测试用例应能看出测试场景、触发行为、输入条件、期望输出和边界含义。
- **一个测试聚焦一个行为**：测试方法可以有多个必要断言，但应验证同一个业务分支或边界。
- **结构清晰**：优先按 Given/When/Then 或 Arrange/Act/Assert 组织，让准备、执行、断言边界清楚。
- **遵循 FIRST**：Fast、Independent、Repeatable、Self-Validating、Timely。
- **边界优先**：正常路径、异常路径、空值、权限、幂等、并发和兼容场景都要纳入考虑。
- **测试数据有语义**：避免无意义的 `foo`、`test1`、`123`；金额、状态、时间、权限等输入应服务测试意图。
- **重构前先有保护**：没有测试保护时，先补最小可用测试或明确验证方式，再重构。

## 9. 类与模块

类和模块是职责边界。边界越清晰，团队协作和长期演进越稳。

- **单一职责**：一个类或模块只有一个主要修改理由。
- **高内聚、低耦合**：相关行为放在一起，不相关能力通过接口、事件或适配器隔离。
- **依赖抽象**：核心业务依赖接口和领域模型，不直接依赖框架、数据库和外部服务细节。
- **组合优先于继承**：继承用于稳定抽象，组合用于灵活扩展。
- **避免上帝类**：当类开始统管流程、规则、状态和外部调用时，应拆分职责。

## 10. 系统与演进

整洁代码最终服务于整洁架构。局部漂亮但破坏边界，不是真正的整洁。

- **构造与使用分离**：对象创建、依赖装配和业务使用应分离，避免业务代码到处 new 复杂依赖。
- **横切关注点集中处理**：日志、鉴权、事务、审计、限流、监控等能力不应污染核心业务逻辑。
- **简单方案优先**：只为已经出现或高度确定的变化建立抽象，不为幻想中的未来制造复杂度。
- **演进而非推倒**：优先通过重命名、拆方法、拆类、移动包、引入接口等小步动作改善结构。
- **架构约束可验证**：重要依赖方向、模块边界和禁止项应尽量沉淀为测试、扫描或脚本。

## 11. 并发

并发代码的首要目标是可理解和可验证，其次才是极致性能。

- **隔离并发复杂度**：多线程、异步、锁、队列等逻辑与业务规则分离。
- **减少共享状态**：优先使用不可变对象、副本、局部变量、消息传递或任务分片。
- **同步区域尽量小**：只保护真正需要互斥的资源，避免把远程调用和复杂计算放进锁内。
- **明确线程模型**：线程池、超时、取消、重试、异常传播和上下文传递必须有清晰约定。
- **测试高风险路径**：并发、幂等、重复消费、乱序和超时场景需要专项验证。

## 12. 评审启发

Review 时优先问这些问题：

- 这段代码的业务意图是否能从命名和结构中直接看出来？
- 它是否把不同抽象层级或不同职责混在了一起？
- 它是否引入了不必要的副作用、全局状态或隐式依赖？
- 它是否让核心业务依赖了框架、数据库、Web 或第三方 SDK 等外部细节？
- 它是否破坏了模块边界、服务分层、依赖方向或对外契约？
- 它是否有足够的测试或验证方式支撑后续修改？
- 它是否为了“看起来高级”牺牲了团队可读、可改、可维护？
