# 演进式架构与遗留系统改造

本文用于遗留系统治理、模块化改造、服务拆分、技术栈升级、核心链路替换、数据迁移和平台能力沉淀。目标是小步、可验证、可回滚地改善系统体质，而不是一次性推翻重来。

## 使用时机

- 遗留系统难以维护、难以测试、边界不清、公共模块膨胀或上线风险高。
- 需要从单体演进到模块化单体、从模块化单体演进到微服务，或反向收敛过度拆分。
- 需要替换核心依赖、SDK、中间件、数据库、搜索、缓存、网关或认证体系。
- 需要迁移历史数据、历史接口、历史客户端或跨团队契约。

## 改造原则

- **先止血再手术**：先加测试、监控、边界和发布保护，再做结构性迁移。
- **小步可逆**：每一步都能独立发布、验证、暂停和回滚。
- **契约先行**：先固定外部行为、接口、消息、数据口径和验收场景。
- **防腐隔离**：新模型不要直接继承旧系统坏味道，用 Anti-Corruption Layer 隔离旧接口和旧数据。
- **业务连续性优先**：迁移期间保证用户体验、数据一致、客服可解释和运营可处理。

## 常用策略

| 策略 | 适用场景 | 关键动作 |
| --- | --- | --- |
| 契约测试 | 外部行为不能破坏 | 固定 API、消息、错误码、字段语义和兼容性。 |
| 防腐层 | 新旧模型差异大 | 旧接口适配为新领域语言，隔离旧枚举、状态和异常。 |
| 绞杀者模式 | 逐步替换大单体能力 | 按业务能力切路由，旧链路逐步收缩，新链路灰度扩大。 |
| 双写 | 新旧数据并行过渡 | 明确主写方、失败处理、幂等、对账和回滚策略。 |
| 回填 | 历史数据迁移 | dry-run、分批、校验、差异修复和审计。 |
| 读切换 | 新读模型验证成熟 | 影子读、结果比对、灰度读、异常自动回退。 |
| 模块化单体 | 边界未成熟或团队运维不足 | 先用模块边界、依赖规则和架构测试治理。 |

## 迁移设计顺序

1. **现状建模**：当前能力、数据、接口、依赖、痛点、事故、维护成本和不可动边界。
2. **目标边界**：目标模块/服务边界、数据归属、接口契约、权限和运行形态。
3. **兼容策略**：历史客户端、老接口、老数据、配置、消息、报表和运营工具如何共存。
4. **迁移路径**：防腐、双写、回填、影子读、灰度切流、停止旧写、下线旧读。
5. **验证与回滚**：契约测试、数据校验、对账、监控、开关、回滚和前滚修复。

## 微服务拆分门槛

只有同时满足以下条件，才建议拆服务：

- 业务边界稳定，领域语言清楚，数据归属明确。
- 调用关系、事务边界、失败路径和一致性方案清楚。
- 团队具备独立发布、监控、告警、容量、值班和故障处理能力。
- 拆分后收益大于网络、运维、数据一致性、测试和协作成本。
- 有灰度、回滚、兼容和迁移方案，不是一次性大爆炸。

不满足时优先模块化单体、包边界治理、依赖规则、领域服务提取和架构测试。

## 遗留系统评审清单

- 是否先定义目标和非目标，避免“顺手重构整个系统”。
- 是否有旧行为的契约测试、回归用例或可比对样本。
- 是否识别公共模块垃圾桶、循环依赖、框架绑架、贫血分层和隐式业务规则。
- 是否有防腐层隔离旧模型，而不是让新代码继续传播旧坏味道。
- 数据迁移是否有 dry-run、分批、备份、校验、对账、审计和回滚。
- 切流是否有开关、灰度比例、监控指标、暂停条件和回退路径。
- 是否定义旧能力下线标准和清理计划，避免新旧两套长期并存。

## 红线

1. 不得以“重构”为名混入无关功能变更。
2. 不得在没有契约测试、灰度和回滚方案时替换核心链路。
3. 不得把边界不清的单体直接拆成多个边界更不清的服务。
4. 不得在双写期间忽略失败处理、幂等、对账和主写方定义。
5. 不得让临时适配层长期无主；必须有下线条件、负责人和清理计划。
