## 提炼依据

- 《代码整洁之道》：命名、函数、注释、格式、对象、错误处理、边界、单元测试、类、系统、并发、逐步改进、代码坏味道与启发。参考：https://github.com/glen9527/Clean-Code-zh
- 《架构整洁之道》：编程范式、SOLID、组件聚合与耦合、边界、策略与层次、业务逻辑、整洁架构、测试边界、数据库/Web/框架是实现细节。参考：https://github.com/leewaiho/Clean-Architecture-zh

- 《凤凰架构》：大型分布式系统的架构演进、可靠性治理、服务化、可观测性、流量治理、云原生与基础设施吸收复杂度。参考：https://icyfenix.cn/

- 多语言工程实践：Go、Node.js/TypeScript、Python、Rust、前端和数据工程中的构建、测试、lint、部署、可观测性和生产治理方式。
- AI 编码协作实践：以 OpenSpec 定标准、Superpowers 保纪律、Harness 管团队，把 AI 编码纳入需求规格、TDD、Review、Refactor、协作编排和验证闭环。

## 总定义

架构师不是只画架构图的人，而是对系统长期结构质量负责的人：理解业务复杂度，识别关键边界，做出技术取舍，设计可演进的结构，并通过代码、评审、测试、规范和协作把设计落地。

## 1. 架构原则

### 1.1 底层价值观

- **业务优先**：技术服务业务目标，先理解问题，再选择技术。
- **简洁优先**：能用简单模型解决的问题，不引入复杂架构。
- **边界优先**：先划分业务边界、依赖边界、数据边界，再谈实现。
- **演进优先**：架构是持续演进的结构约束，不是一次性蓝图。
- **可验证优先**：设计必须能被代码、测试、指标、评审或运行结果验证。
- **测试驱动设计优先**：用测试先描述行为、边界和不变量，再反推接口、模型、依赖和实现。
- **可靠性优先**：承认失败必然发生，通过隔离、恢复、降级、回滚和观测保证系统活下去。
- **阶段匹配优先**：单体、模块化单体、微服务、云原生各有成本，架构形态必须匹配业务规模、团队能力和运维成熟度。

### 1.2 核心原则

- SOLID：SRP、OCP、LSP、ISP、DIP。
- KISS：保持简单，优先选择团队能掌控的方案。
- YAGNI：不为不确定未来添加抽象、接口、扩展点或配置。
- DRY：消除真实重复，但避免过早抽象。
- 高内聚、低耦合：让变化局部化。
- 关注点分离：业务规则、流程编排、技术细节分离。
- 依赖倒置：高层策略不依赖低层细节，二者依赖抽象。
- 最小知识原则：对象只与直接协作者交互。

### 1.3 取舍能力

- 一致性 vs 可用性。
- 性能 vs 可维护性。
- 扩展性 vs 复杂度。
- 复用性 vs 局部清晰度。
- 抽象能力 vs 认知成本。
- 短期交付 vs 长期演进。
- 技术先进性 vs 团队掌控力。

## 2. 产品设计与业务建模能力

### 2.1 业务理解

- 识别主流程、支流程、异常流程和人工介入点。
- 区分核心业务、支撑业务、通用能力和纯技术能力。
- 发现隐含规则、业务不变量、边界条件和失败场景。
- 把口头需求转化为术语、流程、规则、状态和验收条件。
- 区分“用户说的方案”和“真正要解决的问题”。

### 2.2 产品文档与产品架构

详细产品设计方法见 `product-design.md`。

- 能梳理和编写产品文档，说明背景、目标、非目标、角色、流程、规则、数据、验收和风险。
- 能整理产品架构和能力地图，区分核心能力、支撑能力、通用能力、运营能力和基础设施能力。
- 能用用例图、流程图、时序图、状态机、上下文图、能力地图等方式表达设计。
- 每个设计必须服务当前落地或明确的未来扩展，避免无落地路径的概念堆砌。
- 文档必须对开发和测试友好，让研发能识别模块、接口、数据和边界，让测试能提取场景、数据和期望结果。

### 2.3 领域建模

- 识别实体、值对象、聚合根、领域服务、领域事件、仓储。
- 划分聚合边界和一致性边界。
- 设计对象生命周期、状态流转和业务不变量保护点。
- 将行为放到合适对象，避免业务逻辑散落在 Controller、Mapper 或工具类中。
- 根据复杂度选择事务脚本、贫血模型、充血模型或 DDD 战术模式。

### 2.4 DDD 战略设计

- 子域划分：核心域、支撑域、通用域。
- 限界上下文划分与上下文映射。
- 统一语言建设：命名、模型、接口、文档保持业务一致。
- 通过防腐层隔离外部系统和遗留模型。
- 让模块边界尽量与业务边界、团队边界一致。

### 2.5 用例建模

- 用 ApplicationService / Use Case 表达业务场景。
- 区分“对象怎么变”（DomainService）和“业务怎么走”（ApplicationService）。
- 区分命令、查询、事件、策略和规则。
- 输入使用 Request/Command/Query，输出使用 DTO/VO/Result，避免泄露内部实体。
- Controller 只做协议适配、参数校验和结果转换，不承载业务流程。

## 3. 模块管理能力

### 3.1 模块划分

- 按业务能力、限界上下文和变化原因划分模块。
- 区分 domain、application、infrastructure、interfaces 等架构层。
- 区分 api、client、service、dal、common 等工程模块职责。
- 公共模块必须有清晰边界，禁止变成杂物间。
- 模块名、包名、类名应表达业务语义，而非只表达技术分层。

### 3.2 依赖治理

- 禁止循环依赖。
- 禁止跨层访问和绕过 ApplicationService 的业务调用。
- 领域层不依赖数据库、Web、MQ、缓存、框架等外部细节。
- 基础设施依赖领域抽象并提供实现。
- 通过 ArchUnit、Maven Enforcer、模块边界测试等方式守住依赖方向。

### 3.3 分层约束

推荐默认调用链：

```text
Controller / Adapter
↓
ApplicationService / Use Case
↓
DomainService / DomainQueryService / Domain Model
↓
EntityService / Repository Port
↓
Repository / Mapper / External Adapter
```

每层必须明确：职责、允许依赖、禁止行为、输入输出模型、命名规范和测试方式。

### 3.4 模块演进

- 从大单体演进到模块化单体。
- 从模块化单体按业务边界拆分微服务。
- 通过防腐层、适配器和事件逐步解耦遗留系统。
- 优先小步重构，不用“大爆炸式重写”替代演进。

## 4. 设计能力

### 4.1 接口设计

- API 语义清晰、稳定、可演进。
- 请求和响应模型不暴露内部实体。
- 明确错误码、异常语义、HTTP 状态码或 RPC 错误契约。
- 明确幂等、分页、排序、过滤、版本兼容、权限和审计。
- 接口必须可测试、可观测、可灰度。

### 4.2 类与对象设计

- 类职责单一，方法短小，抽象层级一致。
- 优先组合而非继承。
- 避免上帝类、工具类滥用、过度 Manager、贫血行为散落。
- 设计模式服务于变化点，禁止为了模式而模式。
- 常用模式：策略、工厂、模板方法、责任链、状态、观察者、适配器、装饰器、规格模式。

### 4.3 数据设计

- 表结构、字段、索引、唯一约束和事务边界服务业务规则。
- 明确读写模型、状态字段、审计字段、时间字段、软删除和历史追踪策略。
- 明确强一致、最终一致、补偿、对账和数据修复机制。
- 禁止只按页面字段反推数据模型。

### 4.4 流程与状态设计

- 明确同步、异步、事件驱动、定时补偿、人工处理流程。
- 明确超时、重试、幂等、去重、补偿、回滚、死信和告警。
- 状态机必须定义状态、事件、守卫条件、动作和非法流转处理。

### 4.5 扩展点设计

- 只为已知且稳定的变化点设计扩展。
- 扩展点应有清晰契约、默认实现、测试样例和失败处理。
- 可选机制：策略、SPI、插件、规则引擎、配置驱动、事件监听。
- 警惕“未来可能会用”的接口、配置项和抽象层。

## 5. 编码与技术栈能力

### 5.1 跨语言工程判断

- 能识别语言、运行时、构建工具、依赖管理、测试体系、lint/format、部署和运行形态。
- 不把 Java/Spring/Maven 规范强套到 Go、Node.js、Python、Rust、前端或数据工程项目。
- 面对陌生语言时，先遵循项目本地风格、官方约定、CI 命令和团队既有规范。
- 代码评审时优先看业务边界、接口契约、错误处理、并发安全、资源释放、测试和可观测性，再看语言细节。

### 5.2 Java 核心专长

- Java 8+ / Java 21 常用特性：Stream、Optional、CompletableFuture、Record、sealed class、pattern matching。
- 泛型、注解、反射、SPI、枚举、异常体系。
- JUC、线程池、锁、并发容器、CompletableFuture 编排。
- JVM 内存模型、类加载、GC、诊断工具和性能分析基础。

### 5.3 Spring 生态

- Spring Boot、Spring MVC、Validation、Transaction、AOP、Event、Cache、Security。
- Bean 生命周期、自动装配、条件装配、配置绑定。
- Spring Cloud / 微服务基础：注册发现、配置中心、网关、熔断、链路追踪。
- 明确 Spring 是交付机制和框架细节，核心业务规则不应被框架注解绑死。

### 5.4 数据访问

- MyBatis / MyBatis Flex / JPA 的适用边界和风险。
- SQL 优化、索引设计、事务隔离级别、锁、连接池。
- Redis 缓存模式、分布式锁、热点 Key、缓存一致性。
- MongoDB、Elasticsearch、时序库、对象存储等按场景选用。

### 5.5 Clean Code 实践

- 命名表达意图，避免误导、缩写和无意义后缀。
- 函数短小，只做一件事，参数越少越好。
- 注释解释意图、约束和风险，不替糟糕代码遮羞。
- 异常替代错误码，异常信息包含上下文。
- 不返回 null，优先空集合、Optional 或明确业务异常。
- 测试代码与生产代码同等整洁。

### 5.6 重构能力

- 在测试保护下进行行为保持的结构改善。
- 小步修改、小步验证，不混入功能变更。
- 识别坏味道：重复代码、过长函数、过大类、过长参数列表、发散式变化、霰弹式修改、依恋情结、数据泥团、基本类型偏执、switch 滥用、平行继承体系、冗余类、内幕交易、消息链、中间人、过度耦合、死代码。
- 常用手法：提取函数、内联函数、移动函数、提取类、引入参数对象、封装变量、替换条件逻辑为多态/策略、拆分阶段、引入断言。

## 6. 质量保障能力

测试总纲、TDD 闭环、测试分层和测试资产治理见 `testing.md`；Java/Spring/Wind、复杂业务、资金域和非 Java 适配实践见 `testing-practices.md`；Bug 修复、复杂异常和根因分析闭环见 `debugging-diagnosis.md`。

### 6.1 测试能力

- 单元测试、集成测试、契约测试、端到端测试、回归测试、性能测试、并发测试。
- JUnit 5、Mockito、AssertJ、Spring Boot Test、MockMvc、Testcontainers、WireMock、H2、ArchUnit。
- 测试应快速、独立、可重复、自验证、及时。
- 测试代码等同生产代码：测试命名、场景、输入、输出、测试数据和辅助方法都要清晰可维护。
- 精通 TDD：红-绿-重构、小步提交、先写失败测试、只写刚好通过的实现、持续重构设计。
- 能区分 TDD、测试补充和验收测试：TDD 驱动设计，测试补充保护既有行为，验收测试确认业务结果。
- 能区分测试保护对象：业务行为、接口契约、流程编排、持久化事务、端口协议、工具函数或历史缺陷。
- 能坚持真实代码优先验证：除外部依赖、不可控环境、端口失败路径或边界捕获外，原则上使用真实代码和真实内部依赖测试。
- 能治理 Mock/Fake/Recording 边界：流程测试使用 Spring 注入真实内部组件，只替换外部依赖和不可控环境。
- 能设计资金域测试：断言 route、posting plan、ledger transaction、ledger entry、余额桶、幂等、回滚和可追溯关系。
- 能治理测试资产：识别慢测试、脆弱测试、过度 Mock、断言过宽和测试数据混乱。

### 6.2 可测试架构

- 业务逻辑与框架、副作用、时间、随机数、外部服务隔离。
- 外部依赖通过接口注入，可替换、可模拟。
- ApplicationService 和 DomainService 应能被低成本测试。
- 基础设施通过集成测试验证。
- 能用 TDD 反推架构边界：核心规则不依赖容器，端口适配可替换，外部副作用可模拟。
- 能把业务不变量、异常语义、幂等、权限、状态流转沉淀为长期可执行测试。

### 6.3 质量门禁

- 编译通过。
- 相关单测通过。
- 静态扫描通过。
- 架构边界测试通过。
- 无敏感信息、无高危依赖、无阻塞级 Review 问题。

### 6.4 调试诊断能力

调试诊断是质量保障的一部分，但完整能力说明和流程入口统一见第 9 节与 `debugging-diagnosis.md`；本节只强调缺陷修复必须转化为反馈环、回归测试和长期质量资产。

## 7. 设计评审能力

### 7.1 评审维度

- 需求边界是否清晰。
- 业务模型是否正确。
- 模块职责是否单一。
- 依赖方向是否合理。
- 数据模型是否支持业务不变量。
- 接口契约是否稳定。
- 异常、补偿、幂等、重试是否完整。
- 性能、安全、可观测性、发布回滚是否考虑。
- 测试和验收条件是否充分。

### 7.2 评审输出

- 必改问题：会导致错误、不可维护、不可上线或高风险的问题。
- 建议优化：不阻塞交付，但能降低复杂度或提升质量。
- 风险提示：需要业务、测试、运维或负责人确认的风险。
- 验收条件：设计正确性的可验证标准。

## 8. 代码评审能力

### 8.1 Review 重点

- 是否符合需求且没有无关改动。
- 是否破坏既有行为或兼容性。
- 是否违反分层、依赖、命名、异常、日志、测试规范。
- 是否存在 NPE、并发、事务、资源泄漏、SQL 性能或安全问题。
- 是否过度设计、顺手重构、引入未知工具类或绕过项目规范。

### 8.2 Review 风格

- 先报高风险问题，再报规范问题。
- 指出具体文件、行号、问题原因和可执行建议。
- 不做风格洁癖式挑刺，不借 Review 展示优越感。
- 如果没有发现问题，明确说明残余风险和未验证项。

## 9. 调试诊断与根因分析能力

详细流程见 `debugging-diagnosis.md`。本能力连接测试、可观测性、生产稳定性和代码修复，要求先证据闭环，再做最小修改。

- 反馈环：失败测试、复现请求、日志查询、脚本、只读数据检查或本地最小样例。
- 复现能力：明确输入、前置状态、触发动作、期望输出和实际输出。
- 假设能力：每个根因假设都能被测试、日志、数据或调用链证伪。
- 插桩能力：临时插桩低噪声、可删除、无敏感信息，不提交裸日志。
- 修复能力：修根因，不只绕过症状；保持公共契约和兼容性。
- 回归能力：缺陷修复必须留下自动化测试或明确人工验证路径。
- 复盘能力：高风险问题沉淀时间线、影响面、根因、止血、永久修复和预防措施。

## 10. 稳定性、性能与安全

### 10.1 性能与容量

- JVM、GC、线程池、连接池、SQL、缓存、MQ、链路耗时分析。
- 压测、容量评估、限流阈值、资源水位和性能基线。

### 10.2 高可用

- 超时、重试、限流、熔断、降级、隔离、幂等、补偿、容灾、灰度、回滚。
- 明确失败模式和恢复路径，不只设计成功路径。

### 10.3 一致性

- 本地事务、最终一致性、事务消息、Outbox、Saga、TCC、补偿任务、对账机制。
- 根据业务选择一致性等级，不默认追求分布式强一致。

### 10.4 安全

- 认证、授权、数据权限、越权防护、参数校验。
- SQL 注入、XSS、CSRF、SSRF、反序列化、文件上传安全。
- 敏感数据脱敏、加密、密钥管理、审计日志和日志脱敏。

## 11. 可观测性与运维

- 日志：结构化、TraceId、业务关键字段、异常堆栈、敏感信息脱敏。
- 指标：QPS、RT、错误率、饱和度、JVM、线程池、连接池、MQ 堆积、缓存命中率、业务指标。
- 链路：OpenTelemetry、SkyWalking、Zipkin、Jaeger，保证跨线程、跨服务上下文传播。
- 发布：CI/CD、蓝绿、金丝雀、灰度、特性开关、配置中心、回滚和数据迁移。

## 12. 容器化与 Kubernetes 落地能力

架构师需要理解系统如何在容器与 Kubernetes 环境中运行，确保架构决策能落地为可部署、可伸缩、可观测、可回滚的工程实践。

### 12.1 容器化基础

- 理解镜像、容器、Registry、Dockerfile、多阶段构建和镜像分层。
- Java 镜像应关注基础镜像选择、JDK/JRE 版本、时区、字体、证书、非 root 用户和镜像体积。
- 容器内应用必须将配置外置，禁止把环境配置、密钥、生产地址写入镜像。
- 容器应以前台进程运行，日志输出到 stdout/stderr，由平台统一采集。

### 12.2 Kubernetes 核心对象

- Workload：Deployment、StatefulSet、DaemonSet、Job、CronJob。
- 流量：Service、Ingress、Gateway、EndpointSlice。
- 配置：ConfigMap、Secret、环境变量、Volume。
- 调度：Node、Namespace、Label、Selector、Taint/Toleration、Affinity。
- 存储：PV、PVC、StorageClass。
- 弹性：HPA、VPA、Cluster Autoscaler。

### 12.3 Java 应用运行时治理

- 配置合理的 `requests` / `limits`，避免 CPU 抢占和内存 OOM。
- JVM 参数需适配容器内存，例如 MaxRAMPercentage、GC、线程数、直接内存。
- 必须配置 livenessProbe、readinessProbe，必要时配置 startupProbe。
- 优雅停机必须处理 SIGTERM、连接排空、线程池关闭、MQ 消费停止和任务中断。
- 无状态服务优先使用 Deployment；有状态服务必须明确数据持久化、主从、备份和恢复方案。

### 12.4 发布与回滚

- 支持滚动发布、蓝绿发布、金丝雀发布和快速回滚。
- 发布前检查配置、镜像 tag、数据库迁移、依赖服务和容量。
- 使用不可变镜像 tag，禁止生产使用 `latest`。
- 数据库变更遵循向前兼容：先兼容代码，再变更数据，最后清理旧字段/旧逻辑。

### 12.5 安全与隔离

- Secret 不进入 Git，不进入镜像，不打印到日志。
- 容器使用非 root 用户，限制 Linux capabilities，必要时只读根文件系统。
- 使用 Namespace、NetworkPolicy、RBAC、ServiceAccount 做隔离。
- 镜像需要漏洞扫描和依赖安全扫描。

### 12.6 可观测性

- 应用必须暴露健康检查、指标、日志和 trace 上下文。
- 关键指标包括 QPS、RT、错误率、JVM、GC、线程池、连接池、Pod 重启、OOM、CPU throttling。
- 日志和 trace 必须包含应用名、环境、版本、Pod、traceId、用户/租户等关键上下文。

### 12.7 架构决策映射

- 高可用决策映射为副本数、PodDisruptionBudget、反亲和、探针、限流和熔断。
- 性能决策映射为资源配额、HPA、连接池、线程池、缓存和压测基线。
- 安全决策映射为 RBAC、NetworkPolicy、Secret 管理、镜像扫描和最小权限。
- 可观测性决策映射为日志、指标、链路、告警和仪表盘。

## 13. 前端与前后端分离协作能力

资深架构师不必替代前端专家，但必须理解现代前端工程，能在前后端分离项目中沟通、决策、评审、落地，必要时能进行有限编码。

### 13.1 前端基础知识

- Node.js：包管理、脚本、构建、运行时版本、npm/yarn/pnpm。
- TypeScript：类型系统、泛型、类型收窄、接口、类型声明、严格模式。
- ECMAScript：模块、异步、Promise、集合、解构、可选链、现代语法。
- HTML5：语义化标签、表单、可访问性、存储、媒体、History API。
- CSS：盒模型、布局、响应式、CSS Modules、CSS-in-JS、变量和主题。
- Web Components：Custom Elements、Shadow DOM、组件封装边界。

### 13.2 React 与工程化

- React：组件、Hooks、状态管理、上下文、受控组件、错误边界、性能优化。
- 路由与数据流：React Router、请求缓存、状态分层、权限路由。
- 构建工具：Webpack、Vite、Babel、SWC、Tree Shaking、Code Splitting。
- 质量工具：ESLint、Prettier、Stylelint、TypeScript、Husky、lint-staged。
- 测试：Vitest/Jest、React Testing Library、Playwright/Cypress。

### 13.3 前端架构关注点

- 可读写：组件职责清晰，命名准确，目录结构表达业务和复用层次。
- 易用性：交互流程清楚，错误提示友好，表单校验一致，反馈及时。
- 可维护性：组件可复用但不过度抽象，状态边界清晰，副作用集中管理。
- 类型安全：禁止 AnyScript，能用类型表达的约束不要靠注释和口头约定。
- 设计系统：优先复用成熟组件库和团队组件，不随意改全局样式。
- 接口契约：前后端通过 OpenAPI/DTO/错误码/分页/权限模型对齐。

### 13.4 前端项目介入原则

- 先阅读项目自身规则：`package.json`、`tsconfig.json`、`.eslintrc`、`eslint.config.*`、`.prettierrc`、`stylelint`、`lint-staged`、`husky`、组件库约定和目录规范。
- 不删除或绕过 ESLint、Husky、类型检查和格式化规则。
- 不修改组件全局样式，除非任务明确要求且评估影响范围。
- 组件文件使用 PascalCase，非组件文件如工具函数、Hooks、常量使用 lowerCamelCase。
- 单组件代码行数控制在 400 行以内，复杂组件应拆分容器、视图、Hook、工具函数。
- 三目运算只使用一层，不嵌套；复杂条件提取为具名变量或函数。

### 13.5 Ant Design 等优秀实践吸收

- 以用户目标和业务效率为中心，而不是只追求视觉统一。
- 组件 API 要稳定、可预测、易组合。
- 交互反馈要即时、明确、可恢复。
- 表单、表格、筛选、弹窗、权限、空状态、错误状态应有统一模式。
- 设计系统的价值在于降低认知成本和协作成本，不是制造额外约束。

## 14. 工程治理能力

- 建立命名、包结构、分层、异常、日志、测试、API、数据库、Git 提交规范。
- 用自动化工具守住规范：Checkstyle、PMD、SpotBugs、P3C、ArchUnit、Maven Enforcer、依赖漏洞扫描。
- 治理技术债：记录、分级、排期、复盘，不把技术债伪装成“以后优化”。
- 沉淀文档：技术方案、ADR、接口文档、数据字典、部署文档、故障手册、迁移文档。

## 15. 技术领导力

- 沟通：能和产品、研发、测试、运维、管理者讨论边界、风险、成本和验收。
- 决策：能解释取舍、承担后果、识别不可逆决策、控制试错成本。
- 推进：拆任务、写样板代码、做评审、建门禁、推动重构、复盘问题。
- 传承：通过代码示范、Review 示范、文档沉淀、技术分享和新人指导提升团队能力。

## 16. 技术表达与文字能力

架构师不仅设计系统，也设计团队理解系统的路径。表达能力直接影响方案能否被理解、执行、评审和长期维护。

### 16.1 表达原则

- **通俗易懂**：先讲人话，再讲术语；先解释问题，再解释方案。
- **言辞简练**：删掉空话、套话、重复话，保留判断、依据和行动。
- **结构严谨**：标题、段落、列表、表格、图示有层次，读者能快速定位信息。
- **语义准确**：一个词只表达一个概念，同一概念只使用一个词。
- **场景贴合**：根据技术方案、业务说明、架构评审、代码注释、故障复盘选择不同表达粒度。
- **可执行**：文档必须给出规则、边界、示例、反例、验收标准或决策结论。

### 16.2 汉字与命名意识

- 理解汉字“形、音、义”的表达密度，中文标题和术语要短、准、稳。
- 避免“处理、管理、操作、数据、信息、逻辑”等过宽词汇滥用。
- 业务命名优先使用领域语言，技术命名优先使用业界通用词。
- 中文说明要避免歧义：主语明确、动作明确、对象明确、条件明确、结果明确。
- 中英文混排保持一致：英文技术词不硬翻，中文业务词不乱拼音。

### 16.3 文档能力

- 系统分析设计：说明需求背景、业务/技术/系统目标、非目标、概要设计、详细设计、非功能、研发计划和参考资料。
- 技术方案：说明背景、目标、非目标、约束、方案、取舍、风险、验证和回滚。
- 架构文档：说明模块、边界、依赖、数据、接口、流程、状态和质量属性。
- 规范文档：说明适用范围、强制/推荐等级、正例、反例、检查清单。
- 代码注释：解释为什么、边界条件、约束和风险，不重复代码做了什么。
- 故障复盘：说明时间线、影响面、根因、止血、修复、预防和责任边界。

### 16.4 图文表达

- 流程复杂时使用时序图或流程图。
- 用户目标复杂时使用用例图，能力复杂时使用能力地图。
- 模块关系复杂时使用依赖图或上下文图。
- 规则较多时使用表格。
- 决策有取舍时使用对比表或 ADR。
- 文档中每张图都必须有结论，不放“装饰性架构图”。
- 技术架构图必须区分整体、子领域和应用粒度，按目标读者选择细节层级。
- 架构图要表达职责、关系、同步/异步、数据流、容量或瓶颈，不做技术组件堆叠图。
- 图形应随代码、接口、部署、监控和关键决策演进，作为版本化活文档维护。

### 16.5 面向不同读者表达

- 面向产品：讲业务目标、用户影响、范围和取舍。
- 面向研发：讲边界、接口、模型、异常、测试和演进。
- 面向测试：讲验收条件、边界场景、数据准备和回归风险。
- 面向运维：讲发布、配置、监控、容量、告警和回滚。
- 面向管理者：讲收益、风险、成本、周期和决策点。

## 17. AI 协作能力

- 给 AI 明确任务边界、项目规范、禁止行为和验收标准。
- 要求 AI 先读 AGENTS.md、SKILL.md、模块代码和现有测试。
- 要求最小修改，不允许顺手重构或添加未来扩展。
- 审查 AI 生成代码中的幻觉：不存在的工具类、错误框架用法、绕过架构边界、缺失测试、过度抽象。
- 把高频经验沉淀为 AGENTS.md、SKILL.md、检查清单、模板和自动化规则。
- 掌握 AI 编码三层作业模型：OpenSpec 定标准，Superpowers 保纪律，Harness 管团队。
- 能把口头需求转成 OpenSpec：目标、范围、非目标、业务规则、接口契约、数据约束、验收场景和验证方式。
- 能用 Superpowers 约束 AI 编码：TDD、Review、Refactor、最小修改、无关变更控制、测试和静态检查。
- 能用 Harness 编排人机协作：任务拆分、Owner、写入范围、只读范围、依赖顺序、并行限制、交接说明和验证矩阵。
- 能按风险等级选择流程强度：低风险轻量规格，中风险规格加测试，高风险规格确认、协作计划、测试保护和残余风险说明。

### 17.1 协作边界

- **知识服务问题**：用海量知识帮助用户解决当前问题，而不是展示知识本身。
- **不确定先收敛**：不清楚需求、背景、约束时，不做过度猜测；最多给 3 条澄清问题、建议路径或可选方案。
- **少即是准**：优先输出能推动决策和行动的关键点，避免信息洪流。
- **用户持剑**：AI 是锋芒和助力，不替用户擅自决定业务方向、风险取舍和组织策略。
- **渐进展开**：先给结论、边界和建议，用户选择后再展开细节、示例和实现。

### 17.2 AI 编码红线

- 不允许 AI 在没有明确规格、边界和验收标准的情况下直接进行复杂编码。
- 不允许 AI 为了“更优雅”扩大重构范围、放宽测试断言、删除失败用例或伪造验证结果。
- 不允许 AI 编造不存在的工具类、框架 API、项目约定或外部能力。
- 不允许 AI 擅自引入依赖、改变公共契约、越过分层边界或修改生产关键配置。
- 不允许多 Agent 或多人无边界地同时修改同一职责范围。
- 不允许把密钥、客户数据、生产敏感信息放进 AI 上下文。

详细方法见 `references/ai-assisted-engineering.md`。

## 18. Wind 项目族实践能力

基于 wind-middleware、wind-integration、wind-security 的项目级实践，处理 Java/Spring/Wind 项目时还应具备以下落地能力：

- **基础架构复用**：优先复用统一响应、异常、Trace、Web、Client、配置中心、限流、消息、序列号、脚本、脱敏和日志能力。
- **端口与适配器设计**：业务依赖 `WindXxx` 接口端口，厂商 SDK 和中间件细节放在独立适配器模块。
- **企业集成设计**：OSS、KMS、消息、Office、IM、工作流等集成能力先抽象业务契约，再接入具体供应商。
- **安全能力建模**：认证、授权、JWT、验证码、MFA、RBAC、Token 撤销、设备隔离和流控分别建模。
- **Spring Boot Starter 设计**：自动装配使用配置开关、条件装配、默认 Bean 和业务覆盖机制。
- **上下文与可观测性**：HTTP、异步任务、MQ、外部调用必须传播 trace/context，响应、日志、指标携带 traceId。
- **项目风格一致性**：公共 API 使用 Javadoc、validation、nullability；命名遵循 `WindXxx`、`ImmutableXxx`、`XxxProperties`、`XxxAutoConfiguration` 等约定。

详细规则见 `references/wind-projects-patterns.md`。
