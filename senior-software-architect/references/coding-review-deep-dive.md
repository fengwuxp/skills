# 编码约规深化与 Review 方法

本文用于把编码强规约转化为可执行的编码判断、代码 Review 路径和问题定级方式。它不替代 `coding-standards.md`、`project-governance-standards.md` 和 `clean-code.md`，而是在写代码、改代码、审代码时提供更稳定的判断顺序。

## 使用时机

- 用户要求深入编码约规、代码质量、Java/Spring 代码 Review 或编码红线。
- 代码变更涉及领域规则、事务、异常、空值、并发、外部依赖、数据访问、日志、安全或测试。
- Review 发现“看起来能跑，但长期维护风险高”的代码，需要给出可解释判断。
- 需要把团队编码规范沉淀成检查清单、PR 反馈或整改建议。

## 不适用场景

- 用户只要求通用 Clean Code 启发时，优先读 `clean-code.md`。
- 用户要求完整架构方案、系统分析设计或生产变更方案时，先读 `scenario-routing.md` 选择专项 reference。
- 未阅读代码、diff、测试或调用链时，不输出确定性 Review 结论。

## 读取后必须产出

- 问题优先的 Review 判断：业务语义、边界方向、契约完整性、失败路径、工程一致性和复用质量。
- 每个问题说明严重级别、真实风险、证据、最小建议和验证方式。
- 对架构坏味只给可验证整改步骤，不把观察项升级成大范围重构建议。

## 需要继续读取的 reference

- Java/Spring/Wind 强制规则读 `coding-standards.md`。
- Review 输出格式读 `review-and-output-templates.md`。
- 测试问题读 `testing.md`；生产、安全或外部依赖风险读对应专项 reference。

## 按任务读取索引

| 任务 | 优先读取 | 跳过 |
| --- | --- | --- |
| 常规代码 Review | `编码判断顺序`、`重点问题定级`、`整改建议写法` | 不从格式和个人偏好开始 |
| Java/Spring 高频风险 | `Java/Spring 高频检查点` | 不替代 `coding-standards.md` 的完整强规约 |
| 架构坏味扫描 | `Review 问题模式`、`Java 架构坏味启发式扫描` | 不按行数、依赖数机械重构 |
| 复用和方法抽取判断 | `Java/Spring 高频检查点` 中复用与方法抽取 | 不为一行包装或 DTO 构建抽公用工具 |
| Review 交付 | `整改建议写法`、`红线`，再读 `review-and-output-templates.md` | 不输出“优化一下”类不可执行建议 |

## 编码判断顺序

1. **先看业务语义**：名称、模型、方法和测试是否表达真实业务意图。
2. **再看边界方向**：Controller、Application、Domain、Infrastructure、Converter、Repository 是否各司其职。
3. **再看契约完整性**：入参、返回、异常、空值、权限、幂等、分页、兼容和序列化语义是否明确。
4. **再看失败路径**：异常、事务、重试、补偿、日志、告警和人工兜底是否匹配风险。
5. **再看工程一致性**：命名、格式、Lombok、MapStruct、测试、依赖和本地项目风格是否一致。
6. **最后看复用质量**：抽取的方法、工具类和参数对象是否提升业务表达、稳定复用和测试边界，而不是制造跳转和调用链复杂度。

不要从格式或个人偏好开始 Review。格式问题可以指出，但不能掩盖边界、契约、数据和生产风险。

## 分层编码检查

| 层次 | 允许做 | 禁止做 |
| --- | --- | --- |
| Controller / API | 参数校验、鉴权入口、调用用例、响应转换 | 写领域规则、直接访问 Repository/Mapper、拼复杂查询、吞异常 |
| Application / Use Case | 编排业务用例、事务边界、权限校验、调用领域能力和端口 | 堆积细碎业务规则、泄露框架细节、承载复杂 SQL |
| Domain | 业务不变量、状态流转、领域规则、领域事件 | 依赖 Web、ORM、MQ、Redis、HTTP Client、第三方 SDK |
| Infrastructure | Repository、Gateway、Client、MQ、缓存、文件、SDK 适配 | 反向调用领域实现细节、写业务决策、吞外部异常 |
| Converter / Mapper | 模型转换、字段映射、枚举转换、默认值转换 | 权限判断、远程调用、数据库查询、状态流转、审计写入 |
| Test | 表达业务行为、边界、异常、兼容和回归 | 只为覆盖率造数据、过度 Mock、依赖执行顺序、无断言 |

## 重点问题定级

| 等级 | 编码场景 |
| --- | --- |
| P0 | 资金、权限、租户、敏感数据、生产数据、事务一致性会被破坏；可能造成不可逆事故。 |
| P1 | 公共契约破坏、核心流程错误、严重 NPE、事务吞异常、越权风险、重复扣减或状态错乱。 |
| P2 | 边界穿透、缺少幂等/测试/日志、MapStruct 隐式错映射、Lombok 暴露 setter、可维护性明显下降。 |
| P3 | 命名不准、注释欠清晰、局部重复、格式不一致、低风险工具方法可读性问题。 |

Review 输出要说明真实后果，不只写“建议优化”。

## Java/Spring 高频检查点

### 空值与断言

- 内部契约使用 `org.jspecify.annotations` 表达 nullability。
- API 参数、Request、Response、DTO、Query、Command 使用 Bean Validation 表达字段约束。
- 业务前置条件、状态条件和不可达分支优先使用 `AssertUtils`。
- 已由 JSpecify 标注为非空的普通参数、返回值和字段，不再重复写无业务语义的空判断。
- 集合返回空集合，不返回 `null`。
- `Optional` 不用于字段、DTO 属性和序列化模型属性。

### 事务与异常

- 事务方法中捕获异常必须重新抛出或显式回滚。
- 包装异常必须保留 cause 和关键业务上下文，不能丢原始诊断信息。
- 用户提示和内部诊断信息分层，日志不得泄露敏感数据。
- 事务内避免远程调用、长耗时计算和不可控阻塞；确需调用时必须说明补偿与超时。

### Lombok

- Spring Bean 优先 `final` 依赖字段 + `@AllArgsConstructor` 构造注入。
- 有非依赖状态、可选依赖、特殊构造逻辑的 Bean 使用显式构造器。
- 领域对象、聚合根、状态机对象、有业务不变量的类不使用 `@Data` 和全量 `@Setter`。
- `@ToString`、`@EqualsAndHashCode` 避开敏感字段、大字段、双向关联和懒加载对象。

### MapStruct

- MapStruct 只做模型转换，不写业务规则、权限判断、远程调用、数据库查询和状态流转。
- 字段语义差异、枚举、默认值、空值策略和忽略字段必须显式声明。
- 更新已有对象时必须明确 null 覆盖策略。
- 关键转换必须有单元测试，覆盖字段完整性、枚举、空值、默认值和嵌套集合。

### 不可变对象

- 值对象、配置、规则参数、领域事件、审计快照、API 输出模型、缓存 Key、跨线程传递对象优先不可变。
- ORM Entity、MyBatis DO、状态机对象、批量填充模型和强反射框架对象不机械追求不可变。
- DTO、VO、Request、Response、Query、Command、Event 等数据传输对象的成员变量不得使用 `boolean`、`byte`、`short`、`int`、`long`、`float`、`double`、`char` 等 Java 原生基本类型，也不得使用 `AtomicInteger`、`AtomicLong`、`AtomicBoolean`、`AtomicReference`、`LongAdder`、`DoubleAdder` 等并发原子类型；Review 时应检查字段契约是否需要可缺省、默认值、精度和序列化语义，并把并发原子类型视为契约污染，而不是线程安全增强。
- 不可变对象持有集合、数组、`Date` 等可变成员时，必须防御性拷贝。
- 不可变对象跨服务、缓存、MQ、落库时，必须验证 Jackson、Kryo、Protobuf 等序列化/反序列化能力。

### 复用与方法抽取

- 公有方法参数不得超过 5 个；如确需超过，必须先和用户确认调用场景、兼容性和替代方案。
- Java/Spring 项目检查是否手写 `hasText`、`isBlank`、`isEmpty` 等基础工具；此类通用能力按 `coding-standards.md` 优先使用 Spring Framework 或 Apache Commons 成熟工具。
- DTO/VO/Request/Response 等构建逻辑通常不值得提取为公有工具；除非存在稳定映射规则、兼容转换或多处强一致契约。
- 一行代码包装默认不作为有效复用；只有承载领域语言、权限边界、异常语义、审计埋点或策略替换时才有意义。
- 抽取方法后调用链不得明显变深；Review 时要能从主流程读出业务意图、状态变化和副作用。
- 复用判断必须平衡去重、可读性、逻辑内聚、测试边界和未来变化方向，不以“代码行数减少”作为唯一收益。

## Review 问题模式

- **业务规则错层**：规则写在 Controller、Converter、Mapper、Job 或工具类中。
- **贫血用例**：ApplicationService 只搬运数据，领域规则散落在多个私有方法和 if/else 中。
- **模糊契约**：空值、异常、权限、分页、幂等、默认值和兼容语义不清。
- **隐式副作用**：看似查询的方法实际修改状态、发消息、调用外部系统或写审计。
- **转换黑盒**：MapStruct、BeanUtils 或反射拷贝掩盖字段语义差异。
- **伪复用**：为 DTO 构建、一行包装或偶发重复提取公有方法/工具类，导致调用链变深、语义变弱。
- **并发幻觉**：只在单线程路径验证，忽略重复请求、并发提交、重复消息和乱序回调。
- **测试伪覆盖**：测试只覆盖 happy path，没有断言关键业务不变量、异常和边界。

## Java 架构坏味启发式扫描

当用户要求“深度 CR”“架构坏味”“代码质量扫描”“重构评估”或发现长期维护风险时，在常规 Review 后追加架构坏味扫描。扫描结论必须结合业务语义、变更频率和项目规模，以下阈值是提示信号，不是机械定罪。

| 坏味 | 识别信号 | 主要风险 | 优先建议 |
| --- | --- | --- | --- |
| 上帝类 / God Class | 单类同时编排流程、校验规则、持久化、外部调用和状态转换；非模型类过长、注入依赖过多或公开方法过多。 | 修改牵一发动全身，测试困难，业务边界被类名掩盖。 | 先按用例、领域能力、端口适配拆出职责，再用 characterization test 保护行为。 |
| 循环依赖 | 模块、包或 Bean 双向依赖；domain/core 反向依赖 web/infrastructure。 | 发布、测试、初始化和演进受阻，边界失真。 | 调整依赖方向，引入端口接口、领域事件或应用服务编排。 |
| 过长方法 | 方法混合参数解析、业务判断、状态转换、持久化和外部副作用。 | 难以证明失败路径，局部修改容易破坏不变量。 | 先标记阶段边界，再提取有领域语义的小方法或对象。 |
| Feature Envy | 一个类大量读取另一个对象字段或 getter 后做判断。 | 数据和行为分离，规则散落在调用方。 | 把规则移动到拥有数据和不变量的对象、领域服务或策略中。 |
| Data Clumps | 多个方法、类或 DTO 反复携带同一组字段，例如金额/币种、租户/主体/权限、时间窗口。 | 参数漂移，校验重复，契约不稳定。 | 引入值对象、查询对象、命令对象或上下文对象，并补充不变量测试。 |
| 霰弹式修改 | 一个需求需要同步修改多个无稳定边界的类、配置和转换器。 | 变更成本高，容易漏改和产生兼容问题。 | 识别变化轴，收敛到明确扩展点、策略或配置契约。 |
| 公共模块垃圾桶 | `common`、`util`、`manager`、`helper` 聚集业务规则、转换、外部调用和状态判断。 | 复用名义下形成隐式耦合，破坏领域语言。 | 区分通用技术能力和业务能力，业务规则回到所属模块。 |

扫描时建议用四步输出：

```text
坏味：具体文件、类、方法或模块。
证据：依赖方向、调用链、字段组合、方法长度、变更散点或测试缺口。
风险：对业务语义、测试、兼容、发布或生产行为的真实影响。
整改：最小可验证步骤，不把坏味治理扩大成无边界重构。
```

不得只因为代码行数、方法数量或注入数量触发重构建议；如果当前类职责稳定、测试充分、变更少且拆分会增加认知负担，应标为观察项而不是问题。

## 整改建议写法

```text
问题：说明具体代码位置和行为。
风险：说明会导致什么真实后果，例如越权、重复扣减、事务部分提交、兼容破坏。
建议：给出最小可落地修改，不扩大范围。
验证：说明需要补充的单测、集成测试、契约测试或手工验证。
```

示例：

```text
[P1] OrderController 直接扣减库存并发送 MQ
风险：Web 层承载领域规则，事务边界和消息投递失败路径不清，重复提交可能导致库存状态不一致。
建议：将扣减规则下沉到应用服务/领域服务，使用幂等键保护提交，消息投递通过 Outbox 或补偿机制处理。
验证：补充重复请求、库存不足、消息投递失败和事务回滚测试。
```

## 红线

1. 不得以“代码更短”为理由隐藏业务不变量、状态变化、权限判断和副作用。
2. 不得以 Lombok、MapStruct、BeanUtils、反射或自动生成代码替代契约设计。
3. 不得只看风格问题而忽略边界、事务、权限、数据和生产风险。
4. 不得在 Review 中给无法执行的建议，例如“优化一下”“注意一下”“加强校验”。
5. 不得把个人偏好包装成团队规约；没有项目依据时只能作为建议并说明取舍。
