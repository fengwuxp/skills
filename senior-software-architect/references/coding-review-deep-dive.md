# 源码 Review 与风险裁决

本文用于源码级 Review、风险定级和整改裁决。具体 Java/Wind 条目由项目本地规范和 `wind-coding-conventions` 提供；本文只消费规则结果，不复制规则正文。

## 使用时机

- 用户要求代码质量、源码级 Review、风险定级或整改建议。
- 代码变更涉及领域规则、事务、异常、空值、并发、外部依赖、数据访问、日志、安全或测试。
- Review 发现“看起来能跑，但长期维护风险高”的代码，需要给出可解释判断。
- 需要把团队编码规范沉淀成检查清单、PR 反馈或整改建议。

## 不适用场景

- 用户只要求通用 Clean Code 启发时，优先读 `clean-code.md`。
- 用户要求完整架构方案、系统分析设计或生产变更方案时，先读 `scenario-routing.md` 选择专项 reference。
- 未阅读代码、diff、测试或调用链时，不输出确定性 Review 结论。

## 读取后必须产出

- 问题优先的 Review 判断：业务语义、边界方向、契约完整性、失败路径、工程一致性和复用质量。
- 每个问题说明严重级别、真实风险、证据、最小建议和验证方式。
- 对架构坏味只给可验证整改步骤，不把观察项升级成大范围重构建议。

## 需要继续读取的 reference

- Java/Spring 强制规则先读项目本地规范和 `wind-coding-conventions` 通用层；Wind/Nobe 专项按依赖或上下文启用。
- Review 输出格式读 `review-and-output-templates.md`。
- 测试问题读 `testing.md`；生产、安全或外部依赖风险读对应专项 reference。

## Review 导航顺序

处理多文件、大变更或上下文不完整的 Review 时，先判断整体方向，再进入细节：

1. 先看变更说明和整体目的：本次变更是否应该发生，是否属于当前系统方向，是否有清楚的 what / why / 验证信息。
2. 先看主路径和核心文件：从业务规则、接口契约、状态流转、数据写入、外部依赖、并发或权限安全等主要影响面开始。
3. 重大设计问题先反馈：如果主路径存在明显边界、契约或复杂度问题，先给出阻塞意见，避免作者继续在错误方向上堆叠后续变更。
4. 再按逻辑顺序看其余文件：可以先读测试理解行为，再读实现；也可以按调用链、模块依赖或代码审查工具顺序推进，但必须覆盖被分配审查的范围。
5. 明确审查范围：只审查部分文件、只看高层设计、安全、隐私、并发、可访问性或国际化等专项时，要在 Review 结论中说明范围和未覆盖部分。
6. 若用户要求“过度设计 CR / 最小正确实现检查 / Ponytail-style review”，追加一个独立专项 pass，只找可删除复杂度和可复用替代，不把它混成正确性、安全或性能结论。
7. 若用户提供 Open Code Review / OCR 输出，先核对它审查的 diff 范围、业务背景、规则来源和 LLM 连通性，再把每条建议按源码事实、项目规范、通用 Java 约规、已命中的 Wind/Nobe 专项、测试证据和真实风险重新定级；定位失败、缺上下文或只属风格偏好的建议不得直接当阻塞问题。

如果变更过大以至于无法识别主路径，应要求作者拆分或说明审查入口；如果只是缺少上下文，应先要求补充变更说明、设计链接、验证结果或演示材料。

## 编码判断顺序

1. **先看业务语义**：名称、模型、方法和测试是否表达真实业务意图。
2. **再看领域语言**：关键对象、动作、状态和异常是否使用业务能理解的统一语言；同一词在不同上下文中是否被强行统一或混用。
3. **再看边界方向**：Controller、Application、Domain、Infrastructure、Converter、Repository 是否各司其职。
4. **再看契约完整性**：入参、返回、异常、空值、权限、幂等、分页、兼容和序列化语义是否明确。
5. **再看失败路径**：异常、事务、重试、补偿、日志、告警和人工兜底是否匹配风险。
6. **再看工程一致性**：命名、格式、工具使用、测试、依赖和本地项目风格是否一致。
7. **最后看复用质量**：抽取的方法、工具类和参数对象是否提升业务表达、稳定复用和测试边界，而不是制造跳转和调用链复杂度。
8. **追加最小正确实现检查**：是否已经先复用现有代码、标准库、平台原生能力和已安装依赖；是否存在可删除抽象、无主配置、单实现接口或无证扩展点。
9. **追加复杂性投资检查**：这个变更是否让下次理解和修改更容易；简单需求是否出现多文件小改动、浅模块堆叠、直通包装、公共知识泄露或只为测试变绿的战术实现。
10. **追加服务拆分职责检查**：新增接口、Service、ApplicationService、Facade 或 Adapter 是否承载真实业务职责、业务抽象或稳定契约；如果只是透传调用、改名转发或一行包装，应合并回现有职责边界。

不要从格式或个人偏好开始 Review。格式问题可以指出，但不能掩盖边界、契约、数据和生产风险。

## 分层编码检查

| 层次 | 允许做 | 禁止做 |
| --- | --- | --- |
| Controller / API | 参数校验、鉴权入口、调用用例、响应转换 | 写领域规则、直接访问 Repository/Mapper、拼复杂查询、吞异常 |
| Application / Use Case | 编排业务用例、事务边界、权限校验、调用领域能力和端口 | 堆积细碎业务规则、泄露框架细节、承载复杂 SQL |
| Domain | 业务不变量、状态流转、领域规则、领域事件 | 依赖 Web、ORM、MQ、Redis、HTTP Client、第三方 SDK |
| Infrastructure | Repository、Gateway、Client、MQ、缓存、文件、SDK 适配 | 反向调用领域实现细节、写业务决策、吞外部异常 |
| Converter / Mapper | 模型转换、字段映射、枚举转换、默认值转换 | 权限判断、远程调用、数据库查询、状态流转、审计写入 |
| Test | 表达业务行为、边界、异常、兼容和回归 | 只为覆盖率造数据、过度 Mock、依赖执行顺序、无断言 |

## 重点问题定级

Review 的目标是让代码库长期健康持续变好，而不是追求单个变更完美。已经明确改善系统健康、风险可控且验证充分的变更，可以在留下非阻塞建议后通过；确定会降低系统健康、破坏契约或扩大生产风险的变更必须阻塞。

| 等级 | 编码场景 |
| --- | --- |
| P0 | 资金、权限、租户、敏感数据、生产数据、事务一致性会被破坏；可能造成不可逆事故。 |
| P1 | 公共契约破坏、核心流程错误、严重 NPE、事务吞异常、越权风险、重复扣减或状态错乱。 |
| P2 | 边界穿透、缺少幂等/测试/日志、契约映射错误或可维护性明显下降。 |
| P3 | 命名不准、注释欠清晰、局部重复、格式不一致、低风险工具方法可读性问题。 |

Review 输出要说明真实后果，不只写“建议优化”。

评论分级建议：

- 必须修改：会破坏业务语义、公共契约、数据一致性、权限安全、测试可信度或生产可恢复性的 P0/P1/P2 问题。
- Nit：低风险命名、局部可读性、格式或风格微调；可以指出，但不得阻塞健康改善明确的变更。
- Optional / Consider：存在更好方案，但当前方案基于项目证据也可接受时，用于表达取舍建议。
- FYI：只用于知识分享、未来演进提醒或非当前变更范围的问题，不要求作者在本次变更中解决。

评审争议应先回到技术事实、数据、项目规范和工程原则。纯风格问题以项目 style guide、现有代码一致性或格式化工具为准；软件设计问题通常不是个人偏好，必须说明原则、证据和风险。若作者能基于数据或稳固工程原则证明多个方案都可接受，应尊重作者在当前上下文中的选择；长期僵持时应升级给维护者、技术负责人或团队决策，不让变更无限悬空。

评论应针对代码和系统行为，不评价作者个人。要求作者解释复杂代码时，通常应推动代码重写得更清楚，或补充对未来读者也有价值的代码注释；只留在 Review 工具里的解释不能替代可维护代码。

## 规则结果消费

Java 源码任务先读取项目本地规范和 `wind-coding-conventions` 的通用 Java 层；只有存在 Wind/Nobe 高置信度信号时才叠加专项。规则 Skill 只提供适用层级和偏差，架构师负责把每条偏差映射到源码事实、业务不变量、契约、失败路径、测试和生产风险。

规则发现不得直接决定严重级别。Review 时逐条回答：

1. 规则是否适用于当前技术栈、依赖版本和项目上下文。
2. 偏差是否改变外显行为、边界、兼容、一致性、安全或恢复能力。
3. 最小修复在哪里，是否会扩大变更范围或引入新风险。
4. 哪个测试、静态检查、构建或运行证据可以证明修复成立。
5. 结论应回写项目约规、规则 Skill，还是仅作为当前代码 finding。

纯风格或无实际风险的偏差不阻塞；违反业务不变量、公共契约、安全边界或生产恢复能力的偏差按真实后果定级。

涉及外部副作用、异步消息、支付资金、批任务、重试、回调或状态机时追加时间边界三问：重启之后是否保留意图与完成事实，重放之时是否能去重，状态不明时是否能补偿、隔离或人工接管。

## 最小正确实现 / 过度设计专项 CR

该专项只回答“哪些复杂度可以不写、复用或删除”，不替代常规代码 Review。

输出每条建议时保持四要素：位置、可删内容、替代方案、不可删边界。替代方案优先级为：不做、复用仓库已有能力、标准库、平台原生能力、已安装依赖、最小显式实现。

必须标为边界外的问题：正确性缺陷、安全漏洞、性能瓶颈、业务不变量、测试可信度、生产发布风险。这些问题回到常规 Review 或对应专项，不用“过度设计”标签处理。

## Review 问题模式

- **注释与可读性重构**：Review 注释前先问能否通过命名、抽方法、常量/枚举、值对象、参数对象或测试名消除解释；好注释解释 Why / Why not，坏注释复述 What / How；删除或改写注释前先判断是否承载历史决策、兼容背景、临时降级、迁移参考或用户要求保留的信息。
- **业务规则错层**：规则写在 Controller、Converter、Mapper、Job 或工具类中。
- **贫血用例**：ApplicationService 只搬运数据，领域规则散落在多个私有方法和 if/else 中。
- **DDD 战略缺失**：代码堆满 Aggregate、Entity、ValueObject 等名词，但没有通用语言、限界上下文、上下文映射和防腐层，业务人员说的概念在代码里找不到。
- **Repository 退化 DAO**：Repository 以表、SQL 或任意查询为中心，不围绕聚合根存取，导致领域层被持久化模型牵引。
- **模糊契约**：空值、异常、权限、分页、幂等、默认值和兼容语义不清。
- **隐式副作用**：看似查询的方法实际修改状态、发消息、调用外部系统或写审计。
- **契约转换黑盒**：自动转换、反射拷贝或隐式默认值掩盖字段语义差异。
- **伪复用**：为 DTO 构建、一行包装或偶发重复提取公有方法/工具类，导致调用链变深、语义变弱。
- **浅模块堆叠**：新增很多类、接口、方法或配置，但每个只做薄转发，真实复杂性没有被封装，读者必须跨多个文件理解一个业务动作。
- **直通包装**：方法、类或变量只把调用/参数原样传到下一层，没有增加业务语义、边界保护、错误聚合、观测或测试隔离；单行透传方法无论公有或私有，都应优先删除并内联，通常意味着职责重叠或抽象位置错误。
- **透传式应用服务**：ApplicationService、Facade 或接口服务只把请求转给下游接口，没有新增用例编排、事务边界、权限/审计、状态转换、跨资源协调或对外契约隔离；这类拆分会制造似是而非的抽象，应按真实职责合并或重新命名定位。
- **信息泄露**：协议格式、第三方异常、文件结构、缓存键、状态码、金额精度或兼容规则在多个模块重复出现，导致一个规则变化需要多处同步修改。
- **多文件小改动**：简单需求牵动大量文件，每个文件只改几行；优先怀疑变化轴、模块边界或公共知识归属，而不是只按行数判断风险低。
- **过度设计 / 最小正确实现缺失**：未查已有能力就重写 helper，手写标准库 / 平台已提供功能，为单实现抽接口、为单调用建工厂、为没人配置的值加配置项、为未知未来引入依赖或脚手架。
- **注释替代表达 / AI 注释噪声**：代码依赖大段注释才能理解业务意图，或 AI 生成大量复述代码的步骤注释；先做可读性重构，再保留必要 Why 注释。
- **战术 AI 代码**：AI 生成实现只解决眼前测试或编译失败，却扩大接口、复制逻辑、弱化命名、绕过边界或制造后续维护成本；Review 时必须退后一步问设计是否变好。
- **原则误用**：用 DRY 强行抽象不同业务知识，用 KISS 压缩成难懂表达，用 YAGNI 拒绝必要边界，用 Boy Scout Rule 混入大范围重构。
- **伪 Clean 分层**：目录名看似 domain/usecase/adapter，但内层仍 import Controller、ORM Entity、Mapper、SDK 或框架配置。
- **端口倒置失败**：Use Case 依赖具体 Repository、Mapper、HTTP Client 或 SDK，端口定义在外层导致核心规则被实现细节牵引。
- **架构风格误用**：无业务 driver 和运行能力就引入微服务、CQRS、Event Sourcing、Reactive 或复杂端口体系，把问题从代码搬到网络、事件、配置和排障。
- **并发幻觉**：只在单线程路径验证，忽略重复请求、并发提交、重复消息和乱序回调。
- **时间边界缺失**：副作用已经发生但完成事实未可靠落盘，或只靠内存去重、临时令牌、日志文本和调用顺序保证“不重复”；遇到重启、重放、乱序回调或状态不明会重复扣减、重复发货、漏补偿或无法人工接管。
- **测试伪覆盖**：测试只覆盖 happy path，没有断言关键业务不变量、异常和边界。

## 原则误用 CR 提示

原则名不是问题证据，Review 必须把原则落到可观察行为：

| 说法 | 应改成 |
| --- | --- |
| 这里违反 DRY。 | 这两段是否表达同一业务知识、是否必须同步变化；若是，抽成有领域名的能力；若不是，保留重复。 |
| 这里不够 KISS。 | 指出读者需要猜测的路径、隐藏状态、嵌套条件或抽象跳转，并给出更低认知成本的结构。 |
| 这里违反 YAGNI。 | 区分“提前设计接口边界”和“提前实现未来功能”，只阻止未确认实现，不删除必要边界。 |
| 这里有副作用。 | 说明函数名、返回值和调用者预期是什么，实际行为如何违反 POLA/CQS。 |
| 这里用了 DDD 但不像 DDD。 | 指出通用语言、限界上下文、聚合一致性边界、领域事件或 Repository 语义哪个缺失。 |
| 这里应该上微服务/CQRS/Event Sourcing。 | 说明业务 driver、质量属性、收益、代价、前置条件和验证资产；否则只能作为观察或后续评估。 |

## 架构健康扫描契约

架构健康扫描是常规 Review 的补充，不替代业务语义、边界、契约和失败路径检查。开始前先固定范围和强度，选择能回答问题的最小范围：

| 模式 | 适用条件 | 默认输出 |
| --- | --- | --- |
| 最近改动 / diff | 检查当前变更或回归风险时优先使用。 | 只分析变更及其必要上下游。 |
| 指定模块 | 用户明确模块、目录或业务能力。 | 只报告模块内及直接跨界证据。 |
| 全仓 | 用户明确要求，或局部证据指向全局依赖问题。 | 先声明成本和停止条件，再扩展扫描。 |
| 仅架构层 | 只关心依赖方向、循环、模块边界和职责分布。 | 跳过低价值风格问题。 |

默认快速体检只返回 3-5 个最高价值候选；用户明确要求深度扫描才扩展类别、范围或数量。不得因为仓库较小就自动全仓扫描，也不得默认生成独立报告。

扫描结论分两层：

1. **候选信号**：来自语法形态、通用命名、规模阈值或工具输出，只说明“值得核对什么”，不定级、不直接给重构处方。
2. **确认问题**：已经有源码锚点、调用链、变更历史、运行数据或测试证据，并能说明真实后果；只有这一层才进入 P0-P3 和整改建议。

项目已有静态分析、测试、查询日志、profiler 或依赖图时，先使用这些确定性证据定位候选，再由 Review 解释业务影响、误报边界和优先级。证据不足时保留为候选信号或观察项，不补写推测。

跨语言复杂度热点按下表检查，不机械套用语言专属阈值：

| 热点 | 候选信号 | 确认前必须核对 |
| --- | --- | --- |
| N+1 / 循环内 I/O | 循环中访问数据库、HTTP、文件或远程 SDK。 | 实际调用次数，是否已有批量、缓存或预取，以及鉴权、租户、顺序、缺失和错误语义能否保持。 |
| 循环内重复线性扫描 | 循环中反复 `find`、`contains`、`indexOf` 或遍历同一集合。 | 数据上界、热路径、相等性和哈希语义；只有收益可验证时才建议索引结构。 |
| 循环内排序 | 每轮重复排序同一批或高度重叠的数据。 | 中间有序状态是否对业务可见，能否移到循环外且不改变稳定性、并列项和时序语义。 |
| 嵌套或两两比较 | 对同一集合做多层循环或笛卡尔式匹配。 | 实际规模、增长趋势、提前退出条件和算法替换后的等价性。 |
| 错误数据结构 | 用列表承担高频查找、用无序结构承载顺序语义，或以共享可变结构跨并发边界。 | 顺序、重复、内存、并发和序列化约束；不能只按理论复杂度替换。 |
| 渲染路径重复计算 | 每次渲染重复做昂贵派生、格式化、过滤或对象构造。 | profiler、重渲染证据和依赖正确性，避免为冷路径引入失效缓存。 |

小规模有界数据、冷路径、已批量/缓存的调用，或为了清晰度保留的线性实现，可以是合理取舍。此时记录证据和复核条件，不升级为问题。

输出边界：候选只写位置、信号和缺失证据；确认问题写严重级别、位置、证据、真实风险、最小整改和验证。默认不生成健康分数、坏味数量排行榜或 3-12 个月路线图；扫描结果不得自动触发重构、改代码、生成任务文件或启动多 Agent，除非用户另行授权。

## Java 架构坏味启发式扫描

当用户要求“深度 CR”“架构坏味”“代码质量扫描”“重构评估”或发现长期维护风险时，在常规 Review 后追加架构坏味扫描。扫描结论必须结合业务语义、变更频率和项目规模，以下阈值是提示信号，不是机械定罪。

| 坏味 | 识别信号 | 主要风险 | 优先建议 |
| --- | --- | --- | --- |
| 上帝类 / God Class | 单类同时编排流程、校验规则、持久化、外部调用和状态转换；非模型类过长、注入依赖过多或公开方法过多。 | 修改牵一发动全身，测试困难，业务边界被类名掩盖。 | 先按用例、领域能力、端口适配拆出职责，再用 characterization test 保护行为。 |
| 循环依赖 | 模块、包或 Bean 双向依赖；domain/core 反向依赖 web/infrastructure。 | 发布、测试、初始化和演进受阻，边界失真。 | 调整依赖方向，引入端口接口、领域事件或应用服务编排。 |
| Clean Architecture 反向依赖 | core/domain/usecase import web、infrastructure、ORM Entity、Mapper、SDK 或框架配置；只有包名像分层。 | 框架、数据库或第三方变更会迫使业务规则变更，业务测试需要启动容器或真实外部依赖。 | 将端口契约移到内层，外层 adapter 实现端口，补架构测试和不启动 DB/Web/SDK 的核心业务测试。 |
| 过长方法 | 方法混合参数解析、业务判断、状态转换、持久化和外部副作用。 | 难以证明失败路径，局部修改容易破坏不变量。 | 先标记阶段边界，再提取有领域语义的小方法或对象。 |
| Feature Envy | 一个类大量读取另一个对象字段或 getter 后做判断。 | 数据和行为分离，规则散落在调用方。 | 把规则移动到拥有数据和不变量的对象、领域服务或策略中。 |
| Data Clumps | 多个方法、类或 DTO 反复携带同一组字段，例如金额/币种、租户/主体/权限、时间窗口。 | 参数漂移，校验重复，契约不稳定。 | 引入值对象、查询对象、命令对象或上下文对象，并补充不变量测试。 |
| 霰弹式修改 | 一个需求需要同步修改多个无稳定边界的类、配置和转换器。 | 变更成本高，容易漏改和产生兼容问题。 | 识别变化轴，收敛到明确扩展点、策略或配置契约。 |
| 公共模块垃圾桶 | `common`、`util`、`manager`、`helper` 聚集业务规则、转换、外部调用和状态判断。 | 复用名义下形成隐式耦合，破坏领域语言。 | 区分通用技术能力和业务能力，业务规则回到所属模块。 |
| 浅模块 / Classitis | 类和接口数量增加，但接口并不更简单，调用方要理解更多步骤、参数和中间对象。 | 把复杂度从实现内部泄露到使用者，Review 和排障跳转成本升高。 | 合并无语义薄层，收敛为深模块，保留对主用例简单的接口。 |
| 信息隐藏失败 | 文件格式、协议知识、状态码、错误码、缓存键或第三方规则分散在多处。 | 规则变更会形成多文件小改动和漏改风险。 | 让单一模块拥有该知识，对外暴露任务级接口或值对象。 |

扫描时建议用四步输出：

```text
坏味：具体文件、类、方法或模块。
证据：依赖方向、调用链、字段组合、方法长度、变更散点或测试缺口。
风险：对业务语义、测试、兼容、发布或生产行为的真实影响。
整改：最小可验证步骤，不把坏味治理扩大成无边界重构。
```

不得只因为代码行数、方法数量或注入数量触发重构建议；如果当前类职责稳定、测试充分、变更少且拆分会增加认知负担，应标为观察项而不是问题。

## 整改建议写法

```text
问题：说明具体代码位置和行为。
风险：说明会导致什么真实后果，例如越权、重复扣减、事务部分提交、兼容破坏。
建议：给出最小可落地修改，不扩大范围。
验证：说明需要补充的单测、集成测试、契约测试或手工验证。
```

示例：

```text
[P1] OrderController 直接扣减库存并发送 MQ
风险：Web 层承载领域规则，事务边界和消息投递失败路径不清，重复提交可能导致库存状态不一致。
建议：将扣减规则下沉到应用服务/领域服务，使用幂等键保护提交，消息投递通过 Outbox 或补偿机制处理。
验证：补充重复请求、库存不足、消息投递失败和事务回滚测试。
```

## 红线

1. 不得以“代码更短”为理由隐藏业务不变量、状态变化、权限判断和副作用。
2. 不得把过度设计专项 CR 当成正确性、安全、性能或发布准出结论。
3. 不得以框架工具、反射或自动生成代码替代契约设计。
4. 不得只看风格问题而忽略边界、事务、权限、数据和生产风险。
5. 不得在 Review 中给无法执行的建议，例如“优化一下”“注意一下”“加强校验”。
6. 不得把个人偏好包装成团队规约；没有项目依据时只能作为建议并说明取舍。
