# 受控工程执行 Loop（CAD 工程 Profile）

本文定义资深架构师对单个已选定工程任务进行连续多轮执行的工程 profile。`CAD` 只保留为 `cad-mode.md` 的内部路由标识，不再作为面向用户的独立模式、Goal 类型、授权类型或状态机；对外只说明当前使用受控工程执行 Loop、授权边界、验证证据和停止条件。

## 使用时机

- 已选定一个工程任务或阶段切片，业务与架构决策已冻结，但实现、反馈、修正和验证明显需要至少两轮。
- 需要在同一写入范围内连续完成代码、测试、Review、验证和状态回写，并在失败、无进展或风险升级时自动停止。
- GSD-like 大项目编排已经选定单个 Atomic Task，需要把它交给架构师执行，而不是继续规划整个 Roadmap。

## 不适用场景

- 简单、确定、局部、一次可完成的修改；使用 `workflow.md` 的快速编码模式。
- 普通一次性交付；使用 `workflow.md` 的标准工程生命周期。
- 业务语义、公共契约、数据模型、兼容策略或风险取舍仍未确认。
- 用户只要求 Review、方案评估、概念讨论或明确不希望连续执行。

## 读取后必须产出

- 准入结论：`直接执行 / 工程 Loop 条件不足 / 可进入工程 Loop / 已验证`；Goal 状态仍使用统一词表。
- 本任务消费的 Goal / Task、写入范围、验证方式、适用授权、预算、无进展检测和停止条件；适用授权须说明是 Plan Grant 覆盖当前任务，还是单任务 Execution Grant。
- 每轮完成内容、验证证据、失败或跳过原因、残余风险和状态回写位置。

## 权威边界

- Goal、成功标准、预算和跨轮交接读取 `wise-agent/references/goal-governance.md`。
- Loop 状态、反馈、无进展检测、执行前对账和恢复读取 `wise-agent/references/delivery-execution-control.md`。
- 适用授权、Git 与高风险授权策略读取 `wise-agent/references/planning-execution-admission.md` 和项目授权规则。
- 工程生命周期、验证命令和 Git 基础规约读取 `workflow.md`。
- 大项目 Stage、Wave、Atomic Task 和上下文账本读取 `ai-large-project-orchestration.md`。

本文只定义工程 profile 的准入、轮内动作、工程停止条件和回写，不复制 Goal、Loop、Grant、Git 或 GSD 规则。

| 任务 | 优先读取 | 跳过 |
| --- | --- | --- |

## 1. 准入

受控工程执行 Loop 必须同时满足：

1. 单个 Task ID 或阶段切片已经选定，目标、预期行为和非目标清楚。
2. 业务与架构决策已冻结；待确认项不会影响当前实现。
3. 写入范围、只读范围、验证命令和工作区状态清楚，能区分用户已有改动。
4. 预计至少需要两轮实现、反馈、修正或验证；一次可完成时回到快速编码或标准工程流程。
5. Goal / Task、Loop 状态载体、反馈源、验证者、预算、无进展检测、停止条件和适用授权已齐备；适用授权是覆盖当前任务的 Plan Grant，或单任务 Execution Grant，不要求两者并存。不需要跨轮 Goal 时可使用当前任务状态，但不能只靠模型记忆。

准入缺失时只列阻断项，并回到对应权威位置补齐；不得另建 CAD 前缀的状态或授权类型。

## 2. 工程执行闭环

每轮执行：

```text
Pick -> Build/Test -> Review -> Verify -> Record -> Continue/Pause
```

- **Pick**：从已选任务中确定本轮最小目标、验证方式和停止点，不扩大 Task 范围。
- **Build/Test**：按任务测试策略选择先测试、先实现或先建立特征测试；不机械要求每轮都先 Red，但本轮结束前必须产生与风险相称的测试或等价证据。
- **Review**：检查业务语义、架构边界、项目约规、无关修改、幻觉 API、依赖和测试质量。
- **Verify**：运行本轮约定的编译、测试、lint、静态检查或人工验收，并读取真实报告；未运行或 Harness 失败不得写成通过。
- **Record**：回写完成内容、文件范围、证据、失败、残余风险、下一动作和状态。
- **Continue/Pause**：只有新增证据、缺口继续收敛、仍在授权和预算内时进入下一轮；否则暂停或升级。

测试策略由任务性质决定：明确行为变更或 Bug 可用 test-first；实现路线明确且用户要求提高编码效率时可 build-first 后集中补测；遗留行为不清时先做 characterization / diagnostic。无论顺序如何，验证债务未清零前都不能进入 `Verified`、提交或发布判断。

## 3. 工程停止条件

出现任一情况立即停止并回写证据：

- 业务语义、验收、公共契约、状态机、数据模型或兼容策略出现新的未决选择。
- 写入范围扩大、用户已有改动无法隔离、工作区状态不可控或需要改动未授权文件。
- 需要新增依赖、数据库迁移、外部访问、Git 写操作、生产操作或其它未覆盖授权。
- 涉及资金、权限、租户、安全、审计、合规或不可逆风险，需要新的 Owner 确认。
- 验证命令持续失败、Harness 失败、结果无法解释，或连续两轮没有新增证据、缺口没有收敛。
- 达到预算、最大轮次、上下文上限，或用户要求暂停、改向、撤销授权。

用户在任意时点提出暂停或修改约束都立即生效，不依赖固定等待窗口。

## 4. 授权与 Git

受控工程执行 Loop 只消费覆盖当前任务的 Plan Grant，或单任务 Execution Grant，不定义新的授权类型，也不要求重复授权。用户说“继续”“自动推进”或“进入 CAD”只表达执行意图，不能自行扩大写入、Git、联网、生产或高风险业务权限。

默认只在授权范围内读写代码、测试和文档并运行本地验证。`git add`、`git commit`、分支、push、PR、merge、部署、生产、联网、依赖安装、密钥和不可逆操作仍按用户、项目与工具层规则单独判断；平台权限始终优先。

## 5. 回写与收口

每轮最小回写：

```text
关联 Goal / Task:
Goal 状态: Active / Blocked / Verified
执行方式: 受控工程执行 Loop
状态原因 / 恢复条件:
本轮目标:
完成内容与文件范围:
测试策略:
验证命令与真实结果:
失败 / 跳过原因:
残余风险:
授权消耗 / 越界检查:
下一动作或停止原因:
回写位置:
```

大项目任务回写 Stage / Wave 状态、验证矩阵和 handoff；跨轮任务回写 Goal Ledger 或等价状态载体；当前会话可闭环的任务只需交付摘要。`Verified` 只表示当前 Task 的工程证据满足成功标准，不等于整个 Goal 关闭、Git 已授权或生产可发布。
