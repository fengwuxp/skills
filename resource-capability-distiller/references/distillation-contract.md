# 多源资源能力提炼契约

## 使用时机

当任务需要从教程或视频、代码、文档或规范、成功与失败产物中识别可复用能力，处理来源冲突，并判断每个能力单元应进入 Skill、reference、script、asset、fixture 还是任务内时读取。

## 不适用场景

- 只需摘要、翻译、润色、正式成文或资料目录。
- 已明确修改位置和内容的局部 Skill 编辑。
- 只需建立具体业务域、产品域或工程域模型，而非提炼通用 Agent 能力。
- 无法取得正文、源码或真实产物，只有标题、二手转述或模型记忆。

## 读取后必须产出

- 来源矩阵与冲突矩阵。
- 零个或多个完整能力单元。
- 每个能力单元的准入判断、最小落点和理由。
- 可复核的行为样例、验证证据、待确认和拒绝项。

## 需要继续读取的 reference

- 需要核对本方法的外部来源、读取状态和吸收边界时，读取 `source-map.md`。
- 其它材料只按当前任务明确提供的路径或来源读取；本契约不隐式要求任何外部能力包。

## 一、准入判断

同时满足下列条件，才进入能力提炼：

1. 存在跨任务、跨项目或跨人员重复使用的真实场景。
2. 能写出相对稳定的输入、动作、输出、失败边界和验收。
3. 材料提供模型默认不知道、容易反复出错或需要确定性执行的知识。
4. 至少有一种可复核验证方式，例如 fixture、解析断言、行为样例、产物对比或人工验收。
5. 来源、许可、隐私和写入范围足以支持候选产物。

任一条件不成立时，可以输出任务内结论、补证据建议或零能力；不得用“以后可能有用”替代准入证据。

## 二、来源矩阵

### 2.0 整体理解

提取局部能力前，先用自己的话写清资源整体要解决的问题、结构关系、关键术语、主要限制和可能力化范围。单个章节、代码片段或成功样例不能代替整体理解；整体仍无法成立时，停止拆分候选。

### 2.1 材料分工

| 材料类型 | 优先提取 | 常见风险 |
| --- | --- | --- |
| 教程或视频 | 操作顺序、视觉状态、时间节点、讲解中的隐含前提 | 版本过时、演示路径理想化、字幕误差 |
| 代码 | API、命令、参数、数据结构、依赖、异常和真实工具约束 | 示例不可运行、许可不明、密钥或环境耦合 |
| 文档或规范 | 契约、术语、版本、适用范围、禁止项和兼容边界 | 文档与实现漂移、营销描述、缺少生效日期 |
| 成功与失败产物 | 质量基线、不变量、验收差异、反例和恢复条件 | 样本偏差、脱离生成上下文、包含敏感信息 |

不同材料互为校验，不能由单一来源包办全部事实。教程说明怎样做，代码证明工具怎样工作，规范限定允许怎样做，产物显示什么结果才算合格。

### 2.2 来源字段

每个来源至少记录：

```text
source_id:
type:
title_or_path:
source_anchor:
version_or_date:
read_status:
applicable_scope:
license_or_usage_boundary:
freshness_risk:
sensitive_content:
```

结构化 JSON 中，上述来源字段使用非空字符串；没有敏感内容时，`sensitive_content` 也应明确写为 `none observed` 或等价说明，不用布尔值代替。

`source_anchor` 应能回到文件、符号、行号、章节、时间戳、页面或产物位置。能力单元的 `source_anchors` 使用 `<source_id>#<claim>`，其中 `source_id` 必须对应来源矩阵中的真实条目。无法实际读取时，`read_status` 必须写明失败方式，不得进入已确认事实，也不得单独支撑 `accepted` 能力。

结构化候选中的 `read_status` 使用 `read`、`read-with-limitations` 或 `unavailable`；标题、索引和搜索摘要不算已读，不能写成其它模糊状态绕过门禁。

### 2.3 冲突处理

1. 先判断是否为版本、平台、环境、角色或场景差异，不急于合并。
2. 契约事实优先使用当前适用的官方规范与可执行代码互证；产物用于核对真实行为与质量边界。
3. 同版本冲突时记录双方来源锚点、可复现实验和影响面；没有决定性证据就标为待确认。
4. 解决冲突后清除已失效旧值，保留版本适用条件；不得把多版本做法拼成一套不可执行流程。

## 三、能力单元契约

能力单元是最小可独立理解、执行和验证的复用职责。每个单元包含：

```text
capability_id / name:
stable_responsibility:
reuse_scope:
trigger_signals:
non_trigger_signals:
preconditions:
inputs:
ordered_procedure:
tools_and_parameters:
branches_and_failures:
recovery_or_stop_conditions:
outputs:
acceptance_evidence:
source_anchors:
version_and_freshness:
license_and_privacy_boundary:
recommended_destination:
```

提炼时分三层归纳，但不要强制拆成三个文件：

- **规划层**：触发、非触发、阶段顺序、分支、依赖和停止条件。
- **功能层**：稳定任务、输入、输出、工作流、质量标准和可复用方法。
- **原子层**：工具调用、参数、schema、失败模式、负例和确定性断言。

只有术语、观点、目录或背景知识而没有动作与验收的内容，不构成能力单元。

### 3.1 候选生命周期

- **候选**：已识别稳定职责，但证据、步骤、边界或验收尚未闭合。
- **接受**：完整通过校验门禁，可以进入归位与行为验证；接受不等于安装、发布或新建 Skill。
- **拒绝**：缺少复用价值、证据不足、属于常识、无法执行验证或与既有职责重复；记录失败门禁和理由，避免重复提炼。

结构化 JSON 中，`accepted` 能力必须完整提供能力单元契约：描述性单值字段使用非空字符串，触发、输入、步骤、分支、停止条件、输出、验收和来源锚点使用至少包含一个非空字符串的数组。`candidate` 可以保留尚未闭合的字段，`rejected` 至少保留稳定职责、来源锚点、失败门禁和拒绝理由。

候选、接受和拒绝是能力单元的审查状态，不要求为每个状态创建独立文件。长任务只有在确需跨轮恢复时才记录最小进度，不为一次性材料建立流水线目录。

## 四、校验门禁

| 维度 | 必答问题 | 失败处理 |
| --- | --- | --- |
| 完整性 | 输入、动作、失败分支、输出和验收是否闭合 | 补齐或拒绝 |
| 可追溯性 | 每个非通用事实能否回到来源锚点 | 降级为推断或待确认 |
| 去重 | 是否与现有职责、规则或资产语义重复 | 合并到既有权威位置 |
| 跨来源一致性 | 版本、术语、步骤和结果是否相容 | 保留差异并建立冲突矩阵 |
| 可执行性 | 工具、参数、前置条件和停止条件是否足够明确 | 转为参考知识或补实验 |
| 可验证性 | 是否存在程序化断言、对比产物或人工验收 | 不得晋升为可用能力 |
| 证据支撑 | 是否有独立语境、可执行实现或真实产物支持，而非孤立观点 | 补证据或降级为参考 |
| 外推能力 | 能否处理来源未直接回答的新场景，并得到非空泛结果 | 拒绝或保留为示例 |
| 非平庸性 | 是否提供模型默认不知道、容易反复出错或可显著减少遗漏的内容 | 不沉淀为独立能力 |

校验对象是能力行为与产物，不是文档是否写得完整。模型自述、文件数量和篇幅不能作为通过证据。

三项筛选不机械要求“两个来源”或固定通过率：证据支撑可以来自多个独立语境，也可以来自规范与可执行实现互证；外推能力必须用新场景检验；非平庸性判断内容本身，而不是措辞是否新颖。

## 五、归位矩阵

| 落点 | 适用条件 | 不适用条件 |
| --- | --- | --- |
| 新 Skill | 有独立稳定职责、清晰触发边界、完整输入输出、多个复用场景和独立验证 | 只是单篇材料、局部规则或已有能力的别名 |
| 增强既有 Skill | 既有职责覆盖，只缺触发、流程、边界或专项知识 | 会改变其主职责或抢占其他能力触发 |
| reference | 需要按场景读取的方法、知识、模板、清单或复杂分支 | 必须每次确定执行的动作 |
| script | 可确定、重复、易写错、需要真实解析、生成或校验 | 主要依赖语义判断且无法稳定参数化 |
| asset | 最终产物需要复用的模板、样例、媒体或骨架 | Agent 每次都必须读取的规则 |
| fixture | 需要守住触发、边界、回归行为或失败样例 | 只记录一次性探索过程 |
| 任务内 | 一次性、个人偏好、证据不足、许可不明或没有稳定复用价值 | 已经反复出现且可验证的稳定能力 |

先选择最小落点，再考虑组合。一个来源可以产生多个能力单元，每个单元可以有不同落点；不得把整批材料强行封装成一个顶层 Skill。

## 六、候选包契约

只有归位为新 Skill 时，候选包才至少包含：

- 小写字母、数字和连字符组成的唯一 ID。
- 只承担触发识别的名称与描述；描述同时写清正向场景和不适用边界。
- 精简入口正文：定位、核心流程、场景路由、必要红线和按需资源索引。
- 一层直连 reference；确定性动作放入 script；输出模板或媒体放入 asset。
- 至少两个正向行为样例、两个硬负例和一个不含显式名称的正例。
- 结构校验、引用校验、行为验证与安全检查。

候选包是可评审差异，不是安装结果。已有 Skill、reference、script、asset 或 fixture 的增强也必须给出最小修改点和旧值清除检查。

结构化候选使用 JSON 时，运行：

```bash
resource-capability-distiller/scripts/check_capability_candidate.py --file <candidate.json>
```

校验器只检查整体概览、来源字段、来源锚点关系、状态、`accepted` 执行字段和三项筛选证据是否齐备，不替代领域事实、来源真实性和能力价值判断。

## 七、行为验证

1. **正向触发**：覆盖典型多源材料与明确能力提炼请求。
2. **隐式触发**：不出现能力名称，仍能从“多源材料、可复用能力、资产归位”识别任务。
3. **硬负例**：普通摘要、正式成文、领域专家建模和明确局部编辑不得误触发。
4. **边界歧义**：证据不足或只存在单一原子检查时，能够选择增强既有资产或零能力。
5. **匹配对比**：用相同材料和目标比较无候选能力与使用候选能力后的产物，优先检查结构、遗漏、冲突、可执行性和验收证据。
6. **产物优先**：读取真实输出、fixture、脚本结果或人工评审，不把回答中的“已完成”当证据。
7. **相邻能力混淆**：至少一个硬负例应明确属于相邻 Skill 或既有资产，验证当前能力能够克制误触发和重复造 Skill。

验证失败时先判断是触发错误、契约缺失、来源失效、归位错误还是实现问题，只修改能回到失败证据的最小位置。

## 八、安全与停止

- 不默认联网、克隆仓库、安装依赖、执行来源脚本或读取未授权目录。
- 发现密钥、token、客户数据、生产配置或受限内容时停止复制和持久化，只保留脱敏边界说明。
- 许可不明的代码只能记录接口和行为事实，不能复制实现进入候选资产。
- 外部时效知识必须记录核验日期、适用版本和复核条件。
- 用户未授权写入时只交付分析和候选落点；未授权安装、同步、Git 或发布时必须停在候选状态。
- 材料无法支撑稳定能力时，以零能力结束；这不是失败，而是正确停止。
