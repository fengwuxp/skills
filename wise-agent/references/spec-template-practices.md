# Spec 模板最佳实践

本文定义知止者研发流程中 Spec / SDD / OpenSpec / Harness 输入的模板落地方式。它用于把意图压缩成可审、可执行、可验证的约束，不替代 `产品架构专家` 的 PRD，也不替代 `资深架构师` 的系统设计、测试和源码级 CR。

## 使用时机

- 用户要求落地 Spec、SDD、OpenSpec、ASD、SSD 或 AI 编码模板最佳实践。
- 需要把产品上下文、系分、Harness 和测试门禁压缩成一份团队能 Review 的工程 Spec。
- AI 编码经常出现“生成很快，但评审、测试、返工、上线没有变快”，需要让 Spec 直接连接 AC、测试、闸门和知识回流。
- 需要把 `GSD + Goal` 的目标、成功标准、预算 / 时间盒、停止条件和验证证据映射进 Spec。
- 需要让 Spec 成为 AI 生成生产级代码的事实来源，并检查结构化契约、正反例、边界/错误处理、测试计划和五支柱验证是否齐备。
- 需要设计 AC 编号、Given-When-Then、测试函数映射、spec-lint、AC 覆盖、漂移检查或发布门禁。
- 需要判断一个需求应该用轻量任务卡、可评审 Spec、Harness/GSD Spec、工程执行 Loop 候选 Spec 还是人工主导。

## 不适用场景

- 只写普通 PRD、产品方案、Backlog 或 User Story，优先交给 `产品架构专家`。
- 只做具体架构设计、代码修改、补测试、源码级 CR 或生产变更，优先交给 `资深架构师`。
- 只想复制外部 ASD / SSD Harness 的目录、命令、脚本、模板或安装方式；本仓库只吸收可迁移方法，不把外部 Harness 写成默认依赖。
- 缺少业务目标、owner、对象、规则和验收种子时，不直接输出完整 Spec；先回到 Round 0 或产品上下文包。

## 读取后必须产出

- Spec 强度建议：轻量任务卡、可评审 Spec、Harness/GSD Spec、工程执行 Loop 候选 Spec 或人工主导。
- Spec 分层判断：区分 PRD / SDD / 实现 Spec，说明已具备、缺失和不可让 AI 猜的层。
- 一份可复制的 Spec 骨架：背景目标、系统设计、质量保障、交付发布和决策附录。
- 一份规范事实源与可生成性检查：清晰、完整、上下文化、具体、可测试，并能回链 PRD / SDD / 实现 Spec / AC。
- 一份 Goal 映射：Goal ID、成功标准、非目标、预算 / 时间盒、停止条件和 AC / 测试 / 发布证据的对应关系。
- AC 设计规则：全局唯一编号、场景分组、Given-When-Then、测试函数映射和证据 owner。
- 闸门设计：spec-lint、构建、lint、单元测试、AC 覆盖、集成/契约测试、漂移检查、发布观测和人工升级。
- 知识回流计划：哪些事实进入项目知识库，哪些规则进入 `AGENTS.md` / `CLAUDE.md`，哪些约束进入测试、fixture、脚本或模板。
- 轻重切换判断：哪些任务可降级，哪些必须保留完整 Spec / Harness。

## 1. 模板原则

Spec 的目标不是写给 AI 的长文档，而是定义“可接受实现空间”。模板只保留能降低意图损耗、Review 成本或验证成本的内容。

核心原则：

- **最小**：同一判断只表达一次；不要让 PRD、OpenSpec、系分、计划和 Harness 互相复写。
- **显式**：隐性规则、历史坑点、禁止事项、状态不变量和外部依赖必须写出来，或者指向可检索的项目知识。
- **可验证**：每条关键约束都尽量映射到 AC、测试、lint、静态检查、漂移检查、监控或人工确认。
- **实现空间**：Spec 划边界，不复写代码；只在人类需要拍板的设计点保留取舍。
- **Form Follows Reviewer**：模板外形优先对齐团队熟悉的技术方案评审习惯，同时保证 Agent 可继续消费。
- **More Context, Less Control**：能靠项目知识解决的，不加流程层；能靠机器证据解决的，不靠人肉提醒；能一次讲清的，不拆成多份产物。

长期资产不是一次性 Spec 文档，而是测试、知识库、项目规则、fixture、脚本、验证门禁和复盘后的模板改进。

### 正式设计文档与执行资产分界

完整性框架不等于可见目录。`名 -> 事 ->（图）-> 法 -> 器 -> 验`、追踪 ID、AC、Harness 和 Goal/Loop 字段用于作者检查信息是否贯通，不应机械展开成产品设计或系分的章节骨架。正式正文按读者理解顺序组织：先讲背景与问题、目标与非目标、定性与范围，再讲概要设计、详细设计、关键流程、业务规则、接口抽象，最后收口风险、待确认和验收摘要。

三类产物分工如下：

- 产品设计 / PRD：服务业务、产品、设计和工程共同理解，正文末尾只保留可观察业务结果、关键边界和红线等验收摘要。
- 系分 / SDD：服务架构与工程评审，正文末尾只保留设计必须保护的不变量、质量目标、关键风险和验证方向。
- 实现 Spec / 执行计划：承载详细验收矩阵、AC 与测试映射、验证命令、执行 owner 和任务状态，以及 Goal、Loop、Harness、Worker / Checker、发布和回滚动作。

因此，详细验收矩阵、AC 与测试映射、验证命令、执行 owner 和任务状态进入执行计划；正式设计文档只引用其权威路径和适用版本。

只有法规、审计、合同或项目约规要求单文档追踪时，才把详细矩阵作为正式文档末尾的按需附录；不得让附录反过来打断正文主线。正式设计结论与执行资产相互引用，不复制成两个事实源。

## 1.1 PRD / SDD / 实现 Spec 三层边界

AI 编码前先判断上下文处在哪一层，不把三层混成一份大而全文档：

- **PRD**：说明为什么做、为谁做、解决什么问题、核心场景是什么、成功标准和验收摘要是什么；产品语义不足时回到 `产品架构专家`。
- **SDD / OpenSpec / 系分**：说明系统怎么承接，覆盖模块边界、数据/状态、接口抽象与契约、流程时序、技术约束和关键取舍；工程设计不足时回到 `资深架构师`。
- **实现 Spec**：说明本次代码具体写到哪里、怎么拆、用哪些现有库/风格、接口输入输出、边界用例、禁止事项、详细验收矩阵、验证命令和完成标准。

三层可以很轻，但不能假装不存在。小任务可把 PRD 写成一句业务目标、SDD 写成一段模块边界、实现 Spec 写成写入范围和验收清单；如果某层确实不适用，必须写 `N/A` 和原因。

缺层处理：

- 没有 PRD / 产品上下文：不直接开工，先补目标、主体、场景、非目标和验收种子。
- 多模块、契约、状态机、数据、权限或生产行为变更缺少 SDD：不让 AI 猜架构，先路由架构师补系统设计。
- 缺少实现 Spec：不进入编码，先输出写入范围、接口、技术约束、禁止事项和验收清单。

项目规则文件和 Spec 要分工：

- `AGENTS.md` / `CLAUDE.md` 保存长期有效的项目规则、编码约规、命令、安全边界和禁止事项。
- PRD / SDD / 实现 Spec 保存单个需求或变更的当前事实、范围、设计和验收。
- 不把全局规则复制进每份 Spec；只引用规则文件，并补充本次变更特有的约束。

## 1.2 需求 / 设计 / 编码标准门禁

Spec 进入 AI 编码前，先确认本次任务依赖的三类标准是否足够可执行。标准可以很轻，但必须服务当前项目、技术栈和风险等级，不能只引用通用规范名称。

- **需求标准**：P0/P1 需求条目具备原子性、完整性、一致性、正确性、必要性、无二义性、可行性、可验证性、可追踪性，并区分业务事实、衍生需求、待确认和非目标。
- **设计标准**：SDD / 系分能把需求追踪到模块、接口、数据、状态、流程、质量属性、测试和发布；设计方法可重复，关键图表有文字上下文和追踪 ID，不把内部实现写回需求层。
- **编码标准**：编码规则按强制 / 推荐 / 参考分级；每条关键规则说明原因、反例或示例、验证方式和适用范围，避免只写“遵循某标准”而无项目可执行细则。
- **防御式编程**：对输入校验、边界检查、类型转换、金额/精度、除零、异常传播、故障隔离、统一错误处理和无副作用表达式有明确规则，并映射到测试、静态检查或 CR 检查点。

任一类标准缺失时，Spec 只能输出缺口、owner、影响和停止条件；不得让 AI 用猜测补齐需求、设计、代码或测试。

## 1.3 规范事实源与可生成性门禁

用于生产级 AI 编码的 Spec 必须先成为事实来源。Agent 只能消费已确认的 PRD、SDD / OpenSpec / 系分、实现 Spec、AC、项目规则和源码事实；来源冲突、缺失或无法追踪时，先停下补 Spec，不用提示词连续试错。

合格规范至少满足五点：

- **清晰**：目标、主体、范围、非目标和成功标准没有二义性。
- **完整**：功能、非功能、权限、安全、异常、边界、性能、发布和回滚要求不留关键空洞。
- **上下文化**：写清架构约束、依赖、运行环境、项目风格、现有接口、性能/安全要求和历史坑点。
- **具体**：给出输入输出示例、正例 / 反例、禁止事项、状态不变量和错误处理口径。
- **可测试**：P0/P1 AC 能映射到测试、静态检查、监控或人工确认，且有验证命令或替代证据。

涉及代码生成时，还应补充结构化契约：OpenAPI、JSON Schema、TypeScript interface、DDL、接口表、消息契约或项目等价形式。确实不适用时写 `N/A` 和原因，不能让 AI 自行发明字段、路由、依赖或返回结构。

## 2. 强度选择

按风险、耦合度、后果成本和验证难度选择模板强度：

| 强度 | 使用条件 | 最小内容 |
| --- | --- | --- |
| 轻量任务卡 | 单文件、低风险、可快速验证。 | 目标、非目标、写入范围、验证命令、停止条件。 |
| 可评审 Spec | 涉及契约、状态、权限、数据、流程或多模块。 | 背景目标、设计取舍、AC、测试策略、风险自查、发布边界。 |
| Harness/GSD Spec | 长任务、多 Agent、跨模块、上下文易衰减。 | 可评审 Spec + Task ID、owner、依赖顺序、写入范围、验证矩阵和交接。 |
| 工程执行 Loop 候选 Spec | 原子任务边界清楚、验证可运行、授权明确。 | Harness/GSD Spec + Execution Grant 缺口/关联、停止条件和恢复入口。 |
| 人工主导 | 高不确定架构、资金、合规、安全、生产数据、不可逆操作。 | 人工确认、专业审批、dry-run、回滚、审计和人工验收。 |

强度只可向当前风险收敛，不为“看起来专业”加重，也不为“AI 写得快”降级。

### 2.1 渐进式 SDD 四档

渐进式 SDD 的目的不是引入更多文档，而是按风险选择最小约束：

| 需求类型 | 默认强度 | 准出重点 |
| --- | --- | --- |
| 低风险 / 小改动 | 轻量任务卡 + 人确认计划 | 写入范围、验证命令、可回滚。 |
| 中等复杂 / 多人协作 | 可评审 Spec | 边界、关键路径、AC、owner。 |
| 高风险 / 回归风险 | Harness/GSD Spec | `spec.md`、`plan.md`、`review.md`、`verify.md` 或项目等价交接物。 |
| 高不确定 / 关键链路 | TDD 优先 | 失败测试、实现、独立验证和失败回写。 |

Spec 是约束，不是文档；Spec 负责收敛，验证负责裁决。流程变重但证据没有变强时，应先减层、补上下文和机器验证，而不是继续加模板。

写作要求：

- 摘要先给当前能否继续、卡在哪个证据层、下一步 owner 是谁。
- 文档层级必须说明当前是 PRD、SDD、实现 Spec 还是混合交接包；混合交接包也要把三层内容分段呈现。
- 关联 Goal 只写 Goal ID、成功标准和停止条件摘要；详细状态和 Ledger 回 `goal-governance.md`，不要在 Spec 里复制一套目标流水账。
- 系统设计优先用图、流程、时序、边界和取舍，不用长段落堆叠。
- 涉及多对象、多状态、多团队、生产链路或 GSD / 工程执行 的 Spec，必须写系统 DNA；轻量任务可说明不适用原因，但不能让 AI 从功能清单直接推断不变量、状态和边界。
- 方案取舍只写真实分歧和不可逆判断；显而易见的技术选择一句话带过。
- 未确认的信息集中放到待确认和停止条件，不分散成多个“可能”。
- Spec / SDD / OpenSpec 是当前实现约束，不是评审过程记录；讨论过程、迭代草稿、AI 推理轨迹、被拒方案和 Goal Ledger 流水只作为过程资产链接，不复制进正文。

## 4. AC 与测试映射

AC 是 Spec 进入机器验证的主索引。推荐规则：

- 全局唯一编号，例如 `AC-001`、`AC-002`；编号一旦进入测试或 CR，不随文字润色重排。
- 每个 P0/P1 AC 应能回链 Goal 成功标准；如果不能回链，说明这是工程质量 AC、发布 AC、风险 AC 还是范围外项。
- 一个 AC 只验证一个业务事实、契约事实或质量事实。
- 按场景分组：正常路径、校验拦截、异常补偿、权限安全、数据一致性、工程质量、发布观测。
- 用 Given-When-Then 或等价结构表达，不写“体验良好”“性能合理”“尽量稳定”这类不可验证愿望。
- 每个 P0/P1 AC 必须映射到测试函数、fixture、静态检查、漂移检查、监控项或人工确认方。
- 测试函数名、fixture 名或 CR 检查点应包含 AC 编号，例如 `test_ac001_idempotent_request` 或 `AC001_IdempotentRequest`。

AC 表建议：

```text
AC ID:
关联 Goal / 成功标准:
分组:
Given:
When:
Then:
验证方式:
证据位置:
owner:
风险等级:
状态: Draft / Covered / Verified / Waived
```

不得把 `Waived` 当成通过；豁免必须说明原因、影响、替代证据和批准方。

## 5. 闸门管道

闸门不是某个脚本名称，而是一组独立于生成者的准出证据。按项目生态选择实现方式，常见顺序是：

```text
环境 / 依赖检查:
spec-lint:
构建 / 类型检查:
lint / 静态检查:
单元测试:
AC 覆盖清点:
契约 / 集成测试:
漂移检查:
发布观测 / 回滚检查:
```

`spec-lint` 至少检查：

- 必填段落是否存在。
- Spec ID、owner、状态、风险等级和写入范围是否存在。
- 关联 Goal、成功标准、预算 / 时间盒和停止条件是否存在；轻量任务可说明“不适用”。
- 需求或设计变更是否按 Delta Spec 标记 `ADDED / MODIFIED / REMOVED`，并说明影响的 PRD、SDD / OpenSpec、实现 Spec、AC、测试、接口 / 事件、发布风险和通知 owner。
- AC 编号是否唯一、稳定、无跳过误用。
- P0/P1 AC 是否有验证映射。
- 是否存在无 owner 的待确认、开放式 TODO 或执行授权缺口。
- 是否把外部文章、工具宣传或模型自述当作验证证据。
- 是否声明规范事实源，并能回链 PRD、SDD / OpenSpec / 系分、实现 Spec、AC、项目规则或源码事实。
- 用于代码生成时，是否给出结构化契约、输入输出示例、正例 / 反例、边界条件、错误处理和测试计划；缺失时是否写明 `N/A` 与原因。
- 是否把讨论过程、迭代草稿、AI 推理轨迹、被拒方案或完整 Goal Ledger 流水写进 Spec 正文。
- 是否通过需求基线稳定性门禁：系统/产品需求未确认、需求条目不可验证、图文不可追踪或衍生需求无 owner 时，不进入实现或测试。
- 是否通过系统 DNA 门禁：核心不变量、状态流转、责任边界、演化规则和验证方式缺失时，不进入 AI 编码、GSD Wave 或工程执行 Loop 候选。
- 是否通过需求 / 设计 / 编码标准门禁：关键标准没有强制/推荐分级、原因/示例、验证方式或适用范围时，不进入 AI 编码。

AC 覆盖清点至少检查：

- 每个 P0/P1 AC 是否能找到测试、fixture、静态检查、监控或人工确认记录。
- 每个 Goal 成功标准是否至少有一个 AC、测试、CR 检查点、发布监控或人工确认方。
- 测试名、fixture 名或 CR 检查点是否能回链 AC 编号。
- 删除、重命名或豁免 AC 时是否更新原因和批准方。

漂移检查可按项目选择：

- DDL / schema 与 Spec 数据约束是否一致。
- API 路由、错误码、权限、状态枚举、配置键和消息契约是否偏离 Spec。
- OpenSpec / 系分 / Harness 与代码入口、测试和发布脚本是否仍可追踪。
- CR、测试或实现发现漂移时，先分类为代码偏离、Spec 缺失或需求变化；代码偏离修实现，Spec 缺失先补 Spec 并重新 Review，需求变化先走 Delta Spec 和影响清单。

生产级 SDD 闸门至少按五支柱清点：

- **安全验证**：静态安全扫描、依赖漏洞、硬编码密钥、输入校验、SQL 注入 / XSS / 权限越界。
- **测试要求**：单元、契约、集成、关键 E2E、边界、回归和失败路径。
- **代码质量**：lint / format、复杂度、可维护性、命名、文档和项目分层规则。
- **性能验证**：响应时间、资源上限、数据库查询、缓存策略、容量或压测证据。
- **发布就绪**：配置、环境变量、日志、观测、降级、回滚、监控和告警。

失败处理：

- 任一 P0/P1 闸门失败，不进入合并或发布判断。
- Agent 可读取失败输出并修复，但必须有重试上限；超过上限升级人工 owner。
- AI 生成失败时，先把失败场景回写到 Spec / AC / 测试或项目规则，再重新生成或实现；不只换提示词重试。
- 无法执行的闸门必须说明原因、影响、替代证据和残余风险。

## 6. 风险自查

Spec 至少要把下列风险变成可 Review 的显式问题：

```text
资金 / 合规 / 隐私 / 安全:
公共契约 / 兼容性:
状态机 / 数据不变量:
事务 / 一致性 / 幂等:
外部依赖 / 超时 / 重试 / 补偿:
权限 / 租户 / 审计:
性能 / 容量 / 可观测:
发布 / 灰度 / 回滚:
人工兜底:
```

风险自查不是“全都安全”的声明，而是让 Reviewer 判断是否评估过、证据在哪里、还需要谁确认。

## 7. Reviewer 视角

Spec 应让 Reviewer 快速回答：

- 为什么做，什么不做。
- 影响哪些入口、模块、契约、状态、数据和权限。
- 哪些设计点需要人拍板，选项和理由是什么。
- 哪些 AC 覆盖正常、边界、异常、补偿和工程质量。
- 哪些证据已经机器验证，哪些只能人工确认。
- 哪些风险会阻止实现、合并或发布。

如果 Review 者只能看到一份长摘要，却找不到设计取舍、AC 映射、验证证据和源码锚点，模板应回退重写。

Spec 也是实现和 CR 的检查清单：

- AI 编码前先读本次实现 Spec 和项目规则文件，不能只读任务口令。
- 实现后按 Spec 逐项核对写入范围、接口、技术约束、禁止事项、AC 和验证命令。
- 发现偏差时，先判断是实现错了还是 Spec 不完整；Spec 不完整先更新 Spec 并重新 Review，再要求 AI 修改代码。
- 不用“顺手改一下”“补个大概”绕过 Spec；未进入 Spec 的新增范围只能标为待确认或范围外不做。

## 8. 交接和回流

交给 `资深架构师` 前，知止者输出应包含：

```text
Spec 强度:
关联 Goal:
是否可进入 OpenSpec / 系分:
是否可进入 Harness/GSD:
工程执行 Loop 候选缺口:
需要架构师补齐:
需要产品专家补齐:
需要人工确认:
验证门禁:
停止条件:
Goal Ledger 更新:
```

回流位置：

- 项目知识库：领域事实、历史坑点、隐式依赖、迁移策略和事故复盘。
- `AGENTS.md` / `CLAUDE.md`：长期有效的协作规则、编码边界、命令和禁止事项。
- Spec / AC 模板：反复缺失的字段、风险维度和验收槽位。
- 测试 / fixture / 脚本：反复出现且可机器化的问题。
- ADR / 系分：不可逆架构决策和跨团队契约变化。

不要把一次性探索、未验证猜测、客户敏感信息、密钥、生产配置、外部原文、作者口吻或工具宣传写入长期资产。
不要把过程记录回流成最终交付正文。PRD、系分、Spec 只吸收已确认结论；讨论轨迹和被拒方案继续留在 Decision Log、Goal Ledger、评审报告或任务计划。

## 9. 轻重切换

用三问法决定是否降级：

1. 是否跨多个模块、团队、状态机、数据契约、权限边界或生产行为。
2. 是否修改 schema、公共契约、状态机、外部副作用、资金/合规/安全/隐私相关路径。
3. 出错后是否难以靠常规测试、监控或人工操作低成本发现和修正。

任一为“是”，保留可评审 Spec 或 Harness/GSD Spec。三问均为“否”，且验证命令可运行、写入范围清楚、失败可低成本回滚，才可降级为轻量任务卡。

降级后仍必须保留：目标、非目标、写入范围、验证命令、停止条件和残余风险。
