# AI 代码交付闭环

本文定义 AI Native 研发流程中从意图、Spec、Harness、Agent 执行到最终可交付代码的闭环。它用于流程编排和门禁设计，不替代 `资深架构师` 的具体编码、测试、代码 Review 或生产变更 reference。

## 使用时机

- 用户要求优化 AI Coding、SDD、Spec 驱动开发、OpenSpec / Superpowers / Harness 的真实交付效果。
- 编码阶段提速明显，但端到端交付、CR、测试、对齐或返工没有明显改善。
- 需要判断 Spec 写得太重、太轻、不可验证，还是上下文和 Harness 不足。
- 需要把 GSD + Goal 的成功标准、预算 / 时间盒、状态、停止条件和 Ledger 接入交付闭环。
- 需要把 Agent Loop、`/goal`、`/loop`、auto mode 或后台 Agent 接入反馈、验证、预算和停止条件。
- 需要落地 Spec / SDD / OpenSpec 模板、AC 验收、测试映射、spec-lint、AC 覆盖或漂移检查。
- 需要把 CR 高频问题、测试失败、发布问题和复盘发现回流到项目知识、规范、模板、脚本或验证门禁。
- 需要设计“最终代码可交付”的准出条件，而不是只看 Agent 是否生成了代码。

## 不适用场景

- 只做具体代码实现、Bug 修复、补测试或源码级 CR，优先交给 `资深架构师`。
- 只写 PRD、产品方案或 Backlog 决策，优先交给 `产品架构专家`。
- 只评估某个外部 Harness、CLI、IDE 插件或厂商工具是否可用，应先回到 `code-understanding-tools.md` 和当前会话工具状态。
- 高风险资金、合规、安全、生产数据、不可逆操作或外部规则变化，必须保留专业确认、dry-run、回滚和审计边界。

## 读取后必须产出

- 当前瓶颈判断：需求质量、上下文缺失、Spec 层数、Harness 越界、生成质量、CR 认知负荷、测试验证、发布观测还是知识回流。
- 一份代码交付闭环：意图、上下文、Spec 强度、Harness 摘要、独立验证、CR 减负、发布观测和知识回流。
- 一份 Goal 交付闭环：Goal ID、成功标准、GSD Wave / Goal 映射、预算 / 时间盒、停止条件、验证证据、Ledger 更新和是否可 Verified / Closed。
- 一份最小 Spec 强度建议：轻量执行、OpenSpec、Harness/GSD、CAD 候选或人工主导。
- 一份 Spec 模板落地建议：可审骨架、AC 编号、Given-When-Then、测试映射、闸门证据和风险自查。
- 一份 Spec 回写与重试闭环：复现失败、检查规范清晰度、分类 AI 错误模式、回写 Spec / AC / 测试、重新生成或实现、独立验证和人工升级条件。
- 一份 Agent Loop 反馈闭环：状态载体、反馈源、验证者、预算 / 最大轮次、无进展检测、停止条件和回写位置。
- 一份 Harness 传感器清单：测试、lint、静态检查、架构规则、覆盖率、漂移检测、发布监控和人工确认。
- 一份回流计划：哪些经验进入 `CONTEXT.md`、`AGENTS.md` / `CLAUDE.md`、ADR、Spec 模板、测试、fixture、脚本或技能 reference；哪些只是一次性试错。
- 一份学习归位判断：哪些是项目公共知识、哪些是 Skill 方法增强、哪些是用户长期偏好，是否需要 `~/.skill-learning/` 授权。
- 一份效果指标：一次通过率、返工率、CR 轮次、缺陷密度、验证失败恢复成本、上下文重建成本和上线问题。

## 1. 交付闭环判断

AI 编码提速不等于交付提速。进入改造前，先判断瓶颈在哪一段：

```text
业务意图是否明确：
关联 Goal、成功标准和停止条件是否明确：
产品上下文是否可执行：
Spec 是否最小、显式、可验证：
Harness 是否说明 owner、顺序、写入范围、验证和交接：
Agent 是否只在授权范围内执行：
Loop 是否有状态载体、反馈源、验证者、预算和停止条件：
测试 / lint / 静态检查是否独立于生成者：
CR 是否能看懂入口路径、影响模块、源码锚点和边界变化：
发布是否可观测、可回滚、可人工兜底：
复盘是否回流到知识、规范、模板、测试或脚本：
```

判断口径：

- 如果需求、对象、规则和验收不清，先回到产品上下文包或 PRD / 系分预审。
- 如果 Goal 只有口号、没有成功标准、状态证据、预算 / 时间盒或停止条件，先回到 Goal 卡，而不是进入实现。
- 如果 Loop 只有继续执行、没有状态载体、反馈源、验证者、最大轮次、无进展检测或预算上限，先回到 Loop 契约，而不是让 Agent 自动跑。
- 如果 Agent 经常猜错隐式依赖、历史事故或领域术语，优先补上下文资产，而不是继续加流程层。
- 如果 CR 反复发现同类问题，优先把它转成 Harness 传感器、lint、ArchUnit、测试、fixture 或模板约束。
- 如果测试通过但上线后仍失败，优先检查业务验收、发布监控、数据回滚和人工兜底。
- 如果团队只统计代码量、PR 数或生成速度，应切到一次通过率、返工率、CR 轮次、缺陷密度和验证恢复成本。

## 2. 最小 Spec 强度

Spec 的目标不是写长文档，而是定义“可接受实现空间”。合格 Spec 至少满足四点：

- **减层**：从意图到实现尽量减少重复转写；同一判断不要在 PRD、OpenSpec、计划、Review 和 Harness 里反复表达。
- **上下文注入**：领域事实、历史坑点、项目规矩和隐式依赖应可被 Agent 查到，而不是每次靠猜。
- **机器验证**：关键约束必须尽量能被测试、lint、静态检查、架构规则、覆盖率、漂移检测或发布监控验证。
- **自适应强度**：Spec 强度跟风险、耦合度、后果成本和验证难度匹配；小任务轻量，高风险任务加门禁。

模板落地继续遵循 Form Follows Reviewer：Spec 外形应贴近团队已习惯的技术方案评审结构，同时保留 Agent 可消费的 AC、测试映射、写入范围、验证命令和停止条件。完整模板骨架、AC 表和闸门设计读 `spec-template-practices.md`。

强度选择：

| 强度 | 适用 | 最小产物 |
| --- | --- | --- |
| 轻量执行 | 单文件、低风险、快速验证。 | 目标、写入范围、验证命令。 |
| OpenSpec | 契约、状态、权限、数据或多模块。 | 目标、范围、非目标、规则、验收、技术约束。 |
| Harness / GSD | 长任务、多 Agent、跨模块、上下文易衰减。 | Task ID、owner、写入范围、依赖顺序、验证矩阵、交接。 |
| CAD 候选 | 原子任务边界清楚、授权清楚、验证可运行。 | OpenSpec + Harness + Execution Grant + 停止条件。 |
| 人工主导 | 高不确定架构、资金、合规、安全、生产数据、不可逆操作。 | 人工确认、专业审批、dry-run、回滚和审计。 |

## 3. Harness 三层闭环

项目级 Harness 不是工具名，而是最小运行时约束。可按三层设计：

- **Orchestrator**：阶段、owner、任务顺序、写入范围、只读范围、禁止事项、停止条件和恢复入口。
- **Knowledge**：领域术语、隐式依赖、历史事故、架构决策、迁移策略、已知坑点和项目规矩。
- **Delivery**：独立验证、CR、发布监控、回滚、复盘和知识回流。

最小 Harness 摘要应补充：

```text
Task ID:
Goal ID:
Goal 状态:
Goal 成功标准:
目标:
Spec 强度:
Owner:
来源上下文:
写入范围:
只读范围:
依赖顺序:
禁止事项:
验证命令:
独立验证证据:
CR 交接要求:
知识回流位置:
效果指标:
停止条件:
Goal Ledger 更新:
恢复入口:
```

不把外部文章中的 ASD / SSD Harness、命令体系、目录结构或脚本写成本技能默认依赖；本技能只吸收“Orchestrator / Knowledge / Delivery 分层、独立验证和知识回流”的方法。

## 4. 机器验证与独立证据

生成者不能自己给自己签字。Agent 可以生成测试、解释结果和提出修复，但准出证据必须独立可复核：

- 测试：单元、契约、集成、E2E、回归、业务流、失败路径、幂等和权限。
- 静态门禁：lint、类型检查、编译、格式、依赖、ArchUnit、PMD、SpotBugs 或项目等价工具。
- 规格追踪：OpenSpec / AC / Given-When-Then 与测试函数、fixture、接口、状态机或监控项对应。
- Goal 追踪：Goal 成功标准与 GSD Wave、Spec / AC、测试、CR 检查点、发布监控和 Ledger 证据对应。
- 覆盖与漂移：关键业务路径覆盖、上下文文件过期、生成物偏离规范、设计-代码偏差。
- 发布证据：灰度指标、错误率、延迟、告警、人工兜底、回滚演练或 dry-run。

AC 建议全局编号，并尽量映射到测试函数名、fixture 名或 CR 检查点。无法执行的验证必须说明原因、影响和替代证据，不能只写“未验证”。

Spec 闸门至少覆盖三类证据：

- `spec-lint`：检查必填段、owner、风险等级、AC 编号、写入范围和执行授权缺口。
- `ac-coverage`：检查 P0/P1 AC 是否能回链测试、fixture、静态检查、监控或人工确认。
- `drift-check`：检查 DDL、路由、错误码、权限、状态枚举、配置或消息契约是否偏离 Spec。

生产级 SDD 还要按五支柱验收：安全验证、测试要求、代码质量、性能验证和发布就绪。五支柱的具体证据在 `verification-review-release.md` 中清点；本文件只判断它们是否进入闭环。

## 4A. Spec 回写与重试闭环

AI 生成失败不是只换提示词，而是检查规范是否足以生成生产代码：

```text
复现失败:
检查 Spec 是否清晰、完整、上下文化、具体、可测试:
识别 AI 错误模式:
把失败场景回写到 Spec / AC / 测试 / 项目规则:
重新生成或重新实现:
执行独立验证:
达到重试上限时升级人工 owner:
```

常见 AI 错误模式：

- 幻觉依赖、库、API、配置项、字段或项目约定。
- 漏掉边界条件、异常路径、权限、安全或幂等要求。
- 引入 N+1、重复查询、资源泄漏、超时或性能退化。
- 错误处理口径不一致，吞异常、乱打日志或绕过审计。
- 为通过测试放宽断言、修改 fixture、删除失败路径或弱化 AC。
- 不符合项目分层、命名、事务、并发、数据一致性或发布约束。

重试次数必须有上限。超过上限、连续出现同类错误、规范仍有冲突，或涉及资金、合规、安全、生产数据和不可逆操作时，停止自动生成并升级产品 owner、架构 owner、安全或发布 owner。

## 4B. Agent Loop 反馈闭环

Agent Loop 只有在“写了、验证了、读到反馈、能纠正、会停止”时才有交付价值。最小反馈闭环：

```text
读取状态:
选择本轮动作:
执行最小变更:
运行独立验证:
读取反馈:
回写 Spec / AC / 测试 / Harness / Ledger:
判断继续 / 停止:
```

门禁：

- 每轮至少产生一种可复核反馈：测试结果、lint、CR 发现、源码锚点、用户验收、eval 或发布监控。
- 连续两轮没有新增证据、缺口没有收敛或同类错误重复出现时，停止并回写规范或升级人工 owner。
- 预算、最大轮次、上下文预算和无进展检测必须在 Loop 启动前写清。
- Loop 输出不能只写“已完成”；必须写执行轮次、验证证据、失败 / 阻塞、回写位置和下一步。

## 5. CR 减负与可理解交接

AI 生成越快，CR 的认知负荷越容易上升。进入源码级 CR 前，先让变更可理解：

```text
业务意图：
关联 Goal:
Goal 成功标准:
入口路径：
影响模块：
关键调用关系：
边界变化：
源码锚点：
Spec / AC 映射：
测试与静态检查证据：
事实 / 推断：
残余不确定性：
需要人拍板的不可逆判断：
```

CR 减负原则：

- 人审应优先看不可逆架构判断、公共契约、状态机、数据不变量、权限、安全、生产行为和边界变化。
- 高频样式、格式、空值、依赖、分层、命名、测试命名和简单反模式应尽量沉淀成机器门禁。
- 如果 Review 者无法复述“为什么改、影响哪里、怎么证明、还剩什么风险”，不能进入合并判断。
- 如果 PR 说明只有大段总结，没有源码锚点、入口路径、调用关系或验证证据，应回退到理解门禁。

## 6. 知识回流

真正值得长期保存的通常不是一次性 Spec 草稿，而是可复用的知识、规矩和验证资产。

回流位置建议：

- `CONTEXT.md` 或项目等价文件：领域术语、业务对象、关系、歧义、历史坑点、隐式依赖、禁止猜测的事实。
- `AGENTS.md` / `CLAUDE.md` 或项目等价文件：编码规范、架构约束、权限边界、常用命令、禁止操作和协作规则。
- ADR / 系分 / 架构 reference：重要取舍、不可逆判断、边界变化和迁移策略。
- Spec 模板 / AC 模板：反复缺失的业务规则、异常、权限、数据和验收槽位。
- 测试 / fixture / eval：复现缺陷、失败样例、边界样例、回归路径和模型行为样例。
- 脚本 / CI / Hook：重复出现且可机械检查的问题。
- 技能 reference：跨项目可复用的流程、路由、门禁和反模式。

不要回流：

- 一次性探索、临时 workaround、未验证猜测、作者口吻、外部文章原文、工具宣传、客户敏感信息、密钥、生产配置和用户长期学习数据。

## 6A. 授权学习与经验归位

AI 反复踩同一个坑，通常不是模型“不会”，而是稳定上下文、项目规则和用户偏好没有进入合适载体。知识回流要解决“失忆”，但不能绕过授权、隐私和仓库边界。

归位顺序：

| 经验类型 | 进入位置 | 边界 |
| --- | --- | --- |
| 项目事实、业务术语、历史坑点、禁止事项、常用命令 | 项目已有 `AGENTS.md`、`CONTEXT.md`、OpenSpec、ADR、测试或脚本 | 必须是团队公共可提交知识，不含客户敏感信息和密钥。 |
| 跨项目可复用流程、检查项、路由和反模式 | 本仓库对应 Skill 的 `references/`、fixtures 或验证脚本 | 只吸收方法，不复制外部文章原文或一次性偏好。 |
| 可机械检查的重复问题 | `scripts/`、lint、测试、fixture、CI 或本地校验 | 脚本需说明输入、输出、写入范围、失败行为和联网情况。 |
| 使用者稳定协作偏好、个人风格、长期习惯 | `~/.skill-learning/` 或 `SKILL_LEARNING_HOME` | 只有用户明确同意后才能读取或写入，默认关闭，不进仓库和安装目录。 |

回流前先问四个问题：

1. 这是不是同类任务会复用的经验，而不是当前任务噪声。
2. 它属于项目公共知识、Skill 方法、脚本门禁，还是用户个人偏好。
3. 是否已经有权威位置；如果有，只更新权威位置，其他文件只写短指针。
4. 是否涉及隐私、金融、合规、安全、生产上线、权限边界或未来默认行为；涉及则必须显示确认，不能静默学习。

可按三张轻量回流卡输出，不做后台自动扫描：

- **执行经验卡**：记录这轮任务的目标、有效做法、失败动作、正确命令、验证证据和下次可复用检查项；优先进入项目 `AGENTS.md`、脚本、测试或 Skill reference。
- **项目知识卡**：记录团队公共的业务术语、模块边界、历史坑点、常用命令、禁用操作和源码锚点；必须可提交、可复核、可被团队共享。
- **偏好理念卡**：只记录使用者稳定协作偏好、表达风格或长期工作习惯；必须先取得本地协作学习授权，只能进入 `~/.skill-learning/`，不能进仓库、安装目录或公共 Skill。

会话结束、阶段复盘、测试/CR 失败反复出现、GSD Wave 收口或发布复盘时可以触发回流判断。触发不等于写入：没有证据、没有权威位置、只是一时试错、涉及敏感信息或未经用户授权的长期偏好，都只能列为不得回流或待确认。

输出知识回流计划时，应把“已回流、建议回流、需要用户授权、不得回流”分开写。未经授权，不得创建 `~/.skill-learning/`，也不得把用户长期偏好、个人工作方式或对话历史写进本仓库。

## 7. 指标闭环

AI Native 交付指标要从“生成速度”切换到“交付可信度”：

| 维度 | 指标 |
| --- | --- |
| 交付通过 | 一次通过率、返工率、CR 轮次、阻塞项关闭时长。 |
| 工程质量 | 缺陷密度、回归失败、测试覆盖语义、静态门禁失败分布。 |
| 上下文质量 | 上下文重建成本、Spec 缺口、规范命中率、生成成功率、缺陷模式回写率、设计-代码偏差、知识回流命中。 |
| 目标闭环 | Goal 成功标准覆盖率、Goal Verified 证据完整度、Ledger 更新及时性、跨 Wave 目标漂移次数。 |
| 发布质量 | 灰度问题、告警、回滚、人工兜底、用户影响。 |
| Agent 效率 | 有效任务完成率、失败恢复成本、越界次数、人工接管次数。 |
| Loop 稳定性 | 平均轮次、无进展停止次数、预算命中次数、反馈回写率、人工升级率。 |

复盘时至少回答：

- 编码提速是否转化为端到端交付提速。
- Goal 状态是否被独立验证，是否存在 Active 过久、Verified 无证据或 Closed 无回流。
- 返工来自产品语义、上下文缺失、Spec 不可验证、Harness 越界、测试不足、CR 漏洞还是发布观测不足。
- 哪些问题可以用机器门禁前移，哪些必须保留人工 Review。
- 哪些知识已经回流，哪些需要等待更多证据。

## 8. 轻重切换和停止

可用“三问法”判断是否降级为轻流程：

1. 是否跨模块、跨团队、跨状态机、跨数据契约或跨生产行为。
2. 是否涉及资金、合规、安全、权限、隐私、外部依赖、不可逆数据或高失败成本。
3. 是否缺少可运行验证、人工确认方、回滚方案或源码锚点。

任一问题为“是”，不要为了快而轻易降级。停止条件包括：

- 目标、非目标、规则或验收无法验证。
- Goal 成功标准、状态、预算 / 时间盒、停止条件或 Ledger 证据不清。
- 写入范围、只读范围、权限边界或 Execution Grant 不清。
- 知识回流位置不清，或者试图把用户长期偏好写入仓库、安装目录或未授权学习目录。
- 测试、lint、静态检查或发布观测不可执行且无替代证据。
- Agent 输出无法回链源码、Spec、AC 或验证命令。
- Agent Loop 达到最大轮次、预算上限或无进展检测阈值。
- 需要专业确认但确认方缺失。

## 9. 反模式

- 把编码提速当成交付提速，只优化生成代码。
- 用更厚的 Spec 替代更清晰的上下文和验证。
- 多套流程引擎叠加，导致同一判断重复转写、重复 Review。
- 把 Agent 自述“测试通过”当准出证据。
- 为了让测试通过而放宽断言、删除失败测试、修改 fixture 或隐藏失败路径。
- AI 生成失败后只换提示词，不回写 Spec、AC、测试或项目规则。
- Agent Loop 只会继续，不会停；或把 Loop 轮数、PR 数、生成代码量当交付证据。
- 只做 API Spec，不覆盖业务规则、状态机、错误处理、数据约束、权限和架构约束。
- 只写 Goal 标题，不把成功标准映射到 Spec / AC、测试、CR 和发布证据。
- CR 高频问题长期靠人肉提醒，不沉淀为机器门禁。
- 复盘只写总结，不更新知识、模板、测试、脚本或指标。
