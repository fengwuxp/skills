# AI 代码交付闭环

本文定义 AI Native 研发流程中从意图、Spec、Harness、Agent 执行到最终可交付代码的闭环。它用于流程编排和门禁设计，不替代 `资深架构师` 的具体编码、测试、代码 Review 或生产变更 reference。

## 使用时机

- 用户要求优化 AI Coding、SDD、Spec 驱动开发、OpenSpec / Superpowers / Harness 的真实交付效果。
- 编码阶段提速明显，但端到端交付、CR、测试、对齐或返工没有明显改善。
- 需要判断 Spec 写得太重、太轻、不可验证，还是上下文和 Harness 不足。
- 需要把 GSD + Goal 的成功标准、预算 / 时间盒、状态、停止条件和 Ledger 接入交付闭环。
- 需要把 Agent Loop、`/goal`、`/loop`、auto mode 或后台 Agent 接入反馈、验证、预算和停止条件。
- 需要把 AI 工作流 Loop 做成生产可用闭环，明确隔离执行、可复现状态、独立验证、观测审计、人工接管、发布/回滚和责任 owner。
- 需要让 AI 在真实项目中自我挖掘需求和代码上下文、形成计划、完成低风险执行，并判断何时完成、继续、停止或人工确认。
- 需要把多套 AI 工作流、GSD / CAD / GStack / Goal / Harness / 工具链混一为一个生产可用交付口径。
- 需要落地 Spec / SDD / OpenSpec 模板、AC 验收、测试映射、spec-lint、AC 覆盖或漂移检查。
- 需要把 Superpowers v6 SDD 的 task reviewer、file handoff、progress ledger、pre-flight plan review 或最终 broad review 接入现有 AI Native Loop。
- 需要判断 Superpowers、GSD、GStack、Trellis 等 AI 编码框架应作为方法纪律、上下文状态、角色链审查还是仓库级记忆进入交付闭环。
- 需要把 CR 高频问题、测试失败、发布问题和复盘发现回流到项目知识、规范、模板、脚本或验证门禁。
- 需要把调研、早报、教程、培训、行业情报或团队知识传播做成可复用的知识生产 Loop，而不是每次从空白提示词开始。
- 需要设计“最终代码可交付”的准出条件，而不是只看 Agent 是否生成了代码。

## 不适用场景

- 只做具体代码实现、Bug 修复、补测试或源码级 CR，优先交给 `资深架构师`。
- 只写 PRD、产品方案或 Backlog 决策，优先交给 `产品架构专家`。
- 只评估某个外部 Harness、CLI、IDE 插件或厂商工具是否可用，应先回到 `code-understanding-tools.md` 和当前会话工具状态。
- 高风险资金、合规、安全、生产数据、不可逆操作或外部规则变化，必须保留专业确认、dry-run、回滚和审计边界。

## 读取后必须产出

- 当前瓶颈判断：需求质量、上下文缺失、Spec 层数、Harness 越界、生成质量、CR 认知负荷、测试验证、发布观测还是知识回流。
- 一份代码交付闭环：意图、上下文、Spec 强度、Harness 摘要、独立验证、CR 减负、发布观测和知识回流。
- 一份 Goal 交付闭环：Goal ID、成功标准、GSD Wave / Goal 映射、预算 / 时间盒、停止条件、验证证据、Ledger 更新和是否可 Verified / Closed。
- 一份最小 Spec 强度建议：轻量执行、OpenSpec、Harness/GSD、CAD 候选或人工主导。
- 一份 Spec 模板落地建议：可审骨架、AC 编号、Given-When-Then、测试映射、闸门证据和风险自查。
- 一份 Spec 回写与重试闭环：复现失败、检查规范清晰度、分类 AI 错误模式、回写 Spec / AC / 测试、重新生成或实现、独立验证和人工升级条件。
- 一份 Agent Loop 反馈闭环：状态载体、反馈源、验证者、预算 / 最大轮次、无进展检测、停止条件和回写位置。
- 一份自主交付完成判断：已读事实源、候选需求、人工确认边界、执行证据、独立验证、Complete / Loop / Stop / Human handoff。
- 一份生产可用混一模型：一个入口、一个契约、一个准出；内部术语归内部层，准出只看证据。
- 一份生产可用准出卡：真实业务入口、owner、事实来源、Spec / Contract、TDD / 验证、独立 CR、观测回滚、知识回流。
- 一份生产可用 Loop 准出判断：自动化心跳、隔离工作区、上下文资产、连接器权限、Maker / Checker 解耦、可复现状态、观测审计、人工接管和发布/回滚是否齐备。
- 一份 Harness 传感器清单：测试、lint、静态检查、架构规则、覆盖率、漂移检测、发布监控和人工确认。
- 一份 SDD v6 任务闭环判断：计划预检、AI Maker、Task Reviewer、文件化交接、progress ledger、最终整体复核和脚本授权边界。
- 一份 AI 编码框架分层判断：当前任务缺方法纪律、上下文状态、角色链审查还是仓库级记忆；外部框架只作为能力层，不作为默认工具或并列流程。
- 一份回流计划：哪些经验进入 `CONTEXT.md`、`AGENTS.md` / `CLAUDE.md`、ADR、Spec 模板、测试、fixture、脚本或技能 reference；哪些只是一次性试错。
- 一份知识生产 Loop 判断：资料范围、上下文资产、输出标准、自动化触发、来源标识、验收 owner、更新频率和停用条件。
- 一份经验归位判断：哪些是项目公共知识、哪些是 Skill 方法增强、哪些不得回流。
- 一份效果指标：一次通过率、返工率、CR 轮次、缺陷密度、验证失败恢复成本、上下文重建成本和上线问题。

## 1. 交付闭环判断

AI 编码提速不等于交付提速。进入改造前，先判断瓶颈在哪一段：

```text
业务意图是否明确：
关联 Goal、成功标准和停止条件是否明确：
产品上下文是否可执行：
Spec 是否最小、显式、可验证：
Harness 是否说明 owner、顺序、写入范围、验证和交接：
Agent 是否只在授权范围内执行：
Loop 是否有状态载体、反馈源、验证者、预算和停止条件：
Loop 是否具备隔离执行、可复现状态、Maker / Checker 解耦、观测审计、人工接管和发布/回滚：
测试 / lint / 静态检查是否独立于生成者：
CR 是否能看懂入口路径、影响模块、源码锚点和边界变化：
发布是否可观测、可回滚、可人工兜底：
复盘是否回流到知识、规范、模板、测试或脚本：
```

判断口径：

- 如果需求、对象、规则和验收不清，先回到产品上下文包或 PRD / 系分预审。
- 如果 Goal 只有口号、没有成功标准、状态证据、预算 / 时间盒或停止条件，先回到 Goal 卡，而不是进入实现。
- 如果 Loop 只有继续执行、没有状态载体、反馈源、验证者、最大轮次、无进展检测或预算上限，先回到 Loop 契约，而不是让 Agent 自动跑。
- 如果 Loop 能自动跑，但没有隔离工作区、落盘状态、独立 Checker、审计日志、人工接管、发布/回滚和人类 owner 可理解交接，不能声明生产可用。
- 如果 Agent 经常猜错隐式依赖、历史事故或领域术语，优先补上下文资产，而不是继续加流程层。
- 如果 CR 反复发现同类问题，优先把它转成 Harness 传感器、lint、ArchUnit、测试、fixture 或模板约束。
- 如果测试通过但上线后仍失败，优先检查业务验收、发布监控、数据回滚和人工兜底。
- 如果团队只统计代码量、PR 数或生成速度，应切到一次通过率、返工率、CR 轮次、缺陷密度和验证恢复成本。

## 1A. 生产可用混一模型

混一不是把 GSD、CAD、GStack、Goal、Harness、工具链合并成更大的菜单，而是压成生产交付判断链：一个入口、一个契约、一个准出。

- 一个入口：角色协作 Loop，判断当前阶段、owner、能力来源和停止条件。
- 一个契约：Engineering Loop Contract，承接目标、事实、写入范围、验证命令、Checker 和状态回写。
- 一个准出：生产可用准出卡，只看真实业务入口、独立证据、发布 / 回滚和 owner 责任。
- GSD / CAD / Goal / Harness 只做目标、计划、原子执行、状态和授权层；GStack 只做角色链审查；Wisdom Lens 只做取舍校准；工具只做能力补充。

## 2. 最小 Spec 强度

Spec 的目标不是写长文档，而是定义“可接受实现空间”。合格 Spec 至少满足四点：

- **减层**：从意图到实现尽量减少重复转写；同一判断不要在 PRD、OpenSpec、计划、Review 和 Harness 里反复表达。
- **上下文注入**：领域事实、历史坑点、项目规矩和隐式依赖应可被 Agent 查到，而不是每次靠猜。
- **机器验证**：关键约束必须尽量能被测试、lint、静态检查、架构规则、覆盖率、漂移检测或发布监控验证。
- **自适应强度**：Spec 强度跟风险、耦合度、后果成本和验证难度匹配；小任务轻量，高风险任务加门禁。

模板落地继续遵循 Form Follows Reviewer：Spec 外形应贴近团队已习惯的技术方案评审结构，同时保留 Agent 可消费的 AC、测试映射、写入范围、验证命令和停止条件。完整模板骨架、AC 表和闸门设计读 `spec-template-practices.md`。

强度选择：

| 强度 | 适用 | 最小产物 |
| --- | --- | --- |
| 轻量执行 | 单文件、低风险、快速验证。 | 目标、写入范围、验证命令。 |
| OpenSpec | 契约、状态、权限、数据或多模块。 | 目标、范围、非目标、规则、验收、技术约束。 |
| Harness / GSD | 长任务、多 Agent、跨模块、上下文易衰减。 | Task ID、owner、写入范围、依赖顺序、验证矩阵、交接。 |
| CAD 候选 | 原子任务边界清楚、授权清楚、验证可运行。 | OpenSpec + Harness + Execution Grant + 停止条件。 |
| 人工主导 | 高不确定架构、资金、合规、安全、生产数据、不可逆操作。 | 人工确认、专业审批、dry-run、回滚和审计。 |

## 3. Harness 三层闭环

项目级 Harness 不是工具名，而是最小运行时约束。可按三层设计：

- **Orchestrator**：阶段、owner、任务顺序、写入范围、只读范围、禁止事项、停止条件和恢复入口。
- **Knowledge**：领域术语、隐式依赖、历史事故、架构决策、迁移策略、已知坑点和项目规矩。
- **Delivery**：独立验证、CR、发布监控、回滚、复盘和知识回流。

最小 Harness 摘要应补充：

```text
Task ID:
Goal ID:
Goal 状态:
Goal 成功标准:
目标:
Spec 强度:
Owner:
来源上下文:
写入范围:
只读范围:
依赖顺序:
禁止事项:
验证命令:
独立验证证据:
CR 交接要求:
知识回流位置:
效果指标:
停止条件:
Goal Ledger 更新:
恢复入口:
```

不把外部文章中的 ASD / SSD Harness、命令体系、目录结构或脚本写成本技能默认依赖；本技能只吸收“Orchestrator / Knowledge / Delivery 分层、独立验证和知识回流”的方法。

### 3A. SDD v6 任务闭环

Superpowers v6 的 SDD 经验应作为 AI Native 的任务级质量环，而不是新增一个并列流程。最小闭环：

```text
pre-flight plan review:
AI Maker 执行:
实现报告 / 红绿证据:
Task Reviewer:
修复回环:
progress ledger 回写:
final broad review:
Harness v3 状态:
恢复入口:
```

判断规则：

- 计划预检先于编码：检查任务是否互相冲突、接口是否清楚、全局约束是否传递到每个任务、验证命令是否存在、是否命中授权边界。
- AI Maker 只按当前任务写入范围执行，必须交出实现报告、变更文件、测试 / lint / 静态检查证据和未验证项。
- Task Reviewer 是独立 Checker，一次读取任务 diff，同时给出规格符合度和质量结论；不得被 Maker 的解释、成本理由或“先忽略这个问题”影响。
- Reviewer 无法从 diff 判断的要求，必须标为 `can't verify from diff`，回到 controller、架构师或人工 owner 读取更大上下文，不得臆测通过。
- 每轮修复都要回写失败原因、修复动作、验证证据和下一步；同类问题重复或无新增证据时停止。
- 任务级通过后仍需 final broad review，检查跨任务契约、公共 API、错误码、状态机、数据迁移、测试覆盖、风险边界和知识回流。
- Harness v3 要求任务状态只能在 `Ready / Running / Review / Fix / Verified / Blocked / Handoff` 间流转；缺少 owner、写入范围、验证命令、恢复入口或状态回写时，不进入 AI Maker 执行。

文件化交接建议使用当前项目允许的状态载体：任务文档、review package、Goal Ledger、Harness Plan、ADR、验证矩阵或 issue。`task-brief`、`review-package` 和 `sdd-workspace` 已随 vendored SDD v6.0.3 skill 提供；上游 v6.1.0 不改变本仓库这一执行基线。只有用户或项目规则明确授权时，才可创建 `.superpowers/sdd/` 或运行这些 helper。缺少授权时，只采用方法契约，不声明外部 SDD 执行套件已启用。

## 4. 机器验证与独立证据

生成者不能自己给自己签字。Agent 可以生成测试、解释结果和提出修复，但准出证据必须独立可复核：

- 测试：单元、契约、集成、E2E、回归、业务流、失败路径、幂等和权限。
- 静态门禁：lint、类型检查、编译、格式、依赖、ArchUnit、PMD、SpotBugs 或项目等价工具。
- 规格追踪：OpenSpec / AC / Given-When-Then 与测试函数、fixture、接口、状态机或监控项对应。
- Goal 追踪：Goal 成功标准与 GSD Wave、Spec / AC、测试、CR 检查点、发布监控和 Ledger 证据对应。
- 覆盖与漂移：关键业务路径覆盖、上下文文件过期、生成物偏离规范、设计-代码偏差。
- 发布证据：灰度指标、错误率、延迟、告警、人工兜底、回滚演练或 dry-run。

AC 建议全局编号，并尽量映射到测试函数名、fixture 名或 CR 检查点。无法执行的验证必须说明原因、影响和替代证据，不能只写“未验证”。

Spec 闸门至少覆盖三类证据：

- `spec-lint`：检查必填段、owner、风险等级、AC 编号、写入范围和执行授权缺口。
- `ac-coverage`：检查 P0/P1 AC 是否能回链测试、fixture、静态检查、监控或人工确认。
- `drift-check`：检查 DDL、路由、错误码、权限、状态枚举、配置或消息契约是否偏离 Spec。

生产级 SDD 还要按五支柱验收：安全验证、测试要求、代码质量、性能验证和发布就绪。五支柱的具体证据在 `verification-review-release.md` 中清点；本文件只判断它们是否进入闭环。

## 4A. Spec 回写与重试闭环

AI 生成失败不是只换提示词，而是检查规范是否足以生成生产代码：

```text
复现失败:
检查 Spec 是否清晰、完整、上下文化、具体、可测试:
识别 AI 错误模式:
把失败场景回写到 Spec / AC / 测试 / 项目规则:
重新生成或重新实现:
执行独立验证:
达到重试上限时升级人工 owner:
```

常见 AI 错误模式：

- 幻觉依赖、库、API、配置项、字段或项目约定。
- 漏掉边界条件、异常路径、权限、安全或幂等要求。
- 引入 N+1、重复查询、资源泄漏、超时或性能退化。
- 错误处理口径不一致，吞异常、乱打日志或绕过审计。
- 为通过测试放宽断言、修改 fixture、删除失败路径或弱化 AC。
- 不符合项目分层、命名、事务、并发、数据一致性或发布约束。

重试次数必须有上限。超过上限、连续出现同类错误、规范仍有冲突，或涉及资金、合规、安全、生产数据和不可逆操作时，停止自动生成并升级产品 owner、架构 owner、安全或发布 owner。

## 4B. Agent Loop 反馈闭环

Agent Loop 只有在“写了、验证了、读到反馈、能纠正、会停止”时才有交付价值。最小反馈闭环：

```text
读取状态:
选择本轮动作:
执行最小变更:
运行独立验证:
读取反馈:
回写 Spec / AC / 测试 / Harness / Ledger:
判断继续 / 停止:
```

门禁：

- 每轮至少产生一种可复核反馈：测试结果、lint、CR 发现、源码锚点、用户验收、eval 或发布监控。
- 连续两轮没有新增证据、缺口没有收敛或同类错误重复出现时，停止并回写规范或升级人工 owner。
- 预算、最大轮次、上下文预算和无进展检测必须在 Loop 启动前写清。
- Loop 输出不能只写“已完成”；必须写执行轮次、验证证据、失败 / 阻塞、回写位置和下一步。
- 要进入生产可用准出时，继续检查隔离执行、状态持久化、Maker / Checker 解耦、观测 / 审计、人工接管、发布 / 回滚和理解债；缺项先回到 `agent-loop-engineering.md` 的生产可用门禁。

### 4C. 自主交付完成判断

当用户希望 AI 自我规划、执行并完成时，完成判断必须由证据触发，而不是由 Agent 自述触发。

```text
自主交付完成判断
关联 Goal / Task:
用户目标 / 成功标准:
已读事实源:
候选需求 / 问题:
必须人工确认项:
已执行动作:
代码 / 文档 / 测试写入范围:
验证证据:
独立 Checker / CR 证据:
状态回写位置:
残余风险:
判断: Complete / Loop / Stop / Human handoff
理由:
```

判断口径：

- `Complete`：成功标准、验收样例或 AC 已被测试、lint、源码锚点、CR、用户验收、发布监控或人工 owner 覆盖；状态已回写；残余风险有 owner。
- `Loop`：本轮产生新证据且缺口收敛，仍在预算、授权、写入范围和验证边界内。
- `Stop`：验证失败、同类错误重复、缺少权限、目标冲突、范围漂移、成本超限或无法回链证据。
- `Human handoff`：存在需求取舍、公共契约、数据 / 权限 / 安全 / 合规 / 生产 / Git / 联网 / 部署 / 不可逆操作或低置信度判断。

如果“必须人工确认项”非空，AI 可以整理证据、候选方案和推荐问题，但不能把任务标记为完成，也不能继续扩大实现范围。

## 4D. 生产可用准出卡

生产可用只按证据准出，不按 Agent 自述、PR 数、Loop 轮数或演示可运行准出。生产可用准出卡字段：真实业务入口、Owner、事实来源、Spec / Contract、写入范围、TDD / 验证、独立 CR、可用性 / 安全性 / 可靠性、发布观测、回滚 / 人工接管、知识回流、判断。

八项缺一不可：真实业务入口不是 demo / mock / 内存版业务 Service；owner 清楚；事实 / 推断 / 待确认分层；Spec / Contract 含目标、非目标、写入范围和停止条件；TDD、测试或等价验证覆盖外部契约和业务不变量；独立 CR 对齐源码锚点、项目约规、架构原则和红线；发布可观测、可回滚、可人工接管；可复用经验回流到项目上下文、测试、fixture、脚本、ADR 或 Skill。

## 5. CR 减负与可理解交接

AI 生成越快，CR 的认知负荷越容易上升。进入源码级 CR 前，先让变更可理解：

```text
业务意图：
关联 Goal:
Goal 成功标准:
自主交付控制卡:
入口路径：
影响模块：
关键调用关系：
边界变化：
源码锚点：
Spec / AC 映射：
测试与静态检查证据：
事实 / 推断：
残余不确定性：
需要人拍板的不可逆判断：
```

CR 减负原则：

- 人审应优先看不可逆架构判断、公共契约、状态机、数据不变量、权限、安全、生产行为和边界变化。
- 高频样式、格式、空值、依赖、分层、命名、测试命名和简单反模式应尽量沉淀成机器门禁。
- 如果 Review 者无法复述“为什么改、影响哪里、怎么证明、还剩什么风险”，不能进入合并判断。
- 如果 PR 说明只有大段总结，没有源码锚点、入口路径、调用关系或验证证据，应回退到理解门禁。

### 5A. AI 注释去噪与可读性门禁

AI 生成代码常见问题不是“注释太少”，而是用大量注释掩盖代码表达力不足。源码级 CR 前先做一轮可读性门禁：

```text
注释是否复述显而易见的 What / How:
是否存在步骤注释替代方法抽取:
是否存在魔法值、字符串暗号或隐式 Map/Array 契约:
是否能用命名、常量/枚举、值对象、参数对象或测试消除解释:
剩余注释是否解释业务约束、设计取舍、外部规则、历史坑点或 Why not:
删除/改写注释是否会破坏兼容说明、迁移参考或用户要求保留的信息:
```

处理规则：

- 能通过代码结构表达的，先交给 `资深架构师` 做重命名、抽方法、抽常量/枚举、建类型或补测试。
- 必须保留的注释只解释原因、边界、风险和替代方案，不复述代码表面行为。
- 如果代码必须依赖大段注释才能理解，应回到产品上下文、OpenSpec 或架构设计补语义，而不是继续让 Agent 扩写注释。
- 测试名、测试数据和关键断言可以作为可执行文档；边界行为和业务不变量优先进入测试资产。

## 5A. 上下文分层和 update-context 门禁

AI 产研协同不要把所有材料塞进一个提示词、一个 `AGENTS.md` 或一个知识库。上下文按三层组织：

| 层级 | 内容 | 默认载体 | 使用方式 |
| --- | --- | --- | --- |
| L0 项目级上下文 | 项目身份、系统架构、技术栈约束、编码规范、核心 ADR、领域术语、稳定禁止事项。 | 项目 `AGENTS.md`、`CONTEXT.md`、ADR、架构 reference、项目约规 Skill。 | 所有产品、设计、编码、CR 和验证任务都可读取，但只放稳定、公共、重复出现的规则。 |
| L1 模块级上下文 | 模块职责、接口契约、数据模型、状态流转、模块边界、历史坑点。 | 模块 `references/`、系分、OpenSpec、架构图、模块文档、测试资产。 | 只有涉及该模块时读取；不把单模块事实上推成全局规则。 |
| L2 任务级上下文 | 本次 PRD / Spec、任务卡、Harness 摘要、diff、测试结果、CR 结论、发布证据。 | 当前任务文档、Goal Ledger、Review Summary、验证日志、临时计划。 | 只服务本轮交付；完成后只把可复用部分回写到 L0/L1 或 Skill。 |

每个阶段必须有输入输出契约，保证上游产物能被下游直接消费：

- 产品 / PRD 输出业务规则、异常场景、验收种子和待确认项，交给产品专家负责。
- 系分 / OpenSpec 输出接口契约、模块边界、数据变更、风险点和验证入口，交给架构师负责。
- 任务拆解输出输入、输出、依赖、写入范围、验证命令和验收标准，交给 AI Native 编排和架构师 Harness 承接。
- 编码、单测、CR 和修复输出 diff、测试结果、源码锚点、残余风险和是否需要回写上下文，交给架构师和质量门禁承接。

`update-context` 不是新增顶层 Skill，也不是自动写知识库。它是每轮交付后的固定门禁：检查是否有新增业务规则、接口变化、模块边界变化、隐式状态流转、历史坑点、CR 高频问题或验证样例值得回写。可自动整理候选卡，但写入 `AGENTS.md`、`CONTEXT.md`、ADR、模块 reference、Skill、fixture、脚本或 CI 前，必须有证据、owner 和可审查 diff。

不得回写：一次性过程草稿、未验证推断、外部文章原文、作者口吻、个人偏好、私有对话、客户敏感信息、密钥、生产配置、为了本轮凑绿的临时 workaround。

## 6. 知识回流

真正值得长期保存的通常不是一次性 Spec 草稿，而是可复用的知识、规矩和验证资产。

回流位置建议：

- `CONTEXT.md` 或项目等价文件：领域术语、业务对象、关系、歧义、历史坑点、隐式依赖、禁止猜测的事实。
- `AGENTS.md` / `CLAUDE.md` 或项目等价文件：编码规范、架构约束、权限边界、常用命令、禁止操作和协作规则。
- ADR / 系分 / 架构 reference：重要取舍、不可逆判断、边界变化和迁移策略。
- Spec 模板 / AC 模板：反复缺失的业务规则、异常、权限、数据和验收槽位。
- 测试 / fixture / eval：复现缺陷、失败样例、边界样例、回归路径和模型行为样例。
- 脚本 / CI / Hook：重复出现且可机械检查的问题。
- 技能 reference：跨项目可复用的流程、路由、门禁和反模式。

不要回流：

- 一次性探索、临时 workaround、未验证猜测、作者口吻、外部文章原文、工具宣传、客户敏感信息、密钥、生产配置、个人长期偏好和私有对话轨迹。

## 6A. Skill 自我改进外循环

AI 反复踩同一个坑，通常不是模型“不会”，而是稳定上下文、项目规则、验证样例或 Skill 规则没有进入合适载体。知识回流要解决“失忆”，但不能绕过事实、隐私和仓库边界。

归位顺序：

| 经验类型 | 进入位置 | 边界 |
| --- | --- | --- |
| 项目事实、业务术语、历史坑点、禁止事项、常用命令 | 项目已有 `AGENTS.md`、`CONTEXT.md`、OpenSpec、ADR、测试或脚本 | 必须是团队公共可提交知识，不含客户敏感信息和密钥。 |
| 跨项目可复用流程、检查项、路由和反模式 | 本仓库对应 Skill 的 `references/`、fixtures 或验证脚本 | 只吸收方法，不复制外部文章原文或一次性偏好。 |
| 可机械检查的重复问题 | `scripts/`、lint、测试、fixture、CI 或本地校验 | 脚本需说明输入、输出、写入范围、失败行为和联网情况。 |

回流前先问四个问题：

1. 这是不是同类任务会复用的经验，而不是当前任务噪声。
2. 它属于项目公共知识、Skill 方法、脚本门禁，还是不得回流的个人/敏感材料。
3. 是否已经有权威位置；如果有，只更新权威位置，其他文件只写短指针。
4. 是否涉及隐私、金融、合规、安全、生产上线、权限边界或未来默认行为；涉及则必须人工确认，不能自动沉淀。

任务刚验证通过、CR 收口、bug 根因确认或发布复盘完成时，增加一次 Knowledge Harness 判断：这次经验是否会让下次少走弯路。值得沉淀时，只写最小结构化卡片，字段为 `症状 / 根因 / 解决方案 / 预防策略 / 适用条件 / 验证证据`；优先写入项目已有权威位置，不默认新建 `docs/solutions/`。

可按三张轻量回流卡输出，不做后台自动扫描：

- **执行经验卡**：记录这轮任务的目标、有效做法、失败动作、正确命令、验证证据和下次可复用检查项；优先进入项目 `AGENTS.md`、脚本、测试或 Skill reference。
- **项目知识卡**：记录团队公共的业务术语、模块边界、历史坑点、常用命令、禁用操作和源码锚点；必须可提交、可复核、可被团队共享。
- **Skill Improvement Card**：记录目标 Skill、触发样例、错误表现、反馈证据、最小修改位置、验证方式和不得吸收项；只用于生成可 CR 的最小 diff。

聊天交互触发外循环时，先拆成 `目标 Skill / 真实失败模式 / 可复用规则 / 权威落点 / 验证方式`。用户连续指出某模块不符合约规、职责放错位置或命名不清时，不把业务对象写进 AI Native；若结论属于 Wind、架构、产品等专项 Skill，只更新对应 Skill 或 reference，AI Native 只保留如何归位、验证和同步的流程规则。源码级反馈优先沉淀为“模型对象不处理运行时依赖、应用服务组装运行时变量、测试锁住职责边界”这类可迁移职责规则，而不是记录具体类名或当前补丁。修改 Skill 源仓库时，同步更新触发 fixture / validator；有安装副本时，验证后再同步，无法同步则报告“已验证未同步”。

会话结束、阶段复盘、测试/CR 失败反复出现、GSD Wave 收口或发布复盘时可以触发回流判断。触发不等于写入：没有证据、没有权威位置、只是一时试错、涉及敏感信息或个人长期偏好，都只能列为不得回流或待人工确认。

输出知识回流计划时，应把“已回流、建议回流、待人工确认、不得回流”分开写。不得把个人长期偏好、个人工作方式或私有对话历史写进本仓库或安装目录。

## 6B. 知识生产 Loop

知识生产 Loop 用于把高频调研、教程、早报、培训、方案沉淀和团队知识传播做成可复用流程。它不是新工具依赖，优先使用现有资料、项目规则、Skill、reference 和脚本；只有用户明确指定并授权时，才评估外部工具或联网自动化。

最小 Loop：

```text
主题 / 目标读者:
资料范围 / 来源边界:
上下文资产:
输出标准:
自动化触发:
来源标识 / 时效:
人工验收 owner:
更新 / 停用条件:
不得回流内容:
```

门禁：

- 先资产化业务背景、术语、标准格式、常用模板和历史坑点，再生成内容；不要每次从空白提示词重建上下文。
- 调研、教程和早报必须保留来源、时间、链接、读取状态和不确定性；无法核验的材料只能写成待确认。
- 高频流程可以沉淀为 Skill reference、脚本、fixture 或任务模板；不要新增顶层 Skill，除非重复场景、owner、输入输出和验证方式都稳定。
- 自动化只能处理资料收集、分拣、排版、提醒和状态回写；业务结论、产品决策、架构批准、Git、发布和合规仍需对应 owner 确认。
- 知识生产自动化不替代架构判断、CR、测试结果、Git 授权或发布审批。
- 不用字数、生成速度、早报条数或自动化次数衡量价值；看上下文重建成本、复用命中率、返工率、来源错误率和团队可用性。

## 7. 指标闭环

AI Native 交付指标要从“生成速度”切换到“交付可信度”：

| 维度 | 指标 |
| --- | --- |
| 交付通过 | 一次通过率、返工率、CR 轮次、阻塞项关闭时长。 |
| 工程质量 | 缺陷密度、回归失败、测试覆盖语义、静态门禁失败分布。 |
| 上下文质量 | 上下文重建成本、Spec 缺口、规范命中率、生成成功率、缺陷模式回写率、设计-代码偏差、知识回流命中。 |
| 知识生产 | 资料复用命中率、来源错误率、人工修订量、模板复用次数、过期内容清理率。 |
| 目标闭环 | Goal 成功标准覆盖率、Goal Verified 证据完整度、Ledger 更新及时性、跨 Wave 目标漂移次数。 |
| 发布质量 | 灰度问题、告警、回滚、人工兜底、用户影响。 |
| Agent 效率 | 有效任务完成率、失败恢复成本、越界次数、人工接管次数。 |
| Loop 稳定性 | 平均轮次、无进展停止次数、预算命中次数、反馈回写率、人工升级率。 |

复盘时至少回答：

- 编码提速是否转化为端到端交付提速。
- Goal 状态是否被独立验证，是否存在 Active 过久、Verified 无证据或 Closed 无回流。
- 返工来自产品语义、上下文缺失、Spec 不可验证、Harness 越界、测试不足、CR 漏洞还是发布观测不足。
- 哪些问题可以用机器门禁前移，哪些必须保留人工 Review。
- 哪些知识已经回流，哪些需要等待更多证据。

## 8. 轻重切换和停止

可用“三问法”判断是否降级为轻流程：

1. 是否跨模块、跨团队、跨状态机、跨数据契约或跨生产行为。
2. 是否涉及资金、合规、安全、权限、隐私、外部依赖、不可逆数据或高失败成本。
3. 是否缺少可运行验证、人工确认方、回滚方案或源码锚点。

任一问题为“是”，不要为了快而轻易降级。停止条件包括：

- 目标、非目标、规则或验收无法验证。
- Goal 成功标准、状态、预算 / 时间盒、停止条件或 Ledger 证据不清。
- 写入范围、只读范围、权限边界或 Execution Grant 不清。
- 知识回流位置不清，或者试图把个人长期偏好、私有对话轨迹、客户资料或生产敏感信息写入仓库、安装目录。
- 测试、lint、静态检查或发布观测不可执行且无替代证据。
- Agent 输出无法回链源码、Spec、AC 或验证命令。
- Agent Loop 达到最大轮次、预算上限或无进展检测阈值。
- 需要专业确认但确认方缺失。

## 9. 反模式

- 把编码提速当成交付提速，只优化生成代码。
- 用更厚的 Spec 替代更清晰的上下文和验证。
- 多套流程引擎叠加，导致同一判断重复转写、重复 Review。
- 把 Agent 自述“测试通过”当准出证据。
- 为了让测试通过而放宽断言、删除失败测试、修改 fixture 或隐藏失败路径。
- AI 生成失败后只换提示词，不回写 Spec、AC、测试或项目规则。
- Agent Loop 只会继续，不会停；或把 Loop 轮数、PR 数、生成代码量当交付证据。
- 只做 API Spec，不覆盖业务规则、状态机、错误处理、数据约束、权限和架构约束。
- 只写 Goal 标题，不把成功标准映射到 Spec / AC、测试、CR 和发布证据。
- CR 高频问题长期靠人肉提醒，不沉淀为机器门禁。
- 复盘只写总结，不更新知识、模板、测试、脚本或指标。
- 把“混一”理解为把所有模式堆到一个超大流程，而不是收敛为一个入口、一个契约和一个准出。
