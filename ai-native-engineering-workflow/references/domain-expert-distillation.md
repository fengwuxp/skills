# 业务专家蒸馏

把 PRD、系分、产品设计、接口、事件、流程图、测试和代码证据蒸馏成可追溯的领域专家 Skill Pack。本文借鉴 `cangjie-skill` 的长内容蒸馏方法，但不把业务专家做成自由发挥的人设，也不替代产品 owner、架构 owner 或正式评审。

## 使用时机

- 用户要求阅读产品 / 系分 / 设计文档，沉淀某业务域的“业务专家”“领域专家”“可问答专家”或知识库。
- 多个项目、模块或团队反复解释同一业务对象、规则、状态机、接口事件和验收口径，需要把上下文资产化。
- 进入 AI Native 角色协作 Loop 前，现有 PRD、系分或代码证据足以先形成领域理解基线，但还不足以直接执行编码。
- 需要把产品专家的业务语义、架构师的系统边界、测试证据和 owner 追认沉淀为后续可调用 Skill。

## 不适用场景

- 只有一句需求、零散聊天、未读正文的外部文章或模型记忆：先进入知识表达门禁，不蒸馏业务专家。
- 只写一份普通 PRD、产品方案、Backlog 或验收标准：优先路由 `产品架构专家`。
- 只做系分、代码实现、TDD、源码级 CR 或生产排障：优先路由 `资深架构师`。
- 只是想做摘要、会议纪要或文档改写：输出摘要即可，不创建 Skill Pack。
- 含客户数据、合同、密钥、生产配置或内部敏感材料，且没有明确写入范围和脱敏边界：停止。

## 读取后必须产出

- 一份准入结论：资料范围、目标业务域、证据成熟度、owner、写入范围、不可写入内容和是否需要补材料。
- 一份业务证据地图：文档、图、接口、事件、代码、测试、日志或 owner 追认的来源索引。
- 一份领域专家 Skill Pack 草案：领域概览、术语表、流程 / 状态、规则 / 不变量、接口 / 事件 / 数据、反例 / 风险、待确认项和 test-prompts。
- 一份三重验证结果：V1 证据交叉、V2 可回答新问题、V3 领域特异性。
- 一份答题协议：事实 / 推断 / 待确认 / 范围外不做分层表达，且每条关键结论能回链来源。
- 一份压力测试清单：应该触发、应该追问、应该拒答、跨域混淆和过期证据场景。
- 一份下一步：补证据、请 owner 追认、进入产品设计、进入系分、进入 TDD / 编码，或停止。

## 需要继续读取的 reference

- 产品语义、需求夯实、验收种子和产品上下文包读 `product-to-engineering-lifecycle.md`，并按需调用 `产品架构专家`。
- 系统边界、接口事件、代码证据、TDD 和源码级 CR 读 `verification-review-release.md`，并按需调用 `资深架构师`。
- 上下文资产、知识库治理、代码图谱和外部理解工具准入读 `code-understanding-tools.md`。
- 取舍校准、证据反偏和“名实相符”检查读 `wisdom-loop-lens.md`。
- 来源读取状态和不吸收边界读 `source-map.md`。

## 1. 准入

先问这六件事，缺关键项就停在准入：

- 业务域名称：例如支付账务、账户开户、清结算、授信风控。
- 目标用途：问答、PRD 评审、系分评审、TDD 种子、代码接手、培训或知识库。
- 资料清单：PRD、系分、流程图、接口、事件、代码、测试、日志、运营规则和 owner 说明。
- 来源边界：版本、日期、项目、模块、owner、是否已废弃、是否允许写入仓库。
- 输出范围：只生成草案、写入 `references/`、生成候选 Skill，还是仅做评估建议。
- 敏感边界：客户数据、密钥、合同、生产配置和内部材料是否需要脱敏或排除。

准入结论只给四种：

- `D0 不可蒸馏`：没有一手材料或来源不可读。
- `D1 可建证据地图`：能整理事实，但不能形成稳定专家。
- `D2 可生成专家草案`：关键规则有交叉证据，仍需 owner 追认。
- `D3 可进入安装候选`：已通过 owner 追认、压力测试和触发验证。

## 2. 抽取器

把 cangjie 的“长内容抽取”改造成业务文档抽取，不照搬书籍拆解模板：

| 抽取对象 | 目标 | 输出 |
| --- | --- | --- |
| 术语 / 主体 | 定义业务对象、角色、账户、订单、状态、权限和上下游 | `GLOSSARY.md` |
| 流程 / 状态 | 抽出主链路、分支链路、状态机、失败补偿和人工介入点 | `PROCESS_AND_STATE.md` |
| 规则 / 不变量 | 抽出必须恒成立的业务规则、资金规则、幂等、风控、审计和时效约束 | `RULES_AND_INVARIANTS.md` |
| 接口 / 事件 / 数据 | 抽出 API、事件、DTO、字段、枚举、字典、日志和指标 | `INTERFACES_EVENTS_DATA.md` |
| 反例 / 风险 | 抽出不能做、容易混淆、历史事故、过期规则和边界场景 | `RISKS_AND_COUNTEREXAMPLES.md` |
| 问题 / 追认 | 抽出证据冲突、缺失 owner、过期材料和需要业务确认的问题 | `OPEN_QUESTIONS.md` |

## 3. 三重验证

每条关键知识必须过三关：

- **V1 证据交叉**：至少两类来源互相支撑，或一个正式来源加 owner 追认 / 代码 / 测试锚点；单一 AI 摘要不算证据。
- **V2 可回答新问题**：专家草案能回答文档里没有逐字出现、但由规则可推导的业务问题；回答必须标出推断链和待确认项。
- **V3 领域特异性**：结论必须是该业务域特有的对象、规则、流程或风险；通用产品话术、工程常识和模板句不算领域专家能力。

验证失败时不要润色成“专家已成型”，只回写缺口：缺来源、缺 owner、缺反例、缺测试、缺代码锚点或缺触发样例。

## 4. 证据成熟度

- `D0`：只有愿望、口号、标题、二手转述或未读取正文。
- `D1`：已读一手材料，能列事实和待确认项。
- `D2`：关键对象、规则、流程、接口和反例完成交叉验证。
- `D3`：业务 owner / 架构 owner 追认，且通过正例、反例、跨域混淆和过期证据测试。

成熟度只能按证据上升，不按篇幅、图数量、模型信心或输出好看程度上升。

## 5. 产物结构

推荐生成候选目录，不直接安装：

```text
business-domain-expert/
  SKILL.md
  references/
    DOMAIN_OVERVIEW.md
    EVIDENCE_MAP.md
    GLOSSARY.md
    PROCESS_AND_STATE.md
    RULES_AND_INVARIANTS.md
    INTERFACES_EVENTS_DATA.md
    RISKS_AND_COUNTEREXAMPLES.md
    OPEN_QUESTIONS.md
  fixtures/
    test-prompts.json
```

`SKILL.md` 只写触发、角色定位、答题协议、何时读取 reference 和红线；长知识放 `references/`。正式 PRD、系分和 OpenSpec 不在这里改写，业务专家 Skill Pack 只是可复用知识资产。

## 6. 答题协议

业务专家回答任何问题都按四层表达：

- `事实`：来自哪份文档、哪张图、哪段接口 / 事件 / 代码 / 测试或哪位 owner 追认。
- `推断`：基于哪些事实推导，推导链哪里可能断。
- `待确认`：缺 owner、缺场景、缺边界、缺数据、缺测试或来源冲突。
- `范围外不做`：材料没有覆盖，不能替用户或 owner 做决策。

一句话原则：不知道就问，没要求的不写；能回答也要说明证据等级。

## 7. 压力测试

至少准备五类 test-prompts：

- `should_trigger`：问本领域对象、规则、状态、接口、验收和风险。
- `should_ask`：材料不足、owner 不明、日期冲突、规则冲突。
- `should_not_trigger`：普通 PRD 改写、通用编码问题、另一个业务域问题。
- `edge_case`：失败补偿、幂等、逆向流程、人工介入、灰度发布、历史兼容。
- `confusion_test`：相邻领域概念容易混淆时，必须先分域再回答。

压力测试不通过，不进入 D3，不安装，不同步到 Codex。

## 8. 老祖宗反偏问题

- **循名责实**：叫“业务专家”之前，证据是否支撑这个名？是否只是一个会说话的摘要？
- **格物致知**：结论是否来自真实文档、代码、测试和 owner，而不是模型记忆或漂亮话？
- **知行合一**：已沉淀规则是否进入触发、reference、fixture 或验证，而不是停在文档里？
- **治未病**：哪些过期文档、冲突规则、灰区责任会在后续编码和 CR 中变成事故？
- **庖丁解牛**：切分是按业务纹理、状态边界和责任边界，还是按文档标题硬切？
- **阴平阳秘**：专家能力是阳，证据边界是阴；只增强回答能力、不收紧未知边界，会变成幻觉源。

## 红线

- 不用未读取正文、模型记忆、二手摘要或标题推断业务事实。
- 不把业务专家写成自由发挥角色、万能顾问或产品 / 架构 owner 替身。
- 不补齐用户没给、材料没写、owner 未确认的业务规则。
- 不把通用产品方法、工程常识或模板话术冒充领域知识。
- 不把敏感内部材料、客户数据、密钥、合同、生产配置写入公开仓库或可同步 Skill。
- 不自动安装、同步或发布生成的业务专家 Skill；必须先过 owner 追认、压力测试和仓库验证。
