# 验证、CR、发布与复盘闭环

本文定义 AI Native 研发编码流程的验证、代码 Review、发布和复盘闭环。它用于流程编排，不替代 `资深架构师` 的测试、CR、生产变更和安全专项 reference。

## 使用时机

- 用户要求设计 AI 编码后的测试、验证、CR、发布、监控和复盘流程。
- 需要判断 Agent 输出是否可信、是否可合并、是否能进入发布。
- 需要判断某个 Goal、GSD Wave Goal 或 `GSD + Goal` 是否可 Verified / Closed。
- 需要把产品验收、eval、工程测试和生产观测连接起来。
- 需要判断陌生代码库、AI 代码变更、重构计划或 PR 说明是否能被业务 owner、架构师和 Review Agent 共同看懂。

## 不适用场景

- 单纯代码 Review、Bug 修复或生产变更细节，优先交给 `资深架构师`。
- 单纯 PRD 或产品验收模板，优先交给 `产品架构专家`。
- 高风险金融、合规、安全或生产数据变更，仍需专业方、SRE、DBA 或安全负责人确认。

## 读取后必须产出

- 验证矩阵：产品验收、eval、单元测试、契约测试、集成测试、静态检查、CR、发布监控和人工确认。
- 质量 / 测试门禁：测试矩阵、验证顺序、CR 前置条件、失败回退、残余风险交接和架构师测试能力调用点。
- 代码库理解 / 影响可视化门禁：业务意图、入口路径、影响模块、调用关系、边界变化、源码锚点、可视化辅助、owner 复述和残余风险。
- CR 重点：行为、契约、边界、测试、敏感数据、权限、可靠性和发布风险。
- 发布与复盘：灰度、回滚、观测指标、问题复盘和知识沉淀。
- 一份当前可信度判断：可以合并、需要补验证、只能继续 CR，还是必须回退到产品/架构补规格。
- 一份 Goal 完成度判断：Goal ID、成功标准、已验证证据、未验证项、是否可 Verified / Closed、残余风险和下一 owner。
- 一份残余风险清单：未跑的验证、无法确认的业务/合规/安全项、上线前人工确认方和回滚依赖。

## 1. 验证矩阵

AI Native 流程中的验证应覆盖四类证据：

| 验证层 | 证明什么 | 常见证据 |
| --- | --- | --- |
| 产品验收 | 是否解决正确问题。 | 业务 owner 验收、用户反馈、场景验收、运营检查。 |
| Eval | AI 原型、模型能力或智能行为是否稳定。 | 样例集、失败样例、通过率、人工复核、模型版本。 |
| 工程测试 | 实现是否满足契约和边界。 | 单元测试、契约测试、集成测试、E2E、回归测试。 |
| 交付治理 | 是否可合并、可发布、可回滚、可观测。 | lint、静态检查、CR、发布计划、监控、告警、回滚方案。 |

最小验证矩阵：

```text
Goal ID / 成功标准:
需求 / OpenSpec 条款：
Spec / AC 编号：
验收种子：
Eval 样例：
测试用例：
静态检查：
AC 覆盖：
漂移检查：
CR 检查点：
发布监控：
人工确认方：
残余风险：
```

PRD / 系分合议预审进入验证矩阵时，至少把接受项、拒绝项、待定项、分歧项和风险清单映射到验收种子、测试用例、CR 检查点、人工确认方和残余风险。`ACCEPT` 不代表已实现，`REJECT` 不代表风险消失，`PENDING` 不得进入执行授权。

`GSD + Goal` 进入验证矩阵时，至少把父 Goal、Wave Goal、成功标准、预算 / 时间盒、停止条件和 Ledger 最近证据映射到验收种子、测试用例、CR 检查点、发布监控、人工确认方和下一 owner。

易用性要求：

- 先给结论：当前能否继续、卡在哪个证据层、下一步最小动作是什么。
- 验证矩阵只列当前任务需要的证据层；不强制每次输出全量测试百科。
- 对未执行或无法执行的验证，必须说明原因、影响和替代证据，不能只写“未验证”。

## 2. Eval 与测试关系

Eval 和测试不是同一个东西：

- Eval 验证模型、Agent、原型或智能行为在样例集上的表现。
- 工程测试验证代码契约、业务规则、边界、异常、权限和集成。
- 用户验收验证实际工作流是否达成业务目标。

Eval 通过不代表代码可上线；测试通过也不代表产品方向正确。高风险场景必须三者都有明确边界。

## 3. 质量 / 测试门禁

质量 / 测试门禁不是独立测试专家 Skill，而是 AI Native 流程中的协作位：

- **OpenSpec 规定测什么业务事实**：目标、范围、非目标、业务规则、接口契约、数据约束和验收。
- **产品专家提供验收种子**：正常、边界、异常、运营、数据和人工兜底样例。
- **资深架构师设计和实现测试**：测试策略、TDD、补测试、测试分层、测试代码 Review 和工程验证落到 `testing.md`。
- **本技能编排质量门禁**：验证顺序、CR 前置条件、失败回退、残余风险交接、发布前人工确认和复盘沉淀。

门禁清单：

```text
测试矩阵：
验证顺序：
CR 前置条件：
失败回退：
残余风险交接：
架构师测试能力调用点：
```

只有当团队反复需要跨项目测试策略治理、测试资产审计、Flaky Test 治理、覆盖率/变异测试、AI Eval 与工程测试打通、测试数据治理、发布质量门禁模板或质量度量体系时，才考虑独立抽出测试质量专家 Skill。

## 4. 代码库理解 / 影响可视化门禁

AI 生成代码后，Review 的瓶颈常常不是“有没有说明”，而是人是否能看懂代码库结构、源码锚点和影响范围。进入代码 CR 或发布前，先做理解门禁：

```text
业务意图：
入口路径：
变更摘要：
影响模块：
关键调用关系：
边界变化：
数据 / 契约 / 权限影响：
源码锚点：
事实 / 推断：
置信度：
验证证据：
可视化辅助：
owner 复述：
残余不确定性：
```

门禁规则：

- 先让 Agent 把代码库或 diff 从“文件列表”翻译成“业务意图 -> 入口路径 -> 结构影响 -> 源码锚点 -> 验证证据”，再进入源码级 CR。
- 对跨模块、重构、接口迁移、状态机或公共契约变更，必须说明谁调用谁、哪条边界发生变化、哪些调用方需要验证。
- AI 快速阅读、上下文生成和可视化可以是依赖图、调用导览、模块边界图、diff 覆盖层、仓库指令文件或源码锚定的结构视图；它们只辅助理解，不替代源码、测试和人工判断。
- 代码库理解结论包必须区分“源码事实”和“模型推断”，关键结论必须能回链到文件路径、函数、类型、配置、测试或命令证据。
- 人类 owner 或架构师需要能复述：为什么改、影响哪里、风险是什么、哪些证据已验证、哪些仍不确定。
- 如果只能得到大段 Markdown 总结，却无法定位关键源码、调用关系或边界变化，应回退到只读侦察或架构师 CR，不进入合并判断。

适用触发：

- 陌生代码库需要快速形成入口路径、模块职责、运行方式和风险结论。
- AI 一次生成多文件变更，Review 者看不清整体影响。
- 重构计划涉及模块迁移、类型职责变化、调用链调整或公共接口变化。
- 新人接手、跨团队交接或业务 owner 需要参与代码影响确认。
- 需要用 AI 快速阅读、图形、导航、源码锚点或导览方式辅助理解，但当前工具能力、语言支持、权限和联网边界仍需按项目实际核验。
- 直接点名 Gemini CLI、AgentRC 或要求安装 / 调用代码理解工具时，先回到 `code-understanding-tools.md` 做安装准入、只读范围、联网/认证/写入边界和输出交接判断。

不吸收边界：

- 不把任何单一 AI 代码阅读工具、可视化 CLI、IDE 插件、上下文生成器或外部工具写成默认依赖。
- 不默认联网、上传仓库、读取密钥、安装 npm 包或生成会被自动执行的指令文件；需要时必须取得项目和用户授权。
- 不因为图看起来合理就跳过测试、静态检查、代码 Review 或发布门禁。
- 不把 AI 总结、上下文文件或可视化输出当成架构真相；它必须能回链到源码、OpenSpec、Harness 和验证证据。

## 5. CR 顺序

AI 代码 Review 优先顺序：

1. 是否回指 OpenSpec、产品上下文和验收种子。
2. 是否满足关联 Goal 的成功标准、预算 / 时间盒、停止条件和 Ledger 更新要求；Goal 状态不能替代验证证据。
3. PRD / 系分预审中的 `ACCEPT`、`REJECT`、`PENDING` 是否已经映射到实现范围、非目标、验证证据和人工确认方。
4. 是否能说明业务意图、入口路径、影响模块、调用关系、边界变化、源码锚点和残余不确定性。
5. 是否满足当前 Spec 强度和 Harness 摘要，尤其是 owner、写入范围、只读范围、验证命令、独立验证证据、知识回流位置和停止条件。
6. 是否满足 Spec 模板的 AC 编号、Given-When-Then、测试映射、spec-lint、AC 覆盖、漂移检查和风险自查要求。
7. 是否越过写入范围、分层边界或权限边界。
8. 是否破坏公共契约、兼容性、状态机或数据不变量。
9. 是否有正常、边界、异常、权限、幂等和回归验证。
10. 是否泄露敏感信息、吞异常、隐藏外部调用或扩大依赖。
11. 是否引入无主依赖、过度抽象、无用代码或无法维护的生成物。
12. 是否有发布、监控、回滚和人工处理边界。
13. CR 高频问题是否已经沉淀为测试、lint、静态检查、fixture、模板或知识回流计划，而不是继续靠人肉提醒。

## 6. AI 产物风险

常见风险：

- 幻觉 API、工具类、配置项或项目约定。
- 为了让测试通过而放宽断言、修改 fixture 或删除失败路径。
- 局部代码看似正确，但破坏公共契约或跨模块语义。
- 异常、日志、审计、权限、租户和敏感数据处理缺失。
- 生成大量重复或过度抽象代码，提高 Review 和维护成本。
- 未说明无法运行测试的原因，却把结果包装成完成。
- 用冗长文字总结掩盖入口路径缺失、结构影响不清、调用关系不明或边界变化无人确认。
- 只证明“编码完成”，没有证明 Spec 强度合适、Harness 独立验证、CR 减负、知识回流和交付指标。

## 7. 发布门禁

进入发布前至少回答：

- 当前变更服务哪个目标、OpenSpec 条款或验收场景。
- 当前变更服务哪个 Goal、GSD Wave Goal 或成功标准，是否具备 Verified 证据。
- Review 者是否已经理解入口路径、影响模块、关键调用关系、边界变化、源码锚点和残余不确定性。
- 已执行哪些验证命令，结果是什么。
- 哪些验证无法执行，原因和残余风险是什么。
- 是否涉及配置、数据、公共契约、权限、安全、生产行为或外部依赖。
- 灰度、回滚、监控、告警和人工兜底是否明确。
- 谁负责业务验收、工程验收和上线确认。
- 是否可以把 Goal 标记为 Verified 或 Closed；若不能，Ledger 下一 owner 和下一动作是什么。

## 8. 观测和回滚

AI 生成代码进入生产后，更需要观测：

- 核心业务指标：转化、完成率、失败率、处理时长、人工介入率。
- 工程指标：错误率、延迟、吞吐、资源、重试、超时、队列积压。
- 质量指标：缺陷、回归、投诉、回滚、告警、异常路径。
- AI 指标：eval 漂移、模型版本、提示词/上下文变更、误判样例。

回滚不只是一条命令，还包括数据回滚、配置恢复、任务暂停、消息补偿、人工处理和用户沟通。

## 9. 复盘闭环

复盘要回答：

- 问题是否来自产品判断、上下文缺失、OpenSpec 不清、Harness 越界、Agent 失误、测试缺口、Review 漏洞还是发布监控不足。
- Review 是否因为入口路径、结构关系、调用链、边界变化或影响范围没有被共同看懂而漏掉风险。
- 编码提速是否转化为一次通过率提升、返工率下降、CR 轮次下降、缺陷密度下降或验证失败恢复成本下降。
- Goal 是否在 GSD Wave、Spec、CR、发布和复盘中保持同一套成功标准、状态、证据和交接 owner。
- 哪些经验应进入产品 reference、架构 reference、脚本、fixture 或项目文档。
- 哪些只是一次性试错，不应沉淀为长期规则。
- 是否需要更新 eval 样例、测试用例、CR 清单、Harness 模板或验证脚本。
- 是否需要把领域事实回流到 `CONTEXT.md` 或项目等价文件，把编码规矩回流到 `AGENTS.md` / `CLAUDE.md` 或项目等价文件；仓库 Skill 只沉淀跨项目可复用流程和门禁。

长期协作偏好和用户背景不得写入仓库；必须遵守 `AGENTS.md` 的本地协作学习授权。

交付完整性要求：

- CR 结论必须区分已修复问题、剩余阻塞、残余风险和后续建议。
- 发布建议必须同时覆盖业务验收、工程验证、监控、回滚和人工兜底。
- 复盘建议必须说明应沉淀到产品 reference、架构 reference、脚本、fixture、项目文档还是不沉淀。

## 10. 指标

评估 AI Native 流程，不只看速度：

| 维度 | 应看指标 |
| --- | --- |
| 发现质量 | 问题证据强度、机会拒绝率、验证周期、复议命中率。 |
| 交接质量 | 上下文包完整度、OpenSpec 缺口、Harness 越界次数、变更影响复述通过率。 |
| 工程质量 | 测试覆盖语义、CR 问题密度、返工率、缺陷率。 |
| 发布质量 | 灰度问题、回滚、告警、人工兜底、用户影响。 |
| Agent 效率 | 有效任务完成率、失败恢复成本、上下文重建成本。 |
| 学习效果 | eval 样例增长、模板修正、验证脚本新增、反模式减少。 |
