# 产品到工程的 AI Native 生命周期

本文定义从产品发现、AI 原型、eval、PRD-Lite 到工程交接的流程。它用于流程编排，不替代 `产品架构专家` 的产品建模、PRD 和 Backlog 专项 reference。

## 使用时机

- 用户要求设计 AI 时代产品研发流程、产品到研发交接、PRD-Lite 或 Product Builder 工作方式。
- 用户已有 AI 原型、MVP、业务 dogfooding、页面截图、交互稿、客户反馈、需求池或 Backlog，需要判断能否进入研发。
- 用户提到 `pm-skills`、产品判断成流程、产品动作链、路线图取舍、发布复盘或增长实验，需要先判断产品判断动作链是否足以交给工程。
- 用户提到业务架构规划、业务能力地图、战略到项目组合、项目组合治理、投资取舍、重复建设识别或能力-项目-系统映射，需要先路由 `产品架构专家` 形成业务架构结论，再判断是否足以交给工程。
- 用户要求把产品业务目标、验收种子和中大型研发推进串成 Goal 组合，例如 `GSD + Goal`。
- 用户希望把产品专家、架构师、AI 工具和研发团队串成端到端流程。
- 用户要求把文章观点、口头目标、业务知识、非标问题或 AI 生成草稿整理成可进入 PRD、OpenSpec、GSD、CAD、Loop 或实际编码的输入。

## 不适用场景

- 只写完整 PRD、产品方案或 Backlog 决策时，优先读 `product-architecture-expert` 对应 reference。
- 只处理系统设计、编码、测试或生产变更时，交给 `资深架构师`。
- 涉及金融、合规、隐私、安全、外部规则时，必须由产品专家标明待确认方，并由架构师处理工程风险。

## 读取后必须产出

- 当前输入处于哪一层：想法、问题地图、AI 原型、产品候选、Hardened Candidate、工程交接包或执行任务。
- 端到端流程：阶段、输入、输出、owner、AI 工具角色、门禁和停止条件。
- 一份知识表达 / 意图可执行性判断：目标、对象、状态/规则、约束/非目标、验收样例、反馈源和缺口 owner 是否足以进入下一阶段。
- 若是非标问题，输出非标问题处理包：问题主体、影响面、已知事实、关键不确定性、研究路径、候选方案、最小可逆实验、决策标准和停止条件。
- 需要产品专家或架构师继续处理的最小交接项。
- 产品判断 Loop 准入结论：当前是否已有证据、判断动作、取舍结论、不做项、下一产物、owner 和验收 / 停止条件；不足时回 `产品架构专家` 补 `product-judgment-action-chain.md`。
- AI Native 交接结论：当前成熟度、下一步 owner、是否进入 OpenSpec、是否需要 Harness/GSD、CAD 候选缺口和必须停止的原因。
- 若进入中大型或跨轮推进，输出 Goal 组合输入：业务目标、成功标准、非目标、验收种子、预算 / 时间盒、停止条件和产品侧确认方。
- 一份可继续工作的最小产品上下文：目标/非目标、对象/状态、流程/规则、验收种子、风险/待确认和工程交接条件。
- 若信息不足，输出 Round 0 补齐清单，不伪装成已可进入研发。

## 1. 生命周期总览

AI Native 产品研发流程不是“先写长 PRD，再让 AI 写代码”。更稳的链路是：

```text
机会/反馈/业务意图
-> 问题地图
-> AI 原型 / eval / dogfooding
-> 产品上下文包
-> Backlog 决策
-> PRD-Lite / OpenSpec 输入
-> Goal 组合 / GSD Goal
-> 可评审 Spec 模板
-> 架构 hardening
-> Harness / GSD / CAD
-> AI 代码交付闭环
-> 验证、CR、发布和复盘
```

每一层都要回答一个不同问题：

| 层级 | 回答的问题 | 主要 owner |
| --- | --- | --- |
| 问题地图 | 真实问题是什么，证据是什么，失败成本是什么。 | 产品专家 / 业务 owner |
| AI 原型 / eval | 方向是否值得继续，模型或工具能否支撑。 | 产品专家 / 设计 / 业务 owner |
| 产品上下文包 | 工程化必须保留哪些业务事实、对象、规则和验收。 | 产品专家 |
| 业务架构规划 | 战略意图如何落到能力、项目组合、系统支撑和知识库回流。 | 产品专家 / 业务 owner |
| Backlog 决策 | 哪些进入本期，哪些拒绝、延后或继续验证。 | 产品专家 / PO |
| OpenSpec 输入 | 工程要做什么、边界是什么、验收是什么。 | 架构师 |
| Goal 组合 / GSD Goal | 目标如何跨 Wave 持续推进，成功标准、状态、预算、停止和交接如何保持一致。 | AI Native / 产品专家 / 架构师 |
| 可评审 Spec 模板 | 人类能否 Review，Agent 能否执行，机器能否验证。 | 架构师 / 工程 / QA |
| Harness / GSD / CAD | 谁执行、改哪里、按什么顺序、怎么验证、何时停。 | 架构师 |
| AI 代码交付闭环 | Spec 强度是否合适、Harness 是否独立验证、CR 是否减负、知识是否回流。 | 架构师 / 工程 / QA |
| 验证和发布 | 结果是否可信、可回滚、可观测、可复盘。 | 架构师 / 工程 / QA / SRE |

瘦身边界：本流程只给出端到端准入、owner、顺序、停止条件和 AI Native 交接结论；产品上下文包细节回 `产品架构专家`，工程执行细则回 `资深架构师`。

## 1B. 产品判断 Loop 准入

当输入来自访谈、工单、竞品、数据、路线图、PRD、发布复盘或增长实验时，AI Native 不直接把材料改写成系分、TDD 或编码任务。先检查产品专家是否已经形成产品判断动作链：

```text
产品判断 Loop 准入卡

原始材料:
产品判断动作链阶段:
已知事实 / 证据:
判断动作:
取舍结论:
不做项 / 延后项:
下一产物:
产品 owner:
验收 / 停止条件:
工程交接结论:
```

准入规则：

- 有材料但无判断，回 `产品架构专家` 做 `product-judgment-action-chain.md`。
- 有判断但无取舍，不能进入工程排期；先补拒绝 / 延后理由和路线图取舍。
- 有取舍但无验收种子，不能进入 TDD 或编码；先补 Product Context Card。
- 有验收但待确认项影响公共契约、资金、合规、生产、权限或数据口径时，必须标明 owner 和停止条件。
- 只有证据、取舍、owner、验收和停止条件同时具备，才允许进入 OpenSpec、系分、TDD 或编码 Loop。

## 1A. 可开发系统秩序链

当输入是一组散乱想法、页面愿望、AI 原型或“做一个平台/系统”的粗需求时，先把它组织成可开发系统，再进入 OpenSpec 或 Harness。推荐链路：

```text
模糊需求
-> 结构化需求文档
-> 产品 / 系统 DNA
-> 业务流 / 状态 / 规则
-> 原型或页面说明
-> 开发执行任务
-> 验收与发布路径
```

每一层只回答下一层需要的最小信息：需求文档固化目标、范围、角色、前后台分工、数据、安全和验收；产品 / 系统 DNA 固化核心对象、不变量、状态流转、责任边界、演化规则和验证方式；业务流串起主流程、逆向、异常、人工兜底和跨端/跨系统关系；原型说明页面状态、输入输出、权限、校验、错误和埋点；开发任务再拆接口、模型、数据、测试、部署和验收。若前一层只剩口号、愿望清单、页面片段或未确认规则，不能跳到代码生成、GSD Wave 或 CAD 候选。

## 2. 阶段门禁总则

阶段门禁只回答“能否进入下一层”，不替代产品专家的 PRD，也不替代架构师的 OpenSpec、Harness 或代码执行。若缺用户目标、产品边界、验收种子、状态 / 规则、反馈源或 owner，只输出补齐清单和停止条件。

## 2A. Round 0：问题和证据

进入 AI 原型或 PRD 前，至少明确：

- 用户、主体、业务 owner、验收方。
- 业务目标、非目标、成功指标和停止条件。
- 证据来源：客户反馈、工单、数据、访谈、竞品、政策、运营观察或事故复盘。
- 当前替代方式和失败成本。
- 涉及资金、合规、隐私、安全、生产运营或外部规则的待确认方。

若用户输入主要是功能请求、页面愿望、AI 原型描述或二手需求转述，先补一张“需求分析结论卡”，再进入 PRD-Lite、OpenSpec、GSD 或 CAD 候选：

```text
需求分析结论卡

原始诉求：
根源需求：
核心用户 / 被影响主体：
当前替代方式：
产品定义：本产品要提供的对象、能力和可见价值行为
产品边界：本产品负责 / 合作方负责 / 外部系统负责 / 本期不做
稳定点：本期必须保持稳定的对象、规则、流程或验收口径
变化点：可能变化的规则、入口、策略、配置、渠道或上下游约定
边界坐标：从哪个范围判断稳定与变化，例如单功能、产品域、平台、生态或合作链路
进入 PRD / OpenSpec / GSD 结论：
```

规则：

- 功能请求不是需求定义；先说明用户在哪个任务、流程、角色或约束下被卡住，再决定产品能力。
- 产品边界是后续系统拆分、模块边界、接口契约和上下游分工的依据；边界不清时不能跳到系分、代码生成、GSD Wave 或 CAD 候选。
- 稳定点 / 变化点必须带边界坐标；同一事项在单功能、平台或合作生态范围下可能有不同稳定性。
- 只为已确认或高概率演进的变化轴预留扩展点；未发生、无证据、无 owner 的未来想象进入待确认或非目标，防止过度设计。
- 产品专家负责补根源需求、产品定义、边界和验收种子；架构师负责把已确认边界映射到模块、接口、数据、测试和发布风险。

## 2B. Round 0A：需求基线稳定性

进入 PRD-Lite、OpenSpec、SDD、测试设计、GSD Wave 或 CAD 候选前，先判断上游需求是否已形成可验证基线。产品需求、系统需求或外部规则未确认时，不让软件侧用“合理推断”补齐，也不让 AI 直接生成实现、测试或 Harness 任务；应记录缺口、影响范围、确认 owner 和未确认期间默认处理。

最小基线：

- 需求条目说明外部可见行为，而不是内部实现方案。
- P0/P1 需求具备原子性、完整性、一致性、必要性、无二义性、可行性、可验证性和可追踪性。
- 图、流程、原型或表格有正文解释、需求 ID 或验收种子回链；不能只靠图表达业务结论。
- 衍生需求、异常恢复、安全监控、状态异常处理和鲁棒性要求标明来源、owner、验收方式和是否需要回写上游。
- 迭代可以继续，但每轮必须抬升已验证基线；不能在错误、缺失或未确认基线上滚动扩大设计和代码。

## 2C. Round 0B：产品 / 系统 DNA 门禁

进入 PRD-Lite、OpenSpec、SDD、GSD Round 0 或 CAD 候选前，先确认功能不会按“需求 -> 功能 -> 事故 -> 补规则”的顺序野蛮生长。至少回答：

- 产品 DNA：核心对象、业务不变量、生命周期 / 状态、责任边界、演化规则和验收方式是否清楚。
- 系统 DNA：系统要保护哪些不变量、状态流转、接口/数据边界、演进约束和验证门禁。
- 产品 DNA 与系统 DNA 是否能互相追踪：产品不变量是否映射到 AC、状态机、接口契约、数据库约束、测试、监控和 CR。
- 未确认项是否集中列出 owner、影响范围和停止条件，而不是让 AI 在实现阶段补猜。

DNA 门禁不要求一次写成完整架构。轻量需求可以用 5-8 行卡片；但多对象、多状态、多团队、资金/权限/库存/审计、生产链路或中大型 GSD/CAD 任务，必须先过 DNA 门禁，否则只能回退产品专家、架构师或 Round 0 补齐。

## 2D. Round 0C：业务同质性和价值成本门禁

当用户要求“技术平台产品化”“架构升级”“复用一套能力”“进入 GSD/CAD 产研协同”或把 AI 原型快速工程化时，先做业务 / 产品 / 技术交叉准入。目标是确认要交付的是生产可用业务能力，而不是技术自洽、Demo 好看或 AI 随机扩张出来的模块。

最小门禁：

- 主要矛盾：当前产品解决哪个具体主体、具体任务、具体约束下的问题；不要把抱怨、愿望或二手转述直接写成产品价值。
- 价值函数：用户、业务、运营或风险侧获得什么可观察收益；至少有一个核心点必须足够强，而不是所有点平均可用。
- 成本函数：研发、测试、实施、运营、客服、合规、数据、迁移、排障和长期维护的组合成本是否被识别。
- 业务同质性：要复用的客户、场景、对象、状态、规则、流程、权限、数据口径和验收方式是否足够相似。
- 技术复制成本：架构、平台、工具或 AI 原型降低了哪类业务复制成本；若只是增加抽象、配置、治理和排障成本，应回退重审。
- 非标边界：哪些高度个性化、结构特殊或收益不足的部分进入非目标、人工流程、项目化交付或后续观察，而不是强行平台化。

未通过时，只能输出“业务同质性缺口 / 价值成本缺口 / 架构适配风险 / 下一步 owner”，不能进入 GSD Wave、CAD 候选、Execution Grant 或代码生成。通过后，产品专家继续补产品上下文包和验收种子，架构师再把同质业务能力映射到模块、接口、数据、测试、监控和发布风险。

## 2E. Round 0D：问题核心诊断门禁

当输入混有情绪、功能愿望、工具点名、文章观点、管理要求或“先做起来”的战术动作时，先抓核心问题。AI Native 不直接把这些输入改写成 PRD、OpenSpec、GSD Wave、CAD 候选或 Loop 计划；必须先判断它们指向的业务 / 系统病机。

最小诊断顺序：

1. **症状**：当前看到的现象、痛点、失败、投诉、返工、测试失败、交付阻塞或组织摩擦是什么，证据来自哪里。
2. **核心问题**：它真正卡住的是用户任务、业务对象、规则边界、系统结构、协作顺序、工具权限还是验证闭环。
3. **整体边界**：从哪个范围看问题，单功能、产品域、系统域、团队流程、平台生态或生产链路；范围不同，结论不同。
4. **系统结构**：涉及哪些角色、对象、状态、依赖、接口、数据、测试、发布和 owner；不要只看局部动作。
5. **可证伪实验**：下一步用什么最小材料证明判断对错，例如访谈/数据、验收种子、失败测试、只读代码侦察、CR 清单、监控指标或灰度验证。
6. **停止条件**：证据不足、边界不清、owner 不明、验证不可运行、超出授权或风险不可控时停在哪里，交给谁补齐。

可复制诊断卡：

```text
问题核心诊断卡

输入来源:
症状 / 证据:
核心问题:
整体边界:
系统结构:
不是问题核心的噪声:
可证伪实验:
owner / 下一步:
停止条件:
进入 PRD / OpenSpec / GSD / CAD / Loop / 工具准入结论:
```

规则：

- 不用流程勤奋掩盖战略懒惰：会议、模板、GSD、Loop、AI 工具、补测试、CR 或重构都只是手段，必须回到核心问题。
- 不把外部文章观点、工具总结或模型解释当事实；只能作为假设或检查项，必须回链当前用户目标、源码锚点、验收种子或验证证据。
- 产品问题交给产品专家继续收敛为产品定义、价值/意义边界、对象、规则和验收；系统问题交给架构师继续收敛为边界、模块、接口、数据、测试和发布风险。
- 若当前只能看到症状，输出待验证问题地图和下一步证据获取，不扩写成完整方案。

## 2F. Round 0E：知识表达门禁

软件交付不是把自然语言“翻译成代码”这么简单，而是把业务知识表达成可执行、可验证、可维护的系统事实。进入 PRD-Lite、OpenSpec、GSD、CAD、Loop 或实际编码前，先判断输入是否已经具备“意图可执行性”。

Knowledge-to-Execution Card：

```text
原始输入:
业务目标:
业务对象:
状态 / 生命周期:
规则 / 策略:
约束 / 非目标:
验收样例:
反馈源:
需要保留的领域知识:
待确认缺口:
下一 owner:
进入 PRD / OpenSpec / GSD / CAD / Loop / 编码结论:
```

门禁规则：

- 文章观点、管理要求、产品口号、工具能力、AI 草稿和用户愿望只能作为输入，不等于可执行知识。
- 目标没有落到对象、状态、规则、约束和验收样例时，不进入代码修改或自动 Loop。
- 反馈源不清时，Loop 会放大猜测；先补用户反馈、数据、测试、eval、源码锚点或人工验收路径。
- 待确认缺口必须有 owner 和影响范围；不得让 Agent 在实现阶段用“合理推断”补业务事实。

## 2G. Round 0F：非标问题门禁

非标问题通常没有固定流程、固定模板或单一正确答案；它需要先把问题结构化，再决定交给产品专家、架构师或 AI Agent。典型信号包括：老板 / 客户只给方向、跨业务和系统边界、目标含糊、冲突约束多、影响面不明、历史上没有标准解法，或继续执行会把风险转嫁给下游。

非标问题处理包：

```text
问题主体:
影响面:
已知事实:
关键不确定性:
现有替代方案 / 绕行方式:
研究 / 证据路径:
候选方案:
最小可逆实验:
决策标准:
owner / 下一步:
停止条件:
```

处理规则：

- 产品侧非标问题先回到用户任务、业务价值、失败成本、验收种子和解决方案假设；产品专家不是传话筒。
- 工程侧非标问题先回到系统边界、接口契约、状态不变量、验证命令、灰度 / 回滚和生产风险；架构师不是任务转派器。
- AI Native 只负责编排 owner、顺序、门禁和交接；不在证据不足时替产品或架构给出确定结论。
- 非标问题可以进入最小实验，但不能直接进入大范围 GSD Wave、CAD 候选、持续 Loop 或生产代码改造。

## 2H. Round 1：AI 原型 / Eval

AI 原型不是需求本身，只是验证工具。需要记录：

- 原型验证了什么假设。
- 哪些用户或内部团队试用过。
- 已知限制、不可复用实现和模型能力假设。
- eval 样例、通过标准、失败样例和复议条件。
- 模型能力不足时是否进入“下代模型复议池”。

## 2I. Round 2：Hardened Candidate

满足下列条件，才可进入工程交接：

- 业务 owner 和结果责任明确。
- 目标、非目标、停止条件可验证。
- 核心对象、状态、流程、规则、权限和数据口径能支撑主链路。
- 正常、边界、异常、人工兜底、数据和风险验收有种子样例。
- 高风险项已有专业确认方或进入研发前门禁。

## 3. 产品上下文包

产品上下文包比传统长 PRD 更短，但必须比 prompt、原型或聊天记录更可靠。

最小结构：

```text
业务意图：
业务 owner：
目标 / 非目标：
证据来源：
用户与主体：
核心对象与状态：
产品 / 系统 DNA：
流程与规则：
前后台 / 多端 / 运营能力分工：
AI 原型 / eval 结论：
数据、指标与审计：
验收种子：
Goal 输入：
风险与待确认：
工程交接条件：
```

产品上下文包可以嵌入 PRD、OpenSpec、项目任务文档或 `docs/ai-orchestration/`，不强制创建独立文件。关键是让后续工程任务能追溯到业务事实和验收。

可读性要求：

- 先用 3-5 行说明业务意图、当前成熟度和下一步动作，再展开细节。
- 对象、状态、规则和验收用短清单表达；只有需要比较产品侧/架构侧责任时才用表格。
- 不用大而全模板压过当前任务；缺失信息集中放到“待确认”，不要分散在每个段落里。
- 产品上下文包和 PRD-Lite 只保留工程化必须消费的最终业务事实、对象、规则、验收和待确认项；AI 原型讨论、dogfooding 过程、合议争论、迭代草稿和被拒方案放入任务计划、评审报告、Decision Log 或 Goal Ledger，通过链接追溯。

### 3A. 三卡交接协议

端到端流程用三张轻量交接卡连接产品、AI Native 和架构师，避免一个大文档同时承担事实、执行和生产 Loop 三类责任。

**Product Context Card / 产品上下文交接卡** 由产品专家主导，回答“哪些产品事实必须被工程保持”：

```text
Product Context Card:
业务目标 / 非目标:
业务 owner / 验收 owner:
事实证据:
核心用户 / 被影响主体:
核心对象 / 状态 / 不变量:
主流程 / 逆向 / 异常 / 人工兜底:
关键规则 / 权限 / 数据口径:
验收种子:
风险 / 待确认 / 专业确认方:
交给 AI Native / 架构师的边界:
```

**Engineering Handoff Card / 工程执行交接卡** 由 AI Native 编排，交给架构师消费，回答“谁做、改哪里、怎么验证、何时停”：

```text
Engineering Handoff Card:
当前状态:
关联 Goal / Spec / AC:
Wave / Task ID:
写入范围 / 只读范围 / 禁止事项:
依赖顺序:
验证命令 / 人工验收:
授权策略 / Git 策略:
质量 / 测试门禁:
事实 / 推断 / 待确认 / 范围外不做:
失败回写位置:
下一 owner / 停止条件:
```

**Production Loop Card / 生产 Loop 交接卡** 只在需要持续推进、后台 Agent、`/goal`、`/loop`、auto mode 或生产可用 Loop 时使用，回答“循环如何可控、可审计、可接管”：

```text
Production Loop Card:
Loop 目标 / 状态载体:
自动化心跳 / 触发条件:
隔离执行方式:
Maker / Checker 分工:
反馈源 / 独立验证者:
预算 / 最大轮次 / 无进展检测:
观测审计:
人工接管:
发布 / 回滚:
理解债和残余风险:
```

规则：

- Product Context Card 不是 OpenSpec、Harness Plan、GSD Roadmap、CAD 候选、Execution Grant 或上线批准。
- Engineering Handoff Card 不是产品事实来源；产品事实缺失时回产品专家，不由架构师猜测补齐。
- Production Loop Card 不是自动授权；缺状态载体、独立验证、预算、停止条件、人工接管或发布/回滚时，只能停在 Loop Candidate。
- 三卡都不是 Execution Grant、测试通过、CR 结论或上线审批。
- 三卡必须区分事实、推断、待确认和范围外不做；证据不足的内容不能升格为任务、实现或授权。

## 4. 角色协同

| 角色 | AI 时代新增职责 | 不应承担 |
| --- | --- | --- |
| 产品专家 | 问题地图、AI 原型/eval、上下文包、Backlog、验收种子。 | 不直接授权编码上线。 |
| 架构师 | OpenSpec、Harness、验证矩阵、Agent 工作流、工程风险。 | 不替代产品 owner 做业务价值判断。 |
| 工程师 | 系统 owner、Agent 执行管理、复杂问题实现、代码复核。 | 不被动接收模糊 Demo。 |
| QA / 测试 | eval、验收种子、回归矩阵、质量信号和失败样例。 | 不只在末端补测试。 |
| 设计 / UED | 快速原型、服务蓝图、失败恢复、信任和控制体验。 | 不只交静态视觉稿。 |
| AI Agent | 调研、原型、代码、测试、Review、摘要和复盘辅助。 | 不承担最终业务、合规和生产责任。 |

## 5. PRD-Lite

PRD-Lite 不等于低质量 PRD，而是只保留进入研发必须的信息：

- 背景和目标：为什么做，本期成功是什么。
- 范围和非目标：做什么，不做什么。
- 角色和主体：谁使用，谁负责，谁验收。
- 对象和状态：核心对象、字段口径、生命周期和权限。
- 流程和规则：主流程、异常、人工兜底和规则版本。
- 数据和审计：指标、埋点、日志、报表和数据 owner。
- 验收种子：正常、边界、异常、不能接受情况。
- 风险和待确认：专业确认方、工程门禁和发布风险。

如果只剩页面控件说明、交互路径和“让 AI 生成”的提示词，不应进入研发。

PRD-Lite 进入 OpenSpec 或系分前，可用 `prd-system-design-review.md` 做合议预审：挑战者位先从产品价值、交付运营、技术可行性、QA 验收和 UX / 服务体验挑刺；主笔 / 决策 owner 再把每条反馈落成 `ACCEPT`、`REJECT` 或 `PENDING`。通过预审只代表低级问题和分歧已提前暴露，不代表正式产品评审或业务 owner 已批准。

## 6. 工程交接清单

交给架构师前，先形成一段 AI Native 交接结论：

```text
当前成熟度：想法 / 原型 / Hardened Candidate / OpenSpec 输入 / CAD 候选
下一步 owner：产品专家 / 架构师 / 工程 / QA / 业务确认方
是否需要 Goal 组合：是/否，Goal ID / 成功标准 / 预算时间盒 / 状态 / 停止条件：
是否进入 OpenSpec：是/否，缺口：
是否需要 Harness/GSD：是/否，理由：
CAD 候选缺口：Task ID / 写入范围 / 验证命令 / Execution Grant / 停止条件
必须停止的原因：产品语义不清 / 风险确认缺失 / 权限不足 / 验证不可运行
```

交接材料至少说明：

| 交接项 | 产品侧给出 | 架构侧承接 |
| --- | --- | --- |
| 目标和非目标 | 业务目标、成功指标、停止条件 | OpenSpec、质量属性、技术非目标 |
| Goal 输入 | 成功标准、预算 / 时间盒、产品侧确认方、验收种子 | Goal 卡、GSD Wave / Goal 映射、验证矩阵 |
| 产品 / 系统 DNA | 核心对象、业务不变量、状态流转、责任边界、演化规则、验证方式 | 系统 DNA、领域模型、状态机、约束、测试、监控和 CR 门禁 |
| 对象和状态 | 对象、字段口径、状态、不变量 | 领域模型、接口契约、数据模型 |
| 流程和规则 | 主/逆/异常/人工流程、规则矩阵 | 模块边界、事务、一致性、幂等 |
| 验收种子 | 正常、边界、异常、运营、数据验收 | TDD、契约测试、集成验证 |
| 风险和确认 | 合规、资金、隐私、安全、外部规则确认方 | 生产门禁、回滚、审计、安全架构 |
| AI 原型证据 | 原型观察、eval 样例、不可复用实现 | GSD 任务包输入、CAD 候选缺口 |

交接结论必须能回答：

- 是否可以进入 OpenSpec；如果不能，还差哪 3-7 个关键信息。
- 是否需要 `GSD + Goal`；如果需要，继续读取 `goal-composition.md`，补 Goal 卡、GSD Wave / Goal 映射、状态、预算 / 时间盒、验证证据、停止条件和 Ledger 更新。
- 是否需要进入可评审 Spec 模板；如果需要，继续读取 `spec-template-practices.md`，补 Spec 强度、五段式骨架、AC 与测试映射、spec-lint、AC 覆盖和漂移检查。
- 是否触发需求 / 设计 / 编码标准门禁；如果触发，回到 `spec-template-practices.md` 补需求标准、设计标准、编码标准和可验证规则。
- 是否可以进入 Harness/GSD；如果不能，是范围、顺序、权限还是验证缺口。
- 是否存在 CAD 候选缺口；如果存在，只能描述候选边界，不直接授权执行。
- 是否需要 PRD / 系分合议预审；如果需要，先输出决策日志和准出路径，再进入 OpenSpec、Harness/GSD 或回退 Round 0。
- 是否需要代码交付闭环；如果编码提速但 CR、测试、对齐、返工或上线质量没有改善，继续读取 `code-delivery-closed-loop.md`，补 Spec 强度、Harness 独立验证、CR 减负、知识回流和指标。
- 是否通过需求基线稳定性门禁；如果系统/产品需求未确认、需求条目不可验证、图文不可追踪或衍生需求无 owner，先回退产品专家、业务 owner 或架构 owner，不进入 SDD、测试、GSD Wave 或 CAD 候选。
- 是否通过产品 / 系统 DNA 门禁；如果核心对象、不变量、状态流转、责任边界、演化规则或验证方式缺失，先回退产品专家或架构师，不进入实现、GSD Wave 或 CAD 候选。

## 7. 反模式

- 把 AI Demo 直接当需求。
- 系统需求或产品需求未确认时，直接让 AI 生成 SDD、测试、代码或 GSD/CAD 任务。
- 没有产品 / 系统 DNA，就按功能清单拆研发任务，后续再靠事故、补丁、配置和口口相传补规则。
- 只有业务 owner 的主张，没有证据、eval 或验收种子。
- 把长 PRD 缩短成缺少对象、规则和风险的清单。
- 没有拒绝/延后机制，所有机会都进入研发。
- 产品上下文包被误当作 OpenSpec、完整 Harness Plan 或 Execution Grant。
- 产品目标被误当作 Goal 已闭环，或 Goal 状态被误当作 Execution Grant。
- 模型能力变化没有复议机制，旧限制永久留在产品判断中。
- 把 AI 原型讨论、PRD 迭代草稿、合议评审过程和被拒方案混进最终 PRD、系分或 Spec，导致工程侧无法识别当前有效输入。
