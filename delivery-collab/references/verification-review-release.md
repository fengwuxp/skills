# 验证、CR、发布与复盘闭环

本文定义产研协同体系研发编码流程的验证、代码 Review、发布和复盘闭环。它用于流程编排，不替代 `资深架构师` 的测试、CR、生产变更和安全专项 reference。

## 使用时机

- 用户要求设计 AI 编码后的测试、验证、CR、发布、监控和复盘流程。
- 需要判断 Agent 输出是否可信、是否可合并、是否能进入发布。
- 需要判断 AI 找到的 bug、AI 生成的 bug report 或补丁是否可信、是否修到根因、是否能进入 CR 或合并。
- 需要按设计评审、TDD / 测试设计、编码评审、可用性 / 安全性 / 可靠性评估分角色检查，而不是让实现 Agent 自证通过。
- 需要把 Office Hours / CEO Review / Eng Review / Design Review / Review / QA / Ship 角色链落成发布前生产交付审查标准。
- 需要判断“代码写完 / 测试通过”之后，目标环境的版本、配置、关键路径、观测和工单是否真的生效闭环。
- 需要在真实环境证据之外，按实际业务场景做模拟验收，确认实现是否解决真实业务问题。
- 需要判断某个 Goal、GSD Wave Goal 或 `GSD + Goal` 是否可 Verified / Closed。
- 需要把产品验收、eval、工程测试和生产观测连接起来。
- 需要判断陌生代码库、AI 代码变更、重构计划或 PR 说明是否能被业务 owner、架构师和 Review Agent 共同看懂。

## 不适用场景

- 单纯代码 Review、Bug 修复或生产变更细节，优先交给 `资深架构师`。
- 单纯 PRD 或产品验收模板，优先交给 `产品架构专家`。
- 高风险金融、合规、安全或生产数据变更，仍需专业方、SRE、DBA 或安全负责人确认。

## 读取后必须产出

- 验证矩阵：产品验收、eval、单元测试、契约测试、集成测试、静态检查、CR、发布监控和人工确认。
- 质量 / 测试门禁：测试矩阵、验证顺序、CR 前置条件、失败回退、残余风险交接和架构师测试能力调用点。
- 角色协作验证门禁：设计评审、TDD / 测试设计、编码评审、可用性 / 安全性 / 可靠性评估、发布门禁的顺序、owner、能力 / 约规来源、证据和停止条件。
- 代码库理解 / 影响可视化门禁：业务意图、入口路径、影响模块、调用关系、边界变化、源码锚点、可视化辅助、owner 复述和残余风险。
- AI bug / 补丁门禁：复现证据、根因、同类影响、最小修复、独立验证和下一 owner。
- CR 重点：行为、契约、边界、测试、敏感数据、权限、可靠性和发布风险。
- 五支柱验证：安全验证、测试要求、代码质量、性能验证和发布就绪，说明证据、缺口和 owner。
- 生产交付审查卡：目标价值、范围收敛、设计 / 工程 / 代码 / QA / 安全可靠、发布观测、回滚和人工接管证据。
- 生产生效验证卡：版本回读、配置回读、冒烟验证、观测确认、工单 / Goal 回写和残余风险。
- 业务场景模拟验收卡：场景来源、用户角色、前置条件、业务输入、期望结果、模拟证据、未覆盖场景和下一 owner。
- 发布与复盘：灰度、回滚、观测指标、问题复盘和知识沉淀。
- 一份当前可信度判断：可以合并、需要补验证、只能继续 CR，还是必须回退到产品/架构补规格。
- 一份 Goal 完成度判断：Goal ID、成功标准、已验证证据、未验证项、是否可 Verified / Closed、残余风险和下一 owner。
- 一份残余风险清单：未跑的验证、无法确认的业务/合规/安全项、上线前人工确认方和回滚依赖。

## 1. 验证矩阵

产研协同体系流程中的验证应覆盖四类证据：

| 验证层 | 证明什么 | 常见证据 |
| --- | --- | --- |
| 产品验收 | 是否解决正确问题。 | 业务 owner 验收、用户反馈、场景验收、运营检查。 |
| Eval | AI 原型、模型能力或智能行为是否稳定。 | 样例集、失败样例、通过率、人工复核、模型版本。 |
| 工程测试 | 实现是否满足契约和边界。 | 单元测试、契约测试、集成测试、E2E、回归测试。 |
| 交付治理 | 是否可合并、可发布、可回滚、可观测。 | lint、静态检查、CR、发布计划、监控、告警、回滚方案。 |

最小验证矩阵：

```text
Goal ID / 成功标准:
需求 / OpenSpec 条款：
Spec / AC 编号：
验收种子：
Eval 样例：
测试用例：
静态检查：
AC 覆盖：
漂移检查：
CR 检查点：
发布监控：
人工确认方：
残余风险：
```

PRD / 系分合议预审进入验证矩阵时，至少把接受项、拒绝项、待定项、分歧项和风险清单映射到验收种子、测试用例、CR 检查点、人工确认方和残余风险。`ACCEPT` 不代表已实现，`REJECT` 不代表风险消失，`PENDING` 不得进入执行授权。

`GSD + Goal` 进入验证矩阵时，至少把父 Goal、Wave Goal、成功标准、预算 / 时间盒、停止条件和 Ledger 最近证据映射到验收种子、测试用例、CR 检查点、发布监控、人工确认方和下一 owner。

生产级 SDD / AI 生成代码进入验证时，按五支柱清点：

- **安全验证**：SAST、依赖漏洞、硬编码密钥、输入校验、SQL 注入 / XSS、权限/租户/敏感数据。
- **测试要求**：单元、契约、集成、关键 E2E、边界、回归、失败路径和 AC 覆盖。
- **代码质量**：lint / format、复杂度、可维护性、命名、文档、分层和项目规则。
- **性能验证**：响应目标、内存 / CPU、数据库查询、缓存、容量或压测证据。
- **发布就绪**：配置、环境变量、日志、观测、降级、回滚、监控、告警和人工兜底。

易用性要求：

- 先给结论：当前能否继续、卡在哪个证据层、下一步最小动作是什么。
- 验证矩阵只列当前任务需要的证据层；不强制每次输出全量测试百科。
- 对未执行或无法执行的验证，必须说明原因、影响和替代证据，不能只写“未验证”。

## 2. Eval 与测试关系

Eval 和测试不是同一个东西：

- Eval 验证模型、Agent、原型或智能行为在样例集上的表现。
- 工程测试验证代码契约、业务规则、边界、异常、权限和集成。
- 用户验收验证实际工作流是否达成业务目标。

Eval 通过不代表代码可上线；测试通过也不代表产品方向正确。高风险场景必须三者都有明确边界。

## 3. 质量 / 测试门禁

质量 / 测试门禁不是独立测试专家 Skill，而是产研协同体系流程中的协作位：

- **OpenSpec 规定测什么业务事实**：目标、范围、非目标、业务规则、接口契约、数据约束和验收。
- **产品专家提供验收种子**：正常、边界、异常、运营、数据和人工兜底样例。
- **资深架构师设计和实现测试**：测试策略、TDD、补测试、测试分层、测试代码 Review 和工程验证落到 `testing.md`。
- **本技能编排质量门禁**：验证顺序、CR 前置条件、失败回退、残余风险交接、发布前人工确认和复盘沉淀。

门禁清单：

```text
测试矩阵：
验证顺序：
CR 前置条件：
失败回退：
残余风险交接：
架构师测试能力调用点：
```

只有当团队反复需要跨项目测试策略治理、测试资产审计、Flaky Test 治理、覆盖率/变异测试、AI Eval 与工程测试打通、测试数据治理、发布质量门禁模板或质量度量体系时，才考虑独立抽出测试质量专家 Skill。

## 3A. 角色协作验证顺序

产研协同体系的默认验证顺序应服务真实交付，而不是服务某个机制名。GSD、CAD、Goal、Harness 只在对应阶段内部使用。

```text
设计评审
-> TDD / 测试设计
-> 编码实现
-> 编码评审
-> 可用性 / 安全性 / 可靠性评估
-> 发布门禁
-> 复盘回流
```

| 门禁 | 主责视角 | 最小证据 | 停止条件 |
| --- | --- | --- | --- |
| 设计评审 | Design Reviewer + 产品 / 架构 owner | 目标、非目标、对象、状态、规则、验收、边界、风险和待确认项。 | 事实 / 推断混淆，或 owner 无法判断 ACCEPT/REJECT/PENDING。 |
| TDD / 测试设计 | 资深架构师 + 质量门禁 | 失败测试候选、验收种子、验证矩阵、验证命令和不可自动验证项。 | 没有业务不变量、失败路径或可执行验证方式。 |
| 编码实现 | Implementation Maker | 写入范围、只读范围、最小变更、状态回写和提交切片候选。 | 越过 Grant、模拟业务入口、内存版业务 Service 或修改高风险边界。 |
| 编码评审 | Code Reviewer + 架构师 | Spec / AC 映射、源码锚点、测试结果、契约、分层、兼容性和残余风险。 | Maker 自审、测试证据缺失或 Review 者无法复述影响范围。 |
| 可用性评估 | UED / 产品 owner | 用户路径、交互状态、错误恢复、文案可理解性、人工兜底和运营影响。 | 关键路径无法完成、错误不可恢复或 UED / 产品 owner 未确认。 |
| 安全性评估 | 架构师 / 安全 owner | 权限、租户隔离、敏感数据、审计、输入校验、依赖和密钥边界。 | 涉及权限、资金、生产数据或合规风险但无专业确认。 |
| 可靠性 / 发布评估 | 发布 owner / SRE / 架构师 | 监控、告警、灰度、回滚、Runbook、人工接管和发布审批缺口。 | 没有观测、回滚、人工接管或审批。 |

输出要求：

- 当前任务只输出涉及的门禁，不强制全量表格。
- 每个门禁必须有 owner、证据、停止条件和下一角色。
- 每个门禁必须写明调用哪个专项能力：产品 / 交互、验收种子和可用性回到 `产品架构专家`；系分、TDD、编码、CR、安全可靠和发布风险回到 `资深架构师`；结构化 Java Service 生成回到 `java-service-code-generator`。
- 同一 AI 可以辅助多个视角，但输出中必须区分 AI Maker 与 AI Checker；不能让实现者自证完成。

## 4. 代码库理解 / 影响可视化门禁

AI 生成代码后，Review 的瓶颈常常不是“有没有说明”，而是人是否能看懂代码库结构、源码锚点和影响范围。进入代码 CR 或发布前，先做理解门禁：

```text
业务意图：
入口路径：
变更摘要：
影响模块：
关键调用关系：
边界变化：
数据 / 契约 / 权限影响：
源码锚点：
事实 / 推断：
置信度：
验证证据：
可视化辅助：
owner 复述：
残余不确定性：
```

门禁规则：

- 先让 Agent 把代码库或 diff 从“文件列表”翻译成“业务意图 -> 入口路径 -> 结构影响 -> 源码锚点 -> 验证证据”，再进入源码级 CR。
- 对跨模块、重构、接口迁移、状态机或公共契约变更，必须说明谁调用谁、哪条边界发生变化、哪些调用方需要验证。
- AI 快速阅读、上下文生成和可视化可以是依赖图、调用导览、模块边界图、diff 覆盖层、仓库指令文件或源码锚定的结构视图；它们只辅助理解，不替代源码、测试和人工判断。
- 代码库理解结论包必须区分“源码事实”和“模型推断”，关键结论必须能回链到文件路径、函数、类型、配置、测试或命令证据。
- 人类 owner 或架构师需要能复述：为什么改、影响哪里、风险是什么、哪些证据已验证、哪些仍不确定。
- 如果只能得到大段 Markdown 总结，却无法定位关键源码、调用关系或边界变化，应回退到只读侦察或架构师 CR，不进入合并判断。

适用触发：

- 陌生代码库需要快速形成入口路径、模块职责、运行方式和风险结论。
- AI 一次生成多文件变更，Review 者看不清整体影响。
- 重构计划涉及模块迁移、类型职责变化、调用链调整或公共接口变化。
- 新人接手、跨团队交接或业务 owner 需要参与代码影响确认。
- 需要用 AI 快速阅读、图形、导航、源码锚点或导览方式辅助理解，但当前工具能力、语言支持、权限和联网边界仍需按项目实际核验。
- 直接点名 Gemini CLI、AgentRC 或要求安装 / 调用代码理解工具时，先回到 `code-understanding-tools.md` 做安装准入、只读范围、联网/认证/写入边界和输出交接判断。

不吸收边界：

- 不把任何单一 AI 代码阅读工具、可视化 CLI、IDE 插件、上下文生成器或外部工具写成默认依赖。
- 不默认联网、上传仓库、读取密钥、安装 npm 包或生成会被自动执行的指令文件；需要时必须取得项目和用户授权。
- 不因为图看起来合理就跳过测试、静态检查、代码 Review 或发布门禁。
- 不把 AI 总结、上下文文件或可视化输出当成架构真相；它必须能回链到源码、OpenSpec、Harness 和验证证据。

## 5A. 源码质量评审 Loop

目标是用最少流程得到可执行发现，不是把所有工具都跑一遍。CR 的核心是同时完成三件事：知识沉淀、规范统一、质量底线；其中规范统一要求遵循编码规范，质量底线要求不踩红线；既能高屋建瓴看业务语义、业务不变量、架构边界和设计原则，也能深入细节识别代码坏味道，推动整洁代码与架构实现。

```text
准入定界 -> 只读理解 -> 规则装载 / 红线 -> 三线评审 / 业务架构裁决 -> 代码坏味道细查 -> 工具补扫 -> 修复验证 -> 知识回流收口
```

1. **准入定界**：明确 review 对象、业务目标、写入 / 只读范围、风险等级、期望验证和是否允许自动修复；没有 diff、目标或验证要求时，先走决策澄清门禁。
2. **只读理解**：小 diff 用 Codex / `rg` / `git diff` 直接看；陌生代码库、多模块、接口迁移或 AI 多文件 diff，先产出代码库理解结论包。Gemini CLI、AgentRC、Understand Anything 只在已授权、只读、可回链源码锚点时作为辅助。
3. **规则装载 / 红线**：先读项目 `AGENTS.md`、邻近代码风格和项目检查命令；Wind opt-in 时读取 `wind-project-coding-conventions`；源码级判断回 `资深架构师` 的 `coding-review-deep-dive.md`、`coding-standards.md`、`testing.md`。
4. **三线评审 / 业务架构裁决**：先守质量底线，再看架构和维护性，最后处理风格噪音。P0/P1 是逻辑错误、安全漏洞、权限越界、数据一致性、公共契约、资金 / 合规 / 发布风险；P2 是架构设计、性能隐患、可维护性、复用质量和测试可信度；P3 是命名、格式和偏好，除非已写入项目规范或会造成误解，否则不阻塞。评审意见必须能回到业务语义、业务不变量、边界方向、契约完整性、失败路径和源码锚点。
5. **代码坏味道细查**：重点看规则错层、贫血用例、浅模块堆叠、直通包装、隐藏副作用、模糊契约、AI 注释噪声、只为测试变绿的实现和无业务语义的抽象；只给当前目标内可验证整改。
6. **工具补扫**：Open Code Review / OCR 用于覆盖 diff 和行级候选问题，先跑 preview / LLM 连通 / background；Ponytail 只做过度设计和最小正确实现专项 pass；工具输出都是证据源，不是最终 CR。
7. **修复验证**：只要求修复阻塞项和当前目标内 P2；修复后跑最小验证命令，输出残余风险、未覆盖范围和下一 owner。
8. **知识回流收口**：把好的 CR 结论拆成可复用业务事实、项目编码规矩、架构取舍、测试样例、机器门禁或技术债条目；业务知识优先回到项目 `CONTEXT.md` / 模块 reference / ADR / 测试，跨项目方法才回到 Skill，不把一次性意见灌进全局上下文。

技术债只按三类处理：影响稳定、安全、资金、协作效率或正常交付的，立刻还；不影响当前交付但会持续增加修改成本的，进入有 owner 的债务清单、Goal Ledger、ADR 或 issue 后慢慢还；即将废弃、一次性脚本、无真实复用价值的，不还，只标明停止维护或风险边界。CR 不追求“质量越高越好”，只追求当前业务风险下足够高且可验证的质量。

Wisdom Lens 只参与工程取舍：阴阳平衡同时看架构全局和行级细节；先为不可胜先守规范红线、P0/P1 和回退；庖丁解牛沿主路径、风险边界和变化轴下刀；循名责实核对“已完成 / 已修复”是否有代码、测试、日志或 owner 证据；无为而治删掉无价值评论、抽象和流程噪音。

最小输出：

```text
CR 结论：
覆盖范围：
规范 / 红线：
业务 / 架构判断：
关键发现：
- [P级] 文件:行：问题；证据：；最小修复：；验证：
坏味道：
技术债归类：
工具证据：
未采纳工具建议：
整洁实现建议：
知识回流：
回流落点：
验证与残余风险：
下一 owner：
```

红线：

- 不因为 OCR / Gemini / Ponytail 有结论就跳过源码级 CR、测试和项目约规。
- 不把格式、命名、风格噪音排在业务错误、契约破坏、权限安全、数据一致性和发布风险前面。
- 不把架构原则、设计模式、DDD、Clean Code 或老祖宗智慧写成口号；必须落到源码锚点、业务影响、风险等级和最小整改。
- 不把“高屋建瓴”变成脱离代码的宏大判断，也不把“深入细节”变成只挑行级风格问题；两者必须互相校准。
- 不把“请老祖宗做 CR”写成文化化评语；只保留取舍问题、严重级别、证据和修复优先级。
- 不把每条 CR 评论都回流成项目知识；只有已验证、可复用、有 owner、能进入权威载体的结论才回写。

## 5B. AI bug / 补丁门禁

AI 发现 bug 或生成补丁时，先做 triage，再决定是否进入修复、CR 或合并判断。

最小证据：

```text
问题名：
复现 / 触发条件：
源码锚点：
根因假设：
同类影响范围：
最小修复：
验证命令 / 结果：
残余风险：
下一 owner：
```

准入规则：

- 没有复现、日志、失败测试、用户反馈或源码锚点的 AI bug report，只能记为待确认。
- 补丁必须说明根因和同类影响扫描；只在报错点补 guard、吞异常、放宽断言或绕过失败路径的，按创可贴式修复退回。
- 修复必须优先落到共享根因处；改调用方前先查同类调用、公共函数、边界层和测试覆盖。
- 高风险业务、权限、资金、生产数据、公共契约和发布行为，必须有架构师或对应 owner 独立确认。
- Wisdom Lens 只做反偏：循名责实看 bug 名实，庖丁解牛看根因路径，治未病看同类回归防线，先为不可胜看失败回退，无为而治删无价值补丁。

准出结论只允许三类：

- **可进入 CR**：证据完整，修复最小，验证已跑或缺口清楚。
- **继续根因分析**：现象存在但根因、同类影响或验证不足。
- **退回待确认**：缺复现、缺源码锚点、缺业务影响或超出授权。

## 5. CR 顺序

AI 代码 Review 优先顺序：

1. 当前变更处于哪个角色协作阶段，是否已经过设计评审和 TDD / 测试设计，或明确说明跳过理由。
2. 是否回指 OpenSpec、产品上下文和验收种子。
3. 是否满足关联 Goal 的成功标准、预算 / 时间盒、停止条件和 Ledger 更新要求；Goal 状态不能替代验证证据。
4. PRD / 系分预审中的 `ACCEPT`、`REJECT`、`PENDING` 是否已经映射到实现范围、非目标、验证证据和人工确认方。
5. 是否能说明业务意图、入口路径、影响模块、调用关系、边界变化、源码锚点和残余不确定性。
6. 是否满足当前 Spec 强度和 Harness 摘要，尤其是 owner、写入范围、只读范围、验证命令、独立验证证据、知识回流位置和停止条件。
7. 是否满足 Spec 模板的 AC 编号、Given-When-Then、测试映射、spec-lint、AC 覆盖、漂移检查和风险自查要求。
8. 是否越过写入范围、分层边界或权限边界。
9. 是否破坏公共契约、兼容性、状态机或数据不变量。
10. 是否有正常、边界、异常、权限、幂等和回归验证。
11. 是否泄露敏感信息、吞异常、隐藏外部调用或扩大依赖。
12. 是否通过可用性 / 安全性 / 可靠性相关评估，或列出未覆盖范围和人工 owner。
13. 是否有发布、监控、回滚和人工处理边界。
14. 是否需要外部 Checker 作为补充证据：Open Code Review / OCR 只能在 CLI 可用、`ocr review --preview` 的 Will review / Excluded 覆盖清单已确认、LLM provider 已通过 `ocr llm test`、读取范围和模型外发边界获授权、并且 Review 背景包含业务目标 / Spec / 项目约规时调用；其输出必须回到架构师按项目编码规范、Wind opt-in 规则、测试证据和源码事实判读。
15. CR 高频问题是否已经沉淀为测试、lint、静态检查、fixture、模板、`.opencodereview/rule.json` 或知识回流计划，而不是继续靠人肉提醒。

AI 生成代码和人工代码使用同一合并标准；不得因为“是 AI 生成”降低 Spec 符合性、测试、性能、安全、发布和可维护性要求。

## 6. AI 产物风险

常见风险：

- 幻觉 API、工具类、配置项或项目约定。
- 幻觉依赖、不存在的库、过期接口或未经验证的第三方能力。
- 为了让测试通过而放宽断言、修改 fixture 或删除失败路径。
- 局部代码看似正确，但破坏公共契约或跨模块语义。
- 引入 N+1 查询、缓存错用、资源泄漏、超时或性能退化。
- 异常、日志、审计、权限、租户和敏感数据处理缺失。
- 生成大量重复或过度抽象代码，提高 Review 和维护成本。
- 未说明无法运行测试的原因，却把结果包装成完成。
- 用冗长文字总结掩盖入口路径缺失、结构影响不清、调用关系不明或边界变化无人确认。
- 只证明“编码完成”，没有证明 Spec 强度合适、Harness 独立验证、CR 减负、知识回流和交付指标。

## 7. 发布门禁

进入发布前至少回答：

- 当前变更服务哪个目标、OpenSpec 条款或验收场景。
- 当前变更服务哪个 Goal、GSD Wave Goal 或成功标准，是否具备 Verified 证据。
- Review 者是否已经理解入口路径、影响模块、关键调用关系、边界变化、源码锚点和残余不确定性。
- 已执行哪些验证命令，结果是什么。
- 哪些验证无法执行，原因和残余风险是什么。
- 目标环境运行的版本 / commit / build / image 是否已回读确认。
- 配置、开关、环境变量、灰度规则或 prompt 是否已回读确认生效。
- 关键业务路径是否有冒烟、契约或人工验收证据。
- 日志、链路、指标、告警或工单状态是否已回读闭环。
- 是否涉及配置、数据、公共契约、权限、安全、生产行为或外部依赖。
- 灰度、回滚、监控、告警和人工兜底是否明确。
- 谁负责业务验收、工程验收和上线确认。
- 是否可以把 Goal 标记为 Verified 或 Closed；若不能，Ledger 下一 owner 和下一动作是什么。

## 7A. 生产交付审查标准

生产交付审查用于判断“能不能交付生产”，不是替代上线审批、Git / PR / merge 或部署操作。它把 GStack 的 Office Hours、CEO Review、Eng Review、Design Review、Review、QA、Ship 收成一张准出卡，并把生产生效验证作为最后证据链。

审查顺序：

1. 产品价值：核心问题、现状替代、MVP、成功指标和非目标已确认。
2. 范围收敛：CEO Review 已砍掉非必要范围，变更可逆或有止损。
3. 工程设计：数据流、接口契约、edge case、迁移、性能和发布风险有 owner。
4. 交互可用：关键路径、loading / empty / error / permission 状态和人工兜底可用。
5. 实现证据：写入范围内最小实现，有源码锚点、项目约规、真实业务入口；无模拟业务入口或内存版业务 Service。
6. 独立 CR：Staff / 架构师视角已处理 P0 / P1，P2 有 owner。
7. QA 验证：正常、边界、失败、回归、安全可靠验证结果可复现。
8. 发布接管：版本回读、配置回读、冒烟、监控、告警、回滚、Runbook、人工接管、残余风险 owner 齐备。

生产交付审查卡：

```text
结论: Ready / Not Ready / Human Approval Required
阻塞项:
已验证证据:
生产生效证据:
未验证 / 残余风险:
Git/PR/merge/部署授权:
下一 owner:
```

P0 blockers：无真实业务入口；价值 / 范围未确认；公共契约、数据迁移、权限或安全无 owner；测试、CR、版本回读、配置回读、冒烟或发布观测缺失；回滚 / 人工接管不可用；高风险操作未授权。

不把测试通过、PR 数或 Agent 自述当上线批准；版本回读、配置回读、冒烟和观测证据也只是准出证据，不替代发布 owner 审批。

## 7B. 生产生效验证门禁

生产生效验证回答的是“目标环境是否真的按预期生效”，不是“代码是否已经写完”。没有命令输出、日志、工单、监控、截图或 owner 记录等可复核证据时，只能写“待验证”。

路由规则：

- **代码变更**：版本回读 -> 配置回读 -> 冒烟验证 -> 观测确认 -> 工单 / Goal 回写。
- **配置 / 开关 / prompt 变更**：可跳过部署，但必须做配置回读、关键路径冒烟、观测确认和状态回写。
- **业务行为变更或用户可感知变更**：在冒烟和观测之外补业务场景模拟验收；场景不足时先向 owner 问询，或查项目材料、公开资料、业内共识、行业标准规范后标注来源。
- **文档 / Skill-only 变更**：不要求预发环境验证；用仓库校验、diff check、安装态同步或人工审查作为生效证据。
- **资金、权限、安全、公共契约或生产数据变更**：必须有对应 owner 人工确认，AI 只能整理证据和缺口。

生产生效验证卡：

```text
结论: Ready / Not Ready / Human Approval Required
目标环境:
预期版本 / 配置:
版本回读证据:
配置回读证据:
冒烟验证:
业务场景模拟验收:
观测证据:
工单 / Goal 回写:
残余风险:
下一 owner:
```

红线：

- 不把“已部署”“已推配置”“测试通过”“页面能打开”“接口能调用”写成环境 OK。
- 不让实现 Agent 自己宣布生产生效；生产生效至少需要独立 Checker、发布 owner 或可复核工具输出。
- 不因缺少预发环境就伪造替代证据；只能说明缺口、影响和下一 owner。

## 7C. 业务场景模拟验收

业务场景模拟验收用于补偿“环境已生效但业务是否真的可用”的验证缺口。它不替代真实用户反馈、产品 owner 验收、生产观测、合规确认或上线审批。

场景来源优先级：

1. 已确认的 PRD、产品上下文包、验收种子、OpenSpec / SDD、客户反馈、工单或历史事故。
2. owner 问询得到的真实角色、业务目标、前置条件、异常边界和成功标准。
3. 项目内代码、接口、日志、监控、测试用例、Runbook 和业务配置。
4. 公开资料、业内共识、行业标准规范或监管材料；使用时必须标明来源、读取日期和适用边界，不得凭模型记忆补成事实。

最小流程：选 1-3 个代表性场景 -> 写清角色、前置条件、输入、期望结果和不可接受结果 -> 用预发 / 测试环境、mock 外部依赖或只读演练复现关键路径 -> 记录截图、日志、请求响应、数据状态或人工验收结论 -> 标明未覆盖场景和下一 owner。

业务场景模拟验收卡：

```text
结论: Pass / Fail / Need Owner
场景来源:
业务角色:
前置条件:
业务输入:
期望结果:
不可接受结果:
模拟证据:
未覆盖场景:
下一 owner:
```

红线：

- 不把公开资料、行业共识或标准规范写成当前项目事实；只能作为补偿场景来源。
- 不用 happy path 演示替代异常、权限、金额、状态流转、幂等和人工兜底等关键业务风险。
- 不把模拟验收写成生产已生效；它只能补业务可用性证据。

## 8. 观测和回滚

AI 生成代码进入生产后，更需要观测：

- 核心业务指标：转化、完成率、失败率、处理时长、人工介入率。
- 工程指标：错误率、延迟、吞吐、资源、重试、超时、队列积压。
- 质量指标：缺陷、回归、投诉、回滚、告警、异常路径。
- AI 指标：eval 漂移、模型版本、提示词/上下文变更、误判样例。

回滚不只是一条命令，还包括数据回滚、配置恢复、任务暂停、消息补偿、人工处理和用户沟通。

## 9. 复盘闭环

复盘要回答：

- 问题是否来自产品判断、上下文缺失、OpenSpec 不清、Harness 越界、Agent 失误、测试缺口、Review 漏洞还是发布监控不足。
- Review 是否因为入口路径、结构关系、调用链、边界变化或影响范围没有被共同看懂而漏掉风险。
- CR 是否完成知识沉淀、规范统一和质量底线，而不是退化为挑错大会、走过场或权力游戏。
- 技术债是否已按立刻还、慢慢还和不用还分级，并绑定 owner、验证方式或停止维护边界。
- 编码提速是否转化为一次通过率提升、返工率下降、CR 轮次下降、缺陷密度下降或验证失败恢复成本下降。
- Goal 是否在 GSD Wave、Spec、CR、发布和复盘中保持同一套成功标准、状态、证据和交接 owner。
- 哪些经验应进入产品 reference、架构 reference、脚本、fixture 或项目文档。
- 哪些只是一次性试错，不应沉淀为长期规则。
- 是否需要更新 eval 样例、测试用例、CR 清单、Harness 模板或验证脚本。
- 是否需要把领域事实回流到 `CONTEXT.md` 或项目等价文件，把编码规矩回流到 `AGENTS.md` / `CLAUDE.md` 或项目等价文件；仓库 Skill 只沉淀跨项目可复用流程和门禁。

个人长期偏好、用户背景和私有对话轨迹不得写入仓库、安装目录或 Skill 改进材料。

交付完整性要求：

- CR 结论必须区分已修复问题、剩余阻塞、残余风险和后续建议。
- 发布建议必须同时覆盖业务验收、工程验证、监控、回滚和人工兜底。
- 复盘建议必须说明应沉淀到产品 reference、架构 reference、脚本、fixture、项目文档还是不沉淀。

## 10. 指标

评估产研协同体系流程，不只看速度：

| 维度 | 应看指标 |
| --- | --- |
| 发现质量 | 问题证据强度、机会拒绝率、验证周期、复议命中率。 |
| 交接质量 | 上下文包完整度、OpenSpec 缺口、Harness 越界次数、变更影响复述通过率。 |
| 工程质量 | 测试覆盖语义、CR 问题密度、返工率、缺陷率。 |
| 发布质量 | 灰度问题、回滚、告警、人工兜底、用户影响。 |
| Agent 效率 | 有效任务完成率、失败恢复成本、上下文重建成本。 |
| 学习效果 | eval 样例增长、模板修正、验证脚本新增、反模式减少。 |
| 交付闭环 | 一次通过率、CR 轮次、缺陷密度、知识回流命中、上下文漂移、机器门禁覆盖。 |

如果指标只追求代码量、PR 数、自动化轮数或人力节省，流程会诱导团队把低质量输出包装成高效率。
