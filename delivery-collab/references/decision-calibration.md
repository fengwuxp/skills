# 工程决策校准

本文定义如何把 **华夏经世智慧** 接入产研协同体系：以华夏经典智慧校准现实判断、行动取舍和长期修身，经世致用，古法今用。运行时兼容 Skill id / 触发别名为 `huaxia-wisdom`、老祖宗智慧、老祖宗视角、东方判断层、Loop 取舍校准和 Wisdom Lens。它是轻量辅助判断镜片，用来帮助判断该不该动、怎么动、动到什么程度、什么时候停；不是流程主线，不替代事实、证据、测试、代码 Review、授权或上线审批。

命名口径：

- **华夏经世智慧**：正式名称，用于流程、Skill 展示、知识回流和长期传承语境。
- `huaxia-wisdom`：当前安装 / 来源 id，保留为稳定技术标识。
- 老祖宗智慧 / 老祖宗视角：口语触发别名，只用于用户这样表达时路由，不作为正式文档主名。

## 使用时机

- 用户明确要求把华夏经世智慧、Loop 取舍校准、Wisdom Lens、`huaxia-wisdom`、东方智慧、老祖宗视角、东方判断层、周易 / 易经、道德经 / 老子、庄子、论语、中医 / 黄帝内经、五德终始 / 阴阳五德始终、阴阳平衡、中庸之道、先为不可胜、庖丁解牛、循名责实、无为而治、每日三省、知行合一或一张一弛接入产研协同体系 Loop。
- 用户要求混一 AI 工作流、统一 AI 流程、一个入口 / 一个契约 / 一个准出，或用华夏经世智慧 / 老祖宗智慧提升生产可用性。
- Loop 准入、GSD Wave 拆解、Plan Grant / CAD Grant 授权粒度、执行核验、失败回退或复盘回流出现“速度与治理”“自动推进与人工确认”“流程膨胀与真实交付”的张力。
- 产研协同体系需要区分 Agentic Coding、Developer Feedback、External Feedback 三种时间尺度，防止内层 AI 自测替代产品判断或真实用户 / 市场反馈。
- AI 发现 bug、生成补丁或给出修复建议时，需要判断它是在修根因，还是只给症状打补丁。
- 产研协同体系输出已经具备事实、owner、边界和验证路径，但还需要一层取舍、止损、节奏或反偏判断。

## 不适用场景

- 不用于代写 PRD、系分、OpenSpec、Harness Plan、测试策略、代码实现、CR 结论或上线审批。
- 不在正式交付文档中保留文化化讨论过程、AI 推理轨迹、被拒方案或修辞性展开；正式文档只保留当前有效结论和证据边界。
- 不把传统智慧框架当作项目事实、合规结论、架构批准、Execution Grant、测试通过或生产审批。
- 不把中医类比写成医学建议，不把周易 / 五德终始写成占卜、宿命、历史证明、组织合法性或必然趋势判断。
- 不默认切换成“老祖宗口吻”；只有用户明确要求“用某个框架分析”“请一位老祖宗”或“用华夏经世智慧看”时，才按当前运行时可用 Skill 路由到 `huaxia-wisdom`。

## 经典源补足索引

只把经典源转成工程问题，不展开经学、医学或历史考据：

- **周易问变**：目标、对象、边界、风险里，哪些正在变，哪些不能变；是否有小步可逆的太极点。
- **道德经问少**：能不能少写、少管、少审批、少依赖；删掉动作后证据链是否更清楚。
- **庄子问纹理**：业务主路径、代码边界、组织 owner 和验证命令的自然缝隙在哪里。
- **论语问名分**：名不正则言不顺；角色、职责、产物、验收名词是否对得上实际。
- **中医问病机**：当前是表面症状、结构证据还是根因机制；是否已有治未病的预警和回归防线。
- **五德终始问时序**：旧规则、旧接口、旧流程是否已过承载期；新能力是否具备接替证据。

输出时只保留命中的 1-3 个问题，回写到 Loop Contract、验证矩阵、CR 发现、退场计划或知识回流。

## 生产可用混一校准

混一用“体用、阴阳、名实”三问校准：

- **体用**：业务目标、生产边界、验收和风险是体；GSD / CAD / GStack / Goal / Harness / 工具是用。用不能替代体。
- **阴阳**：推进、自动化、产出是阳；边界、证据、停止条件是阴。阳强则越权，阴盛则僵死，阴阳离决则流程脱离 owner。
- **名实**：完成、通过、可发布之名，必须有测试、CR、源码锚点、观测、回滚和 owner 之实。

校准后只回写到 交付契约、生产可用准出卡、验证矩阵或知识回流，不进入正式 PRD / 系分的修辞化过程。

## 三层反馈节奏校准

三层反馈用“阴阳互根、五德问时、循名责实”校准：内层快，是阳；外层慢，是根。阳强不能密，容易把自动化速度误当产品价值；阴盛不动，又会让反馈无法进入行动。目标是阴平阳秘：快慢有序，名实相符。

- **Agentic Coding Loop**：修代码、测试和实现细节，按 Spec / Evals / 失败测试运行；它只能证明“更接近当前 Spec”，不能决定“该做什么”。
- **Developer Feedback Loop**：用产品、业务、架构和交互上下文修正 Vision、范围和 Spec；它能做方向翻译，不能把个人判断写成市场事实。
- **External Feedback Loop**：用真实用户、市场、A/B、工单、监控和复盘修正 Vision；它最慢，但决定价值，不直接改代码，也不能被内层绿灯替代。

反偏三问：

1. 内层是否只证明“做对当前 Spec”，没有越权证明“做对产品”？
2. 中层是否把上下文写成可执行 Spec，而不是只凭品味或想象？
3. 外层是否已有真实反馈入口、观测和回流位置；没有时是否明确为待验证假设？

## AI bug / 补丁取舍

AI 找到 bug 或生成补丁时，用五问收口：

- **循名责实**：bug 之名是否有复现、日志、测试、源码锚点或用户证据之实。
- **庖丁解牛**：补丁是否沿根因、主路径和同类调用点下刀，而不是只挡住报错点。
- **治未病**：是否补了回归测试、规则、fixture 或静态检查，避免同类问题反复出现。
- **先为不可胜**：失败成本、回滚、降级和人工接管是否明确。
- **无为而治**：是否删除多余包装、注释噪音、过度抽象和无价值修复动作。

准出只看证据：复现用例、根因说明、同类影响扫描、最小修复 diff、独立验证和下一 owner。缺一项时只能进入待确认或继续 CR，不写成可合并。

## 四段式调度

1. 准入前：使用周易、阴阳平衡和先为不可胜，判断变与不变、失败成本是否可控、是否需要显式确认。
2. 拆解时：使用庄子、庖丁解牛和中庸之道，判断 Wave / Atomic Task 是否顺着业务与代码纹理拆分，授权粒度是否刚好。
3. 执行中：使用循名责实、治未病和无为而治，检查 AI 自述是否有实证支撑，补同类问题防线，删除无价值干预和假进展。
4. 复盘后：使用论语、知行合一、一张一弛和五德终始，把经验回到 Skill、fixture、脚本、Goal Ledger 或 Decision Log，并判断旧规则是否退场。

### 完整 CR 模式

用户明确要求“请老祖宗做 CR”或“用 Wisdom Lens 做完整 CR”时，必须在取舍校准之外给出可执行发现：

```text
CR 结论：
发现：
- [P级] 位置：问题；证据：；修复优先级：
残余风险：
验证证据：
不替代项：
```

CR 发现必须有严重级别、文件/行号或明确对象、验证证据、残余风险和修复优先级；没有证据的只写待确认，不写成问题。

## 调用华夏经世智慧 / huaxia-wisdom 的边界

- 用户只说“把东方智慧加入产研协同体系 Loop”时，本技能默认只使用本文工程化映射，不读取或展开 `huaxia-wisdom` 全部内容。
- 用户明确说“用阴阳平衡分析”“请老祖宗分析”“用华夏经世智慧输出”或“用 huaxia-wisdom 输出”时，如果当前运行时已安装该 Skill，可按其 Skill 触发规则调用；产研协同体系仍负责把结论转回 Loop Contract、Grant、Verification 和 Stop/Handoff。
- 输出保持工程语言，避免把正式 PRD、系分、Spec、CR 或发布报告写成文化化叙事。
- 不能引用框架名来代替来源材料、源码锚点、测试结果、日志证据或人工确认。

## 反模式

- 用东方智慧替代事实、源码、测试、日志、CR 或用户确认。
- 把中庸之道写成和稀泥，把无为而治写成不作为，把先为不可胜写成永不行动。
- 把阴阳平衡写成两边各退一步，而不是识别互相依存的约束和可验证的太极点。
- 把治未病写成补一个局部 if 或只新增一个凑绿测试，而不追同类问题和回归防线。
- 把周易写成占卜，把五德终始写成宿命或改朝换代口号，把中医写成医学诊断。
- 把经典源堆满输出，导致用户看不到当前 Loop 阶段、owner、证据和停止条件。
- 在正式 PRD、系分、OpenSpec、Harness Plan 或代码评审中保留过程讨论、AI 推理轨迹或修辞性发挥。
- 把 `huaxia-wisdom` 写成默认安装、默认联网、默认审批、默认 Git 操作或默认生产授权。
- 把“混一”写成文化化口号，而没有落到一个入口、一个契约、一个准出和八项证据。
